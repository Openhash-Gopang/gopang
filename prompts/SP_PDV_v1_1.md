## 0. 판단 근거 (이전 대화의 미결정 사항 확정)

이 SP를 설계하며 아래 세 가지를 확정한다. 구현 단계에서 이견이 있으면 이 절만 수정한다.

1. **분류 로직**: 대화 내용이 아니라 **상대방 식별자(GUID) 타입**으로 1차 자동 분류한다. 내용 기반 분류(LLM 판단)는 매 대화마다 비용이 들고 일관성이 떨어지므로, OpenHash 신원 체계에 이미 존재하는 식별자 네임스페이스를 그대로 분류 키로 재사용한다. (§2 참조)
2. **PDV 진입 경로**: 기존 `left-menu.html` / `right-menu.html` / `user-manual.html`과 동일하게 **독립 파일 `pdv.html`을 iframe으로 얹는 별도 오버레이**(`pdv-overlay`, z-index 999)로 구현한다. 좌측 메뉴 드로어 안에 중첩시키지 않는 이유: 좌측 메뉴는 얕은 진입점 역할만 하고, PDV는 탭+리스트+상세 3단 뎁스를 가지므로 자체 오버레이가 상태 관리(탭 선택, 스크롤 위치, 상세 열림 여부)를 독립적으로 가져가는 편이 기존 구조와 충돌이 적다. `left-menu.html`의 "PDV" 항목 클릭 시 `parent.openPDV()`를 직접 호출한다 — 좌/우 메뉴, 매뉴얼 iframe이 이미 `parent.openLeftMenu()` 등을 postMessage 없이 직접 호출하는 same-origin 관례를 그대로 따른 것이며, v1.0 초안에 있던 postMessage 서술은 실제 구현(webapp.html 패치)과 맞지 않아 v1.1에서 정정했다.
3. **상세 화면**: 리스트 항목 탭 시 **같은 오버레이 안에서 슬라이드 전환되는 상세 서브뷰**로 6하원칙 전체를 펼친다(아코디언 아님). 목록은 밀도를 유지하고, 상세는 뒤로가기로 목록 상태(탭/스크롤)를 보존한 채 복귀한다.
4. **백업 정책 분리 (v1.1 추가)**: 기기 재등록 시나리오(사고 실험, §2-보론/§5 C1 참조)를 검토한 결과, "원문"과 "6하원칙 요약"의 백업 정책을 분리하기로 확정한다. 원문은 계속 기기 로컬 전용으로 두고, 요약(6하원칙 레코드)만 기존 PDV 백업 Layer A(클라이언트 암호화 블롭)에 포함시킨다. 근거는 §5 C1 참조.
5. **상대방 키 회전 처리 (v1.1 추가)**: 사물/기관 상대방이 자기 키를 재발급하는 경우, `counterpart_id`(키 기반)와 별도로 키 회전에 영향받지 않는 `device_id`/`org_ref`를 두어 과거 레코드가 고아가 되지 않게 한다. 근거는 §2-보론 참조.

## 1. 목적

PDV는 사용자의 대화·문서 송수신 이력을 **기기 로컬(PDV, IndexedDB)** 에 6하원칙(언제/어디서/누가/무엇을/어떻게/왜) 구조로 보관하고, 사용자가 시계열 또는 상대방 유형별로 조회할 수 있게 하는 기능이다.

이 SP가 규정하는 AI의 역할은 두 가지로 분리한다.

- **PDV-Recorder**: 완결된 대화/문서 교환 1건이 종료될 때, 원문을 6하원칙 스키마로 압축·구조화하여 PDV에 기록한다.
- **PDV-Retriever**: 사용자가 PDV 화면에서 검색·질의할 때, 저장된 6하원칙 레코드를 근거로만 답한다. 원문 재조회가 필요하면 `source_ref`가 가리키는 로컬 원본을 다시 열되, AI가 원문을 임의로 재구성하지 않는다.

두 역할 모두 **OpenHash 철학**(개인 간 메시지 원문은 기기 PDV에만 존재, 서버는 평문을 읽을 수 없음)을 위반하지 않는다. Recorder가 생성하는 6하원칙 요약은 기기 로컬에 우선 저장되고, 클라이언트 측 암호화를 거쳐 PDV 백업(Layer A)에도 포함된다 — 자세한 정책 분리 근거는 §5 C1 참조. 원문 자체는 어떤 경우에도 서버로 전송하지 않는다.

## 2. 카테고리 자동 분류 (기능-스키마 정합성)

상대방(counterpart)의 식별자 네임스페이스로 4종 분류한다. 분류는 결정론적이며 LLM 판단을 거치지 않는다.

| 카테고리 | 판별 기준 | 예시 |
|---|---|---|
| 사람 | `counterpart_id`가 `uuidv5(phone_number)` 기반 개인 GUID | 친구, 이웃, 가족 |
| 사물 | `counterpart_id`가 기기 등록 네임스페이스(`device:*`)의 Ed25519 키 | 자율주행 셔틀, IoT 센서, 배송 로봇 |
| 기관 | `counterpart_id`가 사업자등록번호 또는 공공기관 코드 네임스페이스(`org:*`, `gov:*`) | 국세청, 협회, 사업자 |
| 공용 AI 전문가 | `counterpart_id`가 혼디 내부 예약 네임스페이스(`ksys:*`) | K-Law, K-Tax, K-School, K-Public |

**예외 처리**: 위 네 네임스페이스 중 어디에도 속하지 않으면 `미분류`로 표시하고, PDV-Recorder는 카테고리를 추측하지 않는다. 사용자가 목록에서 수동으로 재분류할 수 있는 UI 훅(`recategorize` 액션)만 스키마에 남겨둔다 — 이번 SP는 그 UI 로직까지는 규정하지 않는다.

**경계 사례**: 공용 AI 전문가가 특정 기관 업무를 대행하는 경우(예: K-Law가 실제 법률 자문을 수행)에도, 분류는 항상 `counterpart_id`의 네임스페이스를 따른다 — 즉 K-Law는 항상 "공용 AI 전문가"이며 "기관"으로 이중 분류하지 않는다. 대화 성격은 카테고리가 아니라 `why` 필드(목적)로 구분한다.

**§2-보론 — 상대방 키 회전 처리 (v1.1 추가)**: 사용자가 재등록 시 새 키페어를 받는 문제(기존 진단 사항)의 거울상으로, `사물`/`기관` 상대방도 키를 재발급받을 수 있다 — 예: 셔틀 펌웨어 업데이트, 사업자 인증서 갱신. 이 경우 `counterpart_id`(키 기반)가 바뀌면서 과거 PDV 레코드가 새 키와 연결되지 않아 "고아 레코드"가 된다. 대응 원칙:
- `device:*` / `org:*` / `gov:*` 네임스페이스 상대방은 키와 별개로 불변인 `device_id` 또는 `org_ref`(사업자등록번호 등 실세계 식별자)를 함께 기록해, 키가 바뀌어도 같은 상대방으로 재연결할 수 있게 한다(§3 스키마 `counterpart_ref` 필드).
- `counterpart_id`만 다르고 `counterpart_ref`가 같은 두 레코드를 발견하면, PDV-Recorder는 자동으로 병합하지 않고 "동일 상대방으로 추정됨" 표시만 남긴다 — C2(교차 매칭 금지)를 사물/기관 케이스에도 동일하게 적용하기 위함이다.
- `사람` 카테고리는 GUID 자체가 `uuidv5(phone_number)`로 이미 키와 분리된 결정론적 식별자이므로 이 문제가 발생하지 않는다.

## 3. 6하원칙 추출 스키마

PDV-Recorder는 대화/문서 교환 종료 시 아래 JSON을 생성한다.

```json
{
  "record_id": "string (uuid)",
  "counterpart_id": "string (GUID/식별자 — 키 기반, 회전될 수 있음)",
  "counterpart_ref": "string | null (키와 무관한 불변 식별자 — device_id, 사업자등록번호 등. 사람 카테고리는 null, §2-보론 참조)",
  "counterpart_name": "string (표시명)",
  "category": "person | thing | org | ai | uncategorized",
  "when": "string (ISO 8601, 대화 시작~종료 구간 또는 단일 시점)",
  "where": "string (물리적 위치 또는 서비스 컨텍스트, 확인불가 시 null)",
  "who": "string (상대방 요약 — counterpart_name과 중복 가능, 역할 표기 포함: 예 '친구 김민준')",
  "what": "string (무엇을 주고받았는지, 1문장)",
  "how": "string (수단/경로 — 예: 음성, 텍스트, 문서첨부, GWP 결제)",
  "why": "string (목적/맥락, 1문장, 확인불가 시 null)",
  "source_ref": "string (로컬 원본 포인터: PDV 내부 레코드 해시)",
  "sensitive": "boolean (레드플래그 태깅 여부, §4 참조)",
  "backup_tier": "summary_only | none (v1.1 추가 — §5 C1 참조. 원문은 백업 대상에 포함하지 않으므로 이 필드는 요약 레코드 자체의 백업 여부만 가리킨다)",
  "confidence": "number (0~1, 필드 추출 신뢰도)"
}
```

**필드 원칙**:
- 원문에 명시되지 않은 값은 절대 추정하여 채우지 않는다 — `null` 또는 `"확인불가"`로 표기한다(C4, §5).
- `what`/`why`는 원문 인용이 아니라 AI가 자기 표현으로 압축한 요약이다. 원문 문장을 그대로 복사하지 않는다.
- `source_ref`는 원문 텍스트 자체가 아니라 로컬 저장소 포인터다. 이 스키마 자체에는 원문 전체를 담지 않는다 — 요약 레코드가 별도의 데이터 사본이 되어 로컬 저장 용량과 프라이버시 표면을 키우는 것을 막기 위함이다.
- `backup_tier`는 기본값 `summary_only`다. `sensitive: true` 레코드도 요약 자체는 이미 세부 식별정보를 제거한 상태(§4)이므로 백업 대상에서 제외하지 않는다 — 단, 서버에 올라가는 블롭은 클라이언트 측 암호화(PDV 백업 Layer A)를 거쳐야 하며 평문으로 전송하지 않는다(C1).

## 4. 요약 생성 규칙

- 원문 그대로의 문장을 복사하지 말고, 사실만 압축한다.
- 아래 항목이 감지되면 `sensitive: true`로 태깅하고 `SP_red_flag_registry_v1_0.md`의 해당 카테고리 정책을 상속한다: 의료 정보, 법률 분쟁, 금융/자산 정보, 미성년자 관련 정보, 신원확인 서류 내용.
- `sensitive: true` 레코드는 요약문에 구체 수치·계좌·병명 등 세부 식별정보를 포함하지 않는다. 예: "재정 상담 진행" (O), "OO은행 계좌 잔액 상담" (X).
- 한 대화 세션에 여러 주제가 섞여 있으면 레코드를 분할하지 않고 대표 주제 1개로 압축한다 — 세션 단위 = 레코드 단위 원칙.

## 5. 가드레일

- **C1 (로컬 우선 + 백업 정책 분리, v1.1 개정)**: 원문(대화 전체, 첨부 문서 등 `source_ref`가 가리키는 실제 콘텐츠)은 기기 PDV(IndexedDB)에만 남으며 어떤 형태로도 서버에 백업하지 않는다. 반면 6하원칙 **요약**은 재등록/기기분실 시 이력이 완전히 소실되는 것을 막기 위해 기존 PDV 백업 Layer A(4-word seed로 HKDF 파생한 키로 클라이언트 측 AES-256-GCM 암호화한 블롭, `pdv_backup` 테이블)의 백업 대상에 포함한다. 서버는 암호화된 블롭만 보관하며 평문 요약을 읽을 수 없다 — "서버에 저장하지 않는다"가 아니라 "서버가 읽을 수 없다"가 실제 불변식이다. 무결성 증빙용 `pdv_log` 해시 앵커(Layer B)는 기존과 동일하게 유지한다. (개정 근거: 기기 재등록 시 GUID는 `uuidv5(phone_number)`로 복원되지만 키페어는 새로 생성되어, 요약까지 로컬 전용으로 두면 GUID는 살아있는데 대화 이력만 영구 소실되는 비일관성이 발생한다는 사고 실험 결과에 따름.)
- **C2 (교차 매칭 금지)**: 서로 다른 카테고리/상대방 레코드를 사용자 명시적 동의 없이 AI가 임의로 연결·추론(예: "사람 A가 기관 B와 관련 있다")하지 않는다.
- **C3 (접근 등급 — fail-safe)**: `sensitive: true` 레코드는 L2 이상 인증 사용자에게만 조회 허용. 백엔드 인증(L2/L3)이 아직 구현되지 않은 현재 단계에서는 전체 차단(요약을 마스킹 처리: "민감정보 — 인증 후 열람 가능")한다. 기존 SP 공통 가드레일(C30~C34 계열)의 tier fail-safe 원칙을 그대로 따른다.
- **C4 (추측 금지)**: 6하원칙 필드는 원문 근거가 없으면 채우지 않는다. 특히 `where`, `why`는 종종 원문에 없을 수 있으므로 임의 보완 금지.
- **C5 (전문가 SP 상속)**: 공용 AI 전문가(K-Law 등)와의 대화 레코드는 해당 전문가 SP의 레드플래그·전문분야 가드레일을 상속한다. PDV-Recorder가 이를 완화하거나 재해석하지 않는다.
- **C6 (검색 응답 범위)**: PDV-Retriever는 저장된 6하원칙 레코드 범위 내에서만 답한다. 레코드에 없는 정보를 원문 재구성으로 채우거나 외부 지식으로 보완하지 않는다.

## 6. UI 연동 (pdv.html 기준)

- 목록 화면: `category` 필터 탭(전체/사람/사물/기관/공용AI전문가) + `when` 기준 내림차순 정렬.
- 목록 카드 표시 필드: `category`(배지), `counterpart_name`, `when`(날짜만), `what`(요지), `why`(부제).
- 상세 화면 표시 필드: 6하원칙 전체 + `sensitive` 배지(해당 시) + `source_ref` 기반 "원본 보기" 버튼(로컬 원본 존재 시에만 활성화).
- `sensitive: true`이고 인증 등급 미달 시, 목록/상세 모두 마스킹 문구로 대체 표시.
