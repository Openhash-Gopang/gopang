# ROUTER-PRIORITY · 통합 라우팅 분류 원칙 v1.0

> **적용 대상**: 사용자 발화 하나가 들어왔을 때 GWP(K-서비스) / JEJU-GOV(정부기관,
> Jejudo) / EXPERT(전문가 페르소나) / AGENT-SUPPLIER(사업자) / AGENT-COMMON(개인
> 기본값) 중 어디로 보낼지 결정하는 상위 원칙. 지금까지 각 계층이 독립적으로
> 발전하며 이 판단 기준이 명시적으로 정의된 적이 없었다.
>
> **배경**: 2026-07-04, "1차 K서비스 → 2차 정부기관 → 3차 전문가 페르소나 →
> 4차 사업자 → 5차 개인" 순서안이 제안됨. 검토 결과 순서(ordinal priority)
> 자체보다 **분류 기준(classification key)**으로 정의하는 편이 실제 구조와
> 더 맞는다고 판단해 아래처럼 재구성함.

## R0. 응급 게이트 — 모든 분류보다 먼저 (cross-cutting)

생명·신체에 대한 긴급한 위험이 감지되면, 아래 R1~R4 분류 절차 전체를 건너뛰고
즉시 응급 대응 노드로 간다. 어느 진입점(K119든 SP-EXP-EMERGENCY든)으로 들어왔든
동일하게 최우선 적용된다.

**주의(기존 중복)**: 현재 `SP-EXP-EMERGENCY`(Jejudo 트랙)와 `SP-02_k119`(K-서비스)가
응급 감지 로직을 각자 독립적으로 구현하고 있다. 통합 라우터 도입 시 이 둘을
최소한 응급 키워드 감지 로직만이라도 공유하도록 리팩터링할 것을 권고한다(별도 과제).

## R1. 핵심 분류축 — 누구의 이익을 대변하는가 (공익 대변 vs 사익 대리)

발화를 분류하는 첫 기준은 순서가 아니라 **역할의 성격**이다.

```
공익 대변(공적 역할)   : K-서비스 + 정부기관(Jejudo)
                        판사·교육부처럼 특정 개인 편이 아니라 제도·기관을 대변.
                        K-Public_common P1, JEJU-GOV-COMMON §1과 동일 원칙.

사익 대리(개인 이익 복무): 전문가 페르소나
                        변호사·개인 교수처럼 사용자 개인의 이익을 위해 복무.
                        SP_common_guardrails C35(적극적 보조 원칙)가 이 계열에만
                        "당사자로서 사용자 편에서 능동적으로 움직여도 된다"는
                        형태로 적용되는 이유가 바로 이것 — 공익 대변자가 특정
                        사용자 편을 능동적으로 들면 그 자체가 위반이지만, 사익
                        대리자는 그게 본연의 역할이다.
```

이 축이 "변호사 질의 → klaw냐 SP_lawyer냐"를 순서가 아니라 **질문의 성격**으로
가른다:
- "이 사안이 법적으로 어떻게 처리되는지, 판결/절차가 어떻게 되는지" (제도 정보,
  제3자적 관점) → **klaw** (공익축)
- "내 상황에서 나한테 유리한 전략이 뭔지, 내 편에서 조언해달라" (개인 이익) →
  **SP_lawyer** (사익축)

같은 논리가 K-School(공익축 — 교육 제도·행정 안내)과 SP_teacher(사익축 — 내
아이 학습을 위한 개인 교사 상담)에도 그대로 적용된다.

## R2. 공익축 내부 구분 — 국가사무 vs 자치사무

R1에서 공익 대변으로 분류됐다면, 다음은 **국가사무냐 자치사무냐**로 나눈다 —
행정법상 실제 사무 구분 기준을 그대로 채택한다. 임의 순위가 아니라 사무의
성격이 결정한다.

```
국가사무 (전국 단일 기관 소관)     → K-서비스 (K-Tax=국세청, K119=소방청 등)
자치사무 (제주도·행정시·읍면동 소관) → 정부기관 (Jejudo 트랙)
```

동일 생활영역이라도 소관 기관에 따라 갈린다 — 예: "세금"이 국세(K-Tax)인지
제주도 지방세(Jejudo 트랙)인지는 세목으로 판단하지 어느 계층을 먼저 검사했는지로
판단하지 않는다.

## R3. 사업자 레이어 — 라우팅 대상이 아니라 상시 배경 레이어 (orthogonal)

`AGENT-SUPPLIER`는 R0~R2, R4와 같은 층위의 "다음 후보"가 아니다. 로그인한
사용자 본인의 프로필에 KSIC 업종코드가 등록돼 있으면, 위 분류 결과와 무관하게
**항상 배경 지식으로 병행 주입**된다(`_compileAgentSP`, 발화 내용을 매칭해서
켜고 끄는 게 아니라 "이 사용자가 어떤 사업자인가"로 결정됨). 따라서 순서
목록에서 "3차에서 못 찾으면 4차로 사업자 검사"처럼 취급하면 실제 구현과
어긋난다 — R0~R2/R4 분류와 **직교(orthogonal)**하는 축으로 취급한다.

## R4. 개인(AGENT-COMMON) — 폴백 기본값

R0~R2, R1의 사익축(전문가 페르소나) 어디에도 매칭되지 않으면 AGENT-COMMON
단독으로 응답한다. 다섯 계층 중 이 지점만이 "순서상 마지막"이라는 표현이
정확히 맞는 곳이다.

## 요약 결정 트리

```
발화 수신
 └─ R0: 응급 신호? ── Yes → 응급 노드 즉시 처리 (분류 종료)
       │ No
       ▼
    R1: 공익 대변이 필요한 질문인가? (제도·기관 정보, 제3자적 관점)
       ├─ Yes → R2: 국가사무인가?
       │           ├─ Yes → K-서비스
       │           └─ No  → 정부기관 (Jejudo)
       └─ No  → 사익 대리가 필요한 질문인가? (개인 이익을 위한 자문)
                   ├─ Yes → 전문가 페르소나
                   └─ No  → R4: AGENT-COMMON (개인 기본값)

  (R3 사업자 레이어는 위 결과와 무관하게, 사용자 프로필에 업종코드가
   있으면 항상 병행 주입된다.)
```

## 변경 이력

- v1.0 (2026-07-04): 최초 제정. "1차 K서비스→2차 정부기관→3차 전문가→4차
  사업자→5차 개인" 순서 제안을 검토, R0(응급 게이트 cross-cutting), R1(공익
  대변/사익 대리 축), R2(국가사무/자치사무 축), R3(사업자=직교 레이어), R4(개인
  =폴백)로 재구성. 순서(ordinal) 대신 분류 기준(classification key) 방식 채택.
