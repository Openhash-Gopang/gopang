## 배경 — 확정된 조직 구조 (2026-07-03 기준)

```
K-Public (모든 국가/공공기관 AI 공통 규칙 원본 — 상속의 최상위)
 ├─ 행정부: K-Province · K-City · K-County · K-Tax · K-Health ·
 │          police · 911 · K-Insurance(공적 영역만)
 ├─ 사법부: 대법원 대변 AI(기존 K-Law는 다음 개정 때 편입) · 헌법재판소 대변 AI

K-Doctor — 독립. K-Health를 상속하지 않는다("보건복지부 : 인간 의사"
           관계가 "K-Health : K-Doctor" 관계와 같다 — 제도 대변과 개인
           전문가 모사는 서로 다른 계열이다).
```

## H1. "상속"은 코드 재사용이 아니라 캐시 공유를 위한 장치다

가장 흔히 하는 오해부터 바로잡는다: LLM API는 상태가 없으므로(stateless),
자식 SP가 부모 SP를 "상속"해도 **매 호출마다 부모 텍스트 전체가 여전히
그대로 전송된다.** 클래스 상속처럼 부모 코드를 안 쓰는 게 아니다.

절약이 일어나는 지점은 오직 하나 — **부모 텍스트가 모든 자식 SP 호출에서
바이트 단위로 완전히 동일하면**, DeepSeek 프롬프트 캐싱이 그 구간을
캐시 히트로 처리해 싸게 청구한다는 것뿐이다(`SP-COMMON-03` T1 참조).
지금까지는 서브시스템마다 SP가 완전히 분리돼 있어 K-Law 사용자와 K-Tax
사용자가 캐시를 전혀 공유하지 못했는데, 계층 구조로 묶으면 **K-Public
공통 규칙 블록 하나를 행정부·사법부·입법부의 모든 기관이 함께 데운다**
— 트래픽이 합쳐지므로 캐시 적중률이 개별 서브시스템일 때보다 오히려
올라간다.

## H2. 상속 텍스트는 반드시 시스템 메시지 맨 앞에, 완전히 동일한 문자열로 배치한다

DeepSeek 캐싱은 메시지 배열의 **맨 앞부터 일치하는 접두사(prefix)** 만
인정한다. 따라서 조립 순서가 정확히 지켜져야 한다:

```
[K-Public 공통 규칙 원문 — 모든 자식이 완전히 동일하게 사용]
  ↓
[중간 계층 공통 규칙, 있다면 — 예: "행정부 공통"]
  ↓
[해당 기관 고유 규칙]
```

자식 SP가 부모 텍스트를 요약하거나, 순서를 바꾸거나, 앞에 자기소개
문구를 붙이는 순간 그 자식은 캐시 공유 대상에서 이탈한다. 부모 블록은
**손대지 않고 그대로**, 반드시 맨 앞에 둔다.

## H3. 상속 원본은 단일 진실 공급원(SSOT)으로 관리하고, 자식은 참조만 한다

`K-Public 공통 규칙`은 `prompts/K-Public_common_v1_1.md` 같은 단일
파일로만 존재한다. 각 자식 SP의 빌드/런타임 코드는 이 파일을 **그대로
가져와 조립**하며(fetch 또는 번들), 손으로 복사해 자기 SP 파일 안에
붙여넣지 않는다. 손 복사는 반드시 시간이 지나면서 미세하게 갈라지고
(오탈자 수정이 한쪽에만 반영되는 등), 그 순간 캐시 공유가 깨진다.

## H4. 부모 계층 개정은 전체 생태계의 캐시를 한 번에 초기화한다는 것을 인지한다

`K-Public 공통 규칙` 원문을 한 글자라도 바꾸면, 그 순간부터 **행정부·
사법부·입법부의 모든 기관 SP가 동시에 캐시 미스로 돌아간다.** 새로
데워지기 전까지 일시적으로 비용이 튄다. 따라서:

- 부모 계층(K-Public 공통, 행정부 공통 등) 개정은 사소한 표현 수정을
  수시로 하지 말고, 필요한 변경을 모아 드물게 배치로 진행한다.
- 반대로 **자식 고유 규칙**(예: K-Tax만의 세무 절차 규칙)은 다른 형제
  서브시스템에 영향을 주지 않으므로 자유롭게 자주 개정해도 된다.
- 이 비대칭성 — "부모는 신중하게, 자식은 자유롭게" — 을 SP 개정 권한
  설계에도 반영하는 것을 권장한다.

## H5. 계층은 "실제로 재사용되는 경우"에만 만든다

행정부 소속 기관들(K-Tax·K-Health·police·911·K-Province/City/County)
에 공통되지만 사법부·입법부엔 맞지 않는 규칙이 실제로 존재한다면
"행정부 공통" 중간 계층을 둔다. 하지만 계층을 과도하게 잘게 쪼개면
계층마다 조립 로직과 관리 대상 파일이 늘어나 유지보수 비용이 커진다.
"이 규칙을 최소 두 개 이상의 형제 SP가 공유하는가?"를 중간 계층 신설의
최소 기준으로 삼는다.

## H6. 캐시 공유는 "같은 API 키를 쓰는 서브시스템 사이"에서만 의미가 있다

프롬프트 캐싱은 벤더 API 키 단위로 격리된다. K-Public 산하 모든 서브
시스템이 캐시를 실제로 공유하려면 **동일한 DeepSeek API 키**를 써야
한다. 특정 서브시스템(예: K-Law)이 전용 키(`KLAW_DEEPSEEK_API_KEY` 등)를
쓰도록 설정돼 있다면, 그 서브시스템은 K-Public 공통 블록을 아무리
정확하게 상속해도 나머지 형제들과 캐시를 공유하지 못한다 — 이 경우
"상속 구조로 토큰을 절약한다"는 전제 자체가 그 서브시스템에는 적용되지
않으므로, 새 서브시스템에 전용 키를 발급할 때는 이 트레이드오프를
인지하고 결정한다.

## H7. 저장소 구조에 계층을 그대로 반영한다

```
prompts/
  K-Public_common_v1_1.md          ← 최상위 공통 규칙 (모든 국가기관 AI)
  K-Public_행정부_common_v1_0.md    ← 행정부 공통 (필요 시)
  K-Public_사법부_common_v1_0.md    ← 사법부 공통 (필요 시)
  K-Public_입법부_common_v1_0.md    ← 입법부 공통 (필요 시)
  K-Tax_v1_0.md                     ← K-Tax 고유 규칙만
  K-Health_v1_0.md
  police_v1_0.md
  ...
```

각 자식 파일은 "고유 규칙만" 담고, 상위 계층 텍스트는 절대 복사해
넣지 않는다 — 조립은 런타임(또는 빌드 시점)에 코드가 수행한다.

## H8. 신규 SP 작성 전 자가 점검 체크리스트 (SP-COMMON-04 M12에 추가)

1. 이 SP가 상속해야 할 상위 계층 문서를 정확히 식별했는가?
2. 상위 텍스트를 요약·재구성 없이, 시스템 메시지 맨 앞에 그대로
   배치했는가?
3. 이 서브시스템은 형제 서브시스템들과 **동일한 API 키**를 쓰는가?
   (아니라면 캐시 공유 전제가 성립하지 않음을 인지하고 진행한다)
4. 이 자식 SP만의 고유 규칙과 상위 계층 규칙이 내용상 중복되지
   않는가? (중복되면 상위로 끌어올리거나 자식에서 제거한다)
