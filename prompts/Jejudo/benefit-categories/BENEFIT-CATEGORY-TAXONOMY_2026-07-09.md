# 복지·공공서비스 카테고리 분류체계 (BENEFIT-CATEGORY)

> 작성일: 2026-07-09
> 배경: `civil-petitions-raw.json`(10,966건, 정규화 후 8,485개 클래스명)에 각각 SP를 쓰면 불필요한 중복이 발생한다는 지적에 따라, 유사 항목을 카테고리로 묶고 카테고리 단위로 SP를 작성하는 방식으로 전환한다.
> 실증: 상위 30개 클래스명(원본 943개 인스턴스)을 시범 분류한 결과 **12개 카테고리로 압축**됐다(약 78배 압축, §3 참조).

---

## 1. 분류 원칙 — 지원대상 × 지원방식 매트릭스

임의로 의미를 보고 묶는 대신, **두 개의 객관적 축**으로 카테고리 ID를 기계적으로 정의한다.

### 1-1. 지원대상 축 (TARGET)

| 코드 | 대상 |
|---|---|
| PREG | 임신·출산 |
| INFANT | 영유아 |
| YOUTH | 청년 |
| SENIOR | 노인·어르신 |
| DIS | 장애인 |
| LOW | 저소득층 |
| VET | 국가유공자·보훈대상자 |
| FARM | 농업인 |
| FISH | 어업인 |
| SMB | 소상공인 |
| MULTI | 다문화·한부모 등 가족형태 특례 |
| FOSTER | 위탁아동 등 아동복지 특수군 |
| GEN | 특정 계층 한정 없음(전 주민 대상) |

### 1-2. 지원방식 축 (MECHANISM)

| 코드 | 방식 |
|---|---|
| CASH | 현금 지급(지원금·장려금·축하금·수당) |
| FEE | 공공요금·이용료 감면 |
| INSURANCE | 보험 가입 지원 |
| SERVICE | 서비스 제공(돌봄·상담·법률 등) |
| VOUCHER | 지역화폐·상품권·이용권 |
| MATERIAL | 물품·자재 지원 |
| LOAN | 융자·보증 |

### 1-3. 카테고리 ID 규칙

```
{TARGET}-{MECHANISM}
예: PREG-CASH(임신출산 현금지원), GEN-FEE(전주민 요금감면)
```

교차 대상(예: 장애인가정 출산지원금 — DIS이면서 PREG)은 **더 구체적인 축을 우선**한다 — 이 경우 PREG-CASH에 포함시키고 §5(동의어 목록)에 "장애인가정 추가지원 있음"으로 각주 처리한다.

## 2. 카테고리 SP 문서 구조 (표준)

```
SP-BENEFIT-{카테고리ID}
§0 상속 위치: JEJU-GOV-COMMON > [본 SP] (기관 계층이 아니라 카테고리 계층 — 독자적 위치)
§1 정체성: 카테고리 정의, 대상, 방식
§CAPABILITIES: 카테고리 공통 개요 안내(직접) / 개별 지자체 시행여부·금액(안내만)
§2 완결 처리 업무: 이 카테고리에 통상 포함되는 자격 요건 패턴, 신청 절차 공통점
§3 DATA_REQUIREMENT: 사용자 거주 지자체의 실제 시행여부·금액은 연동 안 됨(no_interagency_access)
§4 대안 채널: 복지로·정부24 보조금24, 거주 지자체 확인 안내
§5 동의어 목록(alias): 이 카테고리로 흡수되는 정규화된 원본 클래스명 나열 — 검색 라우팅용, living registry
```

**기존 정책기관 SP와의 차이**: 정책기관 SP(SP-NAT-POLICY-*)는 "기관"이 주어였다. 이 카테고리 SP는 "혜택 유형"이 주어다 — 특정 기관 하나에 속하지 않고 전국 지자체가 각자 운영한다는 게 근본적으로 다른 점이다. 그래서 §0 상속 위치도 국가기관 계층이 아니라 독립된 카테고리 계층으로 둔다.

## 3. 실증 — 상위 30개 클래스명 → 12개 카테고리

| 카테고리 | 원본 인스턴스 | 흡수된 클래스명 수 |
|---|---|---|
| GEN-FEE | 263건 | 7개 (체육시설·문화시설·공영주차장·상수도·하수도·장사시설·거주자우선주차 이용료 감면) |
| PREG-CASH | 138건 | 5개 (출산장려금·출산지원금·출산축하금·산후조리비·장애인가정 출산지원금) |
| GEN-INSURANCE | 119건 | 3개 (시민/군민/민안전보험 — **완전 동일한데 "시/군" 접두어 차이로 다른 클래스로 잡혀있던 사례**) |
| GEN-VOUCHER | 111건 | 2개 (지역화폐·지역사랑상품권) |
| PREG-SERVICE | 64건 | 2개 |
| GEN-SERVICE | 57건 | 2개 |
| LOW-CASH | 57건 | 2개 |
| VET-CASH | 47건 | 3개 |
| DIS-SERVICE | 35건 | 1개 |
| FOSTER-CASH | 22건 | 1개 |
| DIS-VOUCHER | 16건 | 1개 |
| FARM-MATERIAL | 14건 | 1개 |

**합계: 원본 943건 → 클래스 30개 → 카테고리 12개.**

가장 극적인 사례는 GEN-INSURANCE다 — "시민안전보험"·"군민안전보험"·"민안전보험"은 지자체가 "시"인지 "군"인지에 따라 접두어만 다를 뿐 완전히 같은 제도인데, 제 정규화 스크립트가 이걸 놓쳐서 별개 클래스로 잡았었다. 카테고리 매트릭스가 이런 표면적 차이를 흡수한다.

## 4. 정직하게 남긴 것

1. **상위 30개만 분류했다.** 나머지 약 8,455개(전체 8,485개 중)는 미분류 상태다 — §1의 매트릭스 원칙만 정의했을 뿐, 전수 적용은 하지 않았다.
2. **자동 키워드 매칭은 89%가 실패했다**(이전 턴에서 확인). 이 12개 카테고리는 전부 사람이 직접 하나씩 확인해 배정한 것이다 — 나머지 8,455개도 이 방식(사람 확인)이 필요하다는 뜻이며, 자동화로 대량 처리할 수 없다.
3. **온디맨드 원칙 적용**: 나머지는 미리 다 분류하지 않고, 실제 사용자 질문이 들어올 때 K-Search가 그 즉시 카테고리를 배정(기존 카테고리에 편입 또는 신규 카테고리 생성)하고 동의어 목록에 추가하는 living registry로 운영한다 — 이건 이 프로젝트가 이미 두 번(민원 3,000건 계획, 정책기관 근거법령) 채택한 원칙과 동일하다.
4. **교차 대상 처리 규칙(§1-3)은 사례 하나(장애인가정 출산지원금)만으로 정한 것**이라, 더 복잡한 교차 사례가 나오면 재검토가 필요할 수 있다.
