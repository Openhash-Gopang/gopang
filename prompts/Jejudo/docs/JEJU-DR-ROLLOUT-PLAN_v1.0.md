## 1. 무엇이 빠져 있나 (현재 상태 재확인)

| 문서군 | 현재 상태 |
|---|---|
| 실국 13개(SP-DO-*) | §CAPABILITIES만 있음. §2(완결)/§3(연계) 항목이 산문 서술이라, "이 항목은 정확히 어느 필드·어느 기관 소관인지"가 구조화 안 됨 |
| 시청 2개(SP-CITY-*) | 위와 동일 |
| 읍면동 템플릿(SP-EMD-TEMPLATE) | §4(접수·안내 업무)가 이미 표 형태(업무영역/실질처리주체/연결SP)라 상대적으로 가깝지만, `connected`/`unavailable_reason` 필드가 명시적으로 선언돼 있진 않음 |
| 국가기관 28개(SP-NAT-*), SP-EXP-WATER | ✅ 이미 필드 단위 DATA_REQUIREMENT 완비 (선례) |

## 2. 방법론 — 어떻게 "필드"를 뽑아낼 것인가

새로 지어내지 않고, **이미 §2/§3에 적어놓은 연계 업무 항목을 구조화하는 것**이 원칙이다(국가기관 SP를 만들 때도 이 방식을 썼다). 즉:

1. 각 문서의 §3(실국) 또는 §3(시청 읍면동 연계 규칙) 항목을 훑어서, "사용자가 물어볼 법한데 지금 이 SP가 답 못 하는 구체적 사실"을 필드 단위로 쪼갠다.
   - 예: 실국 SP-DO-WELFARE의 "기초생활수급 접수" 연계 항목 → 필드화하면 `개인_소득인정액_산정결과`(owner: 시청 복지 담당과), `기초생활수급_심사현황`(owner: 시청) 등.
2. 각 필드에 `owner_agency`/`owner_dept`/`connected`(전부 false로 시작)/`unavailable_reason`(대부분 `no_interagency_access` — 국가기관 사례와 동일 논리: 그 데이터는 어딘가엔 있지만 혼디에 연동이 안 된 것)/`fallback`을 채운다.
3. 이미 있는 §2(완결 처리) 항목은 원칙적으로 DATA_REQUIREMENT 대상이 아니다(이미 답할 수 있는 것이므로) — 단, "개별 처리 현황·확정 여부" 같은 확인성 정보는 §2 항목이어도 실제로는 개별 데이터가 필요하므로 추가로 뽑아야 한다(예: "고향사랑기부제 개요"는 §2로 충분하지만 "내가 낸 기부금 세액공제 얼마 받았어?"는 별도 필드).

## 4. 순서 (우선순위)

1차 배치는 **문의 빈도가 높을 것으로 예상되는 4개**부터: `SP-DO-WELFARE`(기초생활수급), `SP-CITY-JEJU`/`SP-CITY-SEOGWIPO`(건축 인허가), `SP-DO-PLAN`(지방세). 이 4개를 먼저 완료해 패턴을 검증한 뒤, 나머지 12개를 일괄 진행한다(실국 SP 생성기 `gen_do_dept.py`를 확장해 국가기관 때처럼 스크립트로 일괄 처리).

## 5. 실행 방식

`gen_do_dept.py`(실국)와 시청 2개 파일에 각각 `data_requirements` 딕셔너리 인자를 추가하고, 국가기관 템플릿에 이미 쓴 것과 동일한 렌더링 로직(§3 표 생성)을 재사용한다. 읍면동 템플릿은 기존 §4 표에 컬럼 2개(connected/unavailable_reason)만 추가하는 최소 수정이다.

## 6. 검증 방법

각 배치 완료 후 Node 목업 라우터 테스트에 "개별 확인성 질문"(예: "내 기초생활수급 심사 어디까지 됐어?")을 추가해, §10-4(공백 고지) 응답이 실제로 트리거되는 문구를 SP 안에서 찾을 수 있는지 확인한다.

## 7. 승인 대기 사항

이 계획대로 진행해도 되는지, 아니면 우선순위(§4)를 다르게 조정하고 싶으신지 확인 후 착수하겠습니다.
