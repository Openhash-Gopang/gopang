# PDV_HISTORY_REQUEST 파일럿 평가 보고서 v1

## 0. 파일럿 범위

`K-Public_common_v1_2.md` P11로 신설, GOV_AGENCIES 9개 서비스 대상.
`SP-07_ktax_v2.3.txt`에 실증 예시 1건 추가. 새 백엔드 엔드포인트는
만들지 않고 기존 `/pdv/query`(동의 절차·스코프 검증·rate limit 전부
기존 구현 재사용)를 그대로 활용하는 설계.

## 1. 평가 중 발견한 실제 버그 — 설계와 구현의 불일치

시나리오를 실제로 추론해보는 과정에서, 제가 초안(P11-4)에 쓴 "한 번
동의하면 이후 캐시된 동의로 재요청 없이 조회된다"는 전제 자체가 **실제
백엔드와 맞지 않는다는 걸 발견했습니다.**

`handleConsentRespond`를 다시 열어보니: 동의 요청은 생성 시점에
`expires_at`이 5분 뒤로 설정되고(`_storeConsentRequest`), 사용자가
동의(`grant`)해도 이 값을 갱신하지 않습니다. `_verifyConsentToken`은
매 조회마다 `row.expires_at < now`를 검사하므로, **동의를 승인한 바로
그 순간에도 이미 5분이 지났으면 그 동의는 무효**였습니다 — "연속성을
위한 표준 동의"가 아니라 "지금 이 순간의 단발 조회를 위한 5분짜리
승인창"으로 설계돼 있었던 겁니다.

**즉시 수정**: `handleConsentRespond`에서 `grant` 처리 시 `expires_at`을
승인 시점 기준 **90일**로 연장하도록 고쳤습니다. 이제 "이전 기록
조회해줘"에 한 번 동의하면, 90일 안에는 같은 스코프를 다시 물어보지
않고 조회할 수 있습니다.

## 2. 남은 조건 — 클라이언트 책임을 명시했는가

백엔드가 90일을 보장해도, 실제로 "재요청 없음"이 되려면 **클라이언트가
승인받은 `request_id`·`consent_token` 쌍을 scope별로 저장했다가 다음
조회 때 재사용**해야 합니다. 서버는 그 값을 다시 보내주지 않으면
매번 새 동의 요청으로 취급합니다(`!consent_token || !request_id`
분기). 이건 P11-4에 명시했지만, **각 GOV_AGENCIES 서비스의 클라이언트
코드(별도 저장소)가 실제로 이렇게 저장·재사용하는지는 이번 세션에서
확인하지 못했습니다** — SP 텍스트 차원의 지시는 완료됐고, 클라이언트
구현 확인은 후속 과제로 남습니다.

## 3. 시나리오 재추론 — "저번에 상담했던 부가세 신고 이어서 하고 싶어요"

1. 사용자 발화에서 "저번에", "이어서" 감지 → P11-2 트리거 조건 충족
2. `[PDV_HISTORY_REQUEST: scope=ktax, period=180일, ...]` 발화 (SP-07
   v2.3 실증 예시 그대로)
3. 최초 조회라면 동의 화면 안내 → 승인 → (수정 후) 90일 유효한
   consent_token 발급
4. 이후 같은 90일 내 재조회는 즉시 성공 → 이전 과세기간·산정 맥락을
   이어받아 응답
5. 조회 실패/기록 없음이면 "이전 기록이 확인되지 않는다"고 밝히되
   "부가세"라는 이미 언급된 맥락은 다시 안 물음(§10/U2 정직성 + 능동성
   동시 충족)

이 경로 자체는 수정 후 논리적으로 일관됩니다. 다만 4번 단계가 실제로
매끄러우려면 §2의 클라이언트 재사용 로직이 실제로 구현돼 있어야
합니다 — 이게 확인 전까지는 "설계는 맞다, 실제 동작은 조건부"로
남겨둡니다.

## 4. K-School·K-Stock — 파일럿 범위 밖, 여전히 막혀있음

이번 파일럿은 GOV_AGENCIES 9개 한정입니다. K-School·K-Stock은
`VALID_PDV_SCOPES`에 등록조차 안 돼 있어(§ 이전 보고서에서 확인), 이
서비스들에겐 P11이 아직 그림의 떡입니다. 사용자가 원래 제시했던 예시
("수학 공부 이어서")는 **이번 수정으로 아직 해결되지 않았습니다** —
GOV_AGENCIES가 아니라 K-School이 대상이기 때문입니다. 다음 단계로
K-School·K-Stock을 `VALID_PDV_SCOPES`에 추가하고 동일 패턴을 적용하는
작업이 남아있습니다.

## 5. 종합 평가

| 항목 | 상태 |
|---|---|
| 새 프로토콜 설계(P11) | 완료, 기존 인프라 재사용으로 개발비용 최소화 |
| GOV_AGENCIES 9개 적용 | 완료(K-Public_common), 실증 예시 1건(K-Tax) |
| 백엔드 버그 발견·수정 | 완료 (90일 동의 유지) |
| 클라이언트 재사용 구현 확인 | **미확인 — 후속 과제** |
| K-School·K-Stock(원래 예시) | **미해결 — VALID_PDV_SCOPES 등록부터 필요** |

"의도대로 동작하는가"에 대한 결론: **설계 논리는 이제 일관되지만, 사용자가
원래 든 예시(수학 공부)는 이 파일럿 범위 밖이라 여전히 해결 안 됐고,
GOV_AGENCIES 쪽도 클라이언트 구현까지 확인해야 완전하다고 말할 수 있습니다.**
