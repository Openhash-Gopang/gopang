# 리포 상태 감사 및 복구 — 2026-07-14

## 발견한 문제

병행 세션(worker.js 백엔드 작업) 쪽에서 스테일(stale)한 로컬 클론 기준으로
`git pull --rebase` + 커밋을 반복하면서, 이 대화(prompts/SP 작업) 쪽에서
삭제했던 구버전 v1.0 SP 파일들이 두 갈래로 문제를 일으켰다:

1. **enterprises 21개 배치 완전 유실**: 해당 배치의 커밋 자체가 origin/main
   히스토리에 없다(`git log --all`로 확인) — push가 애초에 실패했거나
   누락된 것으로 추정. 원본 v1.0만 남아있었고 §3 반영분(v1.1)은 원격에
   존재하지 않았다.
2. **qgov 58개 구버전 부활**: v1.2(제 반영분)는 정상이지만, 삭제했어야 할
   v1.0이 58개 전부 되살아나 나란히 존재했다. enterprises에서도 HUG·SR
   v1.0 2개가 동일하게 부활했다.

## 조치

1. 컨테이너에 남아있던 생성 스크립트(apply.py)를 재실행해 enterprises
   21개를 처음부터 다시 v1.1로 재생성 — 내용은 최초 배치와 동일(§3 반영,
   헤딩레벨 검증 재통과).
2. qgov 58개 + enterprises(HUG·SR) 2개, 총 **60개의 stale v1.0 중복**을
   git rm 대상으로 정리.

## 검증

- 재생성된 21개 헤딩레벨 전수 PASS.
- qgov v1.2 58개는 내용물 무결성 확인(KDIC 표본 — 1억원 수정 등 반영분 살아있음).

## 재발 방지 권고

- 이후 커밋에서는 `git add -A`처럼 작업 트리 전체를 쓸어담는 방식을
  피하고, 명시적으로 대상 파일만 `git add <path>`할 것 — 이번 사고도
  스테일한 로컬 트리에 남아있던 삭제 대상 파일이 광범위 add 명령으로
  같이 커밋됐을 가능성이 크다.
- 두 세션(prompts 작업 / worker.js 작업)이 같은 리포를 병행 편집 중이면,
  각 세션 시작 시 `git pull`로 최신화한 뒤 작업하고, push 직전에도
  `git status`로 의도치 않은 파일이 스테이징되지 않았는지 확인할 것.
