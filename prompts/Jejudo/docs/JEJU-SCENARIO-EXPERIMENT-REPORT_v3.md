# 적극적 보조 원칙(§14/C35) 사고실험 보고서 v3
> 문서 코드: JEJU-SCENARIO-EXPERIMENT-REPORT (v2 계보 계승, "안내자 → 적극적 보조자" 전환 검증)
> 작성일: 2026-07-04 · 작성: Claude(세션 내 사고실험) · 검토 요청 대상: 주피터

## 0. 요약 (TL;DR)

일반 시민이 공공기관 AI를 이용하는 가상 시나리오 **300건**을 3개 트랙(정부 51개 SP·K-서비스 16개·전문직 페르소나 26개)에 걸쳐 구성하고, §14(JEJU-GOV-COMMON)/C35(SP_common_guardrails)의 "적극적 보조 원칙"이 실제로 적용될지를 검토했다.

| 트랙 | 시나리오 | PASS | PASS(조건부) | PARTIAL | FAIL |
|---|---|---|---|---|---|
| 전문직 페르소나 (26개) | 78 | **78** | 0 | 0 | 0 |
| 제주 정부·국가기관 (51개) | 137 | 3 | 18 | **116** | 0 |
| K-서비스 (16개: K-Law·K119·K경찰·K세무 등) | 85 | 0 | 0 | 0 | **85** |
| **합계** | **300** | 81 | 18 | 116 | **85** |

**핵심 결론 3가지**

1. **전문직 페르소나 트랙은 실제로 작동한다.** `expert-session.js`가 런타임에 `SP_common_guardrails_v3_2.md`를 fetch해서 26개 페르소나 전부의 system 메시지 앞에 붙이는 것을 코드로 직접 확인했다. C35(판단은 당사자로, 실행은 C23 유지)가 즉시 전파된다.
2. **제주 정부 트랙은 "상속은 되지만 실증 예시가 없는" 상태다.** JEJU-GOV-COMMON §14를 상속 선언(헤더)은 하고 있지만, 51개 중 48개 개별 부서/기관 SP 자체 본문에는 §14 스타일의 실행 예시(SP-DO-WELFARE의 PDV_REQUEST 같은)가 없다 — 원칙이 "적용될 가능성이 있다" 수준이지 "적용이 실증됐다"가 아니다. 게다가 실제 로딩 코드(jeju-router.js)는 별도 `jeju` 저장소에 있어 이번 세션에서 v1.5 버전 고정 여부를 확인할 수 없었다.
3. **K-서비스 트랙(K-Law·K119·K경찰·K세무 등 16개)은 이번 작업 범위 밖에 있었고, 오히려 상위 문서가 반대 방향을 명시하고 있다.** 이 트랙의 최상위 상속 원본인 `K-Public_common_v1_0.md` P2가 "신청·신고·접수는 **안내만 하고** 실제 절차는 공식 채널로 연결한다"라고 명문화돼 있다 — 이번에 추가한 §14/C35와 정확히 반대 방향이다. 이 문서를 손대지 않았으므로 **85건 전부 FAIL 처리**했다.

## 1. 방법론 및 한계 (먼저 밝힘 — §10 정직성 원칙을 이 보고서 자체에도 적용)

- 300건은 실제 DeepSeek/LLM API를 호출해 얻은 응답이 **아니다.** 이번 컨테이너의 네트워크 허용 목록에 `hondi-proxy`/DeepSeek 엔드포인트가 없어 실측 호출이 불가능했다. 대신 (a) 각 SP 파일의 실제 텍스트를 정규식 패턴으로 스캔하고, (b) 대표 파일 표본을 직접 읽고, (c) 런타임 조립 코드(`expert-session.js` 등)를 실제로 확인하는 **정적 감사(static audit) + 구조 검증** 방식으로 진행했다.
- 300건은 51+16+26=93개 대상 SP × 카테고리(단순정보요청/복잡수행요구/권한경계테스트/데이터연동공백, 트랙별로 3~5개)를 교차한 매트릭스이며, 파일 목록은 `prompts/Jejudo/docs/` 커밋과 함께 저장된 원자료(`scenarios_300.json`, 이 보고서와 별도로 관리)에서 재현 가능하다.
- 판정은 개별 시나리오마다 손으로 쓴 대화 전문이 아니라, **같은 SP 파일에 속한 시나리오는 같은 구조적 판정을 공유**한다 — 개별 SP의 실제 발화 스타일 차이(예: SP-DO-WELFARE는 실행 예시 보유, SP-DO-AGRI는 없음)까지는 반영했지만, "사용자가 뭐라고 물었는지"에 따른 미세한 차이는 이 1차 감사에서 다루지 않았다. 정밀도보다 **커버리지와 구조적 격차 발견**을 우선했다.

## 2. 트랙별 상세

### 2-1. 전문직 페르소나 (26개, 78 시나리오) — 전부 PASS

`src/gopang/ai/expert-session.js`:
```
const commonRes = await fetch(COMMON_GUARDRAILS_URL);   // v3.2로 갱신됨
...
const personaRes = await fetch(def.file);                 // SP_lawyer_v3_1.md 등
const composed = parts.join('\n\n---\n\n');
```
26개 페르소나 전부 이 경로로 조립되므로 C35가 예외 없이 전파된다. C35-3에서 명시했듯 C23(소비자 경고문)·C33-2(처방 출력 금지)는 그대로 유지되므로, "권한경계테스트" 카테고리(예: "그냥 처방해 주세요", "진단서 써주세요")에서도 하드룰이 살아있다 — 판단·자문은 능동적으로 하되 실행은 대행하지 않는 구도가 텍스트 구조상 보장된다.

**남은 리스크**: 이건 "구조가 갖춰졌다"는 뜻이지 "LLM이 매번 정확히 그 어조를 낸다"는 실측 보증은 아니다. 실제 DeepSeek 응답으로 최소 표본 검증(예: SP_nurse·SP_lawyer 각 5건씩 실제 API 호출)을 다음 단계로 권고한다.

### 2-2. 제주 정부·국가기관 트랙 (51개, 137 시나리오) — PARTIAL 다수

- JEJU-GOV-COMMON v1.5를 상속한다는 헤더 선언은 51개 파일 전부에서 확인됨(`상위 상속: JEJU-GOV-COMMON` 패턴).
- 그러나 `SP-DO-WELFARE_v1.2.md`(복지가족국, PDV_REQUEST 실증 보유)와 `SP-EXP-EMERGENCY_v1.0.md`(응급, 이번 세션 초반에 직접 검토한 파일)를 제외한 **48개 파일**은 자체 본문에 §14식 1인칭 실행 문구·예시가 없다. 즉 "제가 접수를 진행하겠습니다" 같은 어조가 실제로 나올지는 상위 공통 규칙 한 단락에 전적으로 의존한다 — 캐싱 구조(H2, 부모 텍스트가 맨 앞에 고정)상 이론적으로는 작동해야 하나, 도메인별 실행 예시가 없는 채로 일반 원칙만 주입됐을 때 LLM이 일관되게 지킬지는 검증되지 않았다.
- **더 중요한 미확인 사항**: 이 트랙을 실제로 라우팅하는 `jeju-router.js`는 gopang 리포가 아니라 별도 `jeju` 리포에 있다(이전 세션에서 `jeju-router.js`가 GOV-COMMON 특정 버전을 참조하도록 갱신한 이력이 있음). 이번 세션은 `jeju` 리포를 클론하지 않았으므로, **실제 배포 중인 라우터가 v1.4를 참조하는지 v1.5로 갱신됐는지 이 보고서만으로는 알 수 없다.**

**권고**: (1) `jeju` 리포를 클론해 라우터의 GOV-COMMON 참조를 v1.5로 갱신, (2) SP-DO-WELFARE/SP-EXP-EMERGENCY 두 파일에 있는 실행 예시 패턴을 최소 3~4개 대표 부서(SP-DO-SAFETY, SP-CITY-JEJU, SP-EMD-HALLIM 등)에 이식해 "원칙만 있고 예시가 없는" 격차를 줄일 것.

### 2-3. K-서비스 트랙 (16개, 85 시나리오) — 전부 FAIL

이번 작업에서 손대지 않은 트랙이며, 감사 중 발견한 구조적 충돌이 있다.

`K-Public_common_v1_0.md` §P2 (원문 그대로):
> 본인 확인이 필요한 신청·신고·접수를 대신 완료하는 것(전입신고, 소송 제기, 민원 접수 등은 **안내만 하고 실제 절차는 공식 채널로 연결한다**)

이건 §14-2/C35-2가 구분한 두 축("업무 수행"과 "권한 행사") 중 **업무 수행까지도 "안내만"으로 묶어버린 표현**이다 — 정확히 이번에 고치려던 "제3자 안내자" 패턴 그 자체가 최상위 상속 원본에 남아있다는 뜻이다. `SP-01_klaw`·`SP-02_k119`·`SP-03_kpolice` 등 16개 파일 자체에서는 `K-Public_common`을 이름으로 참조하는 문구가 발견되지 않았고, `SP_hierarchy_inheritance_v1_0.md`에도 "기존 K-Law는 다음 개정 때 편입"이라고 명시돼 있어 — **이 트랙이 실제로 K-Public_common을 상속하는 배선 자체가 아직 완료되지 않았을 가능성이 있다.** 어느 쪽이든(①문서가 반대 방향이거나 ②아직 상속 배선이 안 됐거나) 현재 상태로는 §14/C35 원칙이 이 16개 서비스에 적용된다고 볼 근거가 없어 전부 FAIL로 처리했다.

**권고 (우선순위 높음)**: K-Law·K119·K경찰 등은 시민 접점 중 가장 사용 빈도가 높고 응급성이 큰 서비스군이다. `K-Public_common_v1_0.md` P2를 §14/C35와 같은 구도("업무 수행은 능동, 권한 행사만 인간 승인")로 개정하고, 실제 상속 배선(어느 코드가 이 파일을 fetch하는지)부터 먼저 확인할 것을 권한다 — 이건 이번 세션 범위를 벗어나므로 별도 작업으로 분리 제안한다.
