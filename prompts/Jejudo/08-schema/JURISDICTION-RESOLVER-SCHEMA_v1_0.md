## 원칙 — "어디로 갈지"는 "무엇을 할 수 있는지"보다 먼저 결정된다

지금까지 SP-AUTHOR PHASE A(분류)와 PHASE B(템플릿 조회)는 tier가 이미
주어졌다고 가정하고 시작했다. 실제로는 tier 자체를 결정하는 게 가장 먼저
풀어야 할 문제이자 가장 자주 틀리는 지점이다 — "집 앞 상수도 파열"과
"소송"은 둘 다 "정부에 뭔가 요청한다"는 표면 형태가 같지만 계층이
정반대다. 이 판별을 SP-AUTHOR와 사용자 AI 비서가 각자 다른 기준으로
하면 반드시 드리프트가 생기므로, 이 문서 하나를 양쪽이 공유한다.

## 1단계 — 사무 성격 판별 (행정법상 사무 배분 기준)

| 사무 유형 | 판별 기준 | tier |
|---|---|---|
| 국가사무 | 사법(재판·기소·수사 중 국가경찰 사무)·외교·국방·병무·국세·전국단위 등록/인증 기관 소관 | `national`(제주 소재 사무소 있으면) 또는 `kgov`(없으면) |
| 시·도 사무 | 광역 단위 계획·정책, 제주도 조례로 도가 직접 수행 | `do` (do-dept) |
| 시·군·구 사무 | 대다수의 인허가·영업신고·등록 — 실질 처리권한이 시청에 있는 것 다수 | `city` |
| 읍·면·동 사무 | 제증명 발급, 1차 민원 접수, 기초생활 접수 등 | `emd` |

키워드만으로 기계적으로 정확히 가르기 어려운 경우(대부분의 실제 요청)가
많으므로, 이 표는 절대 기준이 아니라 **2단계로 넘어가기 전의 1차 후보
압축**용이다. "소송" 같은 명백한 예는 표만으로 즉시 `kgov`(사법부는
제주도지사 지휘·감독 밖 — `JEJU-NATIONAL-SP` §1 원칙 그대로)로 확정된다.

## 2단계 — 접수 계층과 실질처리 계층은 다를 수 있다

`05-emd/emd-master-data.json`이 이미 42개 읍면동 전체에서 증명한 패턴을
전 계층 공통 어휘로 승격한다:

```json
{
  "intake_tier": "emd",
  "intake_agency": "{읍면동사무소}",
  "substantive_tier": "do",
  "substantive_agency": "제주도 상하수도본부",
  "connected_sp": "SP-EXP-WATER"
}
```

"집 앞 상수도 파열"은 `intake_tier=emd`(신고 접수는 동네 창구)이지만
`substantive_tier=do`(실제 처리는 도 직속 상하수도본부)다 — 둘 다 정답이고,
사용자에게는 `intake_tier`로 응대를 시작하되 처리 주체는 `connected_sp`로
투명하게 안내한다(이미 `emd-master-data.json`이 하고 있는 것과 동일).
이 이원구조를 `do`·`city` 레코드에도 동일 스키마로 확장한다 — 지금까지는
emd에만 있었지만, 시·도 사무 중에도 접수는 시군구가 하고 처리는 도가
하는 경우(반대의 경우도)가 있으므로 계층에 상관없이 이 두 필드 쌍을
표준으로 삼는다.

## 3단계 — city/emd tier의 domain 분기 부재 해소

지난 사고실험에서 발견된 공백: `do-dept`/`national`은 domain별로 레코드가
갈라져 있는데 `city`는 시청 전체 단일 템플릿, `emd`는 읍면동별 단일
레코드(내부에 팀 구성이 배열로만 존재)다. 이 문서가 요구하는 판별을
실제로 수행하려면 `city`·`emd`도 do-dept와 같은 **domain 키 분기**가
필요하다:

```
city-master-data.json  : { domain: "동물미용업등록", 시코드: "jejusi", ... }
emd-master-data.json   : { domain: "상수도", emd_code: "SP-EMD-AEWOL", ... }
```

기존 `emd-master-data.json`의 `접수전용업무` 배열은 이 domain 분기의
원형이었다 — 이번 개정은 그 배열 항목 각각을 독립 레코드로 승격해
`jeju-router.js`의 `_assemble()`이 do-dept와 동일한 방식으로 city/emd도
조회할 수 있게 하는 것이다(스키마 확장이며, 기존 데이터 재입력이
필요한 별도 작업 — 이번 문서는 스키마만 확정).

## SP-AUTHOR와의 관계 — 새 PHASE 0

```
PHASE 0. 관할 계층 판별 (신설, PHASE A보다 먼저)
  1단계·2단계를 실행해 tier_hint 대신 {intake_tier, substantive_tier,
  substantive_agency, connected_sp}를 확정한다. 이후 PHASE A(개인/조직
  분기)·PHASE B-0(입출력 스키마)은 intake_tier 기준으로 진행하되,
  substantive_tier가 다르면 §CAPABILITIES에 반드시 그 사실을
  `mode: intake_only`로 명시한다(HUMAN-AUTHORITY-GATE-SCHEMA의
  CAPABILITIES 어휘 그대로 재사용 — 새 어휘를 만들지 않는다).
```
