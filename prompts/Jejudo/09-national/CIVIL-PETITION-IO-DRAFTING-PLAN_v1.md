# 민원 업무 절차 Input/Output 작성 계획서

> 작성일: 2026-07-09
> 선행 문서: `SP-부서-입출력-마스터목록_v1.md` (82개 부서 단위 input/output)
> 목적: 정부24 기준 온라인 신청 약 2,800종 + 열람·발급 약 760종(중복 포함 약 3,000여 건)의 개별 민원을 input/output 쌍으로 정리하는 작업을 실행 가능한 단계로 분할
> 원칙: 82개 부서를 수작업으로 훑은 이전 방식은 이 규모(3,000여 건)에서 **그대로 반복하면 안 된다** — 자동 수집 + 우선순위 기반 수작업 정밀화를 병행해야 한다.

## 0. 범위 확정 — 근거 수치

| 항목 | 건수 | 출처 |
|---|---|---|
| 정부24 온라인 신청 가능 민원 | 약 2,800종 | 행정안전부 지표서비스(index.go.kr) |
| 정부24 인터넷 열람·발급 가능 민원 | 약 760종 | 상동 |
| 원스톱 서비스(여러 민원 동시 신청) | 12종 | 상동 (예: 임신·출산, 전입신고, 안심상속) |
| 사실·진위 확인 부가서비스 | 20종 | 상동 |

이 문서는 위 총량(약 3,000여 건, 중복·부가서비스 제외 시 실질 약 2,800건)을 대상으로 합니다. data.go.kr에 "행정안전부_대한민국 공공서비스(혜택) 정보"·"행정안전부_통계연보_정부24 민원서비스" 두 Open API가 이미 공개돼 있어, **목록 자체는 수작업 수집이 아니라 API로 긁어올 수 있습니다.**

---

## 1. 핵심 전략 — "전수 수작업"이 아니라 "자동 수집 + 계층적 정밀화"

3,000건을 82개 부서 방식(부서당 몇 시간 수작업)으로 하면 단순 계산으로도 감당할 수 없는 공수가 됩니다. 그래서 세 갈래로 나눕니다.

| 갈래 | 방법 | 대상 규모 |
|---|---|---|
| A. 자동 수집 | data.go.kr Open API로 민원명·소관부처·근거법령 등 메타데이터 일괄 확보 | 전체 ~2,800건 |
| B. 수작업 정밀화 | 사람이 직접 확인해 input/output을 정확히 기술 | 우선순위 상위 건 (아래 §3) |
| C. 온디맨드 생성 | 캐시 미스 시점(실제 사용자가 요청할 때)에만 즉석 조사·등재 | 나머지 저빈도 건 (지난번 PROCEDURE_MAP 설계의 "캐시 미스 → 즉석 조사" 원칙 재사용) |

**즉, 3,000건 전부를 미리 손으로 채우는 게 목표가 아닙니다.** 상위 빈도 건만 선제적으로 정밀화하고, 나머지는 실사용 시점에 채워지는 living registry로 설계합니다 — 이미 C40/eligibility_gate에서 채택한 "완전하다고 보장하지 않는다, 점진적으로 채운다"는 철학과 동일합니다.

## 2. 단계(Phase) 구조

### Phase 0 — 자동 수집 파이프라인 구축 (선행 작업, 1회성)

- data.go.kr "대한민국 공공서비스 정보" API, "정부24 민원서비스 사무 종류" API 연동 스크립트 작성 (`tools/` 디렉토리에 Python 스크립트 추가 — 기존 `build_manifest.py` 등과 같은 패턴)
- 수집 필드: 민원사무명, 민원사무코드, 소관부처(→ 기존 82개 부서 매핑과 연결), 근거법령, 신청/열람/발급 구분
- 산출물: `civil-petitions-raw.json` (약 2,800건, input/output은 아직 비어있는 뼈대)
- **이 단계는 사람이 아니라 스크립트가 하는 일**이라 별도 인건비 없이 진행 가능

### Phase 1 — 최우선 그룹 수작업 정밀화 (Top 50~100건)

선정 기준(§3 참조): 정부24 발급 통계 상위 + Hondi 킬러 유즈케이스와 직결된 건.

- 등본·초본, 인감증명, 각종 세금 납세증명, 병적증명서, 사업자등록 관련 등 지난 대화에서 이미 여러 번 등장한 고빈도 민원부터 시작
- input/output을 "위치기반사업자 등록" 사례 수준으로 정밀하게(선행기관 output=이 기관 input 연쇄까지) 기술
- 산출물: `SP-{민원명}` 개별 문서 또는 `procedure_maps` 레코드로 직접 등재

### Phase 2 — 부서별 상위 N건 확장 (약 1,600~2,400건)

- 82개 부서 각각에서 이용 빈도 상위 20~30건씩 선별해 확장
- 이 단계부터는 사람이 매 건 정밀 조사하지 않고, **기존 SP-AUTHOR PHASE(-1) 클러스터링**(이미 저장소에 구현돼 있는 "수백 개 기관 일괄생성" 프레임워크)을 활용해 반자동 생성 — 사람은 생성 결과를 검수만 함
- 품질 기준을 Phase 1보다 낮춤(핵심 input/output만, 세부 예외사항은 이후 보강)

### Phase 3 — 롱테일 (나머지 저빈도 건)

- Phase 0에서 확보한 뼈대(민원명·소관부처·근거법령)만 `procedure_maps`에 `status: draft`로 미리 등재
- 실제 사용자가 요청할 때(캐시 미스) K-Compose가 즉석 조사해 `status: active`로 승격 — 별도 사전 작업 없음
- 이게 "3,000건을 다 채우지 않고도 3,000건을 커버하는" 방법입니다

## 3. 우선순위 산정 기준

Phase 1·2의 순서를 정하는 기준을 명확히 해둡니다(임의 순서로 진행하면 나중에 재작업이 생김).

1. **정부24 발급 통계 상위** — "행정안전부_인터넷 증명서 발급 현황" API에 민원사무코드별 연도별 발급건수가 있음 → 이 숫자로 정렬
2. **제주 필드테스트 킬러 유즈케이스 직결 여부** — `JEJU-FIELD-TEST-MASTER-PLAN`에서 정한 등본발급·배달주문 등과 겹치는 건 최우선
3. **다른 절차의 하위 구성요소로 반복 등장하는 건** — 예: 사업자등록증은 위치기반사업자 등록·개인파산·창업 등 여러 절차에서 반복 참조됨. 이런 "공통 부품" 성격의 민원을 먼저 정밀화하면 나중 단계 재사용률이 높음
4. **법적 리스크가 큰 건(오류 시 인간 피해 가능성)** — 법원·의료·세무 관련은 정확도 요구가 높으므로 자동생성보다 수작업 비중을 높게 유지

## 4. 스키마 — 기존 설계 재사용

새 스키마를 만들지 않고, 이미 있는 것을 그대로 씁니다.

- 개별 민원 1건 = `atom_row` 1개 (또는 여러 민원이 조합된 경우 `procedure_maps` 1개)
- input/output = 기존 `org_profiles.input`/`output` json 필드
- 근거법령 = `eligibility_gate`에 v3.38에서 추가한 "근거 조문 함께 기재" 원칙 그대로 적용

## 5. 일정 공수 추정 (1인 개발 기준, 개략)

| Phase | 작업 성격 | 추정 소요 |
|---|---|---|
| Phase 0 | 스크립트 개발 1회성 | 수일 |
| Phase 1 (50~100건) | 수작업 정밀화 | 수주 |
| Phase 2 (1,600~2,400건) | SP-AUTHOR 반자동 + 검수 | 수개월(지속) |
| Phase 3 (나머지) | 온디맨드, 별도 사전 공수 없음 | 지속적(사용량에 비례) |

Phase 2는 "한 번에 끝내는 작업"이 아니라 **제주 필드테스트가 진행되며 지속적으로 채워지는 living registry**로 설계하는 게 현실적입니다 — 처음부터 완결을 목표로 하면 착수가 늦어집니다.

## 6. 미확정 · 검증 필요 항목

1. data.go.kr 두 API의 실제 응답 스키마(필드명·형식)를 아직 확인하지 않음 — Phase 0 착수 전 API 문서·샘플 응답 확인 필요
2. "정부24 발급 통계" API가 민원사무코드별 세부 순위를 실제로 제공하는지 확인 필요(§3-1 우선순위 산정의 전제)
3. Phase 1 대상 50~100건의 확정 리스트는 아직 없음 — 다음 단계에서 §3 기준으로 실제 후보 리스트 작성 필요
4. SP-AUTHOR PHASE(-1) 클러스터링이 "부서 단위 일괄생성"용으로 설계됐는지, "개별 민원 단위" 생성에도 그대로 쓸 수 있는지 실제 코드 확인 필요(★검증 필요★)

## 7. 승인 요청 사항

이 계획서에서 실행 전 확인받고 싶은 것 세 가지:

1. Phase 0(자동 수집 스크립트)부터 바로 착수해도 되는지
2. Phase 1의 50~100건을 정부24 발급 통계 상위로 기계적으로 뽑을지, 아니면 제주 킬러 유즈케이스 기준으로 먼저 좁힐지
3. Phase 3(온디맨드)를 지금 설계에 포함하는 데 동의하는지 — 이걸 받아들이면 "3,000건을 언젠가 다 채운다"가 아니라 "당장은 커버리지가 낮아도 되고, 쓰이는 만큼 채워진다"는 걸 전제로 진행하게 됨
