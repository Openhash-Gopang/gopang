# 중앙정부 정책기관(41개) SP 작업 문서화

> 작성일: 2026-07-09
> 계기: "제주도는 지방정부 클래스의 인스턴스일 뿐"이라는 지적 — 제주 필터링 대신 중앙정부 부서 자체의 SP를 작성해야 한다는 방향 전환

## 1. 확인한 사실

기존 `prompts/Jejudo/09-national/agencies/`의 28개 `SP-NAT-*`는 이미 클래스(template)/인스턴스(도코드+지사명) 구조로 잘 설계돼 있었다 — `national-agency-master-data.json`이 그 증거다. 이건 새로 만들 필요가 없었다.

진짜 공백은 따로 있었다: **19개 행정각부·4개 헌법기관·6개 대통령직속·12개 총리직속(총 41개)** 중 어느 것도 자기 이름의 SP가 없었다. 있는 건 그 산하 집행기관(고용노동부의 지방고용노동청 등)뿐이었다 — `national-agency-master-data.json`의 "소속부처" 필드를 집계해 확인했다.

이 41개는 기존 28개와 성격이 다르다 — 대부분 **제주 지사가 따로 없는 전국 단일 창구형** 기관이다(국민권익위원회에 "제주지사"는 없다). 그래서 기존 {지사명} 자리표시자 패턴이 안 맞아, 새 템플릿 유형이 필요했다.

## 2. 만든 것

| 파일 | 역할 |
|---|---|
| `SP-NAT-POLICY-TEMPLATE_v1.0.md` | 41개 정책기관 공통 클래스 (자리표시자: 기관명·소속·input·output·접수채널·근거법령) |
| `policy-bodies-master-data.json` | 41개 전체 인스턴스 데이터 (도코드 없음 — 전국 단일 인스턴스라서) |
| `SP-NAT-POLICY-ACRC_v1.0.md` | 국민권익위원회 — 템플릿의 첫 렌더링 사례 |

## 3. 정직하게 남긴 것

- **28/41개 항목의 접수채널이 TBD다.** 대표전화·URL을 지어내지 않고 비워뒀다 — 공식 홈페이지 확인 후 채워야 한다.
- **jeju-router.js 쪽 렌더링 함수가 아직 없다.** 기존 `_renderNatTemplate()`은 도코드 기반이라 이 새 유형(도코드 없음)에는 그대로 못 쓴다 — 대응 함수(가칭 `_renderPolicyTemplate`) 신설이 필요하다(★미구현★).
- **나머지 40개 인스턴스 렌더링은 하지 않았다** — ACRC 하나만 패턴 검증용으로 완성했다. 나머지는 SP-AUTHOR 파이프라인으로 일괄 생성하는 게 적절해 보인다.
- **41개 중 일부(대통령비서실·국가안보실·대통령경호처·국가정보원 등)는 애초에 "대국민 접수"가 없는 기관이다.** 이런 항목은 실제로는 SP를 만들어도 실사용 빈도가 0에 가까울 수 있다 — AC가 이런 요청을 받으면 "이 기관은 국민이 직접 접촉하는 창구가 없다"고 정직하게 안내하는 것 자체가 §CAPABILITIES의 역할이 될 수 있다.

## 3-1. jeju-router.js 연동 및 검증 중 발견·수정한 버그 3건

`_renderTemplate()`로 실제 렌더링 테스트를 돌려본 뒤에야 발견됐다 — 코드 리뷰만으로는 안 잡혔을 것들이다.

1. **문서 코드 줄의 `{도메인코드}`가 실제 JSON 필드명(`domain`)과 불일치** — unresolved 토큰으로 남았을 것. `<domain 값>` 설명 텍스트로 교체.
2. **"신설 배경" 설명문 안의 `{지사명}`이 옛 템플릿 패턴을 인용한 것일 뿐인데 `{}` 표기 때문에 실제 치환 대상으로 오인식됨** — 따옴표 표기로 교체.
3. **`근거법령` 플레이스홀더가 템플릿에는 있었는데 JSON 레코드에는 해당 필드 자체가 아예 없었음** — 41개 전체에 필드 추가. 확실히 아는 12개(헌법·법원조직법 등 안정적 법령)만 채우고, 최근 정부개편(2025~2026)으로 설치법 명칭이 바뀌었을 위험이 있는 나머지 29개는 지어내지 않고 TBD로 명시.

수정 후 41개 레코드 전부 `_renderTemplate()` 일괄 실행 → unresolved 토큰 0개 확인.
