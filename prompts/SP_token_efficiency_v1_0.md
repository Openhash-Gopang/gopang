# SP-COMMON-03 · 토큰 절약 프로토콜(프롬프트 캐싱 강제 원칙) v1.0

> **적용 대상**: AGENT-COMMON, 모든 EXPERT 페르소나 SP(변호사·간호사·APN 등),
> K-Law/K-Tax/K-Public을 비롯한 모든 서브시스템의 시스템 프롬프트 및 LLM 호출
> 코드. 신규 SP·서브시스템을 추가할 때 이 문서의 원칙을 위반하지 않는지 반드시
> 확인한다 — 위반해도 기능은 정상 동작하지만, 비용이 조용히 수십~수백 배
> 커진다는 점에서 가장 위험한 종류의 결함이다.

## 배경

혼디의 모든 서브시스템은 DeepSeek API 계정 **하나**를 수백 명이 공유한다
(`worker.js`의 `env.DEEPSEEK_API_KEY`). DeepSeek을 비롯한 대부분의 LLM
벤더는 **동일 API 키로 들어오는 요청들의 프롬프트 앞부분(prefix)이 바이트
단위로 일치하면** 그 구간을 캐시 히트로 처리해 훨씬 싼 단가를 적용한다
(`hondi-flash` 기준 캐시 미스 대비 캐시 히트가 약 50배 저렴 — `worker.js`
`HONDI_TIER_MODELS` 가격표 참조).

이 캐싱은 "첫 사용자만 SP를 보내고 이후엔 안 보낸다"는 뜻이 **아니다** —
API는 상태가 없으므로(stateless) 시스템 프롬프트는 매 호출 반드시 다시
보내야 한다. 절약이 일어나는 지점은 **"보내는 내용이 이전과 완전히 같을
때, 벤더가 그 구간을 다시 계산하지 않고 캐시에서 가져와 싸게 청구한다"**는
것뿐이다. 따라서 이 캐싱 혜택을 받을지 여부는 전적으로 우리 코드가
"프롬프트를 얼마나 변하지 않게 구성하는가"에 달려 있다.

## T1. System 메시지 완전 정적성 원칙

시스템 프롬프트(`role: 'system'`)의 내용은 **세션 내내, 그리고 모든
사용자·모든 사건(case)에 걸쳐 완전히 동일**해야 한다. 다음 값은 system에
절대 포함하지 않는다:

- GUID·닉네임·전화번호 등 사용자 식별 정보
- 사건번호·타임스탬프·현재 시각
- 대화별로 달라지는 상태(심급·사건분류 등 누적 파악된 사실관계)
- 그 밖에 "이번 호출에서만" 유효한 어떤 값이든

**위반 예시(실제 발견된 사례)** — K-Law STEP 0~C 판결 생성 프롬프트
(`buildKlawSimPrompt`)가 `사건번호`와 `심급`을 system 문자열 안에 직접
삽입하고 있었다:

```js
// ❌ 위반 — caseNo/level이 매 사건마다 달라지므로 system이 매번 달라짐
return `... [K-Law ${klawVersion} 적용 | 사건번호: ${caseNo} | 심급: ${level}]`;
```

이 경우 방법론 원문(공통 원칙 + STEP 절차문, 수만~수십만 자)이 system에
같이 들어있음에도 불구하고, 사건번호 하나 때문에 **모든 호출이 캐시 미스로
처리된다** — 방법론을 STEP별로 분할해 크기를 줄인 의미가 사건번호 삽입
하나로 무효화된다.

## T2. 동적 데이터는 반드시 user 메시지에

세션마다·턴마다 달라지는 값은 system이 아니라 **매 호출의 user 메시지
앞부분에 병합**한다. `call-ai.js`의 `_buildEnhancedUserContent`가 이미
이 패턴을 정확히 구현하고 있다 — GUID·위치·PDV 요약을 system이 아닌 user
메시지에 넣는다. 새 SP를 만들 때 이 패턴을 그대로 따른다.

## T3. 대화 이력 불변 추가(append-only) 원칙

한 세션(하나의 사건, 하나의 대화) 안에서 이전에 전송한 메시지는 **절대
다시 쓰거나 수정하지 않는다** — 오직 뒤에 추가(append)만 한다. 벤더의
캐시는 "이전 호출과 동일한 접두사(prefix)"를 인식하는 방식이므로, 앞선
메시지 내용이 한 글자라도 바뀌면 그 이후 전체가 캐시 미스로 전환된다.

- ✅ K-Law 인터뷰 SP(`webapp.html`)의 `conversationState.messages`는
  매 턴 `push`만 하고 기존 원소를 수정하지 않는다 — 이 원칙을 준수한다.
- ❌ 예를 들어 "지금까지 파악된 사실관계 요약"을 매 턴 system이나 첫
  user 메시지에 다시 써넣어 "최신화"하는 방식은, 매 턴 접두사가 바뀌어
  캐시를 계속 깨뜨리므로 금지한다. 누적 상태가 필요하면 T1/T2에 따라
  최신 user 메시지 쪽에만 추가하고, 이전 메시지는 그대로 둔다.

## T4. 대용량 참고자료는 "분할하되, 분할된 조각도 정적으로" 유지

K-Law 방법론 원문처럼 큰 참고자료를 STEP·주제별로 나눠 필요한 조각만
투입하는 것은 권장되는 절약 기법이다(`_splitKlawMethodology` 참조). 단,
**분할된 조각 자체는 T1을 그대로 만족해야 한다** — 조각을 system에 넣을
때 그 조각 앞뒤로 사건별 동적 값을 같이 붙이면 T1 위반이 재발한다. 조각은
순수하게 "이 STEP에서는 항상 이 텍스트"인 상태로 system에 넣고, 사건
정보는 user 메시지로 분리한다.

## T5. SP 버전은 세션 내 무중단 고정

한 세션이 진행되는 도중에 더 최신 버전의 SP/방법론이 배포되어도, **그
세션은 시작할 때 로드한 버전을 끝까지 사용**한다. 세션 중간에 system
텍스트를 다른 버전으로 교체하면 그 시점 이후 캐시가 전부 깨지는 것은
물론, 판단 기준이 대화 중간에 바뀌는 정합성 문제도 함께 발생한다.

## T6. 신규 SP 병합 전 자가 점검 체크리스트

새 SP를 추가하거나 기존 SP의 호출 코드를 수정할 때, 병합 전 다음을
스스로 확인한다:

1. 이 system 문자열에 GUID·닉네임·사건번호·타임스탬프 등 **호출마다
   달라지는 값**이 문자열 보간(`${...}`)으로 섞여 있지 않은가?
2. 두 번의 서로 다른 호출(다른 사용자, 다른 사건)에서 이 함수가 반환하는
   system 문자열을 실제로 비교했을 때 **완전히 동일**한가?
3. 대화 이력에 이미 전송된 메시지를 이번 호출에서 수정하고 있지 않은가?
4. 참고자료를 분할했다면, 분할된 조각에 동적 값이 섞여 들어가지 않았는가?

이 네 가지 중 하나라도 "그렇다/아니다"에 걸리면 병합 전에 수정한다.
