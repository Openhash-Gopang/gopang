## 1. 전체 아키텍처 개요

```
[사용자 요청]
     │
     ▼
STEP 0  사전 게이트웨이 (잔액·상한 확인)
     │  통과
     ▼
STEP 1  DeepSeek API 호출 (모델 선택 포함)
     │  usage 필드 수신
     ▼
STEP 2  토큰 → GDC(μT) 환산
     │
     ▼
STEP 3  원장 기록 (fs_ledger 차감, OpenHash 앵커링)
     │
     ▼
STEP 4  이상사용 탐지 (공리 O 확장 트리거)
     │
     ▼
STEP 5  사용자 고지 UX (예상비용/잔액 표시)
     │
     ▼
STEP 6  월간 정산 대사 (DeepSeek 인보이스 ↔ GDC 원장)
     │
     ▼
STEP 7  캐시 최적화 (프리픽스 고정 전략)
     │
     ▼
STEP 8  예외처리 (API 실패·부분응답·환불)
```

## STEP 0 — 사전 게이트웨이

기존 STEP A 게이트웨이(잔액조회/이체/대출 공통 STEP 0-β,0-γ)를
AI 호출 경로에도 동일하게 적용한다.

```
[STEP-0 AI 호출 게이트웨이]
1. 잔액 확인: 사용자 GDC 잔액(μT 단위) ≥ 최소 예약금(reserve_μT)
2. 일/월 상한 확인: 누적 사용액 < 사용자 설정 상한(cap_krw)
   - 기본 상한값: 신규 가입자 월 10,000원 (관리자/사용자 조정 가능)
3. 이상사용 사전 점검: 공리 O 확장 트리거(STEP 4) 재확인
4. 요청 예상 비용 산정(概算): max_tokens 기준 최악의 경우 비용을
   미리 계산해 reserve_μT로 임시 홀드(hold) — 실제 차감은 STEP 3에서
   실사용량 기준으로 확정, 차액은 즉시 반환

[게이트웨이 판정]
  통과 → STEP 1
  잔액 부족 → 충전 안내, 호출 차단
  상한 초과 → 상한 상향 확인 요청 또는 다음 정산주기까지 대기 안내
  이상사용 트리거 → 추가 인증/속도제한 후 재판정
```

reserve_μT 홀드 방식을 쓰는 이유: max_tokens가 큰 요청은 실제
소비 전에는 정확한 비용을 알 수 없기 때문에, "최악의 경우 비용"을
임시로 잠가두고 실제 사용량 확정 후 정산하는 방식(카드사의 승인-매입
구조와 동일한 원리)이 잔액 마이너스를 원천 차단한다.

────────────────────────────────────────────────────────────

## STEP 1 — DeepSeek API 호출

```
[모델 선택 규칙]
- 기본: deepseek-v4-flash (단순 조회, 일반 대화, 대부분의 SP 라우팅)
- 승급: deepseek-v4-pro (복잡한 법률 분석, 다단계 추론이 필요한
  K-Law 서면 작성 등 — 사전에 정의된 SP별 모델 매핑 테이블 사용)
- 승급 시 사용자에게 "이 요청은 고급 모델을 사용하며 비용이
  약 3배 높습니다"를 STEP 0 단계에서 사전 고지
```

호출 후 반드시 응답의 usage 필드를 그대로 로그에 보존한다:
`prompt_cache_hit_tokens`, `prompt_cache_miss_tokens`, `completion_tokens`.
이 세 값이 STEP 2 환산의 유일한 입력이며, 유실되면 해당 요청은
사후 원가 재구성이 불가능해지므로 요청-응답 쌍을 원자적으로
기록해야 한다(트랜잭션 실패 시 요청 자체를 재시도 큐로).

────────────────────────────────────────────────────────────

## STEP 2 — 토큰 → GDC 환산 (핵심 공식)

### 2-1. 정밀도 문제 해결: μT(마이크로-T) 내부 단위 도입

1T = 1원(대사용자 표시 단위)이지만, 개별 요청 원가는 통상 0.2~1원
수준으로 정수 T보다 훨씬 작다. 매 요청마다 정수 T로 반올림하면
누적 손실이 커지므로, **원장 내부적으로만** 1T = 1,000,000μT의
고정소수점 정밀도를 사용한다. 사용자에게 노출되는 잔액 표시는
항상 정수 T로 반올림하되, 실제 차감·적립은 μT 단위로 정확히
수행한다.

### 2-2. 환산 공식

```
cost_usd =
    (hit_tokens   × P_hit  / 1,000,000)
  + (miss_tokens  × P_miss / 1,000,000)
  + (out_tokens   × P_out  / 1,000,000)

cost_krw = cost_usd × FX_rate_KRW_per_USD

charge_μT = round( cost_krw × MARKUP × 1,000,000 )

사용자 잔액(μT) -= charge_μT
```

- `FX_rate_KRW_per_USD`: 원-달러 환율. 실시간 연동이 이상적이나,
  변동성 리스크 흡수를 위해 **주 1회 갱신 + 5% 버퍼**를 기본값으로
  한다(즉 실제 환율보다 약간 높게 고정해 환율 급등 시에도 마진이
  마이너스가 되지 않도록).
- `MARKUP`: 초기 권장값 5배 (원가 대비). 원가가 워낙 낮아 5배를
  적용해도 사용자 체감가는 매우 저렴하다(아래 3번 예시 참조).

### 2-3. 계산 예시 (deepseek-v4-flash, FX=1,400원/$, MARKUP=5)

| 시나리오 | 캐시미스 입력 | 출력 | 원가(USD) | 원가(KRW) | 청구(KRW, ×5) |
|---|---|---|---|---|---|
| 일반 대화 1턴 | 2,000 | 500 | $0.00042 | 0.59원 | 약 3원 |
| 캐시 히트(반복 SP) | 2,000(hit) | 500 | $0.000146 | 0.20원 | 약 1원 |
| K-Law 서면 초안(긴 컨텍스트) | 50,000 | 3,000 | $0.0079 | 11.06원 | 약 55원 |

기본료 1,000원은 대략 이 요금 체계에서 **일반 대화 300여 턴**에
해당하는 완충 재원이 되며, 그 이상 사용 시 초과분만 μT 단위로
정밀 과금된다.

## STEP 3 — 원장 기록

```
[GDC-BILLING 트랜잭션 기록]
tx_type: "ai_usage_charge"
user_id: {GUID}
model: {deepseek-v4-flash|deepseek-v4-pro}
hit_tokens / miss_tokens / out_tokens: {N}
cost_usd / cost_krw / charge_μT: {산출값}
fx_rate_snapshot: {당시 적용 환율}
markup_applied: {배수}
잔액_변경전 / 잔액_변경후: {μT}
OpenHash 앵커링: {tx_hash}
```

fs_ledger에 요청 단위로 append-only 기록하며, STEP 6 월간 대사의
유일한 근거자료가 되므로 삭제·수정 불가(불변 원장 원칙 — 기존
OpenHash 철학과 동일).

────────────────────────────────────────────────────────────

## STEP 4 — 이상사용 탐지 (공리 O 확장)

기존 공리 O(이상거래 트리거) 테이블에 AI 사용량 관련 항목을 추가한다.

| 트리거 | 임계값 | 조치 |
|---|---|---|
| 단시간 대량 호출 | 1분 내 10회↑ | 요청 속도 제한(rate limit) |
| 비정상 출력 길이 요청 | 상위 1% max_tokens | 추가 확인 또는 차단 |
| 캐시 미스율 급증 | 개인 평소 대비 3배↑ | 프롬프트 패턴 점검(캐시 깨짐 원인 조사) |
| 짧은 시간 내 상한 소진 | 일 상한의 80%를 1시간 내 소진 | 임시 차단 + 사용자 알림 |
| 반복 동일 프롬프트 대량 요청 | 10분 내 동일 프롬프트 5회↑ | 캐시 강제 적용 또는 요청 거부(어뷰징 의심) |

공유 키 구조의 특성상, 한 사용자의 이상행동이 전체 서비스
품질(지연·503)과 DeepSeek 측 어뷰징 탐지에 영향을 줄 수 있으므로
이 계층은 개인 보호가 아니라 **플랫폼 전체 보호** 목적임을
명확히 한다.

────────────────────────────────────────────────────────────

## STEP 5 — 사용자 고지 UX (소비자보호)

전자상거래법·통신과금서비스 관련 법령의 사전고지 취지를 반영한다.

- **요청 전 예상비용 표시**: max_tokens가 큰 요청(임계값 이상)은
  "예상 비용 약 N원" 표시 후 사용자 확인
- **잔액·누적사용액 상시 노출**: 앱 상단에 "이번 달 사용액: N원 /
  상한 M원" 형태로 항상 표시
- **상한 도달 임박 알림**: 80% 도달 시 푸시 알림
- **월간 사용내역 명세서**: 요청별 상세 내역(날짜, 모델, 토큰량,
  비용)을 사용자가 조회 가능하도록 제공 — STEP 3 원장을 그대로
  사용자 조회용 뷰로 노출

## STEP 6 — 월간 정산 대사 (Reconciliation)

```
[월간 대사]
DeepSeek 실제 청구액(USD, 인보이스) = A
GDC 원장상 실제 원가 합계(STEP 3 cost_krw 총합, 마크업 제외) = B
GDC 원장상 사용자 청구 합계(charge_μT 총합 → KRW 환산) = C

검증 1: A ≈ B (환율 스냅샷 오차 범위 내 일치해야 함 — 불일치 시
        토큰 카운팅 로직 또는 usage 필드 누락 여부 점검)
검증 2: C - B = 실질 마진 (이 서비스의 순이익 — 마크업 배수가
        적절한지 여기서 재평가)
```

SP-GDC-RESERVE의 준비금 대응비율 검증과 동일한 원리로, 이 대사가
어긋나면(A≠B) 시스템 버그 가능성을 우선 의심한다.

────────────────────────────────────────────────────────────

## STEP 7 — 캐시 최적화 전략

DeepSeek의 자동 프리픽스 캐싱은 프롬프트 시작 부분이 직전 요청과
바이트 단위로 일치할 때만 작동한다(캐시히트 시 입력비용 98%↓).
혼디의 SP/AC 체계는 반복 호출되는 고정 프리픽스라는 점에서
구조적으로 유리하지만, 다음 원칙을 지켜야 캐시 효율이 유지된다.

- SP 프롬프트를 요청마다 동적 삽입(사용자별 개인화 문구 등)으로
  변형하지 말 것 — 개인화는 프리픽스 뒤쪽(캐시 미적용 구간)에 배치
- SP-AUTHOR/AC-AUTHOR의 자기진화 개정 주기와 캐시 히트율은
  트레이드오프 관계 — SP가 자주 바뀌면 캐시가 매번 깨져 원가가
  상승한다. 개정은 배치(batch)로 묶어 정해진 주기에만 반영하는
  것을 권장(예: 주 1회 반영, 상시 반영 지양)
- 캐시 히트율을 STEP 3 원장에서 모니터링 지표로 축적 — 히트율이
  구조적으로 하락하면 SP 구조 자체를 재검토하는 신호로 사용

## STEP 8 — 예외처리

| 상황 | 처리 |
|---|---|
| DeepSeek API 오류(5xx, timeout) | 홀드된 reserve_μT 즉시 반환, 사용자에 재시도 안내, 과금 없음 |
| 부분 응답(스트리밍 중 중단) | 실제 전송된 completion_tokens만큼만 과금(usage 필드 기준) |
| 과금 후 사용자 이의제기 | STEP 3 원장(요청별 상세 기록)을 근거로 소명, 명백한 시스템 오류 시 환급 |
| reserve_μT 홀드 중 잔액 변동 경합 | 홀드 우선순위 큐 적용, 동시 요청 시 선착순 처리(TOCTOU 방지 — 기존
OpenHash BIVM 원칙과 동일하게 처리) |
