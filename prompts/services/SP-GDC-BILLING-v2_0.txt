## 0. 최종 확정 구조

| 항목 | 정책 |
|---|---|
| 결제수단 | GDC 선충전 (일회성 PG 결제 — 카드 단건 또는 계좌이체, 정기결제 아님) |
| 기본료 | 없음 |
| 무료 제공 | 없음 (평생 무료 쿼터 포함 전면 폐지) |
| 사용량 과금 | 잔액 게이트웨이 → 토큰 사용 → μT 차감 → 원장 기록 (v1.0 STEP 0~3 유지) |
| K-서비스(K-Law 등) | 독립 가격표·독립 일일한도 유지, 차감은 공유 GDC 잔액에서 |
| 과금 배수 | 실제 코드에 이미 구현된 `BILLING_MULTIPLIER`(기본 2, env override 가능) — v1.0에서 가정했던 5배가 아니라 이 값이 현재 운영 기준값이다 |

이 구조는 카드 던닝, 계좌 CMS 수수료, 미납 유예, 기본료 우선차감 같은
v1.0~진행 과정에서 논의됐던 복잡한 로직들이 전부 불필요해진다는 점에서
가장 단순하다. 대신 신규 가입자의 온보딩 마찰(충전 전엔 아무것도
못 함)을 사업적으로 감수한다는 전제가 깔려 있다 — 이건 설계 결함이
아니라 명시적으로 선택된 트레이드오프다(미수금·연체 리스크 0 ↔
가입 전환율 리스크).

## 1. STEP 0 게이트웨이 (v1.0 대비 단순화)

v1.0은 "잔액 확인 + 상한 확인 + 이상사용 확인" 3단이었다. 기본료·
무료쿼터가 사라지면서 "상한 확인" 중 월간 상한(cap_krw) 로직은
제거하고, 순수 잔액 확인으로 수렴한다.

```
[STEP-0 AI 호출 게이트웨이 — v2.0]
1. 잔액 확인: 사용자 GDC 잔액(μT) ≥ 요청 예상비용(reserve_μT)
   - 예상비용 산정: max_tokens 기준 최악의 경우 비용을 reserve_μT로
     임시 홀드(v1.0과 동일 원리)
2. 이상사용 사전 점검: 공리 O 확장 트리거(v1.0 STEP 4 그대로 유지)
3. K-서비스 호출인 경우, 해당 서비스의 독립 일일한도(예:
   KLAW_USER_DAILY_KRW_LIMIT)도 함께 확인 — 통과 조건은 AND
   (전역 GDC 잔액도 충분해야 하고, 서비스별 일일한도도 남아있어야 함)

[게이트웨이 판정]
  통과 → STEP 1
  잔액 부족 → 충전 안내, 호출 차단 (유예기간 없음 — 즉시 차단)
  서비스별 일일한도 초과 → "오늘 이 서비스는 한도 소진, 내일 재시도"
                          안내(전역 잔액과 무관하게 별도 소진)
  이상사용 트리거 → 추가 인증/속도제한 후 재판정
```

무료쿼터가 있던 v1.0의 "잔액 부족 시 충전 안내 후 대기 가능" 흐름과
달리, v2.0은 무료 완충재원이 전혀 없으므로 **충전 전까지는 그 어떤
AI 기능도 작동하지 않는다.** 이 사실이 온보딩 화면(가입 직후 첫
화면)에 명확히 드러나야 한다 — "먼저 충전해주세요"가 첫 경험이 된다는
뜻이므로, 최소 충전 단위·결제 UX를 최대한 마찰 없이 만드는 게
중요해진다(별도 UX 과제).

## 2. K-서비스 독립 가격 정책과 전역 잔액의 관계

기존 코드(`worker.js`)에 이미 이 구조의 초안이 구현되어 있었다 —
`HONDI_TIER_MODELS`(일반 챗), `KLAW_TIER_MODELS`(K-Law),
`BUSINESS_TIER_MODELS`(K-비즈니스), `GOV_TIER_MODELS`(K-정부) 4종이
각자 다른 가격표와 한도를 갖는다. 이 문서는 이 구조를 공식화한다.

### 2-1. 원칙
- 모든 K-서비스는 **동일한 GDC 잔액**에서 차감된다 — 서비스별로
  "K-Law 전용 지갑"이 따로 있지 않다.
- 단, K-Law처럼 원가가 높은 서비스는 **서비스 자체의 일일 한도**
  (1인 1일 KRW 한도, 계정 전체 1일 예산 상한, 1인 1일 실행 횟수 한도
  — 이미 KLAW_USER_DAILY_KRW_LIMIT 등으로 구현됨)를 추가로 가진다.
  이건 "한 사용자가 K-Law만 폭주시켜 GDC를 순식간에 소진하는 것"과
  "K-Law 서비스 전체가 예산을 초과 소비하는 것" 두 시나리오를
  각각 방어하기 위함이다.
- 사용자 입장에서는 "GDC 잔액이 있어도 K-Law는 오늘 한도가 다 차서
  못 쓴다"는 상황이 발생할 수 있다 — 이건 버그가 아니라 설계된
  방어선이며, UX에서 그 사유를 명확히 구분해서 안내해야 한다
  ("잔액 부족"과 "오늘 이 서비스 한도 소진"은 다른 메시지).

### 2-2. 원장 기록 확장 (v1.0 STEP 3에 태그 추가)
```
[GDC-BILLING 트랜잭션 기록 — v2.0]
tx_type: "ai_usage_charge"
service_id: "hondi-chat" | "klaw" | "kbiz" | "kgov" | ...  ← 신규 필드
user_id, model, hit_tokens, miss_tokens, out_tokens,
cost_usd, cost_krw, charge_μT, fx_rate_snapshot,
markup_applied(=BILLING_MULTIPLIER 실값), 잔액_변경전/후, OpenHash tx_hash
```
`service_id` 태그가 있어야 사용자가 "이번 달 K-Law에 얼마, 일반
챗에 얼마 썼는지" 조회할 수 있고(STEP 5 UX), 서비스별 수익성도
STEP 6 대사에서 분리 집계할 수 있다.

### 2-3. 동시성 잠금 (v1.0에 없던 구멍 — v2.0에서 보강)
GDC 잔액 하나를 여러 K-서비스가 동시에 노릴 수 있다(예: K-Law와
K-Market을 거의 동시에 호출). STEP 0의 reserve_μT 홀드는 **서비스
종류와 무관하게 사용자의 전역 잔액에 대해 원자적으로** 걸어야 한다.
서비스별 일일한도 체크는 이 원자적 홀드와 별개의 카운터이므로,
두 체크가 서로 다른 시점에 일어나며 경합하지 않도록 같은 트랜잭션
내에서 순서대로(전역 잔액 홀드 → 서비스별 한도 확인) 처리한다.

## 3. 실제 코드와의 정합성 확인 (2026-07-14 기준)

저장소 확인 결과, 다음 사항이 이미 구현되어 있거나 이번에 함께
정리됨:

- `computeBilledKRW()`, `_deepseekUsageToKRW()`, `_billingMultiplier()`
  — 실제 과금액 계산 로직 기존 구현 확인. v1.0에서 가정했던
  MARKUP=5는 폐기하고, 실제 코드의 `BILLING_MULTIPLIER_DEFAULT = 2`를
  기준값으로 채택한다(운영 데이터 쌓이면 재산정 — TODO 참조).
- `USD_TO_KRW = 1500` (고정값, 실시간 미연동) — v1.0의 "주 1회 갱신 +
  5% 버퍼" 권고안은 아직 미구현. TODO로 이관.
- `FREE_QUOTA_KRW_LIMIT`, `_recordFreeSpend()`, `handleFreeQuotaStatus()`,
  `/free-quota-status` 라우트 — 2026-07-14 전면 제거 완료(worker.js).
  제거 시 "게이트가 사라져 사실상 전원 무제한 무료"가 되는 위험한
  공백이 생기므로, 실제 GDC 차감 로직이 연결되기 전까지 **명시적으로
  요청을 차단하는 안전 스텁**(`_billingNotWiredYet`, HTTP 503
  `BILLING_NOT_CONFIGURED`)으로 대체해뒀다. **이 스텁이 그대로 있는
  동안은 AI 채팅 기능 자체가 막힌다** — 프로덕션 배포 전 반드시
  STEP 0/3의 실제 GDC 잔액 조회·차감 로직으로 교체할 것(TODO 최우선).
- L1(PocketBase)에는 `/api/balance`(조회 전용) 엔드포인트만 존재하고,
  잔액 **차감/이체** 엔드포인트는 아직 없다(`handleBizBalance` 참조).
  STEP 0/3 구현의 전제조건.
