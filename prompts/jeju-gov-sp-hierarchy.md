# 제주 행정 도메인 SP(System Prompt) 계층 구조 설계 (가안)

> 목적: "제주도청 전문 AI"가 어떤 질의를 받았을 때, 어떤 SP들을 어떤 순서로 순차 호출(누적 주입)해야 하는지에 대한 위계 규칙을 정의한다.
> 기존 Gopang/Hondi의 AGENT-COMMON → GWP/EXPERT 태그 라우팅 → 도메인별 SP 체인 구조와 동일한 철학을 제주 행정 도메인에 이식한 것으로 가정한다.
> 부서 명칭·조직 개편은 매우 잦으므로(실·국 명칭이 최근 수년간도 특별자치행정국/복지가족국 등으로 여러 차례 개편), 아래 SP ID는 **조직명이 아니라 안정적인 내부 코드**로 설계하고, 실제 조직명은 매핑 테이블(별도 JSON/DB)에서 관리하는 것을 전제로 한다.

---

## 1. 왜 "도청 SP → 시청 SP → 읍면동 SP → 업무 SP"인가 — 위계의 근거

이전에 정리했던 제주 행정 구조(도의회/도청/시청/읍면동)에서 확인했듯, 제주는 **단층제**이지만 실무 행정 전달 체계는 여전히 "도청 → 행정시(제주시·서귀포시) → 읍·면·동" 3단 피라미드로 작동한다. SP 설계도 이 피라미드를 그대로 반영해야 하는 이유는 다음과 같다.

1. **공통 맥락은 상위에서, 특수 맥락은 하위에서 주입** — 도정 전체에 적용되는 법적 고지, 어조, 개인정보 처리 원칙(PDV 등)은 도청 SP 한 곳에서만 관리하고, 하위 SP는 이를 상속받아 중복 정의하지 않는다.
2. **프리픽스 캐싱 유지** — HONDI-FAQ 라우터가 DeepSeek 프리픽스 캐시를 살리기 위해 상위 레이어를 고정하고 하위만 가변시켰던 것과 동일한 원리로, 상위 SP(도청 공통)일수록 변경 빈도가 낮으므로 앞쪽에 고정 배치해야 캐시 히트율이 높아진다.
3. **조직 개편에 대한 격리(decoupling)** — 실·국 명칭이 바뀌어도(예: "보건복지여성국"→"복지가족국") 도청 공통 SP는 영향받지 않고 해당 실국 SP 한 장만 교체하면 된다.
4. **권한·관할의 실제 구조를 반영** — 읍면동은 대개 접수·안내 창구이고 실제 인허가·정책 결정권은 시청 담당과 또는 도 직속기관에 있는 경우가 많다. 이 위계 차이를 SP 체인 순서에도 반영해야 답변의 "책임소재"가 틀어지지 않는다.

## 2. SP 레벨 정의

```
L0  JEJU-GOV-COMMON        (제주 행정 도메인 전체 공통 SP — AGENT-COMMON 하위 확장판)
L1  JEJU-DO-SP             (제주특별자치도청 공통 SP)
L2  ┌─ 실·국 계열 SP        (도청 직속 실/국, 도지사 직할)
    ├─ 직속기관/사업소 SP   (상하수도본부, 자치경찰단, 소방안전본부, 보건환경연구원 등)
    ├─ 행정시 SP            (제주시 SP / 서귀포시 SP)
    └─ 출자·출연기관 SP     (제주개발공사, 제주관광공사, 제주에너지공사 등 — 별도 법인격)
L3  (행정시 SP 하위)
    ├─ 시청 국·과 SP        (자치행정국, 복지위생국, 경제일자리국 등)
    └─ 읍·면·동 SP          (한림읍, 애월읍, 노형동 등 — 제주시 43개 / 서귀포시 12개 읍면동)
L4  (읍면동 SP 하위, 또는 시청 국·과 SP 하위)
    └─ 업무영역(전문) SP    (상하수도, 세정, 복지급여, 농업경영체등록, 인허가 등)
```

핵심 규칙: **L4(업무영역 SP)는 고정된 트리 위치가 아니라, 해당 업무의 실제 처리 주체가 누구냐에 따라 L3(읍면동) 밑에 붙을 수도, L2(시청 국·과 또는 직속기관) 밑에 직접 붙을 수도 있다.** 이것이 사용자가 예시로 든 "한림읍 상하수도"와 "도청 보건복지여성국"이 호출 깊이가 다른 이유다.

## 4. 호출 체인(call chain) 규칙 — 사용자 예시 검증 및 일반화

### 규칙 A. "실·국 직속 업무"는 짧게 (도청 → 실국)
> 사용자 예시: 보건복지여성국(현 복지가족국) 문의

```
JEJU-GOV-COMMON → SP-DO-000(도청) → SP-DO-WELFARE(복지가족국)
```
필요시 그 아래 세부 업무 SP(SP-EXP-CHILDCARE 등)까지 3단으로 확장 가능하지만, 국 단위 SP만으로 답변 가능한 일반 문의는 2단에서 종료.

### 규칙 B. "읍면동 창구 + 광역 사업소 업무"는 길게 (도청 → 시청 → 읍면동 → 업무SP)
> 사용자 예시: 한림읍 상하수도 문의

```
JEJU-GOV-COMMON → SP-DO-000(도청) → SP-CITY-JEJU(제주시청) → SP-EMD-HALLIM(한림읍) → SP-EXP-WATER(상하수도 전문지식, SP-AGY-WATER 참조)
```
이유: 상하수도는 ①정책·요금체계·시설은 도 직속 **상하수도본부**(SP-AGY-WATER) 소관이지만, ②주민이 실제 접촉하는 창구는 **한림읍사무소**이므로, 행정 전달 경로(도청→시청→읍) 그대로 SP를 쌓은 뒤 맨 끝에 전문지식 레이어(SP-EXP-WATER)를 얹는다. 즉 **L4 업무 SP는 "관할 조직 위계"가 아니라 "실제 지식 소스"를 참조하되, 체인 상 위치는 사용자가 접촉하는 창구(읍면동) 뒤에 온다.**

### 규칙 C. "시청 국·과가 직접 처리하고 읍면동은 관여하지 않는 업무"는 읍면동을 건너뛴다
> 예: 건축 인허가, 대규모 개발사업 심의 (제주시 도시건설국/서귀포시 안전도시건설국 소관, 읍면동은 서류 접수만 하거나 아예 관여하지 않음)

```
JEJU-GOV-COMMON → SP-DO-000 → SP-CITY-JEJU → SP-JEJUSI-DOSI(도시건설 담당과) → SP-EXP-BUILDING(건축인허가 전문)
```
읍면동 SP를 생략하거나, 생략하지 않더라도 "접수 창구 안내" 수준의 얕은 역할로만 참조.

### 규칙 D. "도 직속 사업소가 전국/도 전역을 단일 창구로 완결 처리"하는 업무는 행정시·읍면동을 아예 건너뛴다
> 예: 자치경찰 관련 신고, 세계유산(한라산 등) 특별 인허가

```
JEJU-GOV-COMMON → SP-DO-000 → SP-AGY-POLICE(자치경찰단)
```

### 규칙 E. "출자·출연기관" 업무는 도청과 법인격이 다름을 명시하며 분리된 가지로 호출
> 예: 제주삼다수 채용문의(제주개발공사), 관광공사 이벤트 문의

```
JEJU-GOV-COMMON → SP-DO-000 → SP-ORG-JPDC(제주개발공사)
```
이 경우 SP-DO-000 단에서 "본 기관은 제주도가 출자한 별도 법인격 공기업이며 도청 소속 공무원 조직이 아님"이라는 고지 문구를 자동 삽입하도록 설계(사용자 혼동 방지, 실제 K-Law SP의 disclaimer 관행과 동일한 패턴).

### 규칙 F. 서귀포시는 제주시와 조직 구조가 다르므로 하드코딩 금지
서귀포시는 상하수도과가 시청 **안전도시건설국** 내부에 있어(제주시는 도 직속 상하수도본부 체계와 결합), 동일한 "상하수도" 키워드라도 관할 시(市)에 따라 체인이 달라진다.

```
[제주시] 상하수도 문의 → ... → SP-CITY-JEJU → SP-EMD-{읍면동} → SP-EXP-WATER(→SP-AGY-WATER 참조)
[서귀포시] 상하수도 문의 → ... → SP-CITY-SEOGWIPO → SP-SGP-CONSTRUCT(안전도시건설국) → SP-EXP-WATER
```
→ 따라서 라우터(gwp-registry.js 상당)는 "업무영역 키워드"만으로 SP를 고정 매핑하지 말고, **먼저 지역(행정시/읍면동)을 식별한 뒤, 그 지역의 조직표 매핑 테이블을 조회하여 체인을 동적으로 조립**해야 한다. 이는 기존 industry-router.js(77개 KSIC 업종 매핑)와 유사하게, "region-org-map.json" 같은 별도 매핑 리소스를 두는 방식을 권장.

## 5. 라우팅 파이프라인 설계 제안 (기존 아키텍처와의 정합성)

기존 GWP/EXPERT 태그 체계(`[GWP:]`, `[EXPERT:]`)를 그대로 확장한다고 가정하면:

1. **1차 라우팅 — 지역 식별**: 사용자 발화에서 지역명(한림읍, 서귀포시 등) 추출 → 없으면 사용자 프로필(PDV)의 거주지 기본값 사용, 그래도 없으면 "제주도 전역" 기본 SP만 사용
2. **2차 라우팅 — 업무영역 식별**: GWP/EXPERT 태그 방식과 동일하게 키워드 → 업무영역 코드 매핑 (예: "누수", "수도요금" → SP-EXP-WATER)
3. **3차 — 체인 조립**: `region-org-map.json`에서 (지역, 업무영역) 조합에 해당하는 SP 경로를 조회 → 규칙 B~F 중 어느 패턴인지 결정
4. **4차 — 순차 주입**: SP-DO-000(공통, 캐시 고정) → 중간 노드(가변, 캐시 일부 재사용) → 최종 업무 SP(가장 가변적, 캐시 미적용) 순으로 프롬프트에 쌓아 DeepSeek 프리픽스 캐싱 극대화(HONDI-FAQ 라우터와 동일 원칙)
5. **5차 — 관할 불명확 시 처리**: 규칙 C/D처럼 읍면동이 관여하지 않는 업무인데 사용자가 읍면동을 언급한 경우, SP 체인에는 넣지 않되 "귀하가 언급하신 한림읍사무소에서는 이 업무를 접수하지 않으며, ○○시청 △△과로 문의하셔야 합니다"라는 안내를 하위 SP가 아닌 **라우터 단(gwp-registry.js급)에서 미리 검증**하는 것을 권장 — 잘못된 관할 정보를 SP 콘텐츠에 섞으면 추후 조직 개편 시 오류 수정이 어려워짐.

## 6. 조직 개편에 대한 유지보수 전략

- 실·국명은 최근 수년간도 "보건복지여성국→복지가족국", "도시건설국→건설주택국/도시균형추진단" 등으로 자주 바뀜. **SP 파일명·ID는 조직명이 아닌 기능 코드(SP-DO-WELFARE 등)로 고정**하고, "현재 조직명은 무엇인가"는 별도 매핑 테이블 1곳에서만 관리 → 개편 시 SP 본문은 그대로 두고 매핑 테이블 1줄만 수정
- 도지사 임기(민선 교체) 때마다 실·국 수·명칭이 통째로 바뀌는 경향이 있으므로, L2 실국 계열 SP는 "이 SP가 커버하는 정책 영역"을 명칭이 아니라 **업무 키워드 집합**으로 정의해 두면 조직 개편 후에도 라우팅 로직 자체는 수정이 최소화됨
- 43개 읍면동 전부를 개별 SP로 만들면 유지보수 부담이 크므로, 대부분의 읍면동은 **공통 읍면동 템플릿 SP(SP-EMD-TEMPLATE)** + 지역별 변수(지역 특산물, 관할 리 목록, 담당자 연락처 등)만 주입하는 방식을 권장. 한림읍처럼 특수 이슈(마늘·양배추 등 밭작물 비중, 비양도 연륙 여부 등)가 많은 지역만 별도 SP로 분리하는 것이 현실적.
