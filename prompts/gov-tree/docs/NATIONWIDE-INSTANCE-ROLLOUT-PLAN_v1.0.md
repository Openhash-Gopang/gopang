## 0. 요지

제주는 도청(1)·시청(2)·읍면동(26)·국가기관 지역사무소(34)가 전부 인스턴스화돼
있고, 이 네 계층이 각각 **원형 클래스(템플릿)**로 이미 추출돼 있다
(`SP-PROVINCE-TEMPLATE`, `SP-DEPT-*-TEMPLATE`×16, `SP-CITY-TEMPLATE`,
`SP-CITYDEPT-TEMPLATE`, `SP-EMD-TEMPLATE`, `SP-NAT-*-TEMPLATE`×34). 다른 15개
광역시도는 이 원형의 인스턴스가 **도청 실국만 부분 완비**(12개 도, 완비도
8/16~16/16)이고, **시청·읍면동·국가기관 지역사무소는 0개**다. 이 문서는 그
공백을 메우는 작업 계획이다.

## 2. 원칙 (이번 대화 세션 전체에서 확립된 것 — 재확인 없이 그대로 적용)

이 원칙들은 전부 이번 프로젝트 진행 중 실제로 겪은 사고에서 도출된 것이라,
다시 어기면 같은 사고가 재발한다. 작업 착수 전 반드시 숙지.

1. **지어내지 않는다.** 확인 안 된 조직명·조번호·연락처는 "TBD"로 정직하게
   남긴다(GAP-LIST-50의 A/B급 신뢰도 구분과 동일 철학).
2. **약어·코드만으로 기관을 추정하지 않는다.** 작업 2(기타공공기관
   재검증)에서 Research 결과의 다수가 실제 §1 원문 대조 결과 틀렸던 전례가
   있다 — 반드시 1차 출처(해당 지자체 공식 조직도·law.go.kr)로 대조 후
   반영.
3. **복수 독립 출처 교차검증.** 최소 2개 이상(공식 홈페이지 + 나무위키/
   위키백과 등)으로 대조. 최근 개편·통폐합(예: 2026-07-01 전남·광주 통합)
   가능성을 항상 열어둔다(U2-1 원칙).
4. **govType 구분을 반드시 지킨다.** SPECIAL_AUTONOMOUS(현재 제주만)는
   세정 등을 도가 직할하지만, GENERAL(나머지 전부)은 기초자치단체(시/군/구)
   소관이다. 제주 규칙을 다른 도에 그대로 투사하지 않는다.
5. **키워드 과잉일반화 금지.** '복지'·'인허가'·'교통' 같은 지나치게 포괄적인
   단일 단어는 넣지 않는다(자치행정국/복지위생국 오배정 사고 전례). 구체적인
   과 이름·사무 이름 위주로 구성.
6. **행정구역 접미사(시/군/구) 요구 한계를 인지한다.** "서귀포"처럼 접미사
   없는 지명은 도 판별에 안 걸린다는 게 확인된 한계 — 새 데이터를 추가할
   때 이 한계를 악화시키지 않도록(정식 명칭 기준으로 등록) 주의.
7. **실제 코드로 검증한다.** 문서만 쓰고 끝내지 않고, 100건 사고실험과
   동일한 방식(`assembleGovSystemPrompt`를 실제 import해서 시나리오를
   흘려보내는 시뮬레이션)으로 각 배치마다 회귀 검증.
8. **매번 새로 clone → 검증 → patch → 적용 확인 → 재검증** 절차를 지킨다

## 3. 계층별 작업 정의

### 3-1. 도청(광역) 실국 — 이미 부분 진행 중, 나머지 완비 + 신뢰도 낮은 것 재검증

- **완비 대상**: 서울·인천·대전·울산·세종·충북·충남(15/16, 1개만 부족)·
  경북·경남 — 부족한 도메인만 채우면 됨.
- **신규 착수**: 경기·광주·대구·강원 — 처음부터 16개 도메인 실사.
- **재검증(신뢰도 낮음 플래그 해소)**: 인천(예정 조례 통과 여부 재확인),
  경북(조직개편 확정 여부), 경남(스냅샷 재대조).
- **산출물**: 각 도의 `{도명}_L2_TABLE` 상수(키워드) + 실제 안내 내용은
  `SP-DEPT-*-TEMPLATE`(이미 있음)을 인스턴스화하는 `do-dept-master-data.json`
  레코드(도이름/부서명/산하과목록/콜센터명/콜센터번호 등).
- ⚠️ **이 둘은 반드시 쌍으로 추가한다.** 오늘 국가기관 6개(세관·한국은행·
  통계청 등)가 템플릿·마스터데이터는 완비돼 있는데 `JEJU_NATIONAL_TABLE`
  키워드 배선만 빠져서 전부 일반 안내로 떨어지던 버그를 실제로 발견·수정한
  전례가 있다 — "콘텐츠 저작 완료"와 "실제로 라우팅됨"은 별개이고, 둘 다
  안 하면 조용히 실패한다(에러가 안 나서 발견이 늦어짐). 데이터 레코드
  추가 후 반드시 해당 키워드로 실제 발화를 흘려 trace를 눈으로 확인한다.

### 3-2. 시청(기초자치단체) — 전면 신규 착수, 규모가 가장 큼

- 전국 시/군/구는 226개(자치구 포함 기준 다소 변동 가능, sigungu-national-list.json
  기준 230개 레코드 확인됨 — 정확한 수는 착수 시 재확인).
- `SP-CITY-TEMPLATE`(이미 있음)의 인스턴스화. 자리표시자: 시이름, 통치구조_문구,
  행정구역구성_문구, 관할읍면동목록, 상하수도_capability_문구 등.
- ⚠️ 도청과 동일하게 **`{도명}_CITY_TABLE`(키워드 배선) + `city-master-data.json`
  레코드(콘텐츠) 둘 다** 있어야 실제로 라우팅된다(§3-1의 배선 경고 참고).
- **전수 인스턴스화는 비현실적** — 우선순위 방식 채택(§5 참고). 완료 안 된
  시/군/구는 계속 `resolveSigunguDept`(지연조회)로 처리(현행 유지, 안전망
  역할).
- 시청 국(局) 단위(`SP-CITYDEPT-TEMPLATE`, 제주 전례 있음)까지는 1차 범위에서
  제외 — 시청 레벨(광역 정보) 우선, 국 단위는 2차 심화.

### 3-3. 읍면동사무소 — 시청 인스턴스화가 끝난 지역부터 순차 착수

- `SP-EMD-TEMPLATE`(이미 있음)의 인스턴스화. 제주는 26개 읍면동 전부
  완비 — 다른 지역은 인구·행정동 수가 훨씬 많아(예: 서울만 425개 행정동)
  전수 작업이 아니라 **시청 레벨에서 지연조회로 흡수하는 현행 구조를
  기본값으로 유지**하고, 읍면동 단위까지 정적 인스턴스화하는 건 극히
  예외적으로 문의 빈도가 매우 높은 지역(예: 인구 최다 기초자치단체
  1~2곳 파일럿)에만 한정한다. 전국 읍면동 전수 인스턴스화는 이 계획의
  범위 밖으로 명시적으로 제외.

### 3-4. 국가기관 지역사무소 — 부처별 지사 목록 기반, 시청과 별도 트랙

- `SP-NAT-*-TEMPLATE`(34개, 이미 있음)의 인스턴스화. 도별로 각 부처의
  지역 지사명만 다르고(예: "OO지방국세청 XX세무서") 업무 지식은 전국
  공통이므로, 콘텐츠 저작 부담은 시청보다 훨씬 작다 — **국가기관 마스터
  데이터에 도별 레코드(지사명)만 추가하면 되는 구조**(제주에서 이미
  검증된 패턴).
- ⚠️ **바로 이 계층에서 오늘 실제로 배선 누락 버그가 났다** — customs·
  bok·stat 등 6개 기관이 마스터데이터·템플릿은 완비된 채로
  `JEJU_NATIONAL_TABLE`(키워드 배선)에만 안 올라가 있어서 전부 놓치고
  있었다. 도별로 지사 레코드를 추가할 때마다 `{도명}_NATIONAL_TABLE`
  (신설 필요 — 지금은 제주만 있음) 등록을 빠뜨리지 않는다. 체크리스트:
  레코드 추가 → 키워드 배선 추가 → `ROUTE_DESCRIPTIONS`에 이미 있는
  코드인지 확인(방금 전국 공통으로 추상화해뒀으므로 대부분 그대로 재사용
  가능, 신규 기관이면 추가) → 실제 발화로 trace 확인.
- 우선순위: 인구 상위 광역시도부터, 그리고 34개 기관 중 문의 빈도가 높을
  것으로 예상되는 기관(세무서·건강보험공단·국민연금공단·출입국청·경찰청
  등 10개 내외)부터 먼저 전체 도에 채우고, 나머지 24개는 후순위로 넓힌다
  (세로 우선이 아니라 가로 우선 — "이 도 34개 다 채우고 다음 도"가 아니라
  "이 10개 기관을 전국에 다 채우고 나머지 24개로").

## 4. 구조적 선결 과제 (데이터 작업과 별개로 코드 자체를 고쳐야 하는 것)

데이터를 아무리 채워도 아래 두 가지가 해결 안 되면 효과가 반감되거나
사용자에게 잘못된 실패를 노출할 위험이 있다 — **데이터 작업 착수 전에
먼저 처리**.

### 4-1. `candidatesText`(LLM 분류 후보) province-aware 필터링 — 우선순위: 높음, 0단계에서 처리

현재 `ROUTE_DESCRIPTIONS`(방금 추상화 완료)는 여전히 **모든 도에 전체
코드를 다 보여준다.** 국가기관 지사가 제주만 정적 인스턴스가 있는 지금
구조에서는, 비제주 사용자 질문에 LLM이 `SP-NAT-TAX` 같은 jeju 전용 코드를
골라도 `_findTableEntry`가 조용히 못 찾아 실패하고 일반 안내로 떨어진다
(`SP-NATIONAL-LAZY`를 골랐어야 정답). 국가기관 인스턴스가 도별로 늘어날수록
이 문제의 발생 빈도·복잡도가 함께 커지므로, 데이터를 채우기 전에 먼저
`candidatesText`를 `_resolveProvinceCode()` 기준으로 필터링하는 로직을
넣는 게 순서상 맞다(나중에 34개×16도 데이터를 다 채운 뒤 이 필터링을
넣으면 회귀 검증 범위가 훨씬 커진다).

**구체적 수정 방향**: `_classifyFallback` 호출 시 `candidatesText`를
`_l2Table()`/`_cityTable()`/`_nationalTable()`에 실제로 존재하는 코드 +
`SP-SIGUNGU-LAZY`/`SP-NATIONAL-LAZY`(해당 계층 데이터가 비어있는 경우만)로
한정해서 구성.

### 4-2. 라우팅-콘텐츠 이원화(routing=content 단일 소스화) — 우선순위: 중간, 규모 큰 리팩터링

다른 작업자가 이미 지적한 항목. 지금은 라우팅 키워드(`JEJU_L2_TABLE`,
`BUSAN_L2_TABLE` 등)가 `gov-router.js`에 JS 배열로 하드코딩돼 있고, 실제
안내 콘텐츠(마스터데이터 JSON)와 분리돼 있다. 15개 도 분량의 라우팅
키워드를 이번 계획대로 계속 JS 배열에 손으로 추가하면 이 이원화가
구조적으로 더 심해진다.

**이번 계획에서의 처리 방침**: 전면 리팩터링(라우팅 키워드 자체를
마스터데이터 JSON으로 옮기고 런타임에 동적 생성)은 이번 롤아웃 계획의
범위에 포함하지 않는다 — 데이터 물량 자체가 큰 상태에서 아키텍처까지
동시에 바꾸면 실패 시 원인 분리가 어려워진다(이 프로젝트가 반복적으로
확인한 "여러 작업을 한 커밋에 섞지 않는다" 원칙). 대신 **§5의 1단계
파일럿(2개 도)이 끝난 시점에 별도 의사결정 지점을 둔다** — 그때까지 쌓인
키워드 배열 규모를 보고, 계속 JS 배열 방식으로 15개 도를 다 밀 것인지
그 시점에 데이터 이전을 먼저 할 것인지 사용자에게 다시 묻는다. 지금
시점에 미리 정하지 않는다(정보가 부족한 상태에서 큰 결정을 내리지
않는다는 원칙).

## 5. 단계·우선순위

지금까지 이 프로젝트가 항상 "작게 쪼개서 검증 가능한 단위로" 진행해온
방식(작업 1~3, 100건 사고실험도 동일)을 그대로 따른다 — 16개 도 전체를
한 번에 밀지 않는다.

### 0단계 (선행) — §4-1 candidatesText 필터링 코드 수정 + 회귀 검증
데이터 작업 시작 전 1회성 코드 수정. 100건 시뮬레이션 재실행으로 검증.

### 1단계 — 파일럿 2개 도, 도청 실국 완비 + 시청 상위 3~5곳
**부산·서울** 선정 근거: 도청 실국이 이미 16/16, 14/16으로 가장 완비돼
있어 추가 부담이 적고, 인구 규모가 커서 검증 가치가 크다. 이 두 도의
시청 인스턴스화는 "인구 상위 자치구/기초자치단체 3~5곳"으로 범위를
좁혀(예: 부산 해운대구·수영구, 서울 강남구·노원구 등 — 최종 선정은
착수 시 인구·문의빈도 기준으로 재검토) 전체 226개가 아니라 작은
표본으로 방법론 자체를 검증한다.

**이 단계의 목적은 "완성"이 아니라 "패턴 검증"이다** — 시청 인스턴스화
작업이 실제로 얼마나 걸리는지(도청 실국 1개당 걸린 시간과 비교해 몇 배인지)
를 이 파일럿에서 측정해, 2단계 이후 규모 산정의 근거로 쓴다.

### 2단계 — 도청 실국 나머지 완비(경기·광주·대구·강원 신규 + 신뢰도
낮은 3곳 재검증), 국가기관 지사 상위 10개 기관 × 전국 확대

### 3단계 — 시청 인스턴스 확대(인구 상위 순, 목표치는 1단계 측정 결과로
재산정) + 국가기관 나머지 24개 기관 확대

### 4단계(이 계획 범위 밖, 후속 계획으로 분리) — 읍면동 파일럿, §4-2
아키텍처 리팩터링 여부 최종 결정

## 6. 검증 방법론

1. **매 배치(도 1개 또는 기관군 1개)마다** 기존 `gov-router.test.mjs` +
   `100-scenario-thought-experiment.mjs` 재실행 — 실패 0 확인 후 다음
   배치로.
2. **신규 지역·기관 추가 시 시뮬레이션에 해당 지역 케이스 최소 3~5개
   추가** — 지금까지의 관행과 동일(추적 문서 대신 시뮬레이션 파일 자체가
   회귀 자산이 되는 방식으로 이미 전환됨).
3. **기관명·조직명은 §2 원칙대로 1차 출처 대조 없이 반영 금지** — 특히
   시청 데이터는 물량이 많아 유혹이 크지만, 작업 2에서 확인된 "약어 추정
   오답률"을 감안하면 검증 없는 대량 반영이 가장 위험한 지점이다.
4. **규모가 큰 배치는 Research 기능 활용을 우선 검토한다** — 작업 2(기타
   공공기관 254개 재검증)에서 이미 이 방식이 유효했음이 확인됨(파일럿
   5개 직접 검증 후 나머지를 Research로 1차 초안 확보, 이후 §1 원문
   대조로 필터링). 국가기관 지사 10개 기관 × 15개 도(§5 2단계) 같은
   규모는 직접 web_search로 하나씩 하기보다 Research로 초안을 뽑고
   반드시 교차검증하는 방식이 효율적이다 — 단, Research 결과를 그대로
   반영하지 않고 실제 §1/코드 대조로 재확인하는 절차(원칙 2)는 그대로
   유지한다(작업 2에서 Research 추정 5건이 실제로 틀렸던 전례).
5. **govType 가드 회귀 테스트를 도 추가마다 반복** — 새 도의 도청 실국에
   개별 세액 키워드(취득세/재산세)를 실수로 넣지 않는지 매번 확인
   (부산·서울에서 실제로 한 번 재발했던 사고).

## 8. 이 계획에 포함하지 않는 것 (명시적 범위 제외)

- 전국 읍면동/행정동 전수 인스턴스화(§3-3)
- §4-2 라우팅-콘텐츠 아키텍처 전면 리팩터링(1단계 이후 별도 의사결정)
- 이미지 전용 요청 라우팅(이전 세션에서 Phase 4로 분리된 것, 그대로 유지)
- 시청 국(局) 단위 세부 인스턴스화(제주는 있지만, 다른 도는 시청
  레벨부터 — 국 단위는 시청 인스턴스화가 충분히 진행된 후 별도 계획)

## 9. 작업 단위·배포 규칙

이 대화 세션 전체에서 유지해온 규칙을 그대로 적용한다.

- **patch 단위는 "도 1개" 또는 "기관군 1개"를 넘지 않는다** — 여러 도를
  한 patch에 몰아넣지 않는다. 실패 시 원인 분리가 어려워지고, 리뷰도
  힘들어진다.
- **매 patch 전 반드시 새로 clone**(`rm -rf` 후 `git clone`), 적용 전
  `git branch --show-current`로 `main` 확인 후 커밋 — 이 세션에서 브랜치
  실수가 두 번 있었던 재발 방지책.
- **push 후 커밋 해시로 재-clone 검증**하고 나서 다음 배치 착수.
- 파일 3개 이상이면 zip 대신 지금까지처럼 git patch 방식 유지(이 저장소는
  patch 워크플로우가 이미 정착돼 있음).

## 10. 이 계획 자체의 한계 (정직하게 밝힘)

- §5의 "부산·서울 우선" 근거는 도청 실국 완비도와 인구 규모로 정했지만,
  **실제 문의 빈도 데이터(어느 지역·어느 업무가 많이 들어오는지)는
  없다** — 있다면 우선순위가 달라질 수 있다. 데이터가 있으면 알려달라고
  요청하는 게 맞고, 없으면 이 추정 근거로 진행한다.
- 시청 226개(또는 230개) 전수 작업의 소요 시간은 **1단계 파일럿 전에는
  추정 불가**로 정직하게 남겨둔다 — 도청 실국 1개 작업 시간 대비 몇 배
  걸릴지 지금은 근거가 없다.
- 이 계획은 "무엇을, 어떤 순서로, 어떻게 검증하며 할 것인가"를 다루고,
  "언제까지 끝낼 것인가"(일정)는 다루지 않는다 — 소요 시간 추정 자체가
  1단계 결과에 의존하기 때문이다.
