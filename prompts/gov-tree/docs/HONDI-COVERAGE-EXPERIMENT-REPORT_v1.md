# 혼디 시스템 대응 범위 사고실험 보고서 v1 — 다종다양 300건

## 0. 목적과 방법론

이전 라운드들은 특정 SP·라우팅 규칙 하나를 검증했다. 이번엔 "시민이 실제로
무엇이든 물어볼 수 있다"는 전제에서, **혼디 생태계 전체가 어디까지 커버하는지
경계 지도**를 그렸다. 300건을 16개 카테고리(정부민원부터 완전한 범위 밖,
탈옥 시도, 시스템 물리적 한계까지)에 걸쳐 구성하고, 각 건마다:

1. **대응 가능 여부**: 가능 / 부분가능 / 불가능
2. **완전성**(가능한 경우만): 완전 / 부분적

로 판정했다. 방법론은 기존과 동일 — 실측 API 호출이 아니라 지금까지 세션에서
구축·검증한 SP-ARCHITECTURE-MAP·ROUTER-PRIORITY·UNIVERSAL-INTEGRITY 원문
근거에 기반한 정적 판정이다.

## 1. 요약

| 대응 가능 여부 | 건수 |
|---|---|
| 가능 | 216 |
| 부분가능 | 6 |
| 불가능 (정책상 거부) | 73 |

| 완전성 (가능/부분가능 222건 중) | 건수 |

| 부분적 | 70 (32%) |

## 2. 카테고리별 관전 포인트

### 2-1. "불가능"의 두 가지 서로 다른 의미

73건의 "불가능"은 전부 같은 이유가 아니다:

- **정책상 의도적 거부**(위험/어드버서리얼 15건 전부, 범위 밖 28건, 시스템
  능력 밖 10건, 세무 편법 7건, 법률 대행 3건, 의료 처방 4건, 경찰 불법조력
  3건): 시스템이 "못 하는 게 아니라 안 하는 것"이다 — 정직성(U2)·권한한계
  (P2/C23)·탈세금지(K-Tax 규칙3) 등 명문화된 원칙이 실제로 이 요청들을
  막는다는 게 근거로 확인된다.
- **미확인으로 인한 안전측 거부**(K-Cleaner 5건): 저장소 자체를 못 찾아서
  "불가능"으로 분류했다 — 이건 실제로 안 되는 게 아니라 **이번 세션의
  조사 범위 한계**다. 실제로는 서비스가 존재할 수 있다.

### 2-2. "범위 밖" 카테고리의 절반은 사실 커버된다

40건 중 12건("이 시를 번역해줘", "코드 디버깅해줘" 등)은 특정 SP가 없어도
AGENT-COMMON 자체의 일반 지식으로 응답 가능하다 — "혼디 전용 기능은 아니지만
그림자 AI가 못 하는 건 아니다"라는 것도 이번에 명확히 구분했다.

### 2-3. 완전성 "부분적" 70건이 가리키는 반복 패턴

70건 중 다수가 이미 알려진 두 원인으로 수렴한다:
- 정부 트랙 도메인별 §14 실행 예시 부족(기존 v3 권고)
- K-Stock 등 GOV_AGENCIES 미상속 K-서비스의 상속 폭 제한(기존 v6 권고)

새로 발견된 것도 있다: **"K-Market에서 산 물건 환불은 어디서 하나요"**처럼
민간 커머스(kcommerce)와 소비자 보호 관할(공정위/소비자원 등 정부기관)이
겹치는 지점 — 지금 설계엔 이 경계를 다루는 규칙이 없다. R2(국가/자치)도
R1(공익/사익)도 이 케이스를 정확히 분류하지 못한다 — "민간 플랫폼 내부
문제 vs 정부 소비자보호 사무"라는 제3의 축이 필요할 수 있다.

### 2-4. 위기 개입(자살·자해 신호) 경로 — 확인 후 즉시 수정 완료

"우울한데 자해 충동이 있어요" 케이스를 부분가능/부분적으로 표시한 뒤,
추측으로 남기지 않고 실제로 열어서 확인·수정까지 마쳤다. 발견 및 조치:

- `SP_red_flag_registry_v1_0.md`가 "SP-HEALTH-14(정신전문간호 모듈)"라는
  **존재하지 않는 파일**을 가리키고 있었다 → 실존하는 세 페르소나
  (mental-health-professional·clinical-psychologist·school-counselor)로
  참조 수정.
- `SP_common_medical_safety`의 M5(정신건강 연계 프로토콜)엔 "직접적 자원
  안내" 원칙만 있고 실제 전화번호가 없었다 → v1.2 신설, 자살예방상담전화
  1393·정신건강 위기상담전화 1577-0199 등 명시.
- **더 심각한 걸 추가로 발견**: `expert-registry.js`에서 이 세 페르소나의
  `needsMedicalSafety`가 전부 `false`로 돼 있어, M5를 고쳐도 애초에
  로드되지 않는 상태였다 → `true`로 수정.
- 이 조사 과정에서 **페르소나 27개 중 16개가 registry에 v2.2로 등록돼
  있는데 실제 최신 파일은 v2.3**이라는 시스템적 참조 낙후 문제도 함께
  발견해 일괄 수정했다(내용은 책임 조항 문구 완화 수준의 안전한 변경).

이번 세션에서 반복적으로 나타난 패턴 — "내용 파일은 개정됐는데 그걸
불러오는 참조(registry/router)는 갱신 안 됨" — 이 여기서도 그대로
재현됐다. 근본 원인(왜 이런 참조 낙후가 반복되는지)은 별도 프로세스
점검이 필요해 보인다.

## 3. 권고

1. **(안전 최우선, 조치 완료)** 위기개입 경로(M5 전화번호, needsMedicalSafety
   플래그, red_flag_registry 참조) 수정 완료 — 배포만 남음.
2. **(신규, 프로세스)** "내용은 개정됐는데 참조는 안 바뀜" 패턴이 이번
   세션에서 3번째(jeju-router 응급파일, K-Public_common 배포, 페르소나
   16개 버전) 반복됐다 — SP 파일 개정 시 참조처 자동 검증(CI 체크)을
   manifest.json 자동생성 봇과 유사하게 추가하는 것을 검토할 것.
3. 민간 K-서비스(kcommerce 등)와 정부 소비자보호 사무의 경계 규칙을
   ROUTER-PRIORITY에 추가 검토
4. K-Cleaner 저장소 위치 확인(반복 권고)
5. 기존 v3·v6 권고(도메인별 실행 예시, GOV_AGENCIES 확장 여부) 유지
