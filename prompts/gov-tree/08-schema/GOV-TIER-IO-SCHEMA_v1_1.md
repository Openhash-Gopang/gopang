## 왜 5개가 아니라 3개+오버레이인가

"도청/시청/구청/군청/읍면동" 다섯 창구는 이름은 다섯이지만 법적 성격은
셋뿐이다(지방자치법 제2조·제3조):

| 창구 | 법적 분류 | 자치사무 보유 | 묶이는 기본형 |
|---|---|---|---|
| 도청·특별시청·광역시청 | 광역자치단체 | 있음 | **A. PROVINCE** |
| 시청·군청 | 기초자치단체 | 있음 | **B. MUNICIPAL** |
| 자치구청(강남구 등, 특별시·광역시 안) | 기초자치단체 | 있음(단, 일부 권한 제외) | **B. MUNICIPAL + 오버레이 B-1** |
| 읍·면·동 사무소 | 행정기관(자치단체 아님) | 없음 | **C. WARD** |
| 일반구청(수원시 영통구 등, 인구 많은 시 안) | 행정기관(자치단체 아님) | 없음 | **C. WARD + 오버레이 C-1** |

이름이 "구청"으로 같아도 자치구와 일반구는 자치사무 보유 여부가 다르므로
서로 다른 기본형(B vs C)에 속한다. 반대로 도·특별시·광역시는 명칭이
다 달라도 법적 성격이 같으므로 하나의 기본형(A)을 공유한다. 실제 SP는
`{기본형} + {필요시 오버레이} + {그 기관 고유 데이터}` 3단 조립으로
만든다 — SP-DEPT-ECON-TEMPLATE가 "전국 공통 정책 지식 + 지역별 기관명만
자리표시자"로 설계된 것과 동일한 원칙이다.

## A. PROVINCE — 광역자치단체(도·특별시·광역시·특별자치시·특별자치도)

**적용 대상**: 16개 광역시도 전체(서울특별시 1곳, 부산·대구·인천·
대전·울산 광역시 5곳, 경기·충북·충남·경북·경남 일반도 5곳,
세종특별자치시 1곳, 제주·강원·전북 특별자치도 3곳, 전남광주통합
특별시 1곳 — ★ 2026-07-01 전라남도+광주광역시 통합으로 신설된
전국 최초의 "통합특별시" 유형. 서울특별시급 법적 지위·국무회의
참석권 등 다른 광역자치단체에 없는 특례가 있어, 이 문서의 Tier A
정의가 그대로 맞는지는 별도 검증이 필요하다 — TBD).

**입력·출력을 가르는 기준(지방자치법 제14조 사무배분기준)**: 광역자치단체의
사무는 ①2개 이상 시군구에 걸치는 광역적 사무, ②시도 단위로 통일 기준이
필요한 사무, ③국가와 시군구 사이의 연락·조정 사무, ④시군구가 독자적으로
처리하기 어려운 사무, ⑤2개 이상 시군구가 공동 이용하는 규모의 시설
관리로 법정 예시돼 있다. 즉 광역자치단체의 출력은 "주민에게 직접"보다
**"기초자치단체에게, 또는 기초자치단체 여럿에 걸쳐"** 나가는 것이 많다는
게 병원의 "환자→치료"나 기업의 "원자재→상품"과 다른 핵심 차이다.

### 오버레이 B-2 — 행정시 (제주시·서귀포시 등, 2026-07-19 신설)

**적용 대상**: 광역자치단체 직속이지만 법인격·자치권이 없는 행정시(현재
유일한 실사례는 제주특별자치도의 제주시·서귀포시 — 「제주특별자치도
설치 및 국제자유도시 조성을 위한 특별법」 제10조에 따라 시장은 도지사
임명직, 시의회 없음). B-1(자치구)과 혼동하지 않는다 — 자치구는
기초자치단체(자치권 있음)이고, 행정시는 기초자치단체 자체가 아니다.

MUNICIPAL 기본형을 상속하되, INPUT/OUTPUT_SCHEMA의 창구 기능(인허가
접수·복지 신청 등)은 형태상 유사하게 유지하면서 AUTHORITY_SPLIT만
다음과 같이 대체한다:

```json
  "extends": "MUNICIPAL",
  "AUTHORITY_SPLIT_REPLACE": {
    "자치사무": "없음 — 행정시는 법인격이 없는 광역자치단체의 하부 행정기관이다(지방자치법이 아닌 특별법 근거 예외). 실질 정책·예산 결정권은 A(도청) 실·국 또는 도 직속기관(예: 상하수도본부) 소관이다.",

}
```

**주의**: 다른 지역에서 B-2에 해당하는 신규 사례가 발견되면(현재는 제주만
확인됨) 이 오버레이를 그대로 재사용하고 새로 만들지 않는다 — 판별
기준은 "법률 근거가 지방자치법 일반 규정이 아니라 개별 특별법"과
"자치권 부재" 두 가지다.

## C. WARD — 행정창구(읍·면·동, 일반구)

**적용 대상**: 전국 모든 읍·면·동 사무소, 그리고 인구 50만 이상 대도시
안의 일반구청(자치단체 아님 — 예: 수원시 영통구청, 성남시 분당구청).

**입력·출력을 가르는 기준**: 자치사무 자체가 없다 — 법원의 "사건→판결"과
정반대로, 이 계층은 **스스로 결정하는 산출물이 없고 상급 자치단체(B)의
결정을 대행 교부하거나, 접수해서 상급기관에 이첩하는 것**이 출력의
전부다. 병원 비유로는 "접수창구"에 해당하지 "진료"에 해당하지 않는다.

```json
{
  "tier": "WARD",
  "INPUT_SCHEMA": [
    { "doc_type": "DOC-IDENT-RESIDENCE", "required": false, "note": "본인 확인용, 발급 대상 서류 자체이기도 함(순환 아님 — 최초 등록은 별도 절차)" },
    { "pdv_field": "civil_petition_text", "required": false, "note": "생활민원 접수 — 상급기관(B 또는 도 직속기관) 이첩 대상" }
  ],
  "OUTPUT_SCHEMA": [
    { "doc_type": "DOC-IDENT-RESIDENCE",   "note": "주민등록등본·초본 — 즉시발급형" },
    { "doc_type": "DOC-FAMILY-CERT",       "note": "가족관계증명서 — 즉시발급형(전산 등록된 경우)" },
    { "pdv_field": "petition_receipt_no",  "note": "접수확인증 — 처리 자체는 상급기관이 하고, 이 창구는 '접수됐다'는 증표만 출력" }
  ],
  "AUTHORITY_SPLIT": {
    "자치사무": "없음 — 자치단체가 아니라 시/군/구청(자치구 포함)의 하부 행정기관",
    "위임사무": "국가위임사무 중 창구 실무(주민등록·인감·가족관계등록) 대행 — 법적 처분권자는 시/군/자치구청장"
  }
}
```

### 오버레이 C-1 — 일반구 (수원시 영통구 등)

WARD 기본형을 상속하되 다음을 **추가**한다(일반구는 읍면동보다 관할
인구·처리범위가 넓어 일부 인허가성 업무를 창구에서 직접 처리):

```json
{
  "overlay": "C-1_GENERAL_DISTRICT",
  "extends": "WARD",
  "ADD_TO_OUTPUT": [
    { "doc_type": "DOC-LICENSE-{절차슬러그}", "note": "영업신고·등록 등 경미한 인허가 일부 — 시(모구 시청) 조례에 따라 위임 범위가 다르므로 SP-AUTHOR PHASE C에서 개별 확인 필수" },
    { "pdv_field": "local_tax_receipt", "note": "지방세 수납창구(부과는 시청, 수납만 대행하는 경우 있음)" }
  ],
  "SCOPE_NOTE": "일반구는 여전히 자치단체가 아니므로 조례 제개정·독자 예산편성권은 없음 — WARD와 동일. 차이는 오직 '창구에서 처리 가능한 업무 범위'뿐."
}
```

## 갱신 원칙 — 이 스키마는 최초 1회 정의로 끝나지 않는다 (2026-07-13 신설)

AGENCY-AC-COMMON 공리 0(2026-07-13 신설: "AC는 main(), 소속 부서 SP는
submodule")이 명시한 것과 동일한 원칙이 이 문서에도 적용된다 — **여기서
정의한 tier별 INPUT_SCHEMA/OUTPUT_SCHEMA는 프로그램의 함수 시그니처와
같아서, 조직개편·법령 개정·신규 사업 개시가 있을 때마다 다시 검토·갱신돼야
한다.** 개별 기관 SP가 이 문서의 tier 뼈대를 상속해 자신의
§INPUT_SCHEMA/OUTPUT_SCHEMA를 확정한 시점(SP-AUTHOR PHASE B-0)은
"완성"이 아니라 "그 시점 기준 최초 버전"이다.

- 하위 개별 기관 SP는 자신의 §INPUT_SCHEMA/OUTPUT_SCHEMA를 한 번 쓰고
  고정된 사실로 취급하지 않는다 — 최소한 조직개편 확정, 관련 법령 개정,
  신규 업무(doc_type) 발생 시점마다 재검토·갱신한다.
- 이 문서(tier 공통 뼈대) 자체도 예외가 아니다 — 예: 2026-07-01
  전남·광주 통합, 세종특별자치시의 단층제 미분류(§알려진 한계) 같은
  사례가 실제로 이 문서의 재검토를 요구했던 전례가 이미 있다.
- "지금 작성되는 모든 tier 스키마·개별 기관 SP는 이 프로그램의 초기
  버전"이라는 인식을 유지한다 — 향후 갱신은 예외적 유지보수가 아니라
  이 아키텍처가 원래 예정한 정상적인 다음 단계다.

## SP-AUTHOR 연동 방법

개별 기관(예: "충청북도청", "전주시청", "해운대구청") SP를 지을 때
PHASE B-0에서:

1. `JURISDICTION-RESOLVER-SCHEMA`로 이 기관이 A/B/C 중 어느 tier인지,
   B-1/C-1 오버레이가 필요한지(광역시 안의 구인가/일반 시 안의 구인가)
   먼저 판별한다.
2. 이 문서의 해당 tier `INPUT_SCHEMA`/`OUTPUT_SCHEMA`를 뼈대로 상속한다.
3. 그 기관 고유의 개별 doc_type(예: 그 도만의 특산품 인증제도)만 PHASE
   C(세부 절차 조사)에서 추가 조사해 얹는다 — 공통 뼈대까지 매번 새로
   조사하지 않는다(SP-AUTHOR v1.3의 "조사는 기관당 1회가 아니라 템플릿
   패밀리당 1회" 원칙을 지방정부 계층에도 동일 적용).
4. `AUTHORITY_SPLIT`(자치사무/위임사무)은 §CAPABILITIES 작성 시 그대로
   사용 — 위임사무 항목은 "이 창구가 최종 결정권자가 아니라 대행/집행자"
   라는 사실을 반드시 이용자에게 고지하도록 DATA_REQUIREMENT-SCHEMA
   §11(능동적 역량·불가 사유 이원화 원칙)과 연결한다.

## 알려진 한계 (정직하게 남김)

- `DOC-LICENSE-{절차슬러그}`류(영업 인허가 전반)는 업종마다 수백 개
  절차가 있어 이 문서에서 전수 나열하지 않았다 — AGENT-SUPPLIER-XX(KSIC)
  77개 파일의 "업종별 인허가·신고 의무" 섹션과 연동해 재사용하는 게
  맞다(SP-AUTHOR v1.3 근거와 동일한 판단).
- 자치구 B-1의 "상하수도 제외" 여부는 **광역시마다 다를 수 있어 일괄
  단정하지 않고 개별 확인 필수로 명시했다** — 부정확한 확신보다 정직한
  TBD가 낫다는 이 저장소의 기존 원칙(정책기관 41개 근거법령 TBD 처리와
  동일 판단)을 그대로 따랐다.
- 세종특별자치시는 광역·기초 통합형(단층제)이라 A/B 어느 한쪽으로
  단순 분류가 안 된다 — SP-AUTHOR PHASE C에서 별도 처리 필요(이 문서
  범위 밖으로 명시적으로 남김).
