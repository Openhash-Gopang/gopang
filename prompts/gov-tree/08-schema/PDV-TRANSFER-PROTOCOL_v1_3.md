## 0. 기존 규칙과의 관계 — ★사람 검토 필요★

**법률 근거**: 「개인정보 보호법」제17조(개인정보의 제공)·제18조(개인정보의
목적 외 이용·제공 제한)는 정보주체의 별도 동의나 법령상 근거 없이 개인정보를
제3자에게 제공하는 것을 금지한다. 이 금지는 "개인과 기관 사이"뿐 아니라
**"기관과 기관 사이"에도 동일하게 적용된다** — 공공기관이라 해서 서로
간에 개인정보를 자유롭게 주고받을 수 있는 것이 아니며, 기관 간 제공에도
정보주체 동의 또는 법령 근거가 각각 필요하다. 이 문서의 §3-A는 이 법률
요건을 시스템 설계 차원에서 강제하는 장치다 — "기관 SP끼리는 어떤 개인
식별 정보도 서로 직접 주고받지 않는다"는 원칙은 선택이 아니라 준수사항.

`AGENT-COMMON_v3_35.txt` §2 [인출 기준]은 "이용자 정보는 외부 에이전트에
전달하지 않음(데이터 주권)"이라고 예외 없이 못박고 있다. 이 문서는 그
원칙을 뒤집는 게 아니라 **"외부 에이전트"의 범위를 처음으로 정의**한다:

  - **외부 에이전트** (원칙 그대로 적용, 전달 금지 유지): P2P로 연결된
    다른 이용자의 AI 비서, Hondi 생태계 밖의 제3자 서비스
  - **기관 SP** (이 문서가 신설하는 별도 범주): SP-AUTHOR가 작성하고
    Hondi 자신이 호스팅하는, 정부·공공기관을 대표하는 SP. 이용자 관점에서는
    "은행 창구 직원"에 가깝다 — 완전한 제3자가 아니라, 이용자가 스스로
    특정 절차를 위해 자기 정보 일부를 내어주기로 선택하는 대상.

이 구분이 §2 원문과 충돌하므로, 이 문서는 `AGENT-COMMON` 다음 버전(§2에
"단, 기관 SP는 본 문서(PDV-TRANSFER-PROTOCOL)를 따른다"는 한 줄 참조 추가)
에서 명문으로 승인받기 전까지는 **초안 상태**로 취급한다. SP-AUTHOR는 이
문서가 `status: active`로 전환되기 전에는 어떤 GOV_TASK에도 실제 PDV
필드를 실어 보내지 않는다(§3 태그만 정의해두고 실제 전송은 보류).

## 1. 최소수집·목적제한 — "금고를 통째로 안 연다"

기관 SP는 이용자의 PDV 전체에 접근하지 않는다. 접근 가능한 범위는 해당
기관 SP가 `DATA_REQUIREMENT-SCHEMA`의 자매 규격으로 스스로 선언한
**`REQUIRED_USER_FIELDS`**(SP-AUTHOR 설계 PHASE C-② 참조)에 열거된 필드
**그 목록만**이다. 목록에 없는 필드는 그 기관 SP가 요청할 수도, 사용자
AI 비서가 보낼 수도 없다 — 화이트리스트 방식이며, 예외는 §5(오탐 보정)
경로로만 처리한다.

각 `REQUIRED_USER_FIELDS` 항목은 `DATA_REQUIREMENT-SCHEMA`의 `min_level`
(L0~L3) 어휘를 그대로 재사용해 민감도를 선언한다:

  - L0: 이름·연락처 등 낮은 민감도 — 사용자 AI 비서가 자동 매칭·자동 전송
  - L1~L2: 주소·사업자정보 등 — 자동 매칭하되, 최초 1회 이용자 승인 필요
  - L3: 주민등록번호·건강정보 등 — **매 건 명시 동의** 필수(§2, 자동화 불가)

## 2. 동의 절차

L1~L2는 "기관 SP당 최초 1회" 승인이 원칙이다(같은 기관에 같은 필드를
매번 재승인받게 하면 반복입력 제거라는 목표 자체가 무의미해진다). L3는
`AGENT-COMMON` §2 [갱신 원칙]의 기존 원칙("민감 정보는 이용자 명시 동의
후 기록")을 "기록"뿐 아니라 "전송"에도 동일 적용 — 매 건 동의.

```
[PDV_TRANSFER_CONSENT_REQUEST: to_sp={기관 SP id}, task={GOV_TASK id},
  fields=[{key, min_level, purpose_1line}], scope=once|standing]
```

`scope=standing`은 L0~L2에서만 허용(예: "앞으로 위치기반서비스 관련
신고는 이 정보 자동으로 써도 돼"). L3는 `scope=once`만 허용되며 SP-AUTHOR가
이를 어기는 초안을 낼 경우 HUMAN-AUTHORITY-GATE-SCHEMA G5와 동일하게
즉시 반려한다.

## 3. 전송 태그 — GOV_TASK_REQUEST와의 관계

PDV 전송은 `GOV_TASK_REQUEST`에 실려 가는 하나의 페이로드 유형이다(별도
경로 아님 — 통신 프로토콜은 3-2에서 이미 고정됨):

```
[GOV_TASK_REQUEST: from_agent=사용자AI비서, to_agent={기관 SP},
  task={...}, on_behalf_of={사용자 guid}, auth_token={위임 서명},
  pdv_payload=[{key, value, source:"pdv", consent_ref={§2 동의 태그 ID}}]]
```

`pdv_payload`의 각 항목은 §2에서 발급된 동의 참조(`consent_ref`) 없이는
채워질 수 없다 — 이 조건이 없는 `GOV_TASK_REQUEST`는 수신 기관 SP가
`HUMAN-AUTHORITY-GATE-SCHEMA` G4와 동일한 방식으로 거부한다:

```
[GOV_TASK_RESPONSE: status=denied, reason=missing_pdv_consent]
```

## 3-A. 전송 주체 단일화 — "이용자 관련 데이터는 오직 사용자의 AI 비서를 통해서만 전달된다"

이 원칙은 `pdv_payload`에만 적용되는 게 아니라, **특정 이용자를 식별하거나
특정 이용자에 관한 것으로 연결 가능한 모든 산출물**(원본 PDV 필드는 물론,
영수증·신청서 사본·확인서·수리번호처럼 `on_behalf_of={사용자 guid}`가
붙는 모든 `artifact_ref`)에 동일하게 적용된다. §1~2 동의를 거쳐 정당하게
받은 것이라도, 그걸 다른 기관 SP에게 재전달하는 것은 §0의 법률 근거상
금지되는 "제3자(기관) 제공"에 해당한다:

```
# 금지 — 기관 SP가 발신 주체인, 특정 이용자에 연결되는 모든 전달
[GOV_TASK_REQUEST: from_agent={기관SP-A}, to_agent={기관SP-B},
  pdv_payload=[...]]                          ← 금지
[GOV_TASK_REQUEST: from_agent={기관SP-A}, to_agent={기관SP-B},
  on_behalf_of={사용자 guid}, artifact_ref={영수증/서류 등}]  ← 금지
```

기관 B가 이용자의 PDV 필드나, 기관 A가 발급한 영수증·서류가 추가로
필요하면, 기관 A를 경유하지 않고 **사용자의 AI 비서에게 새 요청을
되돌려 보낸다**(`GOV_TASK_REQUEST: from_agent={기관SP-B},
to_agent=사용자AI비서, task=field_or_document_request`). 사용자 AI
비서가 이걸 받아 §1~2 절차(화이트리스트 대조·동의)를 처음부터 다시
적용해 기관 B에게 새로 전송한다. 즉 이용자와 연결되는 모든 데이터
흐름에서 사용자의 AI 비서는 유일한 허브이며, 기관 SP끼리는 서로를
거쳐 개인정보(넓은 의미의 산출물 포함)가 흘러가는 어떤 우회로도
갖지 않는다 — **예외 없음**(v1.1까지 있던 "기관 고유 산출물은 예외"
조항은 v1.2에서 전면 폐지됐다).

이 원칙은 `HUMAN-AUTHORITY-GATE-SCHEMA` G4("기관 간 통신에도 동일 적용")
와 짝을 이룬다 — G4가 "사람의 서명 있는 승인 없이는 기관 간 보증이 안 된다"
는 원칙이라면, §3-A는 그중에서도 개인 식별이 가능한 데이터 자체는 애초에
기관 간 채널에 실리지도 않는다는, 한 단계 더 강한 제약이다.

기관 SP 간 `GOV_TASK_REQUEST`가 실제로 허용되는 범위는 **특정 이용자와
무관한 절차·제도 정보**뿐이다(예: "이 신고에 필요한 서류가 뭔가요",
"수수료 종별이 어떻게 되나요" — K-Search RULE-07의 `official_reference`
카드와 동일한 성격). `on_behalf_of`가 붙는 순간부터는 예외 없이 §3-A가
적용되어 사용자의 AI 비서를 경유해야 한다.

## 4. 전송 이력 — 반복입력 제거와 병목통계가 공유하는 단일 로그

모든 `pdv_payload` 전송은 예외 없이 새 PDV 레코드 타입으로 자기 자신에게
기록된다(기존 `PDV_STORE` 타입 목록에 `transfer` 신설):

```
PDV_STORE {
  "ts": "<ISO8601>",
  "type": "transfer",
  "key": "<전송된 필드명>",
  "value": "<to_sp, task, consent_ref 요약>",
  "source": "agent_report",
  "summary": "<50자 이내: 어느 기관에 무엇을 왜 보냈는지>"
}
```

이 레코드가 두 가지를 동시에 해결한다:

  - **반복입력 제거(이점②)**: 같은 기관·같은 필드 재요청 시, 사용자 AI
    비서는 새로 묻지 않고 이 로그에서 유효한 `consent_ref`(standing)를
    찾아 즉시 재사용한다.
  - **투명성**: 이용자가 "내 정보 어디 어디에 보냈어?"라고 물으면
    AGENT-COMMON이 이 로그만으로 즉답 가능 — 별도 감사 체계 불필요.

## 5. 병목 통계 — PDV 원문 없이 타임스탬프만 집계

`GOV_TASK_REQUEST`/`RESPONSE` 왕복은 이용자 PDV 내용과 무관하게 그 자체로
`{institution, task_type, hop_type, requested_ts, responded_ts, status}`만
기관 횡단 집계 테이블(가칭 `gov_latency_stats`, 개인 식별자 제거)에 함께
쌓인다. `hop_type`은 `new_request`(신규 신청) 또는 `mid_chain_reissue`
(§3-A로 인해 사용자 AI 비서를 경유해 재전달되는, 이미 발급된 문서의
왕복 — 영수증→다음 기관 사례가 전형)로 구분한다. 이 구분이 없으면
"기관이 느려서 걸리는 시간"과 "설계상 필연적으로 발생하는 경유 왕복
시간"이 통계에서 섞여, 진짜 병목 기관을 가려낼 수 없다. 이게 "GDP 수 %"
근거의 실제 데이터원이다 — PDV 내용을 들여다볼 필요 없이 왕복시간만으로
기관별·절차별·hop_type별 지연을 통계낼 수 있고, 개인정보와 완전히
분리되므로 §0의 데이터 주권 원칙과도 충돌하지 않는다.

## 6. 철회·정정

이용자는 언제든 특정 `consent_ref`를 철회할 수 있다(`scope=standing`
항목에 한함 — 이미 기관에 도달해 처리 완료된 건은 회수 불가, 이후
재사용만 차단). 철회는 `PDV_STORE{type:"transfer"}` 로그에 상태 갱신으로
남으며, 별도 삭제(과거 기록 소거)는 하지 않는다 — 이력 자체가 §4·§5의
근거이므로 보존이 원칙, 향후 이용만 차단.

## 7. 위반 처리

- §1 화이트리스트를 벗어난 필드를 요청하는 기관 SP → SP-AUTHOR PHASE E
  검수 항목에 추가, `[SP_DRAFT_REJECTED: reason=field_scope_violation]`
