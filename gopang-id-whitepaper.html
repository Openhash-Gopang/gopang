§1  왜 새로운 주소 체계가 필요한가

지금 우리가 쓰는 인터넷은 주소(IP)와 이름(도메인)이 완전히 분리되어 있습니다. 네이버를 찾으려면 먼저 DNS 서버에 "naver.com의 주소가 뭐야?"라고 물어야 합니다. 이 DNS 서버가 다운되면 전 세계 인터넷이 마비됩니다. 실제로 2016년 미국 Dyn DNS 공격으로 트위터·넷플릭스·아마존이 동시에 접속 불가가 됐습니다.

고팡은 사람·사물·기관 수천억 개 각각에 고유 주소를 부여하는 새로운 인터넷을 지향합니다. 그러면서도 중앙 서버 없이, 누구나 "주피터"라는 아이디만으로 상대방을 찾을 수 있어야 합니다.

📮 주민등록번호와 전화번호에 비유하면:
현재 인터넷은 전화번호부(DNS)가 없으면 아무도 찾을 수 없는 구조입니다. 반면 고팡은 주민등록번호처럼 번호 자체에 정보(지역·생년월일·성별)가 담겨 있어, 번호만 보면 그 사람에 대한 기본 정보를 알 수 있습니다. 고팡 ID는 이 개념을 디지털로 구현합니다.

§2  고팡 ID란 무엇인가

고팡의 모든 사용자(사람, 사물, 기관)는 IPv6 형식의 고유 주소를 받습니다. IPv6는 128비트(0과 1이 128개) 숫자로, 이론적으로 340언데실리온개의 주소를 만들 수 있습니다. 이는 지구상의 모래알 수보다 많습니다.

🏠 도로명 주소에 비유하면:
"제주특별자치도 제주시 한림읍 한림로 7"이라는 주소에는 도→시→읍→도로 순서로 위치가 담겨 있습니다. 고팡 ID도 마찬가지입니다. 주소 안에 국가→광역→기초 지역과 아이디가 모두 인코딩되어 있어, 주소만 보면 이 사람이 어느 나라·어느 지역에서 등록했는지 알 수 있습니다.

고팡 ID의 예시입니다.

고팡 ID 예시
홍길동 (한국 제주 한림읍, 아이디: 주피터):
  fd47:a3c2:3f7a:9c2b:e1d4:KR_JEJU_HALLIM:1:abc
  ├── fd47:a3c2     고팡 ULA 프리픽스 (전 세계 고팡 사용자 공통)
  ├── 3f7a:9c2b:e1d4 SHA256("주피터") 앞 48비트
  ├── KR_JEJU_HALLIM 등록 위치 (한국·제주·한림읍)
  ├── 1              유형 (1=사람, 2=사물, 3=기관)
  └── abc            기기 핑거프린트 (12비트)

성춘향 (소말리아 모가디슈, 아이디: 춘향이):
  fd47:a3c2:ab12:cd34:ef56:SO_MOGA_????:1:xyz

§3  아이디가 곧 주소가 된다

고팡의 가장 혁신적인 설계는 아이디(주피터)를 수학적으로 변환하면 주소의 일부가 된다는 점입니다. 별도의 전화번호부(DNS 서버)가 필요 없습니다.

🔢 계좌번호에 비유하면:
은행 계좌번호 "110-123-456789"를 보면 앞 세 자리(110)만으로 신한은행임을 알 수 있습니다. 고팡 ID도 마찬가지입니다. "주피터"를 수학 함수(SHA256)로 변환하면 항상 동일한 숫자(3f7a:9c2b:e1d4)가 나옵니다. 이 숫자가 주소 안에 포함되어 있어, 누구든 "주피터"를 알면 주소를 계산할 수 있습니다.

한글·영어·소말리아어 등 모든 언어의 문자를 유니코드(Unicode)로 표현하고, 이를 SHA256 함수로 변환합니다. SHA256은 어떤 길이의 입력이든 항상 256비트의 고정 출력을 만들기 때문에, 아이디의 길이에 관계없이 동일한 크기의 주소가 생성됩니다.

"주피터" → IPv6 변환 과정
"주피터" (한글 3글자)
  ↓ 유니코드 인코딩
  주 = U+C8FC,  피 = U+D53C,  터 = U+D130
  ↓ UTF-8 바이트 변환
  EC A3 BC  ED 94 BC  ED 84 B0
  ↓ SHA256 해시
  3f7a9c2be1d45f89a2c1...  (256비트)
  ↓ 앞 48비트만 사용
  3f7a : 9c2b : e1d4
  ↓ IPv6 조합
  fd47:a3c2:3f7a:9c2b:e1d4:0052:0001:0abc
           ↑              ↑              ↑
      아이디 해시       지역코드(KR+JEJU)   핑거프린트

§4  상대방을 찾는 방법 — 우체부 비유

"주피터"를 검색하면 전 세계 수십만 개의 고팡 서버에 동시에 물어봐야 할까요? 그렇지 않습니다. 수학적으로 어느 서버가 담당인지 결정됩니다.

📬 우편 배달에 비유하면:
편지를 보낼 때 우체부는 모든 집을 다 방문하지 않습니다. 우편번호를 보고 해당 지역 우체국으로 바로 보냅니다. 고팡도 마찬가지입니다. "주피터"의 해시값(3f7a...)을 보면, 전 세계 어느 노드가 담당인지 수학적으로 계산됩니다. 그 노드 하나에만 물어보면 됩니다.

이 방식을 DHT(분산 해시 테이블, Distributed Hash Table)라고 합니다. 전 세계 수억 개의 파일을 중앙 서버 없이 공유하는 BitTorrent가 20년 이상 이 방식으로 동작하고 있습니다.

홍길동(제주 한림읍) → 성춘향(소말리아 모가디슈) 검색 흐름
홍길동이 "춘향이" 검색
  ↓
① SHA256("춘향이") 앞 48비트 계산 → ab12:cd34:ef56  (0ms, 로컬)
  ↓
② 이 해시값과 가장 가까운 OpenHash 노드 계산  (0ms, 수학적 결정)
  ↓
③ 해당 노드 1개에만 질의  (~50ms, 네트워크)
  └─ 노드가 꺼져 있으면 → 상위 계층 노드(광역→국가) 순차 질의
  ↓
④ fd47:a3c2:ab12:cd34:ef56:SO_MOGA:1:xyz 반환  (~100ms)
  ↓
⑤ 홍길동 ↔ 성춘향 직접 P2P 연결  ✅

총 소요: ~100ms  |  브로드캐스트 없음  |  중앙 서버 없음

§5  실명제와 프라이버시의 공존

고팡은 실명제를 지향하지만, 모든 서비스에 실명을 요구하지는 않습니다. 서비스의 중요도에 따라 4단계 인증 레벨을 운영합니다.

레벨인증 방법이용 가능 서비스실명 공개
L0 기기 자동 인식 일반 조회, AI 상담 ❌ 비공개
L1 L0 + 얼굴 등록 K-Police 신고, K-Law 소송 ❌ 비공개
L2 L1 + 신분증 사진 K-Market 거래, 금융, 의료 ✅ 실명 인증
L3 L2 + 4단어 시드 계정 복원, 최고 보안 기능 ✅ 실명 인증

예를 들어 "도영민"의 실명은 L2 이상 사용자만 검색할 수 있습니다. 반면 아이디 "주피터"는 누구나 검색 가능합니다. 아이디는 전 세계 유일하며 중복 등록이 불가능합니다.

🔑
핵심: 실명(도영민)과 아이디(주피터)는 분리됩니다. 아이디만으로도 연결이 가능하므로, 실명을 공개하지 않아도 고팡의 대부분 기능을 이용할 수 있습니다. 실명은 오직 본인이 동의한 경우(L2 이상)에만 공개됩니다.

§7  아이디 → IPv6 변환 알고리즘

아이디는 모든 유니코드 문자를 지원합니다. 한글·영어·아랍어·소말리아어 등 어떤 언어든 동일한 알고리즘으로 처리됩니다.

gopang-sso.js · _buildGopangId() JavaScript 
/**
 * 아이디 문자열 → 고팡 IPv6 주소 생성
 * @param {string} username  - 아이디 ("주피터", "jupiter" 등 모든 유니코드)
 * @param {string} fpHex     - 기기 핑거프린트 SHA256 hex
 * @param {object} regionCode - { country, metro, local } 각 비트값
 * @param {string} userType  - 'person' | 'device' | 'org'
 */
async function _buildGopangId(username, fpHex, regionCode, userType) {

  // ① 아이디 → UTF-8 바이트 → SHA256
  const usernameBytes = new TextEncoder().encode(username.normalize('NFC'));
  const hashBuf = await crypto.subtle.digest('SHA-256', usernameBytes);
  const hashHex = Array.from(new Uint8Array(hashBuf))
    .map(b => b.toString(16).padStart(2, '0')).join('');

  // ② 앞 48비트(12 hex chars) = 아이디 해시 필드
  const idHash48 = hashHex.slice(0, 12);

  // ③ 지역 코드 32비트 인코딩
  // 국가 10비트(0~1023) + 광역 11비트(0~2047) + 기초 11비트(0~2047)
  const regionBits =
    ((regionCode.country & 0x3FF) << 22) |
    ((regionCode.metro   & 0x7FF) << 11) |
    ( regionCode.local   & 0x7FF);
  const regionHex = regionBits.toString(16).padStart(8, '0');

  // ④ 유형 코드 4비트
  const typeCode = { person:1, device:2, org:3 }[userType] || 1;

  // ⑤ 기기 핑거프린트 앞 12비트
  const fp12 = parseInt(fpHex.slice(0, 3), 16) & 0xFFF;

  // ⑥ 128비트 조합 → IPv6 8그룹 × 16비트
  const ULA = 'fd47a3c2';   // ULA 프리픽스 고정 (32비트)
  const raw128 = ULA + idHash48 + regionHex
    + (typeCode << 12 | fp12).toString(16).padStart(4, '0');

  const groups = [];
  for (let i = 0; i < 8; i++)
    groups.push(raw128.slice(i*4, i*4+4));

  return groups.join(':');
  // 예: "fd47:a3c2:3f7a:9c2b:e1d4:0052:0001:0abc"
}

§8  지역 코드 인코딩

사용자 등록 시 GPS 좌표를 카카오 역지오코딩 API(gopang-proxy /geocode)로 행정 지명으로 변환합니다. 전 세계 통일 기준으로 ITU-T 국가번호를 사용합니다.

지역 코드 인코딩 예시 비트 구조 
32비트 지역 필드:
┌──────────────┬─────────────────┬─────────────────┐
│  국가 10비트  │   광역 11비트   │   기초 11비트   │
│   (0~1023)   │   (0~2047)      │   (0~2047)      │
└──────────────┴─────────────────┴─────────────────┘

한국 제주특별자치도 한림읍:
  국가: KR = ITU-T 82  → 0000000082 (10비트)
  광역: 제주 = 코드 11  → 00000001011 (11비트)
  기초: 한림읍 = 코드 3 → 00000000011 (11비트)
  조합: 0000000082_00000001011_00000000011
  hex: 0x01458003

소말리아 모가디슈:
  국가: SO = ITU-T 252  → 0011111100 (10비트)
  광역: 모가디슈 = 코드 1 → 00000000001 (11비트)
  기초: 없음 = 0         → 00000000000 (11비트)
  hex: 0x3C800000

미등록 지역 (GPS 없음):
  0x00000000  (전체 0 = 위치 미제공)

§9  분산 검색 — 주소지 노드와 이동성

고팡 네트워크의 모든 사용자는 반드시 하나의 주소지 노드(Home L1)를 갖습니다. 주소지 노드는 행정 최하 단위(읍면동·county 등)를 담당하며, 약 1,000명 내외의 사용자를 관리합니다. 사용자가 전 세계 어디에 있든, 주소지 노드가 현재 위치를 항상 추적합니다. 이것은 이동통신의 HLR(Home Location Register)과 정확히 동일한 원리입니다.

📡 이동통신 기지국에 비유하면:
홍길동이 제주에서 서울로, 다시 뉴욕으로 이동해도 전화를 받을 수 있는 이유는 통신사의 기지국이 홍길동의 현재 위치를 실시간으로 추적하기 때문입니다. 고팡의 주소지 노드가 바로 이 역할을 합니다. 한림읍 주소지 노드는 홍길동이 지구 어느 곳에 있든, 현재 접속 중인 방문 노드(Visit L1)를 알고 있습니다.

3단계 노드 구조 — 성춘향(소말리아) → 홍길동(한림읍 등록, 현재 뉴욕) 호출
Step 1: DHT로 홍길동의 IPv6 + 주소지 노드 파악                  (0ms, 로컬 계산)
  SHA256("홍길동") → Kademlia XOR → 담당 DHT 노드 질의
  응답: fd47:a3c2:????:????:????:KR_JEJU_HALLIM:1:fp
                                         ↑
                              주소지 노드(한림읍 L1) 주소가 인코딩됨

Step 2: 한림읍 주소지 노드(HLR)에 현재 위치 문의              (~30ms)
  "홍길동 지금 어디?"

// 사용자 기기가 새 L1 영역 진입 시 자동 실행
async function _onEnterNewL1Region(newVisitL1, userIPv6) {

  // 1. IPv6에서 주소지 노드 주소 추출
  const homeL1Addr = _extractHomeL1(userIPv6);
  // fd47:a3c2:????:????:????:[KR_JEJU_HALLIM]:1:fp
  //                          ↑ 이 32비트로 한림읍 L1 노드 주소 계산

  // 2. 새 방문 노드에 체크인
  await newVisitL1.checkIn({
    ipv6:     userIPv6,
    homeL1:   homeL1Addr,
    gps:      await getCurrentGPS(),
    checkedAt: Date.now(),
  });

  // 3. 주소지 노드(HLR)에 현재 방문 노드 갱신
  await homeL1Addr.updateVisitNode({
    ipv6:       userIPv6,
    visitL1:    newVisitL1.address,
    updatedAt:  Date.now(),
  });

  // 4. 이전 방문 노드에 체크아웃
  await prevVisitL1?.checkOut(userIPv6);
}

// 30초마다 heartbeat — 기기가 꺼지면 자동 만료
setInterval(() => newVisitL1.heartbeat(userIPv6), 30_000);
💡
주소지 노드의 경제학: 주소지 노드 하나가 약 1,000명을 담당합니다. 각 사용자의 현재 방문 노드 주소(약 32바이트)만 저장하면 되므로, 전체 저장 용량은 약 32KB입니다. 일반 스마트폰으로도 주소지 노드를 운영할 수 있습니다. OpenHash 네트워크의 탈중앙화가 자연스럽게 달성됩니다.
