# 온라인 민원 SP 카테고리 — 제목만 작성 (2026-07-09)

> 요청: 복지 항목과 같은 방식으로 온라인 민원처리 항목을 카테고리로 분류해 SP 제목만 작성. 복지 작업과 구별해 별도 디렉토리에 저장.

## 1. 복지 작업과의 결정적 차이 — 실측 데이터 없음

복지 카테고리 작업은 `civil-petitions-raw.json`(실제 수집된 10,966건 원본 데이터)을 근거로 분류했다. **이번 온라인 민원 작업은 그런 원본 목록이 없다** — 정부24 온라인 민원 2,800/760종의 실제 개별 목록을 data.go.kr에서 두 차례 시도했지만 둘 다 통계만 제공했지 개별 민원명 리스트를 주지 않았다(§CIVIL-PETITION-IO-DRAFTING-PLAN 관련 이전 작업 참조).

**따라서 이번 분류는 실측 검증이 안 된 설계 기반 초안이다** — 82개 부서 소관업무 마스터목록, Phase 1 25건, 일반 지식을 근거로 구성했다. 실제 정부24 목록을 확보하면 재검증이 반드시 필요하다.

## 2. 분류 매트릭스 — 분야 × 처리유형

복지 카테고리(지원대상×지원방식)와 다른 축을 썼다 — 민원은 "누구를 지원하는가"가 아니라 "어느 분야의 무슨 처리인가"가 핵심이라 판단했다.

### 분야(DOMAIN, 18개)
가족관계·부동산건축·세무·자동차운전·보건의료·교육·노동·사업자·출입국·병무·환경·문화체육·농림수산·안전치안·지식재산·금융·복지관련증명서·사법등기

### 처리유형(PROC, 5개)
발급·열람·증명(ISSUE) / 신청·등록·허가(APPLY) / 신고(REPORT) / 정정·변경(CHANGE) / 조회·확인(INQUIRY)

18×5=90 조합 중 의미 없는 6개(병무-변경/신고, 지식재산-신고, 안전치안-발급, 금융-신고, 사법-신고)를 제외해 **84개 카테고리** 생성.

## 3. 산출물 — 제목(스캐폴드)만

요청하신 대로 **§0(예상 흡수 범위 개요) 한 문단만 넣고, §1~§5 본문은 작성하지 않았다.** 각 파일 헤더에 "★ 제목만 작성된 상태, 본문 미작성"을 명시해뒀다 — 나중에 실제 목록 확보 후 본문을 채울 때 이 마커로 진행 상태를 구분할 수 있다.

## 4. 저장 위치 — 복지 작업과 명확히 분리

새 디렉토리 **`온라인 민원 SP`**를 만들어 그 안에 저장한다. 기존 `prompts/Jejudo/benefit-categories/`(복지)와 섞이지 않도록 분리했다.

## 5. 정직하게 남긴 것

1. **84개 카테고리 전부 미검증**이다 — 복지 카테고리처럼 실제 원본 인스턴스로 가중치·우선순위를 매길 근거 자체가 없다.
2. 분야·처리유형 축 자체도 자의적 설계다 — 실제 목록을 보면 이 매트릭스가 안 맞는 민원이 나올 수 있다.
3. 다음으로 실제 정부24 민원 목록을 확보하는 게(예: 국회입법조사처 자료, 정부24 사이트맵 직접 파싱, 또는 별도 data.go.kr 데이터셋 탐색) 최우선 과제다 — 이게 없으면 이 84개는 "그럴듯한 뼈대"일 뿐 신뢰도가 낮다.

## 6. 다음 단계

1. 정부24 실제 민원 목록 확보 경로 재탐색
2. 확보 후 84개 카테고리와 대조 검증(범주 자체의 타당성 포함)
3. 검증된 카테고리부터 본문(§1~§5) 작성
