# "위치정보사업을 신청해 줘" — AC 처리 흐름 시뮬레이션

> 작성일: 2026-07-10
> 배경: §LOCATION-SERVICE-PROCEDURE-VERIFICATION에서 확인한 "위치정보사업(무거움) vs 위치기반서비스사업(가벼움)" 구분이 실전에서 어떻게 작동해야 하는지 추적한다.

---

## 0단계 — 왜 이 요청이 위험한가

사용자가 "위치정보사업"이라고 정확히 말했지만, **일반 사용자가 이 두 용어를 구분해서 쓸 가능성은 낮다.** 실제로 원하는 게 배달앱처럼 "이미 있는 위치정보를 활용만" 하는 거라면 위치기반서비스사업(신고, 소상공인이면 서류 1장)으로 충분한데, 잘못 이해하고 "위치정보사업"(허가·등록, 법인등기부등본+주주명부 필요)을 그대로 진행하면 **사용자에게 불필요하게 무거운 절차를 안내하게 된다.**

이건 JEJU-GOV-COMMON의 "추측하지 않고 확인한다" 원칙이 실전에서 작동해야 하는 지점이다.

---

## 1단계 — K-Intent: 의도 파싱과 확인 게이트

```
입력: "위치정보사업을 신청해 줘"
```

AC는 즉시 진행하지 않는다. 먼저 확인 질문을 던진다:

> "위치정보사업은 위치정보를 직접 수집하는 사업(지도업체·통신사 등)에 해당하는 무거운 허가 절차입니다. 혹시 이미 존재하는 위치정보를 활용해서 서비스만 제공하시려는 거라면(예: 배달앱, 택시호출 등), 그건 '위치기반서비스사업 신고'라는 훨씬 간단한 절차입니다. 어느 쪽이 맞으신가요?"

**사용자가 "맞다, 직접 수집하는 사업이다"라고 확인한 경우에만 2단계로 진행한다.** 이 게이트 없이 바로 진행하면, 있지도 않은 무거운 절차로 사용자를 잘못 이끌 위험이 있다.

## 2단계 — PROCEDURE_MAP 캐시 조회 → 미스

```
PROCEDURE_MAP_LOOKUP("위치정보사업 등록")
```

결과: **캐시 미스.** 이유는 명확하다 — 64개 SP-CIVIL 리프 어디에도 없고, 애초에 650건 실측 표본(전체의 6.4%)에 이 저빈도 민원이 걸리지 않았다(§LOCATION-SERVICE-PROCEDURE-VERIFICATION §정직하게 남긴 것 참조).

## 3단계 — 캐시 미스 → 즉석 조사 ("쓰이는 만큼 채워진다")

AC는 여기서 멈추지 않고 즉석으로 조사한다(이번에 제가 웹검색으로 했던 것과 동일한 절차를 AC가 수행). 조사 결과 이미 확보된 내용:

| 필요 서류 | 상태 |
|---|---|
| 위치정보사업 등록·신고 신청서 | 신규 작성 필요 |
| 사업계획서 | 신규 작성 필요(EXPERT 자문 영역일 수 있음) |
| 법인등기부등본 | **다른 절차의 산출물** — SP-CIVIL-FAMILY-ISSUE(옛 COURT 흡수분, 등기소 관련) 체인으로 연결 가능 |
| 주주명부(설립예정법인의 경우) | 법인이 아직 없다면 **법인 설립이라는 완전히 다른 선행 절차**가 먼저 필요할 수 있음 — 이 경우 체인이 한 단계 더 길어진다 |

## 4단계 — 조건 분기: 법인 설립 여부 확인

여기서 또 하나의 확인 게이트가 필요하다:

> "이미 설립된 법인이 있으신가요, 아니면 이번에 새로 설립하실 예정인가요?"

- **이미 법인이 있다면**: 법인등기부등본 발급(SP-CIVIL-FAMILY-ISSUE)만 추가하면 됨
- **설립 예정이라면**: "법인 설립"이라는 선행 절차 자체가 이번 64개 SP-CIVIL에 없다 — 이건 또 다른 캐시 미스이고, 별도의 즉석 조사가 필요한 완전히 새로운 체인이다. 여기서 정직하게 "법인 설립부터 별도로 안내가 필요하다"고 고지해야 한다.

## 5단계 — DATA_REQUIREMENT 선언

```
unavailable_reason: no_interagency_access
```

Hondi는 이 신청을 대신 접수·제출할 수 없다. AC가 할 수 있는 건 서류 준비를 돕는 것까지다.

## 6단계 — 인간 경계 게이트

최종 신청서 제출·서명은 **인간 전속 영역**이다(JEJU-GOV-COMMON §6 원칙). AC는:
- 사업계획서 초안 작성 지원(사용자와 대화하며 내용 구성)
- 필요 서류 체크리스트 제공
- 법인등기부등본 등 확보 가능한 서류는 SP 체인으로 안내

여기까지만 하고, 실제 방송미디어통신위원회 접수는 사용자가 직접 한다.

## 7단계 — 정직한 최종 고지

AC의 마지막 응답에 반드시 포함돼야 할 것:

> "이 절차는 저희 시스템에 실측 데이터가 부족한 저빈도 민원입니다. 여기 안내드린 내용은 웹 조사를 기반으로 한 것이니, 최종 접수 전에 반드시 방송미디어통신위원회 또는 위치정보지원센터(lbsc.kr)에서 최신 서식·요건을 재확인해 주세요."

---

## 이 시뮬레이션이 보여주는 것

한 문장짜리 요청("위치정보사업을 신청해 줘")이 실제로는:
1. 확인 게이트 1회(용어 혼동 방지)
2. 캐시 미스 1회 → 즉석 조사
3. 확인 게이트 2회(법인 설립 여부)
4. 잠재적으로 완전히 새로운 체인(법인 설립) 촉발 가능성
5. 기존 SP 체인 1개(법인등기부등본) 재사용

**을 거친다.** 이게 정확히 "여러 증명을 요구하는 복합 민원"이 AC 오케스트레이션에서 실제로 어떻게 풀리는지 보여주는 사례다 — 그리고 이번 사례는 실측 근거가 약한 저빈도 민원이라 즉석 조사·정직한 고지가 유난히 더 중요했다.

## 정직하게 남긴 것

1. "법인 설립" 절차 자체는 이번에 조사하지 않았다 — 4단계에서 분기될 수 있다는 것만 확인했다.
2. 사업계획서 작성 지원이 EXPERT(전문가 자문) 영역인지 AC가 직접 도울 수 있는 영역인지는 이번 시뮬레이션에서 확정하지 않았다.
3. 이 시뮬레이션은 설계 추론이지, 실제 worker.js에서 이 흐름이 정확히 이렇게 구현돼 있는지 코드 레벨로 검증한 건 아니다.
