# "위치정보사업을 신청해 줘" — AI-to-AI 실행 모델 (정정판)

> 작성일: 2026-07-10
> 정정 배경: 이전 문서(AC-SIMULATION-LOCATION-INFO-BUSINESS)는 "AC가 안내만 하고 사용자가 직접 접수한다"는 잘못된 전제로 작성됐다. 올바른 모델은 — **정부기관도 각자 AI 비서를 운용**하므로, Hondi의 AC는 상대 기관의 AI와 직접 협상·실행하고, 인간은 최종 확인·서명 단계에서만 개입한다.

---

## 핵심 정정 — 무엇이 바뀌는가

| | 이전(잘못된 모델) | 정정된 모델 |
|---|---|---|
| 서류 준비 | "체크리스트를 드리니 직접 준비하세요" | AC가 각 서류의 발급 주체 AI와 직접 통신해 확보 |
| 신청서 작성 | "여기 양식을 참고해 작성하세요" | AC가 사업계획서 초안까지 작성(사용자 검토용) |
| 접수 | "방송미디어통신위원회에 직접 접수하세요" | AC가 전자민원센터의 접수 시스템(또는 그쪽 AI)과 직접 통신해 접수 절차를 진행 |
| 인간의 역할 | 전체 과정을 인간이 수행 | **최종 확인·서명만** 인간이 수행(human_gate=true는 그대로 유지) |

인간 경계 게이트 원칙 자체는 바뀌지 않는다 — 다만 게이트 **이전**의 모든 단계(서류 수집·조립·초안 작성·접수 준비)가 "사용자에게 안내"가 아니라 "AC가 직접 수행"으로 바뀐다.

---

## 1단계 — 의도 파악 (웹 검색 + 확인 질문, 이전과 동일)

이전 문서 §1에서 확인한 원칙 그대로 유지: 사용자가 "위치정보사업"이라 말했어도 실제로 원하는 게 더 가벼운 "위치기반서비스사업"일 가능성을 먼저 웹 검색으로 재확인하고, 확인 질문을 던진다. **여기까지는 이전 문서와 같다.**

> AC: "위치정보사업은 위치정보를 직접 수집하는 무거운 허가 절차입니다. 이미 있는 위치정보를 활용만 하실 거라면 위치기반서비스사업 신고로 충분합니다. 어느 쪽이 맞으신가요?"

사용자가 "위치정보사업이 맞다"고 확인했다고 가정하고 이어간다.

## 2단계 — 필요 서류의 발급 주체를 각각 특정하고, 해당 기관 AI에게 직접 요청

이전 문서에서 "체크리스트 제공"으로 끝났던 지점을 여기서 실행으로 바꾼다.

| 서류 | 발급 주체 | Hondi AC의 실행 |
|---|---|---|
| 법인등기부등본 | 등기소(대법원 계열, SP-CIVIL-FAMILY-ISSUE 대응) | **등기소 시스템(또는 그쪽 AI 비서)에 발급 요청을 직접 전송**, 결과 수신 |
| 사업계획서 | 신규 작성물, 발급 기관 없음 | AC가 사용자와 대화하며 초안 작성 → 완성본 확보 |
| 주주명부(설립예정법인) | 4단계에서 분기(법인 설립 여부에 따라 별도 처리) | 아래 4단계 참조 |
| 신청서 자체 | 방송미디어통신위원회 / 전자민원센터(emsit.go.kr) | AC가 접수 시스템에 직접 제출 준비(양식 채움까지) |

**이게 §CIVIL-PETITION-IO-DRAFTING-PLAN에서 이미 확립했던 원칙과 정확히 같다** — "선행기관 output=이 기관 input 연쇄"를 AC가 직접 실행하는 것. 위치기반사업자 등록 사례에서 이미 이 체인 구조 자체는 확인했었는데, 이번 정정은 그 체인을 "사용자가 손으로 하나씩" 밟는 게 아니라 "AC가 각 노드의 AI와 통신하며 자동으로 조립"하는 것으로 바뀐다는 뜻이다.

## 3단계 — 기관 간 AI 통신 (GWP 라우팅)

각 정부기관도 자신의 AI 비서를 운용한다는 전제하에:

```
Hondi AC ←→ 등기소 AI비서 (법인등기부등본 발급 요청/수신)
Hondi AC ←→ 전자민원센터 AI비서 (위치정보사업 신고 접수 시스템)
Hondi AC ←→ 방송미디어통신위원회 AI비서 (심사 진행상황 조회)
```

이건 이 프로젝트에서 이미 설계된 GWP(Government Web Protocol 또는 유사 라우팅 프로토콜)와 K-Market의 AI-to-AI 주문 흐름과 같은 패턴이다 — 사람이 개입 없이 기관 간 시스템이 표준화된 방식으로 서로 통신한다는 전제다.

**정직하게 밝힘**: 이 AI-to-AI 통신이 실제로 구현돼 있는지, 아니면 설계 원칙으로만 존재하는지는 이번 세션에서 코드 레벨로 확인하지 않았다. 등기소·방송미디어통신위원회가 실제로 이런 AI 비서 인터페이스를 제공하는지도 확인된 바 없다 — 이건 이 프로젝트의 **목표 아키텍처**이지, 오늘 당장 작동하는 현실은 아닐 수 있다.

## 4단계 — 조건 분기(법인 설립 여부)도 AC가 직접 처리 시도

이전 문서에서는 "정직하게 고지"에서 멈췄지만, 정정된 모델에서는:

- **이미 법인이 있다면**: 2단계에서 등기소 AI와 통신해 등기부등본 확보로 끝
- **설립 예정이라면**: AC가 법인 설립 절차 자체를 새로운 캐시 미스로 인식하고, 즉석 조사 후 **법인 설립도 AC가 직접 조율**(법무사·등기소 등과의 연쇄) — 사용자에게 "따로 알아보세요"라고 떠넘기지 않는다. 다만 법인 설립처럼 훨씬 무거운 연쇄가 새로 열리면, 그 규모를 사용자에게 미리 알리고 진행 여부를 확인하는 게 맞다(인간 경계 게이트가 여기서도 작동 — "이 정도 규모의 일이 될 텐데 계속 진행할까요?"라는 확인).

## 5단계 — 인간 경계 게이트 (유일하게 인간이 개입하는 지점)

모든 서류가 조립되고 신청서가 완성되면, AC는 사용자에게 **최종본 전체**를 제시하고 확인·서명을 요청한다:

> "법인등기부등본 확보 완료, 사업계획서 초안 작성 완료(검토 필요), 신청서 양식 작성 완료. 이 상태로 방송미디어통신위원회에 접수할까요? 최종 제출은 본인 확인이 필요합니다."

여기서만 인간이 개입한다 — 그 이전 전 과정(서류 수집·조립·초안 작성)은 AC가 직접 수행했다.

---

## 이전 문서와의 근본적 차이 요약

이전 문서는 "AC가 지도를 그려주고 사용자가 직접 걷는다" 모델이었다. 정정된 모델은 **"AC가 직접 걸어가고, 마지막 문 앞에서만 사용자에게 열쇠를 건넨다"**다. 이건 이 프로젝트 초기에 확립됐던 원칙("Hondi가 합성 최초 신청자 역할을 해서 콜드스타트 열위를 없앤다", "AI-to-AI 주문 흐름")과 정확히 일치하는 방향이며, 이전 문서가 그 원칙을 충분히 반영하지 못했던 걸 이번에 바로잡은 것이다.

## 정직하게 남긴 것

1. **AI-to-AI 통신 자체가 실제로 가능한지는 검증하지 않았다** — 등기소·방송미디어통신위원회가 이런 인터페이스를 제공한다는 근거를 찾지 못했다. 이건 Hondi의 목표 아키텍처이지 현재 확인된 현실이 아니다.
2. 법인 설립처럼 캐시에 없는 완전히 새로운 연쇄가 열릴 때, AC가 "어디까지 자율적으로 진행하고 어디서 사용자 확인을 다시 받아야 하는지"의 정확한 기준(예: 예상 소요 시간·비용이 일정 수준을 넘으면 재확인?)은 이번 시뮬레이션에서 확정하지 않았다.
3. 이전 문서(AC-SIMULATION-LOCATION-INFO-BUSINESS)를 폐기하지 않고 남겨둔다 — "안내형" 모델이 실제로 필요한 경우(예: AI-to-AI 통신이 아직 안 되는 기관을 다룰 때의 폴백)도 있을 수 있어서, 두 모델을 비교 자료로 함께 보존하는 게 낫다고 판단했다.
