# 사용자 Profile 작성 지침서 (User Profile Authoring Guidelines) v1.0

> personal-assistant SP(현재 v1.1)와 함께 관리되는 데이터 거버넌스 문서.
> "누가, 무엇을, 어떤 우선순위로 Profile에 기록하는가"를 규정한다.

## 0. Profile의 정체성

- **GUID 매핑**: Profile은 사용자의 색상코드 신원(전화번호 → SHA-256 → IPv6 GUID →
  base-9 short_id)에 1:1로 매핑된다.
- **접속 방식**: 혼디 색상코드를 스캔하면 해당 사용자의 Profile — "나만의 미니 웹사이트" —
  로 접속한다. K-Market 등에서의 검색·연결 단위가 곧 이 Profile이다.

## 1. 기록 주체 — Personal Assistant (PA), 단일 기록주체 원칙

- 혼디 가입 시 최초로 호출되는 AI가 PA이며, **PA가 Profile의 유일한 공식 기록자**다.
- 구현체: `personal-assistant-vX.X.txt` (v1.1 → 본 지침 반영 후 v1.2)
- 다른 어떤 Agent·SP도 Profile을 직접 갱신하지 않는다 — 모든 갱신은 PA를 거친다. 이는 데이터
  무결성을 보장하는 단일 기록주체(single writer) 원칙이다.

## 2. Profile 데이터 4대 출처

| 출처 | 설명 | 처리 시점 | 신뢰 우선순위 |
|---|---|---|---|
| ① PA와의 대화 | 온보딩 질의응답(PHASE 1) | 가입 시 | 최우선 — 사용자 직접 응답 |
| ② 그림자(Shadow) AI의 PDV 기록 | 일상 대화 중 언급된 단서(영업시간 변경 등)를 그림자 AI가 PDV에 적재 | 상시 | ①과 동급(사용자가 실제로 한 말) |
| ③ 기존 웹사이트 | 사용자가 URL 제공 시 크롤링·요약 | 온보딩 또는 수정 시 | 보조 — 확인 전 임시값 |
| ④ 물리적 자료 스캔 | 메뉴판·간판·명함 등 이미지 → 비전 AI 추출 | 온보딩 또는 수정 시 | 보조 — 확인 전 임시값 (§4) |

**우선순위 충돌 규칙**:
- ③·④는 AI가 "추출"한 값이므로 사용자 확인 전까지 항상 **임시값**이다. ①을 덮어쓰지 않는다.
- ②는 사용자가 실제로 한 말이므로 ①과 동급. 단, ①·② 사이에 충돌(예: 과거 대화의 옛 영업
  시간 vs 새 답변)이 있으면 **더 최근 시점의 값**을 우선한다.

## 3. 그림자 AI → PDV → Profile 파이프라인 (원칙만 규정)

PA는 온보딩 또는 Profile 수정 요청 시작 시 사용자의 PDV를 조회하여, 아직 답하지 않은 필드의
후보값이 있는지 먼저 확인한다. 있으면 "맞으면 확인만 해주세요" 형태로 질문을 단순화해
재질문을 줄인다(personal-assistant v1.1의 "이미 아는 항목은 다시 묻지 않는다" 원칙의 확장).
구체적 조회 구현은 별도 문서(PDV_QUERY_PROTOCOL) 범위.

## 4. 물리적 자료 스캔 메커니즘

혼디 색상코드 스캔과 **동일한 위치의 파이프라인**이다 — "캡처 → 인식(비전 AI) → 구조화 →
사용자 확인". 상세 명세는 분리된 두 문서에 둔다:

- 비전 프롬프트: `SP-15-IMG_profile-scan_vision_prompt_v1.0.txt`
- 오케스트레이션 모듈: `src/gopang/profile2.0/physical-scan.js`

**핵심 안전원칙(요약)**: 가격·전화번호 등 숫자는 한 글자라도 불확실하면 단정하지 않고
`[확인필요]`로 표시한다 — 메뉴판 스캔에서 숫자를 잘못 읽는 것이 가장 직접적인 금전 피해로
이어지는 환각 유형이기 때문이다. (SP-COMMON-02 C3 출처정직성 원칙을 비전 인식에 적용한 것)

## 5. 거버넌스

갱신될 때마다 본 지침의 §2~§4 반영 여부를 재확인한다.
