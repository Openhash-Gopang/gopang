# PDV 완전보급 가정 하 "구조적으로 불가" 43건 — 4대 유형별 해법
## 현재 구현 가능 기능 vs 향후 설계 필요 기능 통합 정리

작성일: 2026-07-14
근거 문서: `공무원 업무지시 100건 해법 pdv완전보급 가정.md` (2026-07-14 세션)의 후속 심화 논의

---

## 0. 배경

전 국민이 Hondi PDV에 일상·업무·재무제표를 기록하고, 정부24로 모든 증명서를 발급받을 수 있다는 가정 하에서도, 공무원 업무지시 100건 중 43건은 "PDV 보급률과 무관하게" 구조적으로 풀리지 않는다고 분류되었다. 그 43건은 다음 4대 유형으로 나뉜다.

| 유형 | 핵심 문제 | 최종 판정 |
|---|---|---|
| ① 다수인 대상 집계·명단 추출 | "본인 동의" 프레임 자체가 안 맞음 | **완전 해결** |
| ② 강제조사·수사성 업무 | 영장/법적 조사권 기반이라 동의 모델과 충돌 | **부분 해결** |
| ③ 기관 내부 행정/예산 데이터 | 애초에 시민 개인정보가 아님 | **대부분 해결 (망분리 과제 잔존)** |
| ④ 정책용 익명 통계 | 개별 열람이 아니라 별도 익명화 파이프라인 필요 | **부분 해결 (기관별 확장 필요)** |

---

## ① 다수인 대상 집계·명단 추출 (체납자, 위반업소 등) — 완전 해결

### 핵심 논리
명단을 뽑는 주체를 "시민향 Hondi(PDV 에이전트)"가 아니라 **"그 기관 자신의 기관 AC"**로 재정의하면, 이는 "제3자가 동의 없이 접근"하는 문제가 아니라 "데이터 관리자(기관)가 자기 소유 원장을 자기 직원의 지시로 조회"하는, 이미 합법적으로 성립하는 행위의 자동화가 된다. PDV의 "본인 동의" 프레임 자체가 개입할 필요가 없어진다.

### 지금 구현 가능
- 기관 AC를 위한 **GWP_REGISTRY "기관용 AC" 카테고리** 신설 — 시민향 Hondi와 브랜드는 같으나 완전히 별도의 신뢰 영역
- 인증 스택 분리: 시민 = GWP Ed25519 TOFU / 공무원 = GPKI 기반 직무권한 인증
- 트리거를 "담당공무원의 특정 시각·특정 목적 요청"으로 못박는 로그 스키마 설계
- 지방세법·식품위생법 등 개별법이 이미 정한 "누가 어떤 조건으로 명단을 만들 수 있는가"를 그대로 AC 권한 스코프에 매핑 (새 권한 생성 아님, 기존 권한의 자동화)

### 향후 설계 필요
- **목적구속 원칙 강제 장치**: AC 응답 자체에 목적 태그를 부착하고, 후속 사용(예: 다른 목적 전용)을 추적하는 시스템
- **감사 추적 자동화**: AC의 모든 대량조회가 자동으로 감사 로그에 남고, 감사관이 사후에 "누가, 왜, 몇 명분을 뽑았는지" 검증 가능한 체계 (92번 항목과 연결)
- 기관별 AC 권한 스코프를 개별법 조문과 자동 매핑·검증하는 컴플라이언스 레이어

## ② 강제조사·수사성 업무 — 부분 해결

### 핵심 논리
"모든 범죄에 피해자가 있다"는 전제 자체가 틀렸고(무피해자 범죄 존재), 가해자가 자기 PDV에 범죄를 정직하게 기록할 유인이 없으며, 영장주의는 증거수집 기술 문제가 아니라 국가권력 통제를 위한 헌법적 장치이므로, PDV 완비만으로 강제조사·영장 제도 자체가 없어지지는 않는다.

다만 "경찰 소관 원장" 개념을 도입하면 상당 부분이 해결된다. 관건은 **거래성 데이터가 존재하는 경제범죄**와 **거래성 데이터가 없는 대인범죄**를 분리하는 것.

#### ②-A 경제범죄형 (국세청·공정위·감사원 소관) — 사실상 해결
탈세, 부당내부거래, 예산비리 등은 해당 기관이 **이미 법적으로 소관하는 원장**(세금계산서, 공시자료, 피감기관 문서)이 존재하므로, 기관 AC가 이를 상시 스캔·자동화하는 것은 ①과 동일한 논리로 새 권한이 아니다.

#### ②-B 대인범죄형 (검찰·경찰 소관) — 원장이 부분적으로만 존재
폭행, 마약투약, 뇌물의 대가성처럼 상응하는 "기관 소관 원장"이 원천적으로 없는 유형은, 경찰 AC 확장이 곧 혐의 없는 전 국민 상시감시가 되어 헌법상 적법절차·영장주의와 충돌한다.

### 지금 구현 가능
- **②-A**: 국세청·공정위·감사원 AC의 상시 모니터링 자동화 (①과 동일 아키텍처 재사용)
- **②-B 이벤트 트리거형 확장**: 이미 존재하는 개별법상 신고·등록 의무 데이터를 Hondi 표준으로 통합
  - 112/119 신고 접수 기록 (이미 경찰·소방 소관, 실시간 갱신 가능)
  - 병원 기관 AC의 응급의료법상 신고의무(타살의심 손상 등) 자동 표준화 → 경찰 AC로 목적구속 알림
  - 마약류통합관리시스템(NIMS) 데이터를 약국·병원 AC가 실시간 갱신 → 동일인 다중 병원 처방 등 오남용 패턴 자동 포착
  - FIU 의심거래보고(STR)·고액현금거래보고(CTR)를 금융기관 AC가 Hondi 표준 포맷으로 실시간 전송

### 향후 설계 필요
- **망분리 등급별 AC 배포 모델**: 경찰·검찰 AC의 신뢰 경계 분리 설계 (③ 항목과 연결)
- 신고의무 트리거 없는 잔여 영역(합의된 불법행위, 은밀한 대인범죄, 의도·인과관계형 범죄)에 대한 **개별 혐의 기반 영장 청구 프로세스**와 Hondi PDV 열람 연동 인터페이스 — 단, 이는 PDV 인프라가 아니라 사법절차 영역이므로 Hondi가 대체할 수 없고, "영장 발부 시 PDV 열람을 신속화하는 인터페이스" 정도로 범위를 한정해야 함

## ③ 기관 내부 행정/예산 데이터 — 대부분 해결 (망분리 과제 잔존)

### 핵심 논리
예산 편성, 부서 간 공문, 인사 발령, 조달 계약 등은 애초에 국민 개인정보가 아니라 기관이 자기 자신에 대해 생성·관리하는 데이터이므로, "본인 동의" 프레임과 무관하다. 기관 내부 직원이 Hondi를 업무 자동화 도구로 쓰는 문제로 귀결된다.

### 지금 구현 가능
- LLM 해외 서버 사용 제약 → **국내에서 오픈소스 LLM(DeepSeek 등)을 파인튜닝**하여 해결
- 부서 SP(System Prompt) 단위로 내부 업무 자동화 스코프 정의 (기존 GWP_REGISTRY 구조 재사용)

### 향후 설계 필요
- **망분리(network separation) 규제 대응**: 국가정보보안 기본지침상 예산·인사 등 민감 행정데이터가 있는 시스템은 인터넷망과 물리적·논리적으로 분리된 내부망(국가망)에 위치해야 함. LLM을 국내에 파인튜닝해도 해당 서버가 인터넷에 연결되어 있으면 망분리 위반
  - 옵션 A: LLM 가중치를 내부망으로 반입하여 온프레미스 추론 — Hondi의 중앙 SP 레지스트리 실시간 동기화 구조와 충돌 가능, 별도 오프라인 동기화 방식 필요
  - 옵션 B: 국정원 인증 망연계 솔루션(단방향 데이터 다이오드 등)을 통한 제한적 연동
- GWP_REGISTRY "기관용 AC" 카테고리를 **망분리 등급별**(내부망 전용 AC / 인터넷망 AC)로 세분화하는 배포 모델 설계 (②-B의 망분리 과제와 통합 설계 필요)

## ④ 정책용 익명 통계 — 부분 해결 (기관별 확장 필요)

### 핵심 논리
지역별 소득분포, 복지 수급 패턴 등 정책 통계는 개별 PDV 열람이 아니라 집계값만 추출하는 별도의 익명화 파이프라인이 필요하다. 모든 거래가 국세청 등에 보고되므로, 국세청 AC가 익명화된 통계를 API로 공개하는 방식이 유효하나, 이는 **거래성 통계**에 한정된다.

### 지금 구현 가능
- **국세청 AC 익명화 API**: 소득·소비·법인거래 관련 통계(지역별 소득분포, 업종별 매출 추이 등) — 국세청이 이미 전수 보고받는 데이터이므로 즉시 구축 가능한 범위

### 향후 설계 필요
- **기관별 익명화 API 표준화**: 거래 데이터가 아닌 정책 통계 영역은 각 소관 기관이 국세청 사례를 템플릿으로 동일 구조(자기 소관 원장 → 익명화 API) 별도 구축 필요
  - 복지 수급 패턴 → 복지부·사회보장정보원 소관
  - 의료·건강 통계 → 건강보험공단·질병관리청 소관
  - 교육·주거 통계 → 교육부·국토부 소관
- GWP_REGISTRY 차원에서 **동일한 프라이버시 보장 수준**을 모든 기관 익명화 API에 강제하는 표준 스펙 제정
- **차분 공격(differencing attack) 방어**: 단순 집계값 공개가 아니라 k-익명성 또는 차분 프라이버시(differential privacy) 같은 수학적 보장을 API 설계 단계에서 의무화 — 여러 세분화 질의 조합을 통한 개인 역추적 방지

## 종합: 지금 구현 가능한 기능 우선순위 (제안)

1. **GWP_REGISTRY "기관용 AC" 카테고리 신설** (①③④의 공통 기반)
2. **국세청 AC 익명화 통계 API** (④, 독립적으로 즉시 착수 가능)
3. **①의 목적 태그 없는 기본 조회 로그 스키마** (감사 추적 자동화의 전 단계)
4. **②-B 이벤트 트리거형 신고의무 통합** (112/119, NIMS, STR/CTR 연동 — 개별 시스템 연동 난이도 높아 우선순위는 후순위 검토)
5. **DeepSeek 국내 파인튜닝 PoC** (③, 망분리 배포 모델 설계와 병행 착수)

## 향후 설계가 선행되어야 하는 항목 (아키텍처 결정 필요)

- 망분리 등급별 AC 배포 모델 (②-B, ③ 공통)
- 목적구속·감사추적 표준 스펙 (①)
- 기관별 익명화 API 프라이버시 보장 표준 (④)
- 영장 발부 후 PDV 열람 신속화 인터페이스 (②-B 잔여 영역, 사법절차 영역이라 Hondi 단독 설계 불가 — 법원행정처 등 협의 필요)
