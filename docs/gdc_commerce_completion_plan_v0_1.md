# GDC 상거래(P2P 재화·용역 결제) 완성 계획서 v0.1

작성일: 2026-07-18 | 전제: 상용 서비스 시작 전 개발 단계

## 0. 요약

지난 턴 실사에서 확인된 대로, "A가 B에게 짜장면 값 10T를 GDC로 지불"하는
시나리오는 현재 오늘 만든 `sendGdc()`(단순 P2P 이체) 하나로 처리되고
있고, 요청하신 5가지 요건 중 실질적으로 구현된 건 판매자 자격 확인의
일부(전화·생체인증)뿐입니다. 특히 **재무제표 반영은 이미 설계까지 끝난
모듈(M10 `ledger.js`)이 있는데 아무 데도 연결이 안 된 죽은 코드**라는
게 이번 실사의 가장 중요한 발견입니다 — 새로 설계할 필요 없이 "연결"만
하면 되는 부분과, 처음부터 설계해야 하는 부분을 구분해서 계획을
짰습니다.

## 1. 현재 상태 정리

| # | 요건 | 상태 | 근거 |
|---|---|---|---|
| 1 | 송금 vs 상품대금 구분 | ❌ 없음 | `sendGdc()` `memo` 선택 필드 |
| 2 | 가격 공정성 판단 | ❌ 없음 | 카탈로그 매칭(K-Market)만 존재, 비정형 거래엔 미적용 |
| 3 | 판매자 자격(사업자등록/정부24) | ❌ 없음 | 유사 패턴(K-Gov 서류접수)만 있고 GDC와 미연결 |
| 3-1 | 신분증 등록 | ❓ 미확인 | 별도 확인 필요 |
| 3-2 | 전화번호 본인확인 | ✅ 있음 | Solapi SMS OTP (기존) |
| 3-3 | 지문/얼굴 인식 | ✅ 있음(기기 생체) | WebAuthn `userVerification:'required'` (기존) |
| 4 | 거래→재무제표 반영 | ⚠️ 설계는 있으나 미연결(죽은 코드) | `src/profile2.0/ledger.js`(M10), 호출부 없음 |
| 5 | GDC=현금계정 | ⚠️ 원칙은 살아있으나 정식 복식부기 아님 | L1 블록 `bs-cash` 직접 갱신, 감사 가능한 원장 아님 |

## 2. 완성 계획 (Phase별)

### Phase 1 — 거래 목적 분류 (최우선, 법적 리스크 직결)

지난 턴 법적 검토의 핵심 취약점입니다. `sendGdc()`에 `purpose` 필드를
필수화합니다:
- `purpose: 'transfer'` — 단순 송금/증여, memo 선택
- `purpose: 'purchase'` — 재화·용역 대금, **memo(품목명) 필수**, 향후
  Phase 2 판매자 자격과 연동될 필드

L1 블록·PDV 기록에 `purpose`를 함께 남겨, 사후에 "이게 재화 대금
결제였다"는 걸 소명할 수 있는 최소한의 근거를 만듭니다.

### Phase 2 — 판매자 자격 확인 게이트

신규 `seller-verification` 플로우 신설:
1. 사업자등록증 — 기존 K-Gov `REQUIRED_DOCUMENTS_REGISTRY` 패턴
   재사용(`acquisition:'gov24'`), 단 이건 "서류 소지 증명"(SHA-256 해시)
   수준이지 국세청 진위 확인 API 연동은 아님 — worker.js 자체 주석에
   있는 `KNOWN_LIMITATIONS`(원본 파일 저장·진위확인은 범위 밖)이 여기도
   그대로 적용됩니다. 진짜 진위 확인이 필요하면 국세청
   사업자등록정보 진위확인 API 별도 연동이 필요합니다(오픈 이슈).
2. 신분증 등록 — 현재 미구현, 신규 설계 필요
3. 전화번호 재인증 — 기존 Solapi 재사용
4. 생체인증 — 기존 WebAuthn PRF 재사용
5. 네 가지를 취합해 프로필에 `verified_seller: true` 플래그

**정책 결정 필요**: `purpose:'purchase'`인 이체를 받는 쪽이
`verified_seller`가 아니면 (a) 완전 차단, (b) 경고만 표시하고 진행,
(c) 소액(예: 1만원 이하)은 허용하고 그 이상만 차단 — 이 중 뭘 택할지
결정이 필요합니다. (c)를 권합니다 — 법적으로도 소규모 개인 간 거래까지
전부 막으면 서비스 취지와 충돌할 수 있고, 지난 턴에서 확인한 소규모
면제 기준(발행잔액 30억원 미만)과도 결이 맞습니다.

### Phase 3 — 가격 공정성 판단

**여기는 설계 논의가 필요한 지점**입니다. 카탈로그가 있는 K-Market
거래는 이미 가격 대조(`PRICE_MISMATCH`)가 되지만, 노점 짜장면처럼
카탈로그가 없는 완전 비정형 거래의 "공정 가격"을 시스템이 자동
판단하는 건 근본적으로 어렵습니다. 현실적 대안:
- (A) 이상거래 탐지: 동종 카테고리 평균 대비 N배 이상 벗어나면 경고만
  (차단 아님) — 통계 기반 최소 구현
- (B) 사후 신고제: 실시간 판단 대신, 이상 거래는 사후 감사(M11
  `audit.js`가 이미 M10에 의존하는 걸로 봐서 이 방향으로 설계돼 있었을
  가능성) 로 걸러내는 방식
- (C) 애초에 소액(Phase 2의 verified_seller 없는 거래는 상한선 이하만
  허용)으로 리스크 자체를 제한

권장: (C)+(B) 조합 — 소액은 사전 제한, 그 이상은 verified_seller
필수화로 사실상 "가격 공정성"을 "판매자가 사업자로서 책임진다"는
구조로 대체. 완전 자동화된 가격 판단(A)은 정확도가 낮아 오탐이 많을
위험이 커서 1차 범위에서 제외를 제안합니다.

### Phase 4 — 재무제표 통합 (설계는 재사용, 연결만 하면 됨)

**핵심 결정 필요**: 레거시 `ledger.js`(M10, Supabase 기반)를 되살릴지,
아니면 이미 라이브로 도는 L1 블록 체계 위에 복식부기를 새로 얹을지.
후자를 권합니다 — Supabase는 인프라 진화상 이미 L1/PocketBase로
이관 중인 레거시 계층이고(과거 기록 참고), 정본은 이미 L1입니다.

구체안: L1 `/api/tx`가 블록을 저장하는 시점에, `buyer_claim`/
`seller_claim`을 만드는 것과 동일한 원리로 **3행 복식부기 원장**(구매자
차변/판매자 대변/플랫폼 대변, `ledger.js`의 `buildMarketPurchaseRows`와
동일 구조)을 PDV 또는 신규 `ledger_entries` L1 컬렉션에 함께 기록합니다.
`ledger.js`의 `verifyBIVM`/`reconstructBalances`/`detectBalanceAnomalies`
함수는 그대로 재사용 가능합니다(로직 자체는 멀쩡합니다 — 그냥
연결만 안 돼 있었습니다).

**GDC=현금계정 원칙의 실제 검증**: `detectBalanceAnomalies` 패턴을
재사용해, L1 블록에서 역산한 `bs-cash`와 실제 GDC 잔고(L1
`computeBalance`)가 항상 일치하는지 정기 대사(reconciliation) 배치를
추가합니다 — 지금은 이 둘이 같다는 "원칙"만 있고 실제로 같은지
검증하는 코드가 없습니다.

### Phase 5 — 발행잔액 추적 (지난 턴 제안, 재확인)

Phase 4의 원장이 생기면 자연히 "지금까지 발행된 GDC 총량", "현재
미상환 발행잔액"을 원장에서 집계할 수 있습니다 — 법적 검토에서
확인한 30억원/500억원 면제 문턱 대비 현황을 상시 모니터링할 수
있게 됩니다. Phase 4에 종속되므로 같이 진행하는 게 효율적입니다.

## 3. 권장 진행 순서

```
Phase 1 (거래목적 분류) ─── 독립적, 가장 빠르게 착수 가능
Phase 4 (재무제표 연결) ─── 독립적, 설계 재사용이라 상대적으로 빠름
Phase 5 (발행잔액 추적) ─── Phase 4에 종속
Phase 2 (판매자 자격)   ─── 정책 결정(verified_seller 강제 수준) 필요, 신분증 등록 신규 설계 필요해 가장 오래 걸림
Phase 3 (가격 공정성)   ─── Phase 2 정책과 맞물림, 설계 논의 먼저 필요
```

## 4. 결정이 필요한 사항 (구현 착수 전)

1. Phase 2: `verified_seller` 아닌 상대에게 `purpose:'purchase'` 이체 시
   차단/경고/소액허용 중 정책
2. Phase 2: 신분증 등록 방식(어떤 신분증, 어느 정도 수준의 확인)
3. Phase 3: 이상거래 탐지 임계치를 둘지 여부, 둔다면 어느 카테고리
   기준으로 "동종 평균"을 잡을지
4. Phase 4: Supabase 레거시 원장(`ledger.js`)을 완전 폐기할지, 병행
   운영(이관 기간)할지

## 5. 제가 아직 확인 못 한 것

- 신분증 등록 관련 기존 코드가 정말 전혀 없는지(이번 검색으로는
  없는 것으로 보이나, 다른 이름으로 존재할 가능성 배제 못함)
- 국세청 사업자등록정보 진위확인 API 연동 여부/필요성은 법률 자문과
  함께 판단 필요(정부24 서류 해시 소지 증명만으로 "사업자 자격 확인"
  요건을 충족한다고 보기엔 약할 수 있음)

## 6. 2026-07-18 구현 완료 기록

- Phase 1(거래목적 분류): 완료 — `sendGdc()`/`/wallet/gdc-transfer`에 `purpose` 필수화
- Phase 2(판매자 자격): 완료 — 제출(`/seller/verify-submit`) + 승인 절차
  (`/seller/verify-queue`, `/seller/verify-review`, `admin_guids` 재사용,
  service='seller_verification')
- Phase 3(가격 공정성): §2 결론대로 완전 자동판단은 제외, 사후 신고제만
  구현(`/ledger/dispute`, `/ledger/dispute-queue`, service='trade_dispute_review')
- Phase 4(재무제표 통합): 완료 — `ledger_entries` 컬렉션, `/api/tx`·
  `/api/ai-charge`·`/api/mint` 세 곳 모두 기록, `/ledger/reconcile`로 대사
- Phase 5(발행잔액 추적): 완료 — `/ledger/issuance-summary`(단, 연간
  총발행액 500억원 기준은 미구현 — 다음 과제)

**관리자 등록은 이 배치 범위 밖**: `admin_guids`에 실제 사람을 등록하는
작업(예: `services: ['seller_verification', 'trade_dispute_review']`)은
서버 관리자가 DB에 직접 넣어야 한다 — 코드는 등록된 사람만 승인할 수
있게 게이트를 걸어뒀을 뿐, 최초 등록 절차 자체는 만들지 않았다.

**2026-07-18 후반 배치**: 관리자 승인 UI(`admin/seller-verify.html`)
신설로 실제 승인이 가능해짐. 누락돼 있던 `/ledger/dispute-resolve`도
같이 추가.

## 7. I1-2 후속 — 지갑 3세대 드리프트 재확인 (2026-07-19)

`docs/HONDI_MASTER_TEST_PLAN_v1_0.md`(I1-2, 2026-07-17)에 이미 3세대
드리프트 조사·18개 저장소 동기화가 "완료"로 기록돼 있었다. 이번 세션
(GDC 상거래 완성 작업 중) `gopang-wallet.js`에 `sendGdc()`를 새로
추가하면서 그게 허브(gopang)에만 반영되고 나머지 18개 위성 저장소
(users/klaw/911/police/health/school/stock/insurance/market/tax/
traffic/logistics/jeju/public/democracy/qna/security/gdc)에는 아직
전파 안 됐다는 지적을 받아 재확인했다.

**확인 결과**: I1-2의 원래 3세대 문제(v1.0.0 vs v2.0.0 vs 파일 누락)는
어제 동기화로 실제 해소돼 있었다 — users/security/gdc/market 4곳을
직접 클론해 확인, 전부 1,510줄·`IDB_VER=3`(허브의 2026-07-17 기준
최신본)로 일치했다. **이번에 새로 발견된 건 그 이후(오늘) 하루치
증분(sendGdc, 75줄 순수 추가)만 안 퍼진 것** — 훨씬 작고 명확한
범위다. `sync_wallet_18repos.ps1`로 동기화 스크립트를 별도 제공했다
(로컬에 이미 클론된 18개 저장소를 순회하며 gopang-wallet.js 복사 +
git add/commit/push).
