## 부록 E. 알림(Push Notification) 시스템의 구성과 발견된 결함

> 본 부록은 2026년 6월 21일, 혼디(Hondi)의 대화방 초청 알림이 "활성 상태에서는 소리가 안 나고, 비활성(브라우저 종료) 상태에서는 알림 자체가 아예 오지 않는다"는 신고를 진단·수정하는 과정에서 확인된 아키텍처, 발견된 구조적 결함, 검증 방법론을 정리한 것이다. 설계 문서가 아니라 실전 디버깅을 통해 확인된 사실의 기록이며, 같은 실수를 반복하지 않고 향후 유사한 증상을 더 빠르게 진단하기 위한 참고 자료로 작성한다. 부록 C와 마찬가지로 알림은 단일 기능 영역이므로, 개별 결함에 앞서 아키텍처를 먼저 설명한다.

---

### E-1. 전체 아키텍처 개관 — 활성 상태와 비활성 상태는 완전히 다른 두 시스템

혼디의 "대화 요청이 왔을 때 사용자에게 알린다"는 단일한 사용자 경험처럼 보이지만, 실제 구현은 서로 코드를 한 줄도 공유하지 않는 두 개의 독립된 경로로 나뉘어 있다.

**경로 1 — 앱이 활성 상태(포그라운드)일 때.** `src/gopang/ui/p2p-chat.js`의 `startIncomingWatch()`가 Supabase Realtime WebSocket을 직접 구독해(`webrtc_signals` 테이블의 `to_guid` 필터 INSERT 이벤트), 새 `offer` 타입 시그널이 들어오면 `handleIncomingOffer()`를 즉시 호출한다. WebSocket이 끊겼을 때를 대비한 3초 간격 폴백 폴링도 같은 함수를 호출한다. 이 경로는 Push API나 Service Worker와 무관하며, 탭이 실제로 열려 JS가 실행 중이어야만 동작한다.

**경로 2 — 앱이 비활성 상태(탭 닫힘)일 때.** 유일한 경로는 Web Push다. `worker.js`의 `handleSignalPost`가 `offer` 타입 시그널을 저장한 직후 `_sendPushToGuid()`를 호출해 VAPID 서명된 Web Push 메시지를 발송하고, 수신자 기기의 `sw.js`가 `push` 이벤트에서 `self.registration.showNotification()`으로 시스템 알림 배너를 띄운다. 사용자가 그 배너를 클릭하면 `notificationclick` 이벤트가 발생해 앱을 열거나 포커스한다.

이 두 경로가 완전히 독립적이라는 사실이 오늘 진단 전체의 핵심 전제였다. 한쪽을 고쳐도 다른 쪽 증상은 전혀 변하지 않으며, "활성 상태 사운드 없음"과 "비활성 상태 알림 자체가 안 옴"은 처음부터 별개의 버그 두 개(이상)였다.

**교훈:** 동일한 사용자 시나리오("알림이 온다")가 내부적으로 여러 독립된 코드 경로로 구현되어 있을 수 있다. 증상을 하나로 묶어 진단하면 엉뚱한 경로를 고치고도 문제가 해결됐다고 착각하기 쉽다. 진단을 시작하기 전에 "이 경로가 정확히 어느 코드를 타는가"부터 분기별로 확인해야 한다.

### E-2. 결함 ① — 활성 상태 사운드 누락: `Audio` 호출 자체가 존재하지 않았음

`handleIncomingOffer()`를 전수 검색한 결과 `Audio(` 호출이 단 한 곳도 없었다. 이 함수가 사용자에게 제공하는 피드백은 `confirm('📞 ${fromHandle}님의 연결 요청\n수락하시겠습니까?')` 브라우저 팝업 하나뿐이었다. 즉 "활성 상태에서 소리가 안 난다"는 신고는 회귀(regression)나 경쟁 상태가 아니라, 애초에 이 기능이 구현된 적이 없었던 것이다.

수정은 `confirm()` 직전에 `localStorage.getItem('gopang_push_sound')` 설정값으로 `/assets/sounds/{sound}.mp3`를 재생하는 코드를 추가하는 것이었다. `confirm()`은 동기적으로 스레드를 막으므로, 재생 호출을 반드시 그 앞에 두어야 소리가 실제로 시작된다.

**교훈:** "원래 동작하던 게 갑자기 안 된다"와 "애초에 구현된 적이 없다"는 증상만으로는 구분되지 않는다. 코드에 해당 기능의 흔적(`Audio`, `play()` 등 키워드)이 있는지 먼저 전수 검색하면, 회귀 디버깅에 시간을 쓰기 전에 "기능 부재"라는 더 단순한 가능성을 빠르게 배제 또는 확정할 수 있다.

### E-3. 결함 ② — 알림 클릭 시 새 창에서 사운드가 사라지는 경쟁 상태(race condition)

비활성 상태에서 알림을 클릭했을 때 사운드가 나지 않는 별도의 원인이 `sw.js`의 `notificationclick` 핸들러에 있었다.

```js
return clients.openWindow(url).then(w => {
  if (w) w.postMessage({ type: 'PLAY_SOUND', sound });
});
```

탭이 닫혀 있어 `clients.openWindow()`로 새 창을 여는 경우, 이 Promise가 resolve되는 시점은 새 창이 막 생성된 직후이며, 그 창의 JS(`gopang-app.js`)가 아직 로드·파싱되지 않아 `message` 리스너가 등록되기 전이다. `postMessage`는 받는 쪽이 없어도 에러 없이 그냥 사라지므로, 이 실패는 콘솔에조차 흔적을 남기지 않는다. 반면 탭이 이미 열려 있어 기존 창을 포커스하는 경로는 그 창의 리스너가 이미 오래전에 등록을 마친 상태이므로 문제가 없다 — 즉 같은 함수 안에서 한쪽 분기만 결함이 있는 비대칭적 버그였다.

수정은 새 창 분기에서 `postMessage` 대신 사운드 정보를 URL 쿼리 파라미터(`?playSound=ping`)로 실어 보내는 것이었다. 페이지가 로드되자마자(리스너 등록을 기다릴 필요 없이) `location.search`를 직접 읽어 재생하고, 재생 후 `history.replaceState()`로 파라미터를 제거해 새로고침 시 재재생을 방지했다.

**교훈:** `postMessage`는 받는 쪽이 그 시점에 존재하고 리스너가 등록되어 있어야만 동작하며, 실패해도 아무 신호를 남기지 않는다. Service Worker가 새 창을 막 띄운 직후처럼 "받는 쪽의 코드가 아직 실행되지 않았을 수 있는" 타이밍에는 `postMessage` 대신 네비게이션 자체에 데이터를 실어 보내는 방식(URL 파라미터 등)이 경쟁 상태에 안전하다.

### E-4. 결함 ③ — 중복 리스너로 인한 사운드 이중 재생

`gopang-pwa.js`와 `gopang-app.js` 양쪽 모두 `navigator.serviceWorker`의 `message` 이벤트에서 `PLAY_SOUND`를 처리하고 있었다. `gopang-pwa.js` 쪽 코드에는 다음과 같은 주석이 남아 있었다.

> "지금까지 sw.js는 두 메시지를 모두 보내고 있었지만 받는 쪽 리스너가 전혀 없어 둘 다 무반응이었음 — 알림 사운드 미재생 버그의 원인"

이 주석은 과거 세션에서 "리스너가 없어서 무반응"이라는 진단 하에 `gopang-pwa.js`에 리스너를 추가한 이력을 보여준다. 그러나 그 시점에 `gopang-app.js`에 이미 동일한 리스너가 존재한다는 사실을 놓쳤던 것으로 보인다. 결과적으로 탭이 열려 있는 상태에서 알림을 클릭하면 두 리스너가 동시에 `new Audio().play()`를 호출해 소리가 겹쳐 재생된다.

수정은 `gopang-pwa.js`의 리스너에서 `PLAY_SOUND` 분기를 제거하고 `CHECK_FOR_UPDATE`만 남겨, `PLAY_SOUND` 처리를 `gopang-app.js` 한 곳으로 일원화하는 것이었다.

**교훈:** "리스너가 없어서 무반응"이라는 진단으로 새 리스너를 추가하기 전에, 같은 이벤트를 처리하는 리스너가 코드베이스의 다른 위치에 이미 존재하지 않는지 전수 검색해야 한다. 이전 세션의 수정 이력(주석에 남은 진단 과정)을 그대로 신뢰하지 말고, 현재 코드베이스 전체를 기준으로 재검증하는 것이 안전하다.

---

### E-5. 결함 ④ — 탭 판별 로직의 도메인 하드코딩 (사고실험으로 발견)

E-3, E-4를 수정한 뒤 "사고 실험을 진행하라"는 요청에 따라, 수정된 코드가 실제로 모든 시나리오에서 버티는지 시나리오별로 추적하는 과정에서 별도의 잠재 버그가 발견됐다. `notificationclick`이 기존 열린 탭을 찾는 조건이 다음과 같았다.

```js
if (client.url.includes('gopang.net') && 'focus' in client) {
```

이 문자열이 하드코딩되어 있어, 별도로 진행 중인 `hondi.org`/`.net`/`.kr` 도메인 마이그레이션이 완료되면 이 조건이 더 이상 일치하지 않게 된다. 그 시점부터는 이미 열려 있는 탭이 있어도 "없음"으로 오판해 매번 새 창을 추가로 열게 되어, 사용자가 같은 앱이 중복 실행되는 것을 보게 된다.

수정은 `self.location.origin`(Service Worker 자신이 서빙되는 출처) 기준으로 비교하도록 변경하는 것이었다. 이렇게 하면 어느 도메인으로 운영하든 코드 수정 없이 항상 정확하게 동작한다. 같은 패턴이 다른 곳에도 있는지 저장소 전체를 검색했으나, `worker.js`의 `REGISTERED_SERVICES`(서브도메인 허브앤스포크 디렉터리)와 `register-profile.html`의 placeholder 텍스트만 발견됐고, 둘 다 성격이 달라(전자는 별도의 서브도메인 전체 마이그레이션 계획이 필요한 작업, 후자는 동작과 무관한 예시 텍스트) 손대지 않았다.

**교훈:** 버그 리포트 없이도 "지금 수정한 코드가 인접한 미래 변경(도메인 마이그레이션 등 이미 계획된 작업)과 충돌하지 않는가"를 의도적으로 점검하는 사고 실험은, 아직 증상으로 드러나지 않은 잠재 버그를 조기에 발견하는 데 효과적이다. 다만 발견된 모든 하드코딩을 기계적으로 동일하게 고치는 것은 과도할 수 있으며, 각각이 실제로 같은 종류의 위험을 안고 있는지 개별적으로 판단해야 한다.

### E-6. 결함 ⑤(근본 원인) — Web Push 페이로드가 암호화되지 않은 채 발송되고 있었음

비활성 상태에서 알림이 "아예" 오지 않는 증상의 진단은 다음 순서로 좁혀졌다. ① `GET /push/vapid-public-key`를 직접 열어 유효한 키가 반환되는지 확인 — 정상. ② 플랫폼이 iOS standalone 모드 제약과 무관한 안드로이드임을 확인. ③ 발송 함수 `_sendWebPush()`의 구현을 직접 검토.

```js
async function _sendWebPush(env, subscription, payload) {
  const res = await fetch(subscription.endpoint, {
    method: 'POST',
    headers: { ...vapidHeaders, 'Content-Type': 'application/octet-stream', ... },
    body: new TextEncoder().encode(payload),   // JSON 평문을 그대로 전송
  });
  return res.ok || res.status === 201;
}
```

Web Push 표준(RFC 8291)은 페이로드가 있는 푸시 메시지에 대해 구독 정보(`subscription.keys.p256dh`, `keys.auth`)로 `aes128gcm` 방식 암호화를 적용하고 `Content-Encoding: aes128gcm` 헤더를 붙이는 것을 강제한다. 이 코드는 암호화 단계 자체가 없었다. FCM(Android Chrome이 사용하는 푸시 서비스)을 비롯한 표준 준수 푸시 서비스는 이런 요청을 사양 위반으로 거부하며, `_sendWebPush()`는 `res.ok`만 보고 `false`를 반환할 뿐 그 실패가 어디에도 노출되지 않는다. 가입·구독·VAPID 키 발급까지 전부 정상이었던 이유는 그 단계들이 이 결함과 무관했기 때문이고, 정작 메시지를 실제로 내보내는 마지막 단계에서 매번 조용히 실패하고 있었던 것이다.

**구현.** ECDH(P-256) 키 교환 → HKDF 2단계(첫 단계는 `auth_secret`을 salt로, 두 번째 단계는 메시지별 랜덤 salt로) → AES-128-GCM 암호화 순서로 WebCrypto API(`crypto.subtle`)를 이용해 RFC 8291을 직접 구현했다. Cloudflare Workers는 Node.js의 `web-push` 라이브러리를 그대로 사용할 수 없어, 동일한 사양을 표준 Web Crypto API로 재구현하는 방식을 택했다.

**검증.** 손으로 작성한 암호화 코드는 미묘한 실수(HKDF info 문자열의 바이트 하나, 필드 순서 등)가 있어도 구문 검사로는 전혀 드러나지 않으므로, 별도의 독립적인 왕복 검증을 거쳤다.

1. 가상 구독자의 P-256 키쌍과 `auth_secret`을 생성한다.
2. Node.js 표준 라이브러리 `web-push`의 `encrypt()`로 동일 페이로드를 암호화한다.
3. 직접 작성한 복호화 로직(수신자 측, 즉 브라우저가 수행하는 것과 동일한 절차)으로 ②의 결과를 복호화해 원본과 일치하는지 확인한다 — 이는 "RFC 8291을 올바르게 이해했는가"를 `web-push`라는 신뢰할 수 있는 구현을 기준 삼아 검증하는 단계다.
4. `worker.js`에 실제로 넣은 암호화 함수로 같은 페이로드를 암호화하고, ③에서 이미 검증된 복호화 로직으로 풀어 원본과 일치하는지 확인한다 — 일치했다.
5. `web-push` 라이브러리 소스코드에서 `aes128gcm` 인코딩 시 실제로 설정하는 HTTP 헤더 목록을 직접 대조해, 레거시 인코딩(`aesgcm`)에서만 필요한 `Crypto-Key`/`Encryption` 헤더가 `aes128gcm`에는 불필요함을 확인했고, VAPID `Authorization` 헤더의 쉼표 뒤 공백 하나(`vapid t=…, k=…`)가 누락되어 있던 것도 라이브러리 형식에 맞춰 보정했다.

**교훈:** 표준 암호화 프로토콜을 라이브러리 없이 직접 구현할 때는, 구문 오류가 없다는 것과 암호학적으로 올바르다는 것은 전혀 다른 차원의 문제다. 신뢰할 수 있는 독립 구현(이 경우 널리 쓰이는 Node.js `web-push` 라이브러리)을 정답지 삼아 왕복 암복호화 테스트를 거치지 않으면, 겉보기에 그럴듯한 코드가 프로덕션에서 매번 조용히 실패하는 상태로 장기간 방치될 수 있다. 실패가 클라이언트에 노출되지 않는 시스템(웹 푸시처럼 발송 실패가 발신자에게 거의 보고되지 않는 구조)일수록 이런 검증의 가치가 크다.

### E-7. 진단 과정의 함정 — "패치를 전달했다"와 "실제로 적용됐다"는 다른 사실이다

E-3·E-4의 수정 패치(`notification-sound-fix.patch`)를 전달한 직후, 같은 세션에서 별도로 `sw.js` 한 줄만 고치는 패치(`sw-origin-agnostic.patch`, E-5에 해당)를 추가로 전달했다. 이후 "sw.js 변경분을 하나로 합쳐달라"는 요청에 따라 로컬에 누적된 작업본을 기준으로 통합 패치를 만들었는데, 이를 실제 GitHub 원격 저장소 최신 커밋 기준으로 검증(`git apply --check`)하자 적용이 거부됐다.

원인을 추적한 결과, 사용자는 `sw-origin-agnostic.patch`만 실제로 적용·푸시했고 `notification-sound-fix.patch`는 (아마 sw.js 부분이 그사이 다른 패치와 충돌해 `git apply`가 전체를 거부했음에도) 적용되지 않은 채 다음 단계로 넘어갔던 것으로 확인됐다. 이 사실은 로컬 작업 클론을 기준으로 패치를 계속 만들면 발견할 수 없고, 매번 실제 원격 저장소를 새로 내려받아 비교해야만 드러났다. 이후 동일한 패턴(가입 단계 사운드 패치가 부분적으로만 적용된 것)이 한 차례 더 재현됐다.

**교훈:** 작업자(이 경우 AI 어시스턴트)가 패치 파일을 만들어 전달했다는 사실이 곧 "그 변경이 실제 시스템에 반영됐다"는 보장이 되지 않는다. 다중 파일 패치 중 일부 파일만 충돌해도 `git apply`는 전체를 적용하지 않을 수 있고, 그 실패가 사용자에게 눈에 띄지 않게 지나갈 수 있다. 후속 작업을 진행하기 전에 항상 실제 원격 저장소(로컬 작업 사본이 아니라)를 새로 가져와 "이전에 전달한 패치가 실제로 어디까지 반영되어 있는가"를 코드 내용으로 직접 확인하는 습관이, 누적된 가정 위에 잘못된 패치를 쌓는 것을 방지한다. 이는 부록 B-7(GitHub raw URL의 CDN 캐싱으로 인한 배포 검증 함정)과 같은 범주이지만 원인은 다르다 — B-7은 "배포는 됐지만 확인 경로가 옛 캐시를 보여주는" 문제였고, 본 사례는 "애초에 배포(적용) 자체가 안 된" 문제였다.

### E-8. 미해결 — 브라우저 자동재생(autoplay) 정책

E-2~E-6의 코드 수정을 모두 반영한 뒤에도 활성 상태 사운드가 여전히 들리지 않는다는 보고가 있었다. 이 시점에 남은 가장 유력한 원인은 코드 결함이 아니라 모바일 브라우저의 자동재생 정책이다. WebSocket의 `onmessage` 콜백이나 Service Worker의 `postMessage`로 비동기적으로 트리거된 `audio.play()`는, 브라우저가 "사용자의 직접적인 동작에서 비롯된 재생"으로 인정하지 않아 차단할 수 있다 — 이 경우 `Promise` 거부는 코드의 `.catch(() => {})`에 조용히 흡수되어 어떤 흔적도 남기지 않는다.

`notificationclick`처럼 사용자의 실제 클릭에서 비롯된 이벤트 핸들러 안에서 호출된 `clients.openWindow()`는 그 "일시적 활성화(transient activation)"를 새로 열리는 페이지에 물려줄 수 있다는 점이 Chromium 계열에서는 알려져 있으나, 이는 사양 해석에 기반한 것이며 실제 기기에서 직접 검증되지는 않았다. 또한 WebSocket 메시지로 트리거되는 활성 상태 경로(E-2)는애초에 사용자 동작과 무관하므로 이 예외의 적용을 받기 어렵다.

이 한계는 코드만으로 완전히 해소할 수 없다. 가능한 후속 조치로는 ① 크롬 원격 디버깅(`chrome://inspect`)으로 실제 `NotAllowedError` 발생 여부를 직접 확인하는 것, ② 페이지 내 임의의 지점을 한 번이라도 탭하면 이후 자동재생이 허용되는 브라우저의 특성을 이용해 "오디오 활성화" 상태를 기억해두는 우회책을 고려하는 것이 있으나, 둘 다 본 세션 시점까지는 착수되지 않았다.

**교훈:** 알림 사운드처럼 사용자 동작 없이 자동으로 재생되어야 하는 기능은, 그 트리거가 페이지 자체의 직접적인 사용자 입력(클릭, 탭)에서 얼마나 멀리 떨어져 있는지(WebSocket 콜백, Service Worker 메시지, 프로그래밍적 네비게이션 등)에 따라 브라우저의 자동재생 정책에 막힐 위험이 달라진다. 코드 수정으로 "재생을 시도하는 경로"는 만들 수 있어도 "실제로 들리는 것"까지는 보장할 수 없으며, 이 간극은 실제 기기·실제 브라우저에서의 검증 없이는 좁혀지지 않는다.

### E-9. 향후 과제

- E-8의 자동재생 정책 우회 여부를 실기기(안드로이드 Chrome, 아이폰 Safari)에서 직접 검증하고, 필요하다면 "최초 1회 탭으로 오디오 활성화" UX를 도입할지 결정한다.
- E-6에서 직접 구현한 RFC 8291 암호화는 표준 준수 여부를 코드 리뷰 외부의 자동화된 테스트(예: CI에 왕복 암복호화 테스트 고정)로도 지속적으로 검증하는 것이 바람직하다.
- `worker.js`의 `TTL: '60'`(60초)은 기존 코드에 이미 존재하던 값으로 이번 작업에서 변경하지 않았으나, 사용자가 60초 이상 오프라인이면 푸시 자체가 만료되어 폐기된다 — 대화 초청의 실시간성과 메시지 적체 방지 사이에서 이 값이 적절한지는 별도로 재검토가 필요하다.
- iOS Safari의 Web Push는 홈 화면에 추가된 standalone 모드에서만 동작한다는 제약이 있다(본 부록의 범위 밖, 별도 확인 필요) — 정식 앱 출시 전까지 안드로이드와 아이폰 사용자 경험의 비대칭을 어떻게 안내할지 결정해야 한다.
