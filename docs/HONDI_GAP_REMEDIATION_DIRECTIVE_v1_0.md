## 0. 시작 전 필수 확인 사항 (이전 세션에서 얻은 교훈)

### 0.1 Git 동기화 — 매번 origin을 직접 fetch해서 확인할 것
이전 세션에서 두 차례, "패치를 만들어 드렸고 사용자가 적용한 것 같다"고 가정하고 넘어갔다가 실제로는 반영이 안 되어 있던 사고가 있었다(gopang의 `/biz/gdc-deposit-close` 건). **다음 작업을 시작하기 전, 그리고 패치를 만들기 전에는 반드시:**
```bash
git fetch origin main -q && git log origin/main --oneline -5
```
을 실행해서 실제 원격 상태를 확인하고, 그 상태 위에서 작업할 것. `git am`으로 패치를 적용받은 사용자가 "Applying:" 로그만 보여주고 `git push` 결과를 안 보여줬다면, push 성공 여부를 절대 추측하지 말고 `git fetch`로 직접 확인할 것.

### 0.2 패치 검증 — 실제 origin/main 기준으로 clean apply를 확인할 것
패치를 만들면 반드시 `git worktree add /tmp/xxx-verify origin/main -f`로 깨끗한 워크트리를 만들어 그 위에 `git am`이 성공하는지, 문법 검사(`node --check`)가 통과하는지 확인한 뒤에 사용자에게 전달할 것. 로컬에서 만든 커밋 기준으로만 패치를 만들면 origin이 그 사이 움직였을 때(사용자가 병행 세션에서 다른 작업을 했을 때) 패치가 깨진다.

### 0.3 파일 3개 이상은 zip으로
프로젝트 산출물이 3개 이상이면 개별 파일로 제출하지 말고 zip 하나로 묶고, 압축 해제·저장소별 복사·git add/commit/push 명령어를 함께 제출할 것(주피터님 기존 지시사항).

### 0.4 커밋마다 사후 검증
"사후 검증을 반드시 거치세요"라는 지시가 이번 세션 내내 유효했다. 코드를 고치면 반드시:
1. 문법 검사(`node --check`)
2. 실행 기반 테스트(node:test, 필요시 jsdom으로 DOM 재현)
3. 패치가 실제 origin/main에 clean apply되는지 워크트리 검증
세 가지를 전부 거친 뒤에만 완료로 간주할 것.

## 2. 항목별 상세 작업 지시

### 2.1 [최우선] insurance — 실제 보험 심사·지급 파이프라인 설계

**현재 상태**: `submitClaim()`은 이제 정직하게 PDV에 접수 기록만 남긴다("자동 심사 기능 없음"을 사용자에게 알림). 실제 심사·사기탐지·지급은 전혀 없다.

**해야 할 일**:
1. `js/ins-core.js`/`ins-premium.js`/`ins-claim.js`/`ins-risk.js`/`ins-auto.js`(미배선 죽은 코드 클러스터)를 **완전히 폐기**할지, 그 로직을 참고해서 `src/ins/*.js` 쪽에 얇게 재구현할지 먼저 주피터님과 결정할 것. (이전 세션 의견: 폐기 후 재구현 권장 — 죽은 클러스터가 Supabase placeholder에 의존하는 구조 자체가 L1 이관 방향과 안 맞음.)
2. 재구현한다면 최소 범위:
   - PocketBase L1에 `ins_claims` 컬렉션 신설(청구 ID, 사용자 guid, 보험종류, 금액, 상태(접수/심사중/승인/거부/지급완료), 생성/갱신 시각).
   - gopang worker.js에 `POST /biz/ins-claim`(청구 생성, 서명 인증 필수), `GET /biz/ins-claims`(본인 청구 목록 조회) 최소 2개 엔드포인트.
   - **심사·지급 자동화는 이번 범위에 넣지 말 것** — 그 자체가 법적 검토 대상(무심사 자동지급은 보험업법 인가 문제와 직결). 일단 "접수 → 사람이 확인 → 수동 상태 변경" 흐름만 만들 것.
3. `sql/02_ins_rls.sql`에 이미 설계된 RLS 정책(사용자는 본인 것만 SELECT, UPDATE는 service_role 전용)을 실제 Supabase 프로젝트에 적용할지, 아니면 L1 PocketBase 컬렉션 규칙(`listRule`/`updateRule`)으로 옮겨 적용할지 결정 — 플랫폼 전체가 L1로 이관 중인 흐름을 고려하면 후자 권장.

### 2.2 [법적, 대기] gdc LEGAL-HOLD 4종

**현재 상태**: `js/gdc-bank.js`(이자·대출), `js/gdc-credit.js`(신용평가), `js/gdc-pool.js`(법정화폐풀·발행), `src/gdc/escrow.js`(K-Law 자동집행) — 전부 법률 검토 대기 중 배너 주석과 함께 보류.

**해야 할 일**: 코드 작업이 아니라 **의사결정 대기 항목**이다. 새 세션에서는:
1. 주피터님께 변호사 자문 진행 상황을 먼저 확인할 것.
2. 자문 결과가 나온 항목만 개별적으로 활성화 작업 진행. 자문 없이 임의로 활성화하지 말 것 — 이전 세션에서 이미 "법적 인허가 절차가 불필요한 것만 현실화"라는 원칙 하에 DAO·tokenomics는 활성화했고, 나머지 4개는 실질적 인허가 리스크가 있다고 판단해 보류한 상태다. 이 판단을 뒤집으려면 근거(예: 변호사 의견서)가 있어야 한다.
3. 각 항목이 실제로 어떤 인허가와 부딪히는지는 `js/gdc-bank.js`, `js/gdc-credit.js`, `js/gdc-pool.js`, `src/gdc/escrow.js` 파일 상단 배너 주석에 상세히 적혀있으니 먼저 읽을 것.

### 2.3 [중간] tax — 관리자 권한 체계 설계

**현재 상태**: `dashboard.html`에 로그인 게이트는 걸었지만("비로그인 차단"), 로그인한 사람이 실제 세무 담당자인지 확인하는 역할(role) 체계가 없다. gopang의 SSO는 현재 `'user'` role만 지원하고 `'admin'`은 미지원(코드 주석에 명시돼 있음).

**해야 할 일**:
1. gopang의 인증 시스템(`subsystem-auth.js`, `gopang-sso.js`)에 관리자 role 개념을 추가할지 결정 — 이건 tax 하나만의 문제가 아니라 **플랫폼 전체의 관리자 인증 체계 부재** 문제다. 이 세션에서 조사한 다른 저장소(insurance의 admin dashboard 등)에도 같은 구조적 공백이 있을 가능성이 높다 — 새 세션에서는 tax만 고치지 말고, **플랫폼 공통 관리자 인증 패턴을 gopang 레벨에서 설계**하는 것을 권장.
2. 최소 설계안: PocketBase L1에 `admin_guids` 컬렉션(또는 gopang worker.js에 하드코딩된 관리자 guid 목록 + env secret) 신설, `/biz/verify-admin?guid=` 같은 엔드포인트로 검증, 각 K-서비스의 관리자 대시보드가 이 엔드포인트를 호출해서 게이트.
3. tax의 `loadData()`는 이 관리자 검증이 준비되면 지금의 "로그인만 확인" 게이트를 "관리자 확인" 게이트로 업그레이드할 것.

3. 미성년자 데이터라 이 기능 자체가 법적 검토 대상일 수 있음(개인정보보호법상 법정대리인 동의 요건 등) — gdc의 LEGAL-HOLD 패턴과 동일하게 취급할 것.

### 2.5 [낮음] democracy — gdc DAO 엔드포인트 실제 연동

**현재 상태**: `gopang_laws.html`(고팡 평행 법전)이 "1,000표 이상 동의" 등 투표 절차를 상세히 문서화하고 있고, 이 수치(1,000)는 gdc의 `MIN_STAKE_VOTE`와 정확히 일치한다. 그러나 democracy 저장소의 실제 코드는 `/biz/gdc-dao/*` 엔드포인트를 전혀 호출하지 않는다.

**해야 할 일**:
1. `webapp.html` 또는 `desktop.html`에 실제 투표 UI(제안 생성 폼, 투표 버튼, 제안 목록)를 만들고 `src/gdc/dao.js`(v2, 서버 연동 버전)를 가져와 연결할 것. gdc 저장소가 이미 클라이언트 함수(`createProposal`/`vote`/`listProposals`)를 제공하므로 UI만 만들면 됨.
2. `MIN_STAKE_VOTE` 등 숫자가 향후 바뀌면 두 저장소가 다시 어긋날 수 있으니, 가능하면 democracy가 이 값을 하드코딩하지 말고 `/biz/gdc-dao/proposals` 응답이나 별도 정책 조회 엔드포인트에서 가져오게 설계할 것(tax의 수수료율 하드코딩을 `/biz/fee-rate`로 바꾼 것과 같은 패턴).

### 2.7 [낮음] traffic — Kakao API 키 프록시 패치 실배포 확인

**현재 상태**: `traffic/worker-kakao-patch.js`는 "gopang worker.js에 수동으로 추가하라"는 패치 스니펫 문서이지 실제 배포된 코드가 아니다. gopang worker.js에 `GET /kakao/appkey` 라우트가 실제로 등록돼 있는지 미확인.

**해야 할 일**: gopang worker.js에서 `/kakao/appkey` 라우트 존재 여부를 grep으로 확인. 없다면 traffic의 지도 기능이 API 키 없이 실패하고 있을 가능성 — 실제로 확인 후 필요하면 반영.

### 2.8 [낮음] police — 부정어 처리 고도화

**현재 상태**: `classifyCrime()`에 보수적인 규칙 기반 부정어 감지(키워드 앞 6글자 내 '안'/'않' 등)를 추가했다. 이건 임시방편이며, 복잡한 한국어 부정 표현(이중 부정, 먼 거리 부정, 조건문 등)은 여전히 놓칠 수 있다.

**해야 할 일**: 급하지 않음. 여유가 있으면 형태소 분석기(예: 간단한 규칙 기반이 아니라 실제 한국어 NLP 라이브러리) 도입을 검토할 것 — 다만 이건 `classifyCrime()`(법률 문서 생성용)에만 적용해야 하고, `analyzeText()`/`analyzeContext()`(실시간 위험도 판정)의 민감도는 절대 낮추지 말 것(이전 세션에서 여러 번 강조한 원칙).

- **qna**: 특허 문서 위주라 코드 리스크는 낮지만, `docs/` 안의 특허 명세서들이 실제로 출원된 것인지 초안인지 확인이 안 됨(코드 문제는 아니지만 주피터님이 궁금해하실 수 있음).

### 2.10 [중간] RLS/L1 정책 실제 확인

**한계**: 샌드박스 환경이 실제 Supabase 프로젝트 콘솔이나 라이브 L1 PocketBase 관리자 페이지에 접근할 수 없어서, "RLS가 실제로 걸려있는지" 같은 질문은 코드만 보고 확정할 수 없었다.

**해야 할 일**: 주피터님이 직접 Supabase 콘솔에서 다음을 확인해서 새 세션에 알려주시면 그걸 근거로 재평가:
1. `user_profiles`, `school_student_profiles` 등 개인정보 테이블에 RLS가 실제로 ENABLE 되어 있는지.
2. anon key로 이 테이블들을 직접 SELECT했을 때 실제로 몇 행이 반환되는지(전체 노출 여부 실측).

### 2.11 [낮음, 장기] Supabase 레거시 완전 제거

**현재 상태**: 여러 저장소(gdc, insurance, school, stock, tax, security)가 같은 공유 Supabase 프로젝트(`ebbecjfrwaswbdybbgiu.supabase.co`)를 여전히 참조하고 있다. 플랫폼 전체가 L1 PocketBase로 이관 중이라는 방향성과 어긋난다.

**해야 할 일**: 급하지 않음. 장기적으로 각 저장소가 Supabase 직접 호출을 완전히 걷어내고 L1/Worker 경유로 통일할 계획이 있다면, 이번 세션에서 다룬 저장소들의 Supabase 의존 지점을 목록화해서 별도 이관 계획을 세울 것.

### 2.12 [높음, 런칭 전 필수] 혼디 런칭 전 보안 모듈 재점검 체크리스트

5. **크로스오리진 vs 로컬 사본 판단** — gdc escrow.js의 사례처럼, "왜 로컬에 사본이 있는가"를 판단할 때는 실제로 그 값을 공유해야 하는 다른 코드가 지금 존재하는지부터 확인할 것(존재하지 않으면 드리프트 개념 자체가 성립 안 함).
