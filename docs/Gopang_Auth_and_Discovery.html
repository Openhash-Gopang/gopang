§1 개요

고팡은 중앙 서버 없이 사용자가 자신의 PDV(Private Data Vault)에 데이터를 저장하고, OpenHash 5계층 분산 네트워크를 통해 서로를 발견하는 완전 분산형 플랫폼이다.

사용자 정체성은 IPv6 주소 형식의 GUID로 표현되며, 한 번 등록한 정체성으로 K-Market, K-Law, K-Insurance 등 모든 하위 시스템에 SSO로 접근한다.

§2 사용자 등록

2.1 식별자 체계 — 3단계 분리 구조

정체성(Primary GUID), 통신 주소(Current IPv6), 검색 키(Nickname Hash)를 역할별로 분리하되, 하나의 공개키로 묶는다.

공개키 (ECDSA P-256) ← 4단어 시드로 언제든 재생성 가능 │ ├─ ① Primary GUID — 불변 정체성 (DB 기본키) ├─ ② Current IPv6 — 현재 통신 주소 (기기 변경 시 갱신) └─ ③ Nickname Hash — 검색 키 (닉네임 변경 시 갱신)

① Primary GUID — 불변 정체성

Primary GUID = "pguid-" + SHA-256(ECDSA 공개키)[:32] hex

예) pguid-3f7a9b2ec18d45fa...

공개키만을 입력으로 사용하므로 기기 변경·이사·닉네임 변경에 관계없이 항상 동일한 값이 생성된다. 4단어 시드 → PBKDF2 → 공개키 재생성 경로가 불변성을 보장한다.

② Current IPv6 — 현재 통신 주소

Current IPv6 = 2601:db80: + SHA-256(기기 핑거프린트)[:96bit]

예) 2601:db80:bd05:abfe:cf29:fc7f:f5a8:4e5b

기기 핑거프린트 기반 IPv6는 정체성이 아닌 통신 주소로만 사용된다. 기기 변경 시 OpenHash에 USER_UPDATE 블록을 추가하여 Current IPv6만 갱신하며, Primary GUID는 영구 불변이다.

{
  "type"        : "USER_REGISTER",
  "primary_guid": "pguid-3f7a9b2e...",
  "current_ipv6": "2601:db80:bd05:...",
  "public_key"  : "ECDSA 공개키",
  "timestamp"   : "2026-06-08T00:00:00Z",
  "signature"   : "ECDSA(primary_guid + current_ipv6, private_key)"
}

③ Nickname Hash — 검색 키

Nickname Hash = SHA-256(gopang_id)

예) SHA-256("체제수리공") = "a9f3c2..."

user_nicknames 테이블과 L2 이상 노드의 캐시 테이블에 저장되어 닉네임 → Primary GUID 역방향 조회에 사용된다.

식별자 요약

Primary GUIDSHA-256(공개키)영구 불변정체성, DB 기본키

2.3 OpenHash 네트워크 등록 (4단계)

Step 1 — L1 노드 자동 배정
  GPS 또는 IP 기반으로 가장 가까운 읍면동 노드 탐색
  REGISTER 패킷 전송 (GUID, 공개키, L1 주소, 자기 서명)

Step 2 — 상향 전파 Broadcasting
  L1(읍면동) → L2(시군구) → L3(광역) → L4(국가)
  각 계층 라우팅 테이블 갱신

Step 3 — OpenHash 블록 기록
  { type: "USER_REGISTER", guid, l1_node,
    public_key_hash, timestamp, prev_hash }
  SHA-256 해시체인에 연결 → 위변조 불가 영구 기록

Step 4 — 로컬 브로드캐스트
  동일 L1 권역 기존 사용자에게 신규 가입 알림 (mDNS 스타일)
검토 의견
Step 2의 상향 전파는 모든 신규 가입마다 L4까지 전파되므로, 사용자 수 증가 시 네트워크 트래픽이 선형적으로 증가한다. 전파 범위를 L2까지로 제한하고, L3·L4는 주기적 동기화(1시간 배치)로 처리하는 지연 전파(Lazy Propagation) 방식을 검토할 필요가 있다. → §9.1 해결

2.4 인증 레벨 체계 (L0–L3)

레벨수단적용 상황
L0기기 핑거프린트 (자동, 0초)일반 접속·검색·대화

§3 하위 시스템 SSO 인증

등록된 GUID 하나로 모든 하위 시스템에 접근한다. 접근 경로는 두 가지이다.

경로 A — 고팡 앱이 하위 시스템 호출 (GWP)
  고팡 앱 → gwp_token(HMAC 서명, 1시간 유효) 포함 URL 생성
  → 하위 시스템이 토큰 자동 검증 → 즉시 인증 완료

경로 B — 사용자 직접 접속
  ① sessionStorage 유효 토큰 → 즉시 완료
  ② Silent iframe (gopang.net/auth/silent-auth.html) → 토큰 수신
  ③ 미등록 기기 → silent-auth.html 리다이렉트 → 4단어 입력 → 복귀
검토 의견
경로 B-②의 Silent iframe은 브라우저의 서드파티 쿠키 차단 정책(Safari ITP, Chrome Privacy Sandbox)에 영향을 받는다. 모바일 앱(네이티브) 환경에서는 쿠키 대신 Keychain/Keystore 기반 토큰 저장을 병행하는 것을 권장한다. → §9.2 해결

§4 사용자 검색 (GDUDA)

4.1 OpenHash 5계층 구조

L5 글로벌
 └─ L4 국가 (대한민국)
     └─ L3 광역 (제주특별자치도)
         └─ L2 시군구 (제주시)
             └─ L1 읍면동 (이도1동) ← 사용자 PDV

각 L1 노드는 라우팅 테이블(local_users, neighbor_l1, kbucket)을 유지한다.

4.2 검색 5단계

가장 가까운 계층부터 순서대로 질의하며, 결과를 찾는 즉시 캐시하고 반환한다.

단계범위방식예상 지연
1로컬 캐시캐시 직접 조회즉시
2동일 읍면동 (L1)mDNS 직접 조회~1ms
3동일 시군구 (L2)BGP 라우팅 테이블~5ms
4동일 광역 (L3)Kademlia XOR 룩업~15ms
5국가·글로벌 (L4·L5)글로벌 인덱스~50ms
 
GDUDA_SEARCH(target_id):
  1. 로컬 캐시 확인 → 있으면 즉시 반환
  2. L1 노드 직접 질의 → 있으면 반환
  3. L2 라우팅 테이블 → target의 L1 주소 확인 → L1 직접 질의
  4. L3 Kademlia 룩업 (SHA-256(target_id) 기반)
  5. L4/L5 글로벌 인덱스 조회
  → 모든 단계 실패 시 NOT_FOUND
검토 의견
닉네임(@체제수리공) 검색과 GUID(해시값) 검색이 동일한 경로를 사용한다. 닉네임은 DHT 라우팅 키로 직접 사용하기 어려우므로, 별도의 닉네임→GUID 매핑 테이블을 L2 이상 노드에도 캐시하는 구조가 필요하다. 현재 GDUDA 문서에는 이 매핑 경로가 명시되지 않았다. → §9.3 해결

4.3 검색 완료 후 — P2P 직접 통신

검색으로 상대방의 L1 노드 주소와 공개키를 획득하면, 서버를 거치지 않고 PDV ↔ PDV 직접 E2E 암호화 통신을 시작한다.

PDV-A                                    PDV-B
  │ 1. GDUDA로 상대 L1 노드 탐색          │
  │ 2. 상대 공개키 획득                    │
  │ 3. 메시지 공개키로 암호화              │
  │──────── DIRECT_MESSAGE ─────────────→│
  │         { to, from, payload(암호화),  │
  │           signature }                 │
  │                       4. 복호화·검증  │
  │←──────── ACK + 수신 서명 ────────────│

상대가 오프라인이면 L1 권역 DB 노드가 최대 72시간 메시지를 임시 보관 후 전달한다.

검토 의견
PDV ↔ PDV 직접 연결은 NAT 환경(가정용 공유기, LTE 등)에서 P2P 홀펀칭이 필요하다. STUN/TURN 서버가 전제되어야 하며, 모바일 환경에서는 FCM/APNs 푸시 알림으로 PDV를 깨우는 웨이크업 메커니즘이 별도로 필요하다. → §9.4 해결

5.4 가중 평균 계산

평균 검색 시간
  = (2ms × 0.40) + (7ms × 0.30) + (20ms × 0.15)
  + (70ms × 0.10) + (200ms × 0.05)

  = 0.8 + 2.1 + 3.0 + 7.0 + 10.0

  ≈ 약 23ms

캐시 효과(파레토 법칙)를 반영해도 L5 글로벌 검색 5%가 200ms로 평균을 끌어올려 약 24ms로 수렴한다.

§9 기술 문제 해결 방안

9.1 등록 전파 범위 — 지연 전파 (Lazy Propagation)

[즉시] L1 → L2 전파 (수ms 이내)
  · 동일 읍면동·시군구 사용자는 즉시 검색 가능
  · 일상적 검색의 70%가 L1·L2 범위

[배치] L2 → L3 동기화 (10분 주기)
  · 신규 가입 후 최대 10분 내 광역 검색 가능

[배치] L3 → L4 → L5 동기화 (1시간 주기)
  · 타 국가 사용자에게는 1시간 내 검색 가능
방식L4 전파 요청 수 (하루 100만 가입 기준)
현행 (즉시 전파)100만 건/일 (실시간)
개선 (배치 동기화)24건/일 (1시간 배치)
감소율약 99.998%

9.2 Silent iframe 차단 — Keychain/Keystore 토큰 병행

인증 성공 시 토큰 저장 전략:

웹 브라우저 환경
  · 1차: sessionStorage (탭 단위)
  · 2차: localStorage   (도메인 단위, 30일)

§12 엔티티 유형별 키 발급 체계

12.1 배경 — 4단어 시드의 적용 범위 문제

고팡 플랫폼은 한국의 사람 5천만 명, 기관 1천만 개, 사물 1억 개 등 총 약 1.6억 개의 엔티티를 대상으로 한다. 4단어 시드 방식이 이 전체를 커버할 수 있는지 수학적·구조적으로 검토한다.

12.2 수학적 가용성 분석

4단어 시드의 엔트로피는 단어 사전 크기에 달려 있다.

단어 사전 크기4단어 조합 수비고
1,000단어1,000⁴ = 10¹² (1조)한국어 기초 어휘
2,048단어2,048⁴ ≒ 1.76 × 10¹³BIP-39 표준
10,000단어10,000⁴ = 10¹⁶국립국어원 기본어휘

총 등록 대상 1.6억 개 대비, 1,000단어 사전만으로도 약 6,250배의 여유가 있다. 순수한 경우의 수로는 충분하다.

최종 GUID는 SHA-256(공개키)로부터 256비트로 생성된다. 생일 역설(Birthday Problem)에 의해 충돌 확률이 50%에 도달하려면 약 2¹²⁸ ≒ 3.4 × 10³⁸개를 생성해야 하므로, 1.6억 개 규모에서 충돌 위험은 실질적으로 제로이다.

12.3 구조적 한계 — 엔티티 유형별 문제

수학적 가용성과 별개로, 4단어 시드 방식은 사람 중심 인증을 전제로 설계되어 있어 기관과 사물에는 구조적 문제가 발생한다.

엔티티문제핵심 원인
사람 (5천만)없음4단어를 종이에 보관 가능
기관 (1천만)키 소유권 단절대표자 교체·폐업 시 4단어 보관자 소멸
사물 (1억)입력 불가센서·기기는 4단어를 직접 입력할 수 없음
설계 제약
법인 GUID가 대표자의 개인 시드에 종속되면 기관 정체성의 연속성이 깨진다. 또한 IoT 기기 1억 개에 개별 4단어 시드를 할당하는 것은 운영상 불가능하다. 엔티티 유형별로 키 발급 파이프라인을 분리해야 한다.

12.4 엔티티 유형별 권장 키 발급 구조

사물의 경우 계층적 키 위임(Hierarchical Deterministic Key)이 현실적이다. 제조사가 마스터키를 보유하고, 생산 라인에서 각 기기에 서명된 자식키를 발급하는 방식이다. 이렇게 하면 4단어 시드의 역할은 사람에게만 한정되고, 기관·사물·AI는 각자의 등록 파이프라인이 담당한다.

사물   : 제조사 마스터키
           └─ HD Key 파생 → 기기별 자식키 → Primary GUID
           └─ 공장 출하 시 자동 발급, 사람 개입 없음
AI     : 소유자(사람·기관) 키로 서명된 파생키 → Primary GUID
           └─ 소유자 GUID와 계층 관계 OpenHash에 기록

AI Agent의 서명은 반드시 소유자의 개인키로 이루어져야 한다. AI 자신이 자신의 GUID를 자가 발급하는 구조는 허용하지 않는다. 소유자가 서명하지 않은 AI GUID는 OpenHash 노드가 등록을 거부한다.

GDUDA에서 AI Agent 검색

AI Agent는 GDUDA에서 일반 엔티티와 동일한 5계층 경로로 검색된다. 단, 검색 필터에 entity_type = 'ai'와 owner_guid를 추가하여 소유자 귀속 조회를 지원한다.

GDUDA_SEARCH(target_id):
  일반 경로 (캐시 → L1 → L2 → L3 → L4/L5)

추가 쿼리 패턴:
  · "@체제수리공의 AI"    → owner_guid 조회 후 하위 aguid 목록 반환
  · "K-Market AI 비서"   → scope='k-market' + entity_type='ai' 필터
  · aguid 직접 입력      → 일반 GUID 검색과 동일 경로

13.4 동명이인 처리

"홍길동"은 흔한 이름이므로 제주도 내에서도 복수 결과가 반환될 수 있다. 고팡은 이를 프라이버시 보호 원칙 하에 다음과 같이 처리한다.

결과 수처리공개 정보 범위
0건NOT_FOUND 반환—
1건즉시 연결 시도상대의 is_public 설정에 따름
2–10건후보 목록 표시 (닉네임 + 공개 프로필만)사용자가 공개한 항목만 노출
11건 이상추가 지역 조건 요구 (읍면동까지)범위 미축소 시 목록 미표시
프라이버시 설계 원칙
검색 결과에서 노출되는 정보는 반드시 해당 사용자가 is_public = TRUE로 설정한 항목에 한정된다. 닉네임이 실명인 경우에도 연락처·주소·직장 등의 정보는 사용자 동의 없이 노출되지 않는다. 11건 이상일 때 목록을 표시하지 않는 것은 대량 크롤링 방지(enumeration attack 차단) 목적이다.

13.5 L3 닉네임 캐시 테이블 확장

§9.3에서 설계한 닉네임 캐시 테이블에 l3_node 인덱스를 추가하면 경로 A를 단일 쿼리로 처리할 수 있다.

-- L3 노드 닉네임 캐시 (§9.3 확장)
{
  "nickname_hash" : "b7d2f1...",
  "primary_guid"  : "pguid-...",
  "l1_node"       : "KR-JEJU-JEJU-IDO1",
  "l2_node"       : "KR-JEJU-JEJU",
  "l3_node"       : "KR-JEJU",           ← 신규 추가
  "ttl"           : 86400,
  "updated_at"    : "2026-06-08T00:00:00Z"
}

-- 복합 검색 쿼리 (Supabase Phase 1 구현 예시)
SELECT primary_guid, display_name, public_profile
FROM   nickname_cache
WHERE  nickname_hash = SHA256('홍길동')
  AND  l3_node       = 'KR-JEJU'
  AND  is_public     = TRUE
ORDER BY updated_at DESC;

§14 속성 기반 검색 — 이력·시간 조건

14.1 GDUDA의 구조적 한계

"서울대학교를 90년 중반에 입학한 홍길동"은 §13의 지역+닉네임 검색과 근본적으로 다른 유형이다. 학력과 시간 범위는 GDUDA의 라우팅 계층(지역 기반)에 존재하지 않는 속성(Attribute)이다.

조건유형GDUDA 처리 가능 여부

"서울대학교 입학"이력 속성불가 — 라우팅 키 없음
"90년 중반"시간 범위 (~1993–1997)불가 — 라우팅 키 없음
세 조건의 교집합속성 복합 쿼리미설계
설계 한계
GDUDA는 위치 기반 계층 라우팅이다. 학력·직업·이력·시간 범위 같은 속성은 라우팅 키로 존재하지 않으므로, GDUDA 단독으로는 이 검색을 처리할 수 없다. 별도의 속성 인덱스 레이어가 반드시 필요하다.

14.4 속성 인덱스 서비스 (AIS) 설계

AIS(Attribute Index Service)는 사용자가 자발적으로 공개한 속성만을 저장하는 별도 레이어이다. 고팡의 PDV 원칙과 충돌하지 않으려면 세 가지 조건을 반드시 충족해야 한다.

조건구현
자발적 등록사용자가 명시적으로 "검색 허용" 체크 후 제출한 항목만 인덱싱
원본 비보관AIS는 GUID + 속성 해시만 저장, 원본 텍스트(졸업증명서 등)는 PDV에만 보관
철회 권한사용자가 언제든 특정 속성을 AIS에서 삭제 가능, 삭제 즉시 검색 불가
 
AIS 레코드 구조:
{
  "primary_guid"  : "pguid-3f7a9b2e...",
  "attr_type"     : "education",
  "attr_value_hash": SHA-256("서울대학교|입학|1995"),  ← 원본 비보관
  "school_guid"   : "oguid-snuXXXX...",               ← 기관 GUID 참조
  "enroll_year"   : 1995,
  "is_public"     : true,
  "submitted_at"  : "2026-06-08T00:00:00Z",
  "owner_signature": "ECDSA(..., 소유자 개인키)"       ← 자발적 제출 증명
}

14.5 "90년 중반" — 시간 범위 파싱

자연어 시간 표현을 정형 범위로 변환하는 파싱 레이어가 클라이언트에 필요하다.

자연어 입력          → 정형 범위
──────────────────────────────────────
"90년 중반"          → 1993–1997
"90년대 초"          → 1990–1993
"90년대 말"          → 1997–1999
"2000년대"           → 2000–2009
"작년"               → 현재연도 - 1
"최근 3년"           → 현재연도 - 3 ~ 현재연도
모호성 처리
"90년 중반"은 1990년대 중반(1993–1997)과 1890년대 중반(불가능)으로 파싱될 수 있다. 현재 맥락에서 1990년대로 확정되지만, "60년대 입학"처럼 복수 세기가 가능한 경우 사용자에게 확인을 요청한다. 또한 "입학"과 "졸업"을 구분하지 않은 입력은 두 경우를 모두 포함하여 검색하되, 결과 목록에 구분 표시한다.

§15 IPv6 기반 분산 공개 속성 레코드

15.1 출발점 — §14 AIS의 중앙화 문제

§14에서 설계한 AIS(Attribute Index Service)는 사용자가 자발적으로 제출한 속성을 저장하는 별도 레이어이다. 기능적으로는 작동하지만 중앙 서버가 필요하다는 점에서 고팡의 완전 분산 원칙과 긴장 관계에 있다.

고팡의 모든 사용자는 이미 고유한 IPv6 주소를 보유하고 있다. 이 주소를 통신 주소로만 사용하는 대신, DHT 공개 레코드의 키로 활용하면 외부 중앙 서버 없이 OpenHash 네트워크 자체가 속성 인덱스가 된다.

15.2 IPv6 주소 공간 재검토

IPv6 전체 공간   : 2¹²⁸ ≈ 3.4 × 10³⁸
고팡 사용 대역   : 2601:db80::/32 (고정 접두어)
                 + SHA-256(기기 핑거프린트)[:96bit] (가변)
가변 공간        : 2⁹⁶ ≈ 7.9 × 10²⁸

→ 한국 전체 등록 대상 1.6억 개는 이 공간의 극히 일부.
→ 각 IPv6는 사실상 전 우주적으로 유일한 주소.

현재 문서(§2.1)에서 Current IPv6는 통신 주소로만 정의되어 있다. 그러나 이 주소는 동시에 OpenHash DHT에서 해당 사용자를 가리키는 전역 유일 키이기도 하다. 이 이중 역할을 명시적으로 활용하는 것이 이 섹션의 핵심이다.

15.3 IPv6 기반 공개 레코드 구조

각 L1 노드는 관할 사용자의 공개 속성 레코드를 IPv6 키로 인덱싱하여 캐시한다. PDV가 오프라인이어도 L1 노드가 TTL 기간 동안 레코드를 제공한다.

공개 속성 레코드 (Public Attribute Record, PAR):
{
  "ipv6"          : "2601:db80:bd05:abfe:cf29:fc7f:f5a8:4e5b",
  "primary_guid"  : "pguid-3f7a9b2e...",
  "published_at"  : "2026-06-08T00:00:00Z",
  "ttl"           : 86400,
  "attrs"         : [
    {
      "type"      : "education",
      "school"    : "oguid-snuXXXX...",   ← 서울대학교 기관 GUID
      "role"      : "student",
      "year_from" : 1995,
      "year_to"   : 1999
    },
    {
      "type"      : "location",
      "l3_node"   : "KR-SEOUL",
      "l2_node"   : "KR-SEOUL-GN"
    },
    {
      "type"      : "occupation",
      "field"     : "software_engineering"
    }
  ],
  "signature"     : "ECDSA(SHA-256(attrs), 소유자 개인키)"
}
프라이버시 설계 원칙
PAR에 포함되는 속성은 오직 사용자가 명시적으로 선택한 항목만이다. 등록 시 속성별로 공개 범위를 설정할 수 있다(전체 공개 / 고팡 사용자에게만 / 상호 연결된 사용자에게만). PAR 자체는 언제든 철회 가능하며, 철회 시 L1–L5 노드 캐시에서 TTL 만료와 함께 자동 소거된다.

15.4 OpenHash 블록 확장

기존 USER_REGISTER 블록에 PAR 게시를 위한 블록 유형을 추가한다. 기존 블록 구조는 변경하지 않으며, 신규 블록 유형만 추가한다(additive-only 원칙).

기존 블록 유형 (변경 없음):
  USER_REGISTER · USER_UPDATE · AI_REGISTER · AI_REVOKE

신규 추가 블록 유형:
  PAR_PUBLISH  — 공개 속성 레코드 게시 또는 갱신
  PAR_REVOKE   — 공개 속성 레코드 전체 철회
  PAR_UPDATE   — 특정 속성 항목 추가·수정·삭제

PAR_PUBLISH 블록:
{
  "type"        : "PAR_PUBLISH",
  "ipv6"        : "2601:db80:bd05:...",
  "primary_guid": "pguid-3f7a9b2e...",
  "par_hash"    : SHA-256(PAR 전체),
  "prev_hash"   : "이전 블록 해시",
  "timestamp"   : "2026-06-08T00:00:00Z",
  "signature"   : "ECDSA(..., 소유자 개인키)"
}

§16 언어 코드 기반 닉네임 검색 최적화

16.1 현재 구조의 문제

§9.3에서 정의한 Nickname Hash는 SHA-256(닉네임)으로 생성된다. 이 방식에서는 언어 정보가 해시 생성 시점에 소멸되어 두 가지 문제가 발생한다.

문제 1 — 언어 간 충돌 가능성
  SHA-256("rose")      = "a4f2..."   ← 영어 닉네임
  SHA-256("로즈")      = "c9d1..."   ← 한국어 닉네임
  → 해시는 다르지만, 검색자가 "rose"를 입력할 때
    한국어 사용자 노드까지 전수 조회해야 할 이유가 없음

문제 2 — 검색 범위 과잉
  "홍길동" 검색 시 현재 경로:
    SHA-256("홍길동") → 전 세계 L4/L5 닉네임 인덱스 전체 조회
  → "홍길동"은 한국어 닉네임이므로 한국(KR) 노드 이외에서
    발견될 가능성이 통계적으로 극히 낮음
    전 세계 인덱스를 조회하는 것은 불필요한 비용

16.2 언어 코드 포함 해시 — 설계 변경

Nickname Hash 생성 공식을 다음과 같이 변경한다. 기존 해시와 하위 호환되지 않으므로 신규 필드로 추가하고 구버전을 병행 지원하는 전환 기간을 둔다.

기존 (v1):
  nickname_hash = SHA-256("홍길동")
                = "b7d2f1..."

개정 (v2):
  nickname_hash = SHA-256("ko:" + "홍길동")
                = SHA-256("ko:홍길동")
                = "3e9a7c..."   ← 완전히 다른 해시 공간

  언어 코드는 BCP 47 표준 사용:
    ko   — 한국어
    en   — 영어
    zh   — 중국어
    ja   — 일본어
    ar   — 아랍어
    ...

  다국어 별칭 등록 시:
    nickname_hash_ko = SHA-256("ko:홍길동")
    nickname_hash_en = SHA-256("en:Hong Gildong")
    nickname_hash_zh = SHA-256("zh:洪吉童")
    → 세 해시 모두 동일 primary_guid를 가리킴

16.3 언어 코드 → 지역 노드 사전 매핑

언어 코드와 GDUDA 계층 노드 사이의 확률적 매핑 테이블을 L2 이상 노드에 정적으로 배포한다. 이 매핑은 검색 시작점을 전 세계 L5가 아닌 해당 언어 주요 사용 국가의 L4로 즉시 낮춘다.

언어 코드우선 조회 노드근거
koKR (대한민국 L4)한국어 사용자 99% 이상이 KR 노드
enL5 글로벌 (분산)영어권 국가 다수 — 단일 L4 우선 불가
zhCN, TW, HK (복수 L4)중국어권 3개 주요 노드
jaJP (일본 L4)일본어 사용자 집중
arSA, EG, AE 등 (복수 L4)아랍어권 다수 분산

{

CREATE INDEX idx_nickname_lang

16.6 다국어 별칭 — 동일 GUID 연결

사용자가 언어별 닉네임을 복수 등록하면, 각 닉네임 해시가 모두 동일한 primary_guid를 가리키는 별칭 레코드로 저장된다. 검색자는 어느 언어로 입력해도 동일 인물을 찾을 수 있다.

동일 인물의 다국어 닉네임 캐시 레코드:

  { lang: "ko", hash: SHA-256("ko:홍길동"),      guid: "pguid-3f7a..." }
  { lang: "en", hash: SHA-256("en:Hong Gildong"), guid: "pguid-3f7a..." }
  { lang: "zh", hash: SHA-256("zh:洪吉童"),      guid: "pguid-3f7a..." }

OpenHash PAR_PUBLISH 블록에 alias 배열로 기록:
{
  "type"     : "PAR_PUBLISH",
  "aliases"  : [
    { "lang": "ko", "nickname_hash": "3e9a7c..." },
    { "lang": "en", "nickname_hash": "d1f4b8..." },
    { "lang": "zh", "nickname_hash": "a2e5c9..." }
  ],
  "primary_guid": "pguid-3f7a9b2e...",
  "signature"   : "ECDSA(..., 소유자 개인키)"
}

  · 전환 완료 기준: v1 조회 비율이 1% 미만으로 감소
  · 전환 완료 후 v1 컬럼 deprecated → 6개월 유예 → 제거
언어 코드를 Nickname Hash에 포함하고 언어→노드 매핑 테이블을 도입하면, "홍길동" 검색의 조회 범위가 전 세계 80억에서 한국 5천만으로 약 160분의 1로 축소된다. 이 최적화는 GDUDA 프로토콜 변경 없이 해시 생성 공식과 캐시 테이블 스키마 수정만으로 구현 가능하며, §13 지역 조건과 결합하면 읍면동 수만 명 수준까지 즉시 좁힐 수 있다.

§17 중복 닉네임 허용 — 표시 이름 모델

17.1 설계 전제의 변화

§9.3 이후 문서는 닉네임이 사실상 유일하다는 암묵적 전제 위에 설계되었다. 중복 닉네임을 허용하면 이 전제가 무너지고, 닉네임은 식별자(Identifier)에서 표시 이름(Display Name)으로 역할이 내려앉는다.

Primary GUID유일 식별자유일 식별자 (불변)

최종 식별 부담이 닉네임에서 빠져나와 다른 곳으로 이동한다. 이 이동을 어떻게 설계하느냐가 이 섹션의 전부이다.

17.2 보조 식별 수단 3계층

① 고팡 핸들 — @닉네임#태그

닉네임에 짧은 숫자 태그를 결합하여 동일 닉네임 내 유일성을 보장한다. 태그는 GUID에서 자동 파생되므로 사용자가 직접 선택하지 않는다.

handle    = "@" + display_name + "#" + GUID[:4] (16진수)

예)
  홍길동 A의 handle : @홍길동#3f7a
  홍길동 B의 handle : @홍길동#c18d
  홍길동 C의 handle : @홍길동#45fa

특성:
  · 태그는 GUID에서 결정론적으로 파생 → 충돌 없음
  · 사용자는 닉네임만 기억, 태그는 시스템이 관리
  · handle 직접 입력 시 1:1 즉시 검색 가능
  · 닉네임 변경 시 태그는 불변 (GUID 기반이므로)

② 검색 결과 내 맥락 필터 — PAR 기반

중복 닉네임 검색 결과를 반환할 때 §15의 PAR(공개 속성 레코드)를 함께 표시한다. 검색자가 지역·직업·상호 연결 등 맥락으로 대상을 선택한다. §13·§14의 동명이인 처리 구조를 그대로 재사용한다.

검색: "홍길동"
결과 (PAR 포함 표시):

  @홍길동#3f7a  |  제주시 이도1동  |  소프트웨어 개발자
  @홍길동#c18d  |  서울 강남구    |  변호사
  @홍길동#45fa  |  부산 해운대구  |  (비공개)
  @홍길동#9b2e  |  ✓ 검증됨       |  제주대학교병원 의사

③ 검증 배지 (Verified Badge)

기관·공인·유명인은 신뢰할 수 있는 기관 GUID의 서명으로 닉네임에 검증 표시를 부여받는다. 동명의 일반 사용자와 명확히 구분된다.

검증 배지 구조 (PAR attrs 내):
{
  "type"       : "verified",
  "issuer_guid": "oguid-jeju-hospital...",  ← 검증 기관 GUID
  "role"       : "의사",
  "issued_at"  : "2026-06-08T00:00:00Z",
  "expires_at" : "2027-06-08T00:00:00Z",
  "signature"  : "ECDSA(..., 기관 개인키)"
}

검증 계층:
  국가 기관 서명   → ✓✓ (최상위 신뢰)
  공인 기관 서명   → ✓  (기관 신뢰)
  상호 보증 누적   → ◎  (커뮤니티 신뢰)

17.3 닉네임 캐시 테이블 재정의

1 hash : N GUID 구조를 수용하도록 §16의 닉네임 캐시 테이블을 재정의한다. 기본키가 nickname_hash 단독에서 (nickname_hash, primary_guid) 복합키로 변경된다.

-- 개정 닉네임 캐시 (복합 기본키)
{
  "nickname_hash" : "3e9a7c...",        ← SHA-256("ko:홍길동")
  "primary_guid"  : "pguid-3f7a...",    ← 복합 기본키의 두 번째 컬럼
  "handle"        : "@홍길동#3f7a",     ← 신규: 직접 검색 지원
  "lang_code"     : "ko",
  "script"        : "Hang",
  "verified"      : true,               ← 신규: 배지 여부
  "issuer_guid"   : "oguid-...",        ← 신규: 검증 기관 (nullable)
  "l1_node"       : "KR-JEJU-JEJU-IDO1",
  "l2_node"       : "KR-JEJU-JEJU",
  "l3_node"       : "KR-JEJU",
  "ttl"           : 86400,
  "updated_at"    : "2026-06-08T00:00:00Z"
}

-- handle 직접 검색 인덱스
CREATE UNIQUE INDEX idx_handle ON nickname_cache (handle);

-- 검색 쿼리: 닉네임 중복 허용
SELECT handle, primary_guid, l3_node, verified, issuer_guid
FROM   nickname_cache
WHERE  nickname_hash = SHA256('ko:홍길동')
ORDER BY verified DESC, updated_at DESC
LIMIT  50;

17.4 handle 직접 입력 검색 경로

검색자가 @홍길동#3f7a처럼 handle을 완전히 입력한 경우, 닉네임 해시 조회 없이 handle 인덱스로 즉시 단일 GUID를 반환한다.

입력 유형 판별:
  "@홍길동#3f7a"  → handle 직접 입력
                   → idx_handle 인덱스 조회 → 1건 즉시 반환 (~2ms)

  "@홍길동"       → 닉네임만 입력
                   → nickname_hash 조회 → N건 반환 → PAR 필터

17.5 §16과의 관계 — 변경되는 것과 유지되는 것

§16 설계중복 허용 후 상태
SHA-256("ko:닉네임") 해시 공식유지 — 언어 파티셔닝 효과는 그대로
언어 → 노드 우선 매핑유지 — 검색 범위 축소 효과 유지
1 hash : 1 GUID 전제폐기 — 복합 기본키로 대체
닉네임 = 식별자폐기 — 닉네임 = 표시 이름
다국어 별칭 연결유지 — handle 단위로 연결

17.6 중복 허용의 최종 검색 흐름

입력                검색 경로                        예상 소요
─────────────────────────────────────────────────────────────
@홍길동#3f7a    handle 인덱스 직접 조회             ~2ms
홍길동          lang→노드 매핑 → nickname_hash      ~30ms
                → PAR 맥락 필터 → 사용자 선택
제주도 홍길동   L3 노드 → nickname_hash             ~20ms
                → PAR 필터 → 사용자 선택
서울대 홍길동   nickname_hash → PAR edu 필터        ~40ms
중복 닉네임 허용은 식별 부담을 닉네임에서 handle(자동 태그) + PAR 맥락 + 검증 배지의 3계층으로 분산한다. Primary GUID의 유일성은 완전히 유지되므로 인증·결제·계약 등 핵심 기능에는 영향이 없다. 검색 UX 관점에서는 오히려 사용자가 자유롭게 원하는 이름을 쓸 수 있어 진입 장벽이 낮아지는 이점이 있다.

§18 닉네임 혼동으로 인한 거래 상대방 오인 — 구조적 차단

18.1 본 문서의 범위 외 사안

닉네임 중복(§17)으로 인한 거래 상대방 오인 문제 — 예컨대 "제주도 사는 홍길동"에게 송금하려다 "서울 사는 홍길동"에게 잘못 입금하는 시나리오 — 는 본 문서(인증·검색 아키텍처)의 범위를 넘는다. 해당 문제의 해결 방안은 아래 세 하위 시스템에서 다룬다.

하위 시스템역할

18.2 구조적 차단 근거 — Give & Take 원칙

위 세 하위 시스템에는 다음 원칙이 공통 적용된다.

모든 거래는 반드시 한 쌍의 가치를 서로 주고받는 Give & Take로 구성되며, Give와 Take는 상호 동등하다.

이 원칙에 의해 고팡의 모든 거래는 단방향 송금이 아니라 쌍방이 각자의 Give와 Take를 명시적으로 서명한 교환 계약으로 성립한다. 트랜잭션 구조는 다음과 같다.

거래 = {
  give : { party: pguid-A, value: "GDC 100만원" },
  take : { party: pguid-B, value: "서비스 또는 재화" },
  signatures: [
    ECDSA(give + take, A의 개인키),   ← A의 서명
    ECDSA(give + take, B의 개인키)    ← B의 서명
  ]
}

→ 두 서명 모두 동일한 { give, take } 페이로드에 대해 이루어짐.
→ B의 서명은 B의 기기에서 B의 개인키로만 생성 가능.
→ "서울 사는 홍길동"의 기기에서 생성된 서명은
   "제주도 사는 홍길동"의 pguid를 포함한 페이로드에
   절대 일치하지 않음.

18.3 오인 차단이 구조적인 이유

닉네임 혼동으로 잘못된 상대를 선택하더라도, Give & Take 원칙 하에서 트랜잭션이 완결되려면 실제 수신자(B)가 본인의 개인키로 해당 거래에 서명해야 한다. 이 서명은 B의 PDV 내 개인키로만 생성되므로, 엉뚱한 상대가 선택된 경우 다음 중 하나가 반드시 발생한다.

상황결과
잘못 선택된 B가 거래 요청을 수신B는 예상치 못한 거래 내용에 서명하지 않음 → 트랜잭션 미완결
잘못 선택된 B가 실수로 서명B가 본인이 동의한 가치 교환 → 법적으로 유효한 거래 성립
A가 서명 전 수신자 GUID 확인페이로드에 pguid 명시 → 선택 오류 인지 가능

세부 구현(수신자 신원 카드, QR 기반 확정, 지연 실행, 사후 구제 절차 등)은 각 하위 시스템 문서에서 정의한다. 본 문서는 닉네임 검색 결과의 오인이 거래 완결로 이어지지 않음을 구조적으로 보장한다는 사실만을 명시한다.
