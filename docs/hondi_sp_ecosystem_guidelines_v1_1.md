# 혼디 SP 생태계 운용 지침 (Hondi SP Ecosystem Operating Guidelines) v1.1

> 본 문서는 페르소나 SP가 아니라 **거버넌스 문서**다 — 애플의 "App Store Review Guidelines"와
> 같은 위치에 있다. 혼디 생태계 안의 모든 SP(K-Law 판단 SP, 출력형식 SP, 평가기준 SP 등)는
> 본 지침을 따른다. v1.0은 골격이며, 8장의 미해결 과제는 의도적으로 미완성으로 남겨둔다.

---

## 0. 기본 전제: SP 생태계 = 앱 생태계

> "아이폰앱과 아이폰의 관계는 혼디 SP와 LLM의 관계다."

| 애플 생태계 | 혼디 생태계 | 비고 |
|---|---|---|
| iPhone (하드웨어) | LLM(Claude·DeepSeek 등 기반모델) | SP가 실행되는 "런타임" |
| iOS (운영체제) | SP-COMMON-02/03(C1~C28, M1~M7) | 모든 SP가 상속하는 공통 커널 |
| 개별 앱 | 개별 SP(K-Law, K-Tax, 변리사 SP 등) | L1 판단 SP에 해당 (§2) |
| App Store | 혼디장(Hondi Market) / `gwp-registry.js` | 등록·발견·심사 창구 |
| Xcode/SDK | SP 작성 도우미 (미구현, §8) | 진입장벽을 낮추는 도구 |
| App Review Guidelines | **본 문서** | 메타 거버넌스 |

**구조적 차이 — 이게 혼디의 진짜 기회이자 진짜 위험이다.** 앱은 설치되는 독립 실행파일이라
샌드박스로 분리되지만, SP는 **같은 LLM 위에서 즉시 호출·전환**된다. 설치·컴파일이 없으니
진입장벽이 압도적으로 낮다(자발적 참여가 현실적으로 가능한 이유). 그러나 모든 SP가 동일
런타임을 공유하므로, 한 SP의 결함이 다른 SP·LLM 자체의 신뢰까지 끌어내릴 수 있다 — 그래서
앱스토어보다 공통 안전장치(C1~C28)의 비중이 훨씬 커야 한다.

## 1. SP의 3계층 분류 — K-Law에서 추출한 핵심 모델

K-Law를 분해하면 SP가 1개가 아니라 **3개 계층**이 함께 작동하고 있었다. 이 3계층 모델이 본
지침의 중심축이다.

| 계층 | 역할 | K-Law 사례 | 지금까지 만든 38개 SP의 현재 상태 |
|---|---|---|---|
| **L1. 판단(추론) SP** | 도메인 추론을 실제로 수행하는 본체 | `klaw_v15.1.md` | ✅ 38개 전부 존재 |
| **L2. 출력형식 SP** | L1의 출력이 반드시 따라야 하는 강제 서식·완료태그 | `가상판결_출력형식_v13.3` | ❌ 미존재(STEP 태그는 있으나 강제출력 서식 분리 안 됨) |
| **L3. 평가기준 SP** | L1 산출물을 실측 정답(있는 경우)과 대조해 채점하는 메타기준 | `일치도_평가기준_v1.6` | ❌ 미존재 |

**원칙**: L1만 있으면 "구조적으로 완성됐지만 미검증" 상태다. 지금까지 만든 SP들이 정확히
이 상태임을 솔직히 인정한다. **L1+L2+L3 세 계층이 모두 갖춰져야 "검증된 SP"로 부른다.**

## 2. SP 등록 절차 — 누가/언제/어디서/무엇을/어떻게

| 항목 | 기준 |
|---|---|
| **누가(Who)** | 현업 경력 증빙 보유자(자격·경력 메타데이터를 색상코드 신원체계에 연계) 또는 기존 SP 개선 기여자(PR 방식) |
| **언제(When)** | 상시 제출. 단 최소 L1 + Tier 0 구조게이트(§3) 통과가 제출 전제조건 |
| **어디서(Where)** | 혼디장(Hondi Market) 제출 인터페이스 → `gwp-registry.js` 등록 큐 |
| **무엇을(What)** | 표준 제출 패키지: L1 본문 + (해당시) L2 출력형식 + (해당시) L3 평가기준 + 메타데이터(직종분류·위험등급·근거자격) |
| **어떻게(How)** | SP 작성 도우미(§8, 미구현) 활용 권장, 또는 SP-COMMON-02/03 직접 준수 작성 |

## 3. 심사 파이프라인 — Tier 0 → Tier 1-provisional → Tier 1-certified

```
제출 → [Tier 0: 자동 구조게이트] → [Tier 1-provisional: 시장 노출+실사용 데이터 축적]
     → [실질평가 분기] → [Tier 1-certified] 또는 [반려/재작성]
```

**3.1 Tier 0 — 자동 구조게이트** (즉시·자동)
STEP 태그 존재, 미확인정보 체크리스트, 반대논거 위험등급, 직역별 안전모듈(M1~M7 등 해당시)
포함 여부를 자동 검사. 이번 대화에서 25개 SP에 실제로 적용한 grep 검증과 동일한 방식.

**3.2 Tier 1-provisional 진입**
Tier 0 통과 즉시 "검증중" 배지로 시장 노출. 실사용 데이터(C22 배치분석 대상) 축적 시작.

**3.3 실질평가 — 분야별 분기**
- **정답지 존재 분야**(세무사→국세심판원 결정문, 관세사→품목분류 사례, 노무사→노동위 판정 등):
  L3 평가기준을 적용해 K-Law식 일치도를 주기적으로 산출
- **정답지 부재 분야**(학예사, 사서, 경영지도사 등): 동일분야 인증전문가 2인 이상의
  동료심사(C28 교차검증 프로토콜의 인간 버전)

**3.4 Tier 1-certified 승급 조건**
누적 사용횟수 N건 이상 + 심각 실패사례 0건 + (해당시) 일치도 X% 이상 (N, X는 분야별 §6 위험
등급에 따라 차등 — 구체적 수치는 추후 결정)

**3.5 강등·퇴출 조건**
사용자 불만족률 임계치 초과, 안전모듈 위반 발견, 미유지 기간 초과(§5.2)

## 4. K-Law를 참조 구현체(Reference Implementation)로

신규 SP 작성자는 K-Law를 다음 순서로 학습한다:
1. `klaw_v15.1.md`의 공리체계·STEP파이프라인 구조를 본다 (L1 작성법)
2. `가상판결_출력형식_v13.3`의 강제태그·완료선언 방식을 본다 (L2 작성법)
3. `일치도_평가기준_v1.6`의 배점·면책조항 구조를 본다 (L3 작성법)
4. v1.5→v1.6 변경이력처럼, 실측 데이터로 SP와 평가기준을 **함께** 갱신하는 패턴을 본다 (C22 운용법)

**✅ v1.1 갱신**: 위 4단계의 추상 템플릿은 별도 문서로 완성됨 →
`SP_authoring_standard_L1L2L3_v1_0.md` 참조. Tier 0 게이트(§3.1)는 이 표준을 기준으로 검사한다.

## 5. 보상 체계

**5.1 사용료 분배** — GDC 기반, 기존 수익자부담원칙과 정합. 분배율은 추후 결정.

**5.2 유지보수 연동** — 보상 지속조건을 "최초 등록"이 아니라 "C22 패치노트 갱신 주기 준수"에
연동. 방치된 SP가 계속 수익을 내는 모럴해저드 방지.

**5.3 책임·면책 프레임워크** — ⚠️ **미정, 법률 검토 필요.** 작성자의 책임범위와, 각 SP에 이미
넣은 면책고지가 작성자도 보호하는지를 먼저 확정해야 현업 전문가(잃을 게 있는 사람들)의
실질적 참여가 가능하다.

**5.4 평판·순위** — 색상코드 신원체계에 누적 트랙레코드(사용횟수·확신도 실측정확도·불만족률)를
연동해 공개. 동일분야 복수 SP 경쟁 시 이용자의 선택 기준이 된다.

## 6. 위험등급 차등 운용

모든 분야에 동시에 K-Law 수준을 요구하지 않는다. SP-COMMON-03(M1~M7)에서 이미 쓰던 위험등급
사고를 등록 절차 자체에 적용한다.

| 위험등급 | 예시 분야 | 요구 계층 | provisional 기간 |
|---|---|---|---|
| 고위험 | 의료·법률·금융·세무 | L1+L2+L3 전부 필수, 동료심사 필수 | 길게 |
| 중위험 | 건축·기술·교육 | L1+Tier0, L3는 점진 도입 | 중간 |
| 저위험 | 사서·학예사·기상예보사 | L1+Tier0만으로 신속 출시 | 짧게(사후 모니터링 중심) |
