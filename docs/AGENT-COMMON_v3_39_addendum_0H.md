## 0-H-2에 추가할 원칙 — 목표 경계 판단 기준

새 goal을 정의하거나(0-H-3 최초 조사 시) 기존 goal에 조건부 분기를
추가할지 판단할 때(0-H-2 목표적합성 확인 중 하위 변형이 드러났을 때),
아래 기준으로 가른다:

  - **같은 goal 안 조건부 분기로 표현**: 이미 시작한 하나의 신고·신청
    행위 안에서 선택지만 달라지는 경우(예: 사업자등록 신청서 안에서
    "법인/개인" 체크박스만 다름 — 관공서에 대한 의사표시 자체는 하나).
  - **별도 goal로 분리**: 갈래마다 별도의 신고·신청·등록 행위(관공서에
    대한 새로운 의사표시)가 필요한 경우(예: 법인전환은 "개인사업자
    폐업신고"라는 별도 행위와 "법인설립등기"라는 또 다른 별도 행위가
    필요하다 — 사업자등록 신청서 하나로 끝나지 않는다).

  판단이 애매하면(드물게 있을 수 있다) 별도 goal 쪽으로 기운다 — 조건부
  분기 안에 억지로 밀어넣으면 나중에 더 복잡해진다.

## PROCEDURE_MAP steps 확장 — sub_goal 합성 (0-H-8-B의 한계 해소)

기존 `steps` 항목은 `{seq, atom_id, expert_advisor, condition}`만
지원했다(atom 단위 실행만 가능). 아래를 추가한다:

```json
{"seq": 1, "sub_goal": "개인사업자 폐업신고"}
{"seq": 2, "sub_goal": "법인설립"}
```

`atom_id` 대신 `sub_goal`이 있으면, 그 seq는 atom을 직접 실행하지 않고
**다른 PROCEDURE_MAP을 그대로 재귀 조회·실행**한다(0-H-2부터 그 목표로
다시 들어간다 — 이미 캐시가 있으면 곧장 재사용). 이렇게 하면:

  - "법인전환"이라는 흐릿한 요청을 별도의 완전히 새로운 goal로 처음부터
    설계할 필요 없이, 이미 있는 "폐업신고"·"법인설립" 두 goal을 순서대로
    엮은 **합성 goal**로 즉시 만들 수 있다.
  - 0-H-8-B(목표 전환)의 "단순 교체만 다룬다"는 한계가 자연히 풀린다 —
    "전환"처럼 보이는 요청은 실제로는 새로운 합성 goal(sub_goal 두 개로
    구성된)을 만나는 것이지, 기존 goal을 다른 goal로 갈아치우는 게
    아니었다는 걸 이번에 알게 됐다.
  - 창업처럼 경계가 흐릿한 영역에서 계속 새 goal이 필요할 때마다, 완전히
    새로 설계하지 않고 기존 goal들을 조합하는 방식으로 대응할 수 있다 —
    goal 자체가 레고 블록처럼 재사용된다.

★ 순환 참조 방지 — `sub_goal`이 자기 자신이나 상위 goal을 다시 참조하면
(A→B→A) 무한 루프가 된다. 0-H-3(최초 조사)에서 새 sub_goal 조합을 만들
때 이 순환을 확인하는 절차가 필요하다(★ 미구현 ★, worker.js 쪽 검증
로직으로 다음 순서에 추가).

## 0-H-8-B 갱신 — "전환"의 재정의

기존 0-H-8-B("진행 중 목표 자체가 바뀌는 경우")를 아래로 갱신한다:

> "사실 개인회생이 나을까요?"처럼 **완전히 다른, 서로 무관한 목표**로
> 바뀌는 경우는 기존 0-H-8-B 그대로(새 goal로 교체, 확보 자료 재사용).
> "법인전환"처럼 **기존 목표를 공식 종료하고 새 목표를 잇는** 경우는
> 별도 처리하지 않고, 위 sub_goal 합성으로 그 자리에서 새 합성 goal을
> 만들어(또는 이미 있으면 재사용해) 0-H-2부터 그 합성 goal로 다시
> 진입한다 — "전환"은 특수 케이스가 아니라 "두 개 이상의 goal을 이어
> 붙인 합성 goal"의 한 사례였을 뿐이다.
