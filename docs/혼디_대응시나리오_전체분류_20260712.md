# 혼디 — 사용자 발화 대응 시나리오 전체 분류 (2026-07-12)

250건 사고실험(실제 코드·SP 대조 + hondi-proxy 실사 7건)을 종합해 정리한
"AC가 사용자 발화를 받은 뒤 실제로 갈 수 있는 모든 경로"의 전체 지도.
원래 9개 패턴(A~I) 분류를 실사로 검증하며 재구성했다 — 패턴 자체보다
"AC가 그 순간 어떤 종류의 판단을 내리는가"를 기준으로 재편했다.

---

## 1. 단일 서비스 위임 — `[GWP: id]`, 새 탭

AC가 gwp-registry의 21~24개 서비스 중 하나를 판단해 새 탭으로 위임.
가장 흔한 경로(250건 중 압도적 다수).

### 1-A. 활성 서비스 즉시 위임
`status: 'active'`인 서비스로 바로 이동. 도착 후에는 각 서비스의
자체 SP(K-Health, K-Tax 등)가 대화를 이어간다.
- 예: "짜장면 시켜줘"→kcommerce, "재산세 납부해줘"→ktax

### 1-B. 방향 반전 — 구매가 아니라 판매/등록
같은 카테고리(예: K-Market)라도 이용자가 "사려는" 게 아니라 "팔려는/
등록하려는" 방향이면 다른 GWP id로 가야 한다.
- 예: "중고 캠핑용품 등록해줘"→`kcommerce_seller`(오늘 신설)
- 이 반전을 놓치면 AC가 구매자 전용 SP로 보내 되묻기 루프에 빠진다
  (#48/#57에서 실제 확인).

### 1-C. 승인 대기(pending_review) 서비스 — 사실상 미작동
등록은 됐지만 `status`가 `active`가 아니라 라우팅이 코드 레벨에서
차단된다(오늘 신설한 안전장치). AC는 §9 표에 없는 서비스로는 애초에
가지 않으므로, 사용자에게는 "그런 서비스가 없다"로 보인다 — 아래
4번(SP_DRAFT_REQUEST)과 사용자 입장에서는 구분이 안 된다.
- 현재 해당: `kbank`, `ktelecom`, `kestate`(전부 오늘 신설, 배포 저장소
  미생성)

## 2. AC 자체 시스템 전환 — 새 탭 아님, 같은 스레드 안에서 전환

`_forwardSwitchSP`로 시스템 프롬프트 자체를 교체. 사용자는 탭 전환을
못 느낀다(대화가 이어지는 것처럼 보임).

### 2-A. K-Search 핸드오프 — `[KSEARCH_HANDOFF]`
"펫시터 구해줘"류. 실사(#175)로 실제 발동 확인됨.

### 2-B. 오케스트레이션 착수 — `[CALL_KINTENT]`
아래 5번으로 이어짐.

---

## 3. AC 직접 처리 — 라우팅 없음(R4)

어떤 서비스로도 안 넘기고 AC 자신이 답한다.

### 3-A. 단순 상태 저장 — PDV_STORE류
"주차 위치 저장해줘", "당뇨약 알림 설정해줘". SP 불필요, 태그 하나로
끝남.

### 3-B. 상식적 즉답 — §0-A 적용
잡담, 정서적 토로, 일반 지식 질문. §0-A(상식 최우선 원칙)가 SP 확인
5단계보다 먼저 적용된다.

### 3-C. 응급 게이트 — §0-G, 최우선
"살려줘", 명백한 위기 신호. **코드 레벨 사전 게이트가 없고 순수
LLM 판단**(이전 세션에서 확인된 설계 특이점) — R0로 분류되면 나머지
판단을 생략하고 곧장 처리.

---

## 4. 신규 SP 필요 신호 — `[SP_DRAFT_REQUEST]`

AC가 "이 요청을 처리할 서비스가 없다"고 판단할 때. 실사로 두 가지
전혀 다른 원인이 섞여 있다는 게 드러났다:

### 4-A. 진짜 커버리지 갭
정말로 대응하는 서비스가 없던 경우. 250건 사고실험에서 은행(kbank)·
통신(ktelecom)·부동산(kestate) 세 카테고리를 발견해 SP-Author를
대행 호출, 초안 작성·등록까지 완료(위 1-C 참조 — 배포는 별도 필요).

### 4-B. "가짜" 커버리지 갭 — 문서 결함으로 인한 오판
서비스는 있는데 §9 표의 설명이 부정확하거나 누락돼 AC가 못 찾는
경우. 오늘 실제로 여러 건 발견·수정:
- kcommerce에 "예약" 키워드 누락 → "미용 예약해줘"가 SP_DRAFT_REQUEST로
  잘못 감(#173 실사로 확인, v3.41~42로 수정)
- kgov에 "등록·신고" 키워드 누락 → "반려동물 등록해줘"가 잘못 감(#180
  실사로 확인, 같은 커밋으로 수정)
- K-Traffic 설명이 실제 SP와 완전히 다름("길찾기"라 적혀 있었지만
  실제로는 "과태료·단속·운전면허 행정") → v3.43으로 수정
- **AGENT-COMMON 안에 서비스 요약표가 최소 3곳(890행대 서술형,
  1965행대 정본 ID표, 2024행 예시) 흩어져 있어, 한 곳만 고치면 다른
  곳이 낡은 채로 남는 구조적 위험**이 있음을 스스로 재발견(v3.41→42
  경위) — 근본 해결(단일 소스화)은 아직 안 됨.

4-B는 코드 결함이 아니라 **AC가 참조하는 문서 자체의 결함**이라는 점이
중요하다 — 아무리 라우팅 코드를 잘 짜도 표가 틀리면 소용없다.

## 5. 오케스트레이션 — 다기관·다절차 조합, §0-H

`[CALL_KINTENT]` → K-Intent(의도 명확화) → `[HANDOFF_TO_KCOMPOSE]` →
K-Compose(여러 SP 순차/병렬 실행, `[ORCHESTRATION_PROGRESS]`로 실시간
진행상황 — 오늘 신설) → `[HANDOFF_TO_KDELIVER]` → K-Deliver(결과조립+
`pending_user_action` 분리+PDV기록+메타테이블 갱신) → AC로 복귀.

### 5-A. 진입 판별 — 단일 위임과 오케스트레이션의 경계
§0-H 기준(조건문+예시, 키워드 표보다 안전한 형태)으로 판별하지만
**실사 검증은 아직 안 함**(1번 카테고리처럼 hondi-proxy 실제 호출로
확인 안 된 상태 — 다음에 확인할 후보: "개인파산 신청 도와줘"=#250,
"이사하면서 전입신고랑 인터넷 설치까지"류 경계 케이스).

### 5-B. 알려진 구조적 약점
- 매 홉(AC→Intent→Compose→Deliver→AC)이 시스템 프롬프트 전체 교체라
  DeepSeek 캐시가 매번 깨짐 — "복잡한 요청일수록 캐시 이점을 가장 못
  받는" 역설(주피터님 확인, 별도 조치 예정)
- K-Compose 안에서 하위 SP를 실제로 몇 번 호출하는지(순차 시 지연시간
  누적)는 아직 실측 안 됨

## 6. 경계·판별 실패 위험 지대 — "정상 경로"가 아니라 위험 신호

250건을 순서대로 짚으며 실제로 애매했던 지점들. 심각한 결함까지는
아니지만 §9 표에 명시가 없어 AC 판단이 흔들릴 수 있는 회색지대.

| 유형 | 예시 | 경계 |
|---|---|---|
| 카테고리 월경 | #154 학원 등록, #160 캠프 신청 | kedu(학습지도 자체)인가 kcommerce(예약형 서비스)인가 |
| 행정 vs 의료 | #138 건강보험 자격득실확인서 | khealth인가 kgov인가 |
| 배송 vs 상거래 | #205 여행 짐 배송 | klogistics인가 kcommerce인가 |
| 응급 vs 정보성 | #126 응급실 안내 | kemergency(§0-G)인가 khealth인가 |
| 회사 vs 개인 | #152 직장인 심리상담 | khealth인가 kbusiness(EAP)인가 |
| 트리거 유사 다른 서비스 | K-Cleaner(환경신고) vs "청소업체 예약"(#230) | 실제로는 안 겹침(§9 표에 K-Cleaner 자체가 없어 위험 낮음), 그러나 gwp-registry triggers엔 "청소"가 겹쳐 있음 |

이 지대는 **정적 추적만으로는 확답 불가** — hondi-proxy 실사로만
검증된다(오늘 4/7건 실측 성공 사례가 이미 이 방법론의 유효성을
보여줌).

## 요약 — 경우의 수 트리

```
사용자 발화
├─ 1. 단일 서비스로 즉시 위임 (가장 흔함)
│   ├─ 1-A. 활성 서비스 (즉시 작동)
│   ├─ 1-B. 방향 반전(구매↔판매) 판별 필요
│   └─ 1-C. 승인대기 서비스 (사실상 미작동 — 사용자는 "없다"로 인식)
├─ 2. AC 시스템 전환(새 탭 아님)
│   ├─ 2-A. K-Search 핸드오프
│   └─ 2-B. 오케스트레이션 착수 → 5번으로
├─ 3. AC 직접 처리
│   ├─ 3-A. 상태 저장(PDV_STORE)
│   ├─ 3-B. 상식적 즉답(§0-A)
│   └─ 3-C. 응급 게이트(§0-G, 최우선 — LLM 판단 전담, 코드 게이트 없음)
├─ 4. SP_DRAFT_REQUEST
│   ├─ 4-A. 진짜 커버리지 갭 (은행/통신/부동산 → 오늘 해소)
│   └─ 4-B. 가짜 갭(문서 결함) — §9 표 누락/오류로 오판 (오늘 여러 건 발견·수정)
├─ 5. 오케스트레이션 (§0-H, 다기관 조합)
│   └─ 5-A/5-B. 진입판별 미검증 + 캐시 구조 역설(별도 조치 예정)
└─ 6. 경계 회색지대 (실사로만 검증 가능, 8~9곳 미확정)
```

## 이 분류가 시사하는 것

250건을 실제로 훑어보니, "완전히 새로운 기능이 없어서 실패"하는
경우(4-A)보다 **"있는데 AC가 못 찾아서 실패"하는 경우(4-B, 1-B, 6번)가
더 많았다.** 오늘 고친 것 중 절반 이상이 코드나 SP 자체가 아니라
AGENT-COMMON의 요약표(§9) 정합성 문제였다 — 표가 여러 곳에 흩어져
있어 한쪽만 고치면 다른 쪽이 낡은 채로 남는 구조가 근본 원인이다.
이 구조를 단일 소스화하지 않으면, 남은 750건에서도 같은 유형의
결함이 계속 나올 가능성이 높다.
