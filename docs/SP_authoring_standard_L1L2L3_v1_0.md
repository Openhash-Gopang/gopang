# SP 작성 표준 — L1/L2/L3 템플릿 명세 (K-Law 전형 기반) v1.0

> 본 문서는 「혼디 SP 생태계 운용 지침 v1.0」§4("L1/L2/L3 작성 템플릿, 별도 추출·문서화 예정")의
> 실행 문서다. K-Law(klaw_v15.1 / 가상판결_출력형식_v13.3 / 일치도_평가기준_v1.6)는 **법률
> 고유 내용과 구조가 뒤섞여 있다.** 본 문서는 그 구조만 뽑아내 도메인 불문 적용 가능한 표준으로
> 고정한다. **K-Law는 본 표준의 적용 사례가 아니라, 본 표준이 K-Law로부터 역추출된 것**이므로
> 표준-사례 관계가 거꾸로다 — 그래서 부록 A에 K-Law를 "최초 준수 사례"로 명시한다.

## 0. 표기 원칙

아래 모든 명칭(예: "판례법리군", "역방향추론")은 **K-Law가 실제로 쓴 표현**이다. 각 도메인은
**기능은 그대로 유지한 채 이름만 자신의 용어로 바꿀 수 있다.** 예: 의료 SP라면 "판례법리군"
→ "임상가이드라인군", "역방향추론" → "감별진단 반대증거"로 명명. **기능의 존재 자체는 협상
불가**이며, 이것이 Tier 0 게이트(§4)의 검사 대상이다.

## 1. L1 판단(추론) SP 표준 템플릿

### 1.1 공리체계 (Axiom System)
- 단일 출처 원칙: 동일 규범은 단 하나의 공리에만 존재 (중복 금지)
- 계층 구조: 기초원칙층(절대 원칙) → 해석규칙층(모호함을 푸는 규칙) → 절차규칙층(실행 순서)
- 각 공리는 고유 식별자(알파벳/번호)를 갖고 본문 전체에서 그 식별자로만 참조됨

### 1.2 강제규칙 체계 (이미 SP-COMMON-02 C21에 일반화 완료 — 재인용)
번호 매김 + 3행 구조(발동조건/위반효과/근거공리) + 등급(★필수/▲중요/○권고)

### 1.3 STEP 파이프라인 — 기능 4종, 명칭은 자유

| STEP | 필수 기능 | K-Law 명칭(참고) |
|---|---|---|
| STEP 0 | 동일성확인 + 준거틀확정 + **직관-격리**(초기 직관을 먼저 선언하고 정식분석과 대조) + 쟁점 3층위 명제추출 | 0-α 직관 / 0-β 격리 / 명제 |
| STEP A | 도메인지식 수집(군) + 의미해석 + **반대논거(역방향추론)** + 게이트웨이(전제조건 분기) + 요인분해 + 숨은쟁점탐지(명제정교화) + 충돌지점식별 | A-0~A-5 |
| STEP B | 최종확정 + **조건부모듈**(발동조건 명시, 미해당 시 1줄 선언 의무) + 상식필터·반전논거 배척 + 최종결론문 | B-1~B-7, B-5-A/B/C |
| STEP C | 보호항목점검(해당 도메인의 약자·위험 보호) + 인용검증 + 형식점검 + **완결게이트웨이**(체크리스트) | C-1~C-α |

- 각 STEP·서브스텝은 예외 없이 `[STEP-X-COMPLETE | 핵심요약]` 완료태그로 종료
- 조건부모듈은 "발동/미해당"을 **반드시 선언** — 침묵 금지(이게 누락되면 강제규칙 위반)

### 1.4 확신도+복잡도 (SP-COMMON-02 C19에 이미 일반화 — 재인용)
이원확신도(전문지식×사실관계) + 난이도지수(DI) 공식. 도메인별 산식은 자유이나 산출과정은 투명해야 함.

## 2. L2 출력형식 SP 표준 템플릿

L1이 "어떻게 사고할지"를 정한다면, L2는 "그 사고를 어떤 형태로 반드시 출력할지"를 정한다.
**L1과 L2는 별도 문서다** — 이게 K-Law 구조의 핵심이다.

| 구성요소 | 내용 |
|---|---|
| 메타데이터 | 변경이력 표(버전/기준 L1 버전/주요변경) — L1 버전과 L2 버전이 항상 매핑되어야 함 |
| 응답헤더 | `[SP명 vXX.X 적용 \| 사건/사례 식별자 \| 심급·단계 등 추가맥락]` |
| STEP별 강제서식 | §1.3의 4개 STEP 각각에 대해, 어떤 항목을 어떤 순서·형식으로 출력해야 하는지 필드 단위로 명시. 조건부 항목은 `[조건부]` 표시 후 미해당 시 1줄 선언으로 대체 |
| 완료태그 | STEP·서브스텝마다 `[STEP-X-COMPLETE \| 요약]` — 누락 시 응답 불완전으로 처리 |
| 완결게이트웨이 | C-α처럼 항목별 체크리스트(보통 8~10항목)로 출력 전 자가점검 |
| 확신도+메타지표 블록 | §1.4 수치를 표준 표기로 출력 |
| 자기검증 블록 | 강제규칙 준수여부 자가선언 + 미준수 규칙 명시 + 재생성 필요여부 |
| 요약 블록 | 핵심결론 2~3문장(주문·핵심근거·확신도 포함) |
| 부록: 면책고지 | 해당 도메인의 한계 고지(C24/M7과 연동) |

## 3. L3 평가기준 SP 표준 템플릿

L1·L2가 "SP가 어떻게 작동할지"를 정한다면, L3는 **"그 작동이 잘 됐는지 누가, 어떻게 채점할지"**
를 정한다. 정답지가 없는 도메인은 §4(혼디 SP 생태계 운용 지침)의 동료심사로 대체하되, 정답지가
있는 도메인은 아래 7장 구조를 따른다.

| 장 | 내용 |
|---|---|
| 제1장 목적·적용범위 | 평가대상 정의(L1 산출물 ↔ 실측 정답 쌍), 적용 L1 버전 호환성 |
| 제2장 배점체계(N점 만점) | 하위항목 분해, 등급구분, **조건부모듈(L1 §1.3) 결과의 가산·감산 반영규칙** |
| 제3장 정보부족처리·면책조항 | 판단유보면책, 도메인특유 면책(예: 판례미인용면책), 정보부족 가산점 체크리스트(상한선 명시 — 만점 초과 금지), **면책 남용 방지장치**(필수정보가 모두 주어졌는데도 판단을 유보하면 면책 불가, 정상감점) |
| 제4장 강제추정규칙 | "정보가 부족해서 평가가 불가능한 경우는 (거의) 없다" — 최소한의 결론 방향은 항상 부여하도록 강제(평가 회피 방지) |
| 제5장 평가출력형식 | 숫자 / N/A / [면책] 3종 표기만 허용, 자유서술 금지 |
| 제6장 특별규정 | 면책-가산점은 독립적으로 병용 가능, 면책사건은 통계(방법론 갱신 트리거)에서 분리집계, 조건부모듈 미발동(누락) 시 처리 — 강제규칙 위반 기록 + 감점 |
| 제7장 버전이력 | 본 L3가 어느 L1 버전과 함께 쓰이는지 명시(L1이 갱신되면 L3도 같이 검토) |

**제6장이 가장 중요하다.** 면책조항이 "평가를 피하는 핑계"가 되지 않도록 막는 장치(제3장
면책남용방지, 제6장 결정회피위반처리)가 없으면 L3 자체가 무력화된다.

| 위험등급 | L1 | L2 | L3 |
|---|---|---|---|

## 5. 본 표준과 SP-COMMON-02/03의 관계

혼동 방지를 위해 명확히 한다:
- **SP-COMMON-02/03**(C1~C28, M1~M7) = §0(앱생태계 비유)의 "iOS 커널" — 모든 SP가 무조건
  상속하는 공통 안전·추론 장치
- **본 표준(L1/L2/L3 템플릿)** = "iOS 위에 앱을 어떤 골격으로 짜야 하는지"를 규정하는 **상위
  메타규칙** — SP-COMMON을 상속한 위에서, L1/L2/L3 세 문서를 어떻게 나눠 쓸지를 정함

즉 SP-COMMON 준수는 "안전성"의 문제이고, 본 표준 준수는 "검증가능성"의 문제다. 둘 다
필요하다.

## 6. 준수 검증 — Tier 0 게이트 갱신

혼디 SP 생태계 운용 지침 §3.1(Tier 0 자동 구조게이트)에 다음을 추가한다:
- L1 문서에 §1.3의 4개 STEP 필수기능이 (명칭 무관하게) 전부 존재하는가
- 위험등급(§4)에 따라 L2/L3 제출이 요구되는 경우, 해당 문서가 §2/§3 구조를 따르는가
- L2가 L1과, L3가 L1·L2와 버전 매핑이 되어 있는가(메타데이터 헤더 확인)
