## 표기 규칙

각 사고실험 항목은 `[결과]`를 셋 중 하나로 표기한다.
- **PASS** — 설계대로 정확히 동작
- **PASS(경계)** — 동작은 하지만 계획이 암묵적으로만 다루던 경계 사례
- **ISSUE** — 실제 결함·미정의 지점 발견 → 계획 수정 필요

## 1. 계획 (설계안)

### 1-1. 아키텍처

```
[Hondi 프런트/SP 레이어] → [hondi-proxy Worker] → [KOSIS OpenAPI]
                              (KOSIS_API_KEY는
                               여기에만 존재)
```

- 프런트/SP는 `hondi-proxy`가 노출하는 자체 엔드포인트만 호출한다 (예: `/api/stats/{stat_id}`).
- `hondi-proxy`가 요청을 받으면 (a) KV 캐시 확인 → (b) 캐시 미스 시 KOSIS 호출 → (c) 응답을 KV에 TTL과 함께 저장 → (d) 클라이언트에 반환.

### 1-2. 데이터 구조 — `KOSIS_API_MAP`

```javascript
KOSIS_API_MAP = {
  stat_id,        // Hondi 내부 식별자, 예: 'welfare-recipient-by-region'
  org_id,         // KOSIS 기관코드
  tbl_id,         // KOSIS 통계표ID
  item_ids,       // itmId — 조회할 통계 항목 (미확인 시 null, 임의로 채우지 않음)
  update_cycle,   // 통계 갱신 주기 — 통계표설명 API로 확인한 값만 기재
  publish_lag,    // 조사기준년도 대비 실제 공표 시점 (예: "익년 7월")
  scope_hint,     // §3 트랙4 어느 항목(66-75)과 매칭되는지
  track4_status,  // 'confirmed' | 'not_found' — KOSIS에 대응표가 없으면 자체 익명화 파이프라인(④ 항목 원안) 대상으로 재분류
}
```

### 1-3. 캐싱

- 캐시 키: `kosis:{stat_id}`
- TTL: `update_cycle` 기준 (예: 1년 주기 통계는 TTL을 갱신 주기에 맞춤)
- 캐시 미스 시에만 KOSIS 호출 → 분당 호출 제한(2026-07-09 신설 공지) 대응

### 1-4. 보안

- `KOSIS_API_KEY`는 Worker Secret로만 존재, 코드/레포/`.env`에 평문 노출 금지
- 이미 스크린샷으로 1회 노출된 구 키는 폐기 후 재발급된 키로 교체 완료 상태를 전제
- 클라이언트(hondi.net)는 KOSIS 도메인이나 키를 절대 참조하지 않음

### 1-5. 트랙4 6건 대응 절차

1. KOSIS 통합검색 API(`statisticsSearch.do`)로 후보 키워드 검색
2. 결과 중 지역·집단 단위(시도별 등) 통계표만 선별
3. 대응표 있으면 `KOSIS_API_MAP`에 등록(`track4_status: 'confirmed'`)
4. 대응표 없으면 국세청 AC/복지부 AC 등 자체 익명화 API 트랙(④ 항목 원안)으로 재분류

## 2. 사고실험

### A. 캐싱·호출 제한 관련

**A1.** 인기 통계표(예: 복지 수급자수)에 사용자 요청이 동시에 100건 몰렸는데 마침 캐시가 만료된 직후다. → 100개 요청이 전부 캐시 미스로 판정되고 동시에 KOSIS를 호출한다. 분당 호출 제한에 그대로 걸릴 뿐 아니라, 정작 캐시를 갱신하려던 목적과 반대로 순간적 과호출(cache stampede)이 발생한다.
**[ISSUE-1]** — §1-3 캐싱 설계에 "캐시 미스 시 최초 1건만 실제 호출하고 나머지는 그 결과를 기다리는" 락(lock) 또는 `stale-while-revalidate`(만료 직후에도 잠시 구 데이터를 반환하며 백그라운드에서 갱신) 로직이 없다. KV는 단순 get/put이라 분산 락을 직접 못 주므로, KV에 "갱신 중" 플래그 키(`kosis:{stat_id}:lock`, 짧은 TTL)를 별도로 두는 방식이 필요.

**A2.** KOSIS 서버가 일시 장애(5xx)거나 타임아웃일 때 프록시가 그대로 에러를 클라이언트에 전달한다. → §1-1 설계에 이 경우의 처리가 명시돼 있지 않다.
**[ISSUE-2]** — 캐시가 만료됐어도 "완전 삭제"가 아니라 이전 값을 유지해 두고, KOSIS 호출 실패 시 그 구 데이터를 stale 표시와 함께 반환하는 폴백이 필요. 통계는 초 단위로 바뀌는 데이터가 아니므로 약간 오래된 값을 보여주는 게 에러를 그대로 노출하는 것보다 사용자 경험상 낫다.

**A3.** TTL을 `update_cycle`(예: "1년")로만 설정하면, 실제로는 매년 특정 시점(예: 7월)에만 갱신되는 통계인데 임의의 날짜에 캐싱을 시작해서 엉뚱한 시점에 TTL이 끝나거나, 갱신 시점이 지났는데도 구 데이터를 계속 캐싱해서 보여줄 수 있다.
**[ISSUE-3]** — §1-2에 `publish_lag`("익년 7월" 같은 표현)를 필드로 넣어둔 건 맞는 방향이지만, 이걸 실제 TTL 계산 로직에 반영하는 부분이 §1-3에 빠져 있다. 단순 "N초 후 만료"가 아니라 "다음 예상 공표일까지"로 TTL을 계산하는 함수가 필요 — 안 하면 신규 데이터가 나온 뒤에도 옛 캐시를 계속 제공하는 조용한 오류가 생긴다.

### B. 트랙4 매핑 관련

**B1.** 트랙4 6건 중 실제로 KOSIS에 대응 통계표가 없는 항목이 나올 수 있다(예: Hondi가 원하는 조합이 국가승인통계로 지정 안 된 경우). → §1-5 절차 4번대로 자체 익명화 API 트랙으로 재분류하면 되지만, 이 경우 `KOSIS_API_MAP`에는 애초에 등록되지 않고 조용히 빠진다.
**[ISSUE-4]** — "KOSIS에서 못 찾음"과 "아직 검토 안 함"을 구분할 기록이 없다. `track4_status: 'not_found'`로 남겨서, 나중에 진행상황 확인 시 "6건 중 몇 건 KOSIS 확인 완료/몇 건 자체구축 필요/몇 건 미착수"를 구분할 수 있게 명시적으로 기록해야 한다 (이미 §1-2에 필드는 반영해뒀으나, 이 값을 실제로 채우는 절차가 §1-5에 없음 — 문서 내 자체 불일치).

**B2.** 트랙4 원본 100건 목록(개별 항목명)이 레포에 없어서, 지금까지 확인한 "복지 수급자수(시도)"가 66~75번 중 정확히 몇 번인지 특정할 수 없었다.
**[ISSUE-5 — 해소됨, 2026-07-16]** — 원본 `공무원_업무지시_100건_해법_PDV완전보급_가정.md` 확보로 6건 확정:

| # | 업무 | 해법(원문) |
|---|---|---|
| 66 | 지역별 1인가구 비율 통계 | 차등 프라이버시 집계 필요 |
| 67 | 평균 가계부채 통계 | 66과 동일 방식 |
| 68 | 관내 자영업 폐업률 추이 | 국세청 사업자 통계 연동, 익명 집계 |
| 69 | 청년 실업률 지역별 분석 | 고용보험 데이터 익명 집계 |
| 71 | 관내 빈집 비율 조사 | 건축물대장·전기사용량 등 복수소스 익명결합 |
| 73 | 미세먼지 노출도-호흡기질환 상관분석 | 의료데이터 익명결합, **IRB 절차 별도 필요** |
| 75 | 지역별 평균소득 5분위 분석 | 66과 동일 방식 |

**주의**: 앞서 확인한 "국민기초생활보장 수급자수(시도)"(`DT_1YL13801E`)는 위 6건(66,67,68,69,71,73,75) 그 어디에도 해당하지 않는다 — 원본 100건 목록에는 복지 수급자수가 66-75 범위 밖의 별도 항목(11번, 조건부)으로 이미 존재한다. 지난 세션에서 "복지 수급 관련 1건"이라고 잠정 매핑했던 것은 **오매핑이었다**. `KOSIS_API_MAP`에서 이 항목을 제거하고 `financial-statement` scope 기반 개별 심사(§B 11번)용으로 재분류해야 한다.

73번은 원문에 "IRB 절차 별도 필요"라고 명시돼 있어 상관분석 자체가 복수 원자료 결합·자체 계산이 필요한 연구 성격이다 — KOSIS 단일 통계표로 해결될 가능성이 낮으므로, 검색 전부터 `track4_status: 'not_found'` 후보로 잠정 분류한다(검색은 확인 차원에서 별도 진행).

### C. 보안 관련

**C1.** `KOSIS_API_KEY`가 이미 한 번 스크린샷으로 노출됐다. §1-4는 "재발급된 키로 교체 완료를 전제"라고만 적어뒀는데, 실제로 재발급했는지 확인하는 절차가 계획에 없다.
**[ISSUE-6]** — 전제로만 남겨두지 말고, 배포 전 체크리스트에 "구 키 폐기 확인" 항목을 명시해야 한다. KOSIS는 회원당 키 1개이므로, 재발급 시 반드시 기존 키가 무효화되는지 KOSIS 공유서비스 정책을 확인해야 한다(자동 무효화가 아니라 별도 폐기 신청이 필요할 수도 있음 — 미확인).

**C2.** Worker Secret는 코드에서 `env.KOSIS_API_KEY`로만 접근하므로 로그에 실수로 찍힐 위험은 낮지만, 에러 핸들링 코드에서 `console.log(kosisUrl)`처럼 전체 요청 URL을 로깅하면 키가 그대로 로그에 남는다.
**[ISSUE-7]** — 코드 작성 시 "요청 URL 전체를 로그에 남기지 않는다"는 규칙이 계획에 명시돼 있지 않다. 에러 로깅 시 URL에서 `apiKey` 파라미터를 마스킹하는 유틸리티가 필요.

### D. 스코프·컴플라이언스 관련

**D1.** GOV24_API_MAP(트랙1/2, 58건)은 PDV 동의 기반이라 매 호출마다 사용자 동의·목적 태그가 필요하다. KOSIS_API_MAP은 이미 공표된 익명 통계라 개인 동의 절차가 필요 없는데, 이 차이가 프록시 코드 레벨에서 명확히 분리돼 있지 않으면 나중에 두 매퍼가 섞여서 KOSIS 호출에도 불필요한 동의 체크 로직이 붙거나, 반대로 GOV24 쪽에 동의 체크가 빠지는 실수가 생길 수 있다.
**[ISSUE-8]** — `hondi-proxy`의 라우팅 레벨에서 `/api/stats/*`(KOSIS, 동의 불필요)와 `/api/gov24/*`(PDV 동의 필수)를 코드 구조상 완전히 분리된 핸들러로 두고, 하나의 공통 미들웨어에서 동의 체크를 일괄 적용하는 실수를 방지해야 한다.
