# ④번 항목(정책용 익명 통계) 갱신 — 국가데이터처 KOSIS 연동

작성일: 2026-07-15
선행 문서: `PDV-4대유형-해법-현재구현vs향후설계_2026-07-14.md` §4
근거: 국가데이터처(mods.go.kr, 옛 통계청 2025-10-01 개편)·KOSIS 공유서비스(kosis.kr/openapi) 실사

## 0. 기관 정정

`PDV-4대유형-해법` §4 작성 시점엔 "④번은 각 소관 기관이 국세청 AC 사례를 템플릿으로 자체 구축해야 한다"고 정리했다. 이후 스크린샷으로 확인한 `data.go.kr`(공공데이터포털, 행안부/NIA 소관)과는 별개로, **국가데이터처**(옛 통계청, 2025-10-01 개편, mods.go.kr)가 이미 완성된 익명화 통계 파이프라인(KOSIS)을 운영 중이라는 걸 확인했다 — 이번 문서는 그 발견을 §4에 반영한다.

## 1. KOSIS 공유서비스(OpenAPI) 실제 사양

- 주소: `kosis.kr/openapi` — 회원가입 + OpenAPI 활용신청 후 `apiKey` 발급받아 사용
- 제공 서비스 7종: 통계목록·통계자료·대용량통계자료·통계설명자료·통계표설명·KOSIS통합검색·통계주요지표
- 응답 포맷: JSON·XML·SDMX·XLS, REST 방식
- 호출 URL 패턴(예): `kosis.kr/openapi/Param/statisticsParameterData.do?method=getList&apiKey={key}&orgId={기관코드}&tblId={통계표ID}&itmId=...&objL1=ALL&format=json&prdSe=Y&startPrdDe=...&endPrdDe=...`
- 최근 공지(2026-07-09): 분당 호출건수 제한 신설, HTTP(비-HTTPS) 프로토콜 제공 종료(2026-02) — Gopang 연동 시 이 제한을 감안한 캐싱·재시도 설계 필요
- **`data.go.kr`에도 "국가데이터처_KOSIS 공유서비스"라는 이름으로 동일 API가 교차 등록돼 있음** — §3 레이어 B(정부24 API 매퍼) 구축 시 data.go.kr 카탈로그 검색만으로도 발견 가능

## 2. ④번 항목 재평가

| 원안(§4, 07-14) | 갱신(이번 발견) |
|---|---|
| 각 기관이 국세청 AC 방식을 자체 구축 | **국가승인통계로 이미 존재하는 항목은 KOSIS 연동으로 대체** — 자체 구축 범위가 줄어듦 |
| "복지 수급 패턴 → 복지부·사회보장정보원 소관" | KOSIS에 사회복지 관련 국가승인통계(기초생활보장 수급자 현황 등)가 이미 있는지 확인 후, 있으면 그쪽을 우선 사용 |
| "의료·건강 통계 → 건강보험공단·질병관리청 소관" | 마찬가지로 KOSIS 교차 확인 후 우선순위 결정 |

**단, KOSIS로 완전히 대체되지는 않는다** — KOSIS는 **이미 국가승인통계로 지정·공표된 항목만** 다룬다. Hondi/Gopang이 PDV 데이터 기반으로 새로 만들고 싶은 **맞춤형 집계**(예: "Hondi 가입자 중 특정 동 단위 소득 분포" 같은, 국가승인통계에 없는 조합)는 여전히 §4 원안대로 자체 익명화 파이프라인이 필요하다. 즉 ④번은 "KOSIS로 완전 대체"가 아니라 **"국가승인통계 커버 영역은 KOSIS 우선 사용, 그 외 영역만 자체 구축"**으로 범위가 좁아지는 것이 정확하다.

## 3. Gopang 연동 설계 — 레이어 B(정부24 API 매퍼)에 KOSIS 소스 추가

기존 §3 레이어 B(`GOV24_API_MAP`)는 정부24 단일 소스를 전제로 설계돼 있었다. KOSIS는 인증 방식(apiKey, 회원제)과 조회 방식(orgId+tblId 조합, 통계표 단위 조회)이 달라 같은 테이블에 억지로 합치지 않고 **별도 소스 매퍼**로 분리하는 것을 제안한다:

```
KOSIS_API_MAP = {
  stat_id,        // Gopang 내부 식별자 (예: 'welfare-recipient-by-region')
  org_id,         // KOSIS 기관코드
  tbl_id,         // KOSIS 통계표ID
  item_ids,       // itmId — 조회할 통계 항목
  update_cycle,   // 통계 갱신 주기 (KOSIS 통계마다 다름 — 지어내지 않고 통계표설명 API로 확인)
  scope_hint,     // 어느 §3 트랙4(익명통계 6건: 66-69,71,73,75) 항목과 매칭되는지
}
```

- `org_id`/`tbl_id`는 KOSIS "통계표설명" API로 검증 없이 채우면 안 된다 — 실제 존재하는 통계표인지 KOSIS 통합검색 API로 먼저 조회해서 확인해야 한다(SCOPE-MAPPING-TRACK1-2의 "지어내지 않는다" 원칙과 동일).
- 분당 호출 제한이 있으므로, Gopang 쪽에서 통계표별로 캐싱(예: 통계 갱신 주기에 맞춘 TTL)을 두는 게 필수다 — 매 사용자 요청마다 KOSIS를 직접 호출하지 않는다.

## 4. 다음 단계 (미착수)

- [ ] §3 트랙4의 6건(66-69,71,73,75 — 지역단위 통계)을 KOSIS 통합검색 API로 실제 대응하는 통계표가 있는지 하나씩 확인
- [ ] `apiKey` 발급(회원가입) — 실제 발급은 피터님 계정으로 진행 필요, Claude가 대신할 수 없는 부분
- [ ] `KOSIS_API_MAP` 최초 항목 1~2개로 파일럿(예: 복지 수급 관련 통계표) 연동 후 캐싱 설계 확정
