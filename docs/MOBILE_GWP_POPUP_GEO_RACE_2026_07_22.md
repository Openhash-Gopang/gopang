# 모바일 실기기에서 GWP 새 탭이 안 열리는 문제 (2026-07-22)

모바일 실기기(브라우저·설치된 웹앱 모두)에서 "제주도청을 호출하겠습니다"
등 GWP(K-서비스) 호출 응답은 오는데, 실제로 새 탭이 전혀 열리지 않고
폴백 링크도 뜨지 않는 문제를 조사·수정한 기록입니다. 데스크톱 크롬의
기기 흉내(에뮬레이션) 모드에서는 재현되지 않았습니다.

## 1. 원인

`sendMessage()`(`ui/send-message.js`)는 첫 메시지 전송 시
`_initLocation()`을 호출해 GPS 권한을 요청합니다. 이 함수는
`navigator.geolocation.watchPosition()`을 호출하는데, 이 세션에서
처음 호출되는 경우 **네이티브 위치 권한 다이얼로그**를 띄웁니다.
`_locationReady`/`_locationPending`은 페이지 로드마다 초기화되고
영속화되지 않으므로, 매번 새 세션의 첫 메시지마다 이 프롬프트가
재발합니다.

문제는 이 호출이 GWP 새 탭 예약(`window.open('', '_blank')`)보다
**먼저** 실행됐다는 점입니다. 모바일 브라우저는 네이티브 권한
다이얼로그가 뜨는 순간 페이지가 시스템 UI로 포커스를 뺏긴 것으로
취급해, 그 직후 같은 클릭 핸들러 안에서 실행하는 `window.open()`을
"사용자가 직접 요청하지 않은 팝업"으로 오판하고 조용히 차단합니다.
`_gwpLaunch()`의 폴백(탭하여 연결 링크)조차 안 뜬 것도, 사용자 활성화가
이미 소진된 상태에서 재시도용 `window.open()`까지 연쇄로 실패했기
때문으로 보입니다.

데스크톱 크롬 에뮬레이션에서 재현되지 않은 이유는, 테스트 계정이
데스크톱에서는 이미 위치 권한을 허용해둔 상태라 재프롬프트 자체가
없었기 때문으로 추정됩니다.

## 2. 수정

`send-message.js`에서 GPS 요청을 `setTimeout(_initLocation, 0)`으로
다음 매크로태스크로 미뤄, 같은 클릭 핸들러 안의 `window.open()`
호출들이 권한 다이얼로그보다 먼저 실행되도록 했습니다.
`_initLocation()`은 원래도 `await` 없는 fire-and-forget이라 이
지연 자체는 GPS 기능에 영향이 없습니다.

## 3. 남는 교훈

모바일에서 "사용자 제스처 컨텍스트 안에서 실행돼야 하는 API"
(`window.open`, `navigator.share`, 결제/생체인증 트리거 등)를 여러 개
같은 핸들러 안에서 호출할 때는, 그 사이에 **네이티브 권한 프롬프트를
띄우는 API**(geolocation, notification 등)가 끼어들지 않도록 순서를
의식적으로 관리해야 합니다. 데스크톱 에뮬레이션만으로는 이런 클래스의
버그가 재현되지 않으므로, 모바일 관련 리포트는 실기기 재현 여부를
우선 확인하는 편이 진단 속도가 빠릅니다.
