## 사고실험 (KSIC 문서와 동일 관례 — 표 구성 중 드러난 것)

1. **24/27을 "중분류 통째로 Tier1"에 넣은 게 맞는 결정인지는 의문이
   남는다.** KSIC은 업종(사업자) 단위라 자격규제가 업종 전체에 걸리는
   경우가 드물지만, KSCO는 같은 중분류 안에 완전히 다른 규제수준의
   세분류가 섞여 있는 경우(24=간호사~의사, 27=변호사~판사)가 구조적으로
   흔하다. 이건 KSIC 방법론을 그대로 가져다 쓰면서 발견한 차이점이다 —
   **실제 작성 단계에서 세분류 단위 재분류가 추가로 필요할 가능성이
   높다.**
2. **AGENT-SUPPLIER(KSIC, 사업자)와의 수요 중복을 Tier2에서 처음
   고려했다** — 예를 들어 농부 개인의 KSCO 직업 특화(L1)와 그 농부가
   운영하는 사업자의 KSIC 특화(AGENT-SUPPLIER)가 실질적으로 겹치는
   정보를 담을 수 있다. 이 경우 굳이 둘 다 만들 필요가 있는지, 아니면
   `job_ksco`가 있고 그 사람이 동시에 `entity_type='business'`(같은
   업종)라면 KSIC AGENT-SUPPLIER 하나로 충분한지는 이번 표에서
   결론짓지 않았다 — 실제 작성 우선순위를 정할 때 다시 볼 문제다.

### 서버 게이트 반영

위 Tier3 코드를 `worker.js`에 `TIER3_REGULATED_KSCO_CODES`로 신설해
`industry_fields.schema_id`의 `TIER3_REGULATED_SCHEMA_IDS`와 동일한
원칙(review_due 이전엔 `job_ksco.visibility`를 강제로 `private`) 적용.
Tier2 두 항목(법무사·집행관, 정부 행정 전문가)은 이번엔 게이트에
넣지 않고 이 표에 재검토 대상으로만 남긴다 — Tier1/Tier3 이분법으로
당장 서버 로직을 만들되, Tier2는 사람이 다시 볼 때까지 Tier1과 동일
(비공개 강제 없음)하게 둔다.
