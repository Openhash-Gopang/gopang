# EXPERT 페르소나 실행성 사고실험 종합 보고서 (2026-07-19)

**범위**: `EXPERT_REGISTRY`의 60개 전문가 페르소나가 "안내"가 아니라 "지시 이행"을
실제로 수행하는지에 대한 감사·시나리오 검증·수정 전체 이력.
**최종 커밋**: `7a23a0b` (origin/main과 동기 확인, 테스트 11/11 통과)
**출발점**: 법무사 페르소나 10개 시나리오 사고실험 요청 → 감정평가사 시나리오에서
사용자 지적("페르소나가 안내에만 치중하고 실행을 안 한다") → 코드 레벨 근본원인
추적으로 확대.

---

## 1. 방법론

1. **시나리오 기반 트레이스**: 가상의 사용자 발화를 실제 SP 원문(STEP 0~D)과
   상위 상속 계층(UNIVERSAL-INTEGRITY/UNIVERSAL-common/PROFESSIONAL-common/
   SP_common_guardrails)을 그대로 따라가며 어떻게 반응할지 단계별로 추적.
2. **코드 대조 검증**: SP 문서가 "이렇게 한다"고 서술한 태그·메커니즘이 실제
   클라이언트 코드(`pages/expert-chat.html`, `src/gopang/ai/*.js`)에 처리
   로직이 있는지 `grep`으로 직접 확인 — 문서만 읽고 판단하지 않음.
3. **실제 조립 프롬프트 재현**: `manifest-loader.js`/`expert-session.js`의
   조립 로직을 Node 스크립트로 그대로 재현해, 실제로 모델에 전달되는 시스템
   프롬프트 원문을 로컬에서 재구성하고 검사.
4. **회귀 테스트**: 각 코드 수정마다 `expert-session-switch.test.mjs`에 케이스
   추가, 매 수정 후 전체 스위트 재실행.

## 2. 발견 사항 (연대기 순)

### 발견 1 — AC→페르소나 핸드오프 시 대화 맥락 소실
[EXPERT:] 태그 발동 시 "이번 발화" 원문 한 줄만 전달되고, AC와 이미 여러 턴
확인된 맥락(당사자·경위·이미 진행된 절차)이 소실돼 사용자가 같은 내용을
반복 진술해야 했다. → `handleExpertTag`에 `summarizeHandoffContext6W` 신설로 해결.

### 발견 2 — 감정평가사 시나리오에서 "안내에 머문다"는 지적
"주소·평수·층수를 묻고, 검색하고, 추정치를 내야 한다"는 지적에, 처음엔 웹검색
도구 자체가 없다고 오판(검색어를 이 리포 명명 관례에 안 맞춰 찾음). 재검색
결과 `[WEB_SEARCH: query=...]`가 이미 존재했으나 `pages/expert-chat.html`
(페르소나 실행 환경)엔 연결돼 있지 않았음을 확인.

### 발견 3 — SP_lawyer STEP R도 phantom
지난 감사에서 "정상 커버됨"이라고 잘못 평가했던 STEP R(K-Search 위임·기관
자동조회)이 실제로는 `expert-chat.html`에 `[KSEARCH_HANDOFF]` 처리 코드가
없어 전혀 작동하지 않는 상태였음. → `KSEARCH_HANDOFF`를 실제 작동하는
`WEB_SEARCH`로 교체.

### 발견 4 — UNIVERSAL-common/PROFESSIONAL-common 완전 미로드 + 3중복 (★ 최상위 근본원인)
"모든 페르소나가 상속받는 상위 SP는?" 질문에 문서만 읽고 잘못 답변했던 것이
실사로 드러남: `UNIVERSAL-common`(U0 의도특정·U1 권한의 한계·U7 업무처리
파이프라인 — "안내로 끝내지 않는다"는 원칙의 실제 본문)과 `PROFESSIONAL-common`
(전문가 정체성 계층)이 **60개 EXPERT 페르소나 전원의 실제 조립 프롬프트에
한 번도 로드된 적이 없었다.** 동시에 `UNIVERSAL-INTEGRITY`가 조립 함수의
반복 호출로 3중복 삽입되는 버그도 발견. → `_composeExpertPrompt` 재구성,
길이 84,620자 → 64,280자(24% 감소), 전 계층 정상 포함 확인.

### 발견 5 — CI가 그 수정을 자동으로 삭제
`tools/build_manifest.py`가 `SP_` 접두사 파일만 스캔해, 발견 4에서 추가한
두 키가 다음 push 때 조용히 사라짐(commit `f111f85`). → 스캔 규칙 신설로
근본 수정.

### 발견 6 — WEB_SEARCH는 독립 도구가 아니라 K-Search RULE-07의 하위 도구
`[WEB_SEARCH: query=...]`는 AC 범용 기능이 아니라 K-Search(SP-18) 자신이
RULE-07 [7-B]에서 신설한 태그(v1.3, 2026-07-11)였다. EXPERT 페르소나에
직결시킨 것은 K-Search 본체의 안전장치(RULE-01: 전수조사·감시추적·민감정보·
미성년자·전화번호역조회 금지)를 우회하는 지름길이었다. → `SP_common_guardrails`
C43-1+ 신설로 안전장치 상속 의무 명문화, `expert-chat.html` 재주입 텍스트도
K-Search RULE-07 [7-A] 형식으로 정정.

### 발견 7 — AC 핸드오프 요약이 위험 신호를 희석시킬 수 있음
`summarizeHandoffContext6W`의 "80자 요약" 압축 과정에서 감시·추적 등을
암시하는 표현이 "이사 계획 있음, 주변 정보 관심"처럼 밋밋하게 순화될 위험을
사고실험으로 발견. → `risk_signals` 필드 신설, 위험 신호는 요약하지 않고
원문 그대로 별도 블록(`⚠️`)으로 분리 보존.

### 발견 8 — C41(오케스트레이션 하위 호출) 미반영 현황 재확인
지난 감사에서 "lawyer 외 26개 미반영"이라던 낡은 문서 메모를 근거 없이
인용했던 걸 실제 grep 실사로 정정: **42/60 반영, 18/60 미반영**(의료 핵심
14개 + 상담 4개 — social-worker 포함). 이번 세션에서도 해결하지 못하고
남겨둠(§4 참조).
