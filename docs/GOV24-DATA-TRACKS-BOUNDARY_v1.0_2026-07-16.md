# 정부24 데이터 2계열 — 경계 정의 (2026-07-16)

> 같은 세션에서 "혜택 카탈로그(10,966건)"와 "발급형 민원(배치1·2, 사용자 수기 제공)"을
> 같은 것으로 착각해 재조준을 제안했다가 정정한 사고가 있었다. 앞으로 같은 혼동이
> 반복되지 않도록 경계만 짧게 못박아둔다. 상세 설계는 각 계열 작업자가 별도 문서로.

## 계열 A — 공공서비스 혜택 카탈로그

- 원본: `tools/civil-petitions-raw.json` (odcloud.kr `gov24/v3/serviceList`, 10,966건, 전량 `status:'draft'`)
- 성격: 지원금·감면 등 자격요건 기반 복지 서비스. 59.5%가 시군구 단위(같은 이름 사업이
  지자체마다 별도 운영 — 중복 아님).
- 하류 목적지: `org_profiles` / `atom_rows` / `procedure_maps` (K-Intent→K-Compose→K-Deliver
  오케스트레이션, `AGENT-COMMON §0-H`가 이미 라이브로 연결).
- 담당: 별도 작업자 진행.

## 계열 B — 발급형 민원(증명서 발급·신청·신고·조회)

- 원본: 사용자가 세션마다 수기로 붙여넣는 정부24류 목록(현재 2배치, 100건). **이 계열은
  계열 A와 달리 저장소 내 공식 API 수집 스크립트가 아직 없다** — `tools/fetch_civil_petitions.py`와
  동일한 역할을 하는 스크립트가 이 계열에는 존재하지 않음을 확인함(2026-07-16).
- 성격: 단일 기관(대부분 중앙행정기관) 창구에서의 사본 발급·접수. 자격요건 게이트보다
  절차·구비서류 안내가 핵심.
- 하류 목적지: `kgov`(`SP-10_kpublic`) / `REQUIRED_DOCUMENTS_REGISTRY` / `gov_task_schema_drafts`.
- 담당: 이 스레드(Claude, kgov 트랙) 계속 진행.
- 로컬 분류 도구: `classify_batch.py`(R1~R7 이상탐지) — 아직 저장소에 커밋되지 않은 상태.

## 공유 원칙 (양쪽 다 적용)

- U0(의도 특정 후 해법 제시)·U2-1(생소함≠오류, 웹검색 우선 검증) — `UNIVERSAL-common` 상속이므로
  두 계열 모두 자동 적용.
- 자동화 수준(얼마나 자동 처리 가능한가) 저장 위치는 **`org_profiles` 하나로 통일** —
  계열 B가 별도 레지스트리를 새로 만들지 않는다(2026-07-16 `RESOLUTION_TIER_REGISTRY` 폐기 참조).
