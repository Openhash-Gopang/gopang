# BENEFIT_CANDIDATE_SEARCH 100건 사고실험 (STEP 0-DISCOVER 검증)

> 작성일: 2026-07-16 | 근거: worker.js `handleBenefitCandidateSearch`의
> 실제 쿼리 로직(SQLite `~` LIKE, 대소문자 무시 부분일치)을 파이썬으로
> 재현해, 실제 시딩된 10,310건(656건 재시도분 제외) 위에서 100개 시나리오를
> 직접 실행. 가정이 아니라 **실행 결과**다.
>
> **한계 고지**: PocketBase에 직접 붙어서 돌린 게 아니라 로컬 시뮬레이터로
> LIKE 연산자를 재현한 것이다. 쿼리 매칭 로직(문자열 포함 여부)은
> worker.js와 동일하게 재현했지만, 실제 PocketBase 성능·정렬(`sort=-created`)
> 등은 재현 대상이 아니다 — 이번 사고실험의 목적은 "매칭 결과가 의도대로
> 나오는가"이지 인프라 성능 검증이 아니다.

## 0. 결론 먼저

**의도대로 동작하지 않는다.** 12개 클러스터 중 3개(A·H·K)에서 설계의
핵심 전제가 실제 데이터 규모에서 무너지는 걸 확인했다. 가장 심각한 건
**STEP 0-C의 "속성 하나씩 좁혀나가기"가 구조적으로 작동 불가능**하다는
점이다 — 이건 부차적 버그가 아니라 이번 기능 전체의 존재 이유였다.

## 2. 치명적 결함 — 이번 기능의 존재 이유가 무너짐

### 2-1. STEP 0-C "속성 하나씩 좁히기"가 실제로는 작동하지 않는다 (클러스터 K)

STEP 0-C의 설계는 "후보가 많으면 속성 하나를 확인해 재검색"이다. 그런데
재검색이 실제로 어떻게 구현되는지 보면 — `q`는 **단일 문자열**이고
LIKE는 **그 문자열 전체가 통째로 들어있는지**만 본다. "청년"과 "30세"를
따로따로 아는 상태에서 "청년 30세"로 재검색하면, `goal+eligibility_gate`
텍스트 어디에도 "청년 30세"라는 연속된 문자열이 존재할 리 없다(각
조건이 `conditions[]` 배열의 서로 다른 항목에 따로 적혀 있으므로).

실측 결과:
```
"청년 30세"          -> 0건  (기대: 만 18~34세 청년 다수)
"제주 농업"          -> 0건  (기대: 제주 지역 농업 지원 다수)
"다자녀 저소득 주거"   -> 0건
"청년 창업 제주"      -> 0건
"저소득 한부모"       -> 30/71건 (우연히 원문에 붙어서 쓰인 경우만 성공)
```

5건 중 4건이 실패했고, 유일하게 성공한 1건도 "설계가 의도한 AND
조건 결합"이 아니라 "원문에 우연히 그 두 단어가 붙어 등장했을 뿐"이다
— 즉 **재현 가능한 성공이 아니라 우연**이다. STEP 0-C는 여러 회
반복하며 후보를 좁혀가는 걸 전제하는데, 그 반복의 매 단계마다 쓰는
재검색 메커니즘 자체가 신뢰할 수 없다.

### 2-2. 흔한 단어 하나가 카탈로그의 2/3를 삼킨다 (클러스터 H)

```
q="지원" -> 7,585건 매칭 (전체 10,310건의 73.6%)
q="이"   -> 6,814건 매칭 (66.1%) — 단순 조사/어미
```

"이"는 한국어에서 조사·어미로 거의 모든 문장에 등장한다. STEP
0-DISCOVER가 goal에서 "핵심어 1~3개"를 뽑으라고만 지시했지, 이렇게
흔한 단어를 걸러내라는 지침이 없다 — 모델이 이용자 발화에서 부주의하게
조사를 핵심어로 뽑으면 사실상 "전체 카탈로그 무작위 30건"을 반환하는
것과 같아진다.

### 2-3. K-Intent의 자유텍스트 domain과 procedure_maps의 고정 10종이 안 맞는다 (클러스터 I)

SP-19_kintent는 domain을 "참고용 분류(자유 텍스트)"라고 명시한다.
그런데 procedure_maps.domain은 gov24 원본의 고정된 10종
문자열(`"행정·안전"`처럼 가운뎃점까지 정확히 일치해야 함)이다.

```
domain="행정 안전"(가운뎃점 없이) -> 0건
domain="보건의료"(가운뎃점 없이)   -> 0건
domain="취업"                    -> 0건
domain="창업지원"                 -> 0건
```

**더 심각한 건**: worker.js 코드에서 `q`와 `domain` 절이 `&&`로
결합된다(`clauses.join(' && ')`). 즉 K-Compose가 정확한 `q`를 줬어도
domain 표기가 조금만 어긋나면 **매칭 가능했던 q 결과까지 전부
0으로 사라진다** — domain 파라미터가 필터가 아니라 저격수 역할을
하는 셈이다.

## 3. 심각한 결함 — 규모 문제

### 3-1. limit(30/50)이 "narrow" 알고리즘의 전제를 깬다 (클러스터 A·C·G)

STEP 0-C는 "후보 리스트를 전부 읽고 배제·비교"를 전제로 설계됐다.
그런데 흔한 도메인 질의는 매칭이 수백~수천 건이고, 서버는 그중 30건만
(실제로는 `sort=-created`, 즉 최신순 임의 순서로) 잘라서 준다. 즉
K-Compose가 "전체 후보군"이라고 믿고 배제·비교하는 건 실제로는
**전체의 1~4%에 불과한 임의 부분집합**이다. 최근에 시딩된 30건만
계속 보이고, 나머지 수백~수천 건은 이용자에게 존재조차 알려지지
않는다.

```
q="생활안정" -> 2,117건 매칭, 30건만 반환(1.4%)
q="의료"    -> 1,578건 매칭, 30건만 반환(1.9%)
q="장애인"   -> 1,275건 매칭, 30건만 반환(2.4%)
```

### 3-2. 656건 재시도분 제외로 인한 실제 데이터 공백 (클러스터 B)

정확한 사업명(B클러스터)조차 10건 중 3건이 0매칭이었는데, 확인해보니
전부 지난번 처리에서 **명시적으로 제외한 656건(max_tokens 부족으로
재시도 대기 중)**에 속한 레코드였다(`174000000013`·`479000000106`·
`611000019619`). 이건 시스템 결함이 아니라 예상된 공백이지만, 656건이
merge되기 전까지는 이런 "실존하지만 아직 카탈로그에 없는" 사업이
계속 miss로 잡힌다는 걸 재확인했다.

## 4. 중간 결함 — 텍스트 매칭의 근본적 한계

### 4-1. 동의어·완곡어 (클러스터 E, 5/10 실패)

"노인" 대신 "어르신"(224건, 우연히 성공), "MZ세대"(0건, 실패),
"농사꾼"(0건), "장사하는 사람"(0건), "몸이 불편한 분"(0건) — 성공
여부가 순전히 **LLM 추출 단계에서 그 표현을 우연히 썼는지**에
달려 있다. 체계적 동의어 사전이 없다.

### 4-2. 실존 사업명조차 띄어쓰기 하나로 깨진다

시나리오 #20 "친환경농산물 인증비용"(붙여쓰기)이 0건이 나와서 원본을
직접 대조해봤다 — 실제 사업명은 `"친환경 농산물 인증비용 지원"`
(**친환경**과 **농산물** 사이에 띄어쓰기가 있음)이었다. 이용자가
띄어쓰기를 하나 빠뜨리기만 해도, **실제로 카탈로그에 존재하는
사업조차 못 찾는다.** 한국어 띄어쓰기 관용은 사람마다 다른데, LIKE
완전일치는 이걸 전혀 흡수 못 한다.

### 4-3. 오타는 예외 없이 전멸 (클러스터 F, 5/5 실패)

"첫년"(청년 오타) 등 5건 전부 0건 — 예상된 결과이지만, 실사용
환경에서 모바일 타이핑 오타는 드물지 않다는 걸 감안하면 실사용
체감 실패율에 상당히 기여할 걸로 보인다.

## 5. 경미한 결함 — 파라미터 검증 공백

- `limit=0` — JS의 `parseInt('0',10) || 30` 패턴에서 `0`은 falsy라
  암묵적으로 30으로 치환된다. 의도한 동작인지 우연인지 코드만 봐선
  불분명하고, 최소한 주석이 없다.
- `limit=-5` — `Math.min(-5, 50)`은 `-5`를 그대로 통과시킨다.
  PocketBase에 `perPage=-5`가 그대로 전달되는데, 이때 PocketBase가
  어떻게 반응하는지 이번 사고실험에서는 검증하지 못했다(로컬
  시뮬레이터는 이 분기를 실제로 재현하지 않음) — **미검증 위험으로
  남겨둔다.**

## 6. 잘 작동한 것 (정직하게 함께 기록)

- **할루시네이션 없음** — 클러스터 D(존재하지 않는 혜택 10건)가 전부
  정확히 0건으로 나왔다. 서버가 없는 걸 있다고 지어내지 않는다.
- **domain 단독 조회는 견고함** — 클러스터 L 5건 전부 의도대로
  작동(빈 쿼리 400 에러 포함).
- **limit 상한(50) 강제는 정상 작동** — `limit=100` 요청이 정확히
  50건으로 잘렸다.
- **일부 동의어는 우연히 잡힘** — LLM 추출이 원문을 기계적으로 베끼지
  않고 자연스러운 한국어로 풀어 쓴 덕에("어르신"·"홀몸노인"·"새터민"
  등), 순수 사업명 매칭보다는 검색 성공률이 더 높다. 완벽하진 않지만
  전혀 없는 것보단 낫다.

## 7. 종합 판정과 권고

**"의도대로 동작하는가?"에 대한 답: 부분적으로만.** domain 단독 조회나
정확한 사업명 검색(존재하는 레코드에 한해)은 쓸 만하다. 하지만 이
기능의 핵심 가치 제안이었던 "발견형 의도 처리"(STEP 0-DISCOVER
전체)는 후보가 조금만 많아져도(4건 이상) 실제로는 설계대로 작동하지
않는다 — 임의의 30건 부분집합 위에서 우연에 기댄 매칭을 반복할 뿐이다.

**권고 우선순위:**

1. **(치명적, 최우선)** STEP 0-C의 재검색 메커니즘을 "단일 문자열
   재검색"에서 "다중 조건 AND 필터"로 바꿔야 한다 — `q` 하나가 아니라
   `q[]` 배열을 받아 각각 개별 LIKE 절로 만들고 `&&`로 묶는 식.
   지금 상태로는 STEP 0-C의 반복 좁히기가 이론상으로만 존재한다.
2. **(치명적)** 흔한 단어(조사·"지원"·"신청"·"확인" 등) 불용어 목록을
   만들어 `q`에서 걸러내거나, 최소 2글자 이상 의미어만 허용.
3. **(심각)** domain 파라미터를 K-Intent의 자유텍스트 그대로 쓰지
   말고, 서버 또는 K-Compose 단에서 10종 고정값으로 정규화하는
   매핑 레이어를 추가. 매핑 실패 시 domain 절을 아예 빼고 q만으로
   검색(현재처럼 AND로 묶어 전체를 죽이지 않게).
4. **(심각)** limit 30을 "전체 후보"로 오인하지 않도록, 응답에
   `total_match_estimate`(전체 매칭 건수, count만) 필드를 추가해
   K-Compose가 "30건 중 30건"과 "2,117건 중 30건"을 구분할 수 있게
   해야 한다 — 지금 응답에는 이 정보 자체가 없다.
5. **(중간)** 띄어쓰기 정규화(공백 제거 후 비교) 최소한이라도 추가 —
   "친환경농산물"과 "친환경 농산물"을 같은 걸로 취급.
6. **(경미)** `limit` 파라미터 0/음수 입력에 대한 명시적 검증(400
   에러 또는 기본값 강제) 추가, 현재는 falsy 함정에 암묵적으로 의존.
