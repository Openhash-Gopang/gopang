# 공무원 직책 코드(role) 표준화 설계 (v1.0)

작성일: 2026-07-15
배경: `access_cert.role`이 검증되지 않는 자유 텍스트로 확인됨 — 사고실험 재확인,
`EMERGENCY_ELIGIBLE_ROLES` 즉시 비활성화 조치와 함께 진행

---

## 1. 결론부터 — 새 코드 체계를 만들지 않는다

대한민국은 이미 이 문제를 위한 국가 표준을 3중으로 갖추고 있다. 새로 발명하는 대신 기존 것을 채택하는 게 맞다.

| 계층 | 실제 시스템 | 하는 일 |
|---|---|---|
| 인증서 발급/검증 | **GPKI**(gpki.go.kr, 행정전자서명 인증관리센터) | 공무원 개인·기관용 전자서명 인증서 발급. 정부24 등 1만여 개 행정업무가 이미 이 인증서로 로그인·서명 |
| 직원 정보(직급·직위) 디렉터리 | **정부디렉터리시스템**(dir.go.kr) | 각 기관이 인증서 발급 전 반드시 여기 직원정보(이름·직급·직위·소속·인증서 정보 등 38개 항목)를 등록해야 함. 표준 LDAP API로 조회 가능 |
| 코드값 표준 | **행정표준코드관리시스템**(code.go.kr, 행안부) | 직위코드(629건), 직급코드, 직렬·직류 코드를 이미 표준화해 공시 — 공공데이터포털에서 파일로도 받을 수 있음 |

즉 "누가 어떤 직책인가"를 검증 가능한 코드로 관리하는 인프라가 **이미 전국 단위로 존재**한다. Hondi의 `access_cert`(Ed25519 자체서명 + TOFU)는 이 GPKI 체계와 암호학적으로 호환되지 않는 별도 발명품이지만, **"직책을 나타내는 코드값이 무엇이어야 하는가"** 부분만큼은 이 기존 표준을 그대로 가져다 쓸 수 있고, 그래야 한다 — 나중에 실제 GPKI/정부디렉터리와 연계할 여지도 남는다.

## 2. 단계적 채택 계획

### 즉시 (이번 패치, §2 코드 반영)
`_verifyAccessCert`에 role 화이트리스트 검증을 추가하되, 지금 실제로 쓰이고 있는 값만 우선 인정한다:
- `staff`, `manager` — worker.js 9374행 affiliation 필드에서 이미 쓰이던 두 값, dept-task-handler.js 1260행에서 이미 `'manager'` 문자열로 실사용 중
- 이 두 값 외의 role은 전부 거부(코드가 없으므로 EMERGENCY_ELIGIBLE_ROLES처럼 무의미한 대조를 만들지 않음)

이 조치만으로 "기관장이 아무 문자열이나 서명해 넣을 수 있다"는 구멍은 완전히 막힌다.

### 후속 (별도 데이터 작업, 이번 세션 범위 밖)
1. 공공데이터포털에서 **행정안전부_행정표준코드_직위코드** 파일(629건)을 내려받아 `NATIONAL_ROLE_REGISTRY` 형태로 저장소에 반입
2. 92번 감사·EMERGENCY_ELIGIBLE_ROLES처럼 도메인 특화 권한이 필요한 지점은, 이 629건 중 실제로 해당하는 코드값(예: 사회복지직렬 관련 직위코드)을 찾아 매핑 — `welfare_officer` 같은 임의 영문 식별자 대신 **행안부가 공시한 실제 코드값**을 그대로 쓴다
3. 장기적으로는 `_verifyAccessCert`의 서명 검증 자체를 GPKI 인증서 체인 검증으로 대체하는 안을 검토할 수 있다 — 다만 이건 Ed25519/TOFU 기반의 현재 access_cert 설계를 근본적으로 바꾸는 큰 작업이라 별도 세션에서 다뤄야 한다

## 3. 이번 설계가 EMERGENCY_ELIGIBLE_ROLES 재활성화에 주는 의미

`EMERGENCY_ELIGIBLE_ROLES`를 다시 채우려면, §2 후속 작업에서 실제로 반입된 행안부 직위코드 중 "복지direct·아동보호·경찰연락 등 긴급조치 자격이 있는 직위"를 코드 담당 부서와 협의해 골라야 한다 — 지금처럼 개발자가 임의로 영문 식별자를 지어내는 방식으로는 다시 채우지 않는다. 이 협의 자체가 별도 과제이므로, 지금은 §2의 최소 검증(staff/manager)까지만 반영하고 EMERGENCY_ELIGIBLE_ROLES는 계속 빈 Set으로 둔다.

## 4. 미결 사항

- 정부디렉터리시스템(dir.go.kr) LDAP API 실제 연동 가능 여부(기관 대상 서비스라 Hondi 같은 민간/시민 플랫폼이 접근권을 받을 수 있는지) — 별도 확인 필요
- 행안부 직위코드 629건 파일의 실제 구조(코드-명칭 매핑 방식)는 아직 확인 못함 — 공공데이터포털에서 직접 내려받아야 함
