## 1. 임베딩 인프라 — Cloudflare Workers AI (bge-m3) + Vectorize

"DeepSeek 계열 오픈소스 임베딩(BGE 등) 자체 호스팅"을 검토한 결과,
**별도 GPU 서버를 새로 세우지 않고 이미 이 프로젝트가 쓰는 Cloudflare
Workers 인프라 안에서 해결 가능**하다는 걸 확인했다:

- **임베딩 생성**: `@cf/baai/bge-m3` — Cloudflare Workers AI가 이미
  호스팅하는 BAAI의 BGE-M3 모델. 100개 이상 언어(한국어 포함) 지원,
  8,192 토큰 컨텍스트, dense/sparse/multi-vector 검색을 동시 지원.
  `env.AI.run('@cf/baai/bge-m3', {text: [...]})`로 호출한다 — API 키·
  별도 계정 불필요, 기존 Cloudflare 계정 안에서 바로 쓸 수 있다("자체
  호스팅"의 정신에 부합 — 외부 벤더 임베딩 API 요금제에 종속되지
  않는다).
- **벡터 저장·검색**: **Cloudflare Vectorize** — Workers AI 임베딩과
  짝을 이루도록 설계된 네이티브 벡터DB. `env.VECTORIZE.upsert(...)`/
  `.query(...)`로 유사도 검색을 한다. PocketBase(SQLite)는 벡터 검색을
  지원하지 않으므로, `procedure_maps`는 지금처럼 원본 데이터(진실의
  원천)로 PocketBase에 두고, Vectorize에는 `{id: petition_id, vector:
  embedding, metadata: {goal, domain}}`만 별도 인덱스로 둔다 — 두
  저장소가 동기화돼야 하므로 procedure_maps 갱신 시 Vectorize도 같이
  갱신하는 훅이 필요하다(§3-2 참조).

**wrangler.toml 변경 필요**:
```toml
[ai]
binding = "AI"

[[vectorize]]
binding = "VECTORIZE"
index_name = "hondi-benefit-catalog"
```

**인덱싱 파이프라인 개요**: 10,310건(추후 656건 merge 후 10,966건)의
`goal + eligibility_gate 요약`을 bge-m3로 임베딩해 Vectorize에 일괄
upsert. 배치당 몇 건까지 되는지, 임베딩 1회 생성 비용(무료 티어
한도 등)은 실제 배포 시점에 Cloudflare 문서로 재확인이 필요하다 —
이번 세션에서 가격표까지 확정하진 않았다.

## 2. 5단계 파이프라인 재설계

### 2-1. 현재(3단계) vs 제안(5단계)

```
[현재]  K-Intent(목표 확정) → K-Compose(캐시 조회+구성+실행까지 전부) → K-Deliver(결과 조립+보고)
[제안]  K-Intent(목표 확정) → K-Compose(캐시 조회+구성만) → K-Execute(실제 수행) → K-Deliver(결과 조립+전달) → K-Report(이해당사자 통지)
```

현재 K-Compose(SP-20)는 "어느 절차를 쓸지 결정"과 "그 절차를 실제로
실행"을 한 SP 안에서 다 한다(RULE-02 STEP4가 atom 실행까지 포함).
이걸 분리하는 이유: **계획(planning)과 실행(execution)을 같은
에이전트가 하면, "이 절차가 맞는지" 판단과 "지금 이 절차를 실행해도
되는지"(권한·전제조건 재확인) 판단이 섞여서 각각의 책임 경계가
흐려진다.** 분리하면 K-Compose는 순수하게 "무엇을, 어느 순서로"만
확정하고, K-Execute가 "지금 이걸 실행한다"는 행위 자체에만 집중한다
— 이건 이미 이 프로젝트의 다른 곳에 있는 원칙(§1-0 "국세청 등
정책관할과 실무처리기관을 구분", HUMAN-AUTHORITY-GATE의 "판단과 행사
분리")과 같은 정신이다.

### 2-2. 단계별 역할 재정의

**K-Intent (SP-19, 변경 없음)** — "무엇을" 확정. 그대로 둔다.

**K-Compose (SP-20, 축소)** — procedure_maps 캐시 조회 또는 신규
구성까지만 한다. 지금 RULE-02 STEP4(atom 실행)를 들어내 K-Execute로
옮긴다. K-Compose의 산출물은 "실행 계획"(steps 배열, 각 step의
atom_id·org_id·순서·병렬여부)이지 실행 결과가 아니다.

**K-Execute (신설, SP-22)** — LLM이 AI Agent로서 계획을 실제로
수행한다. K-Compose가 넘긴 계획의 각 step을 순서대로(또는 병렬로)
실제 실행한다:
  - `atom_rows`가 가리키는 실제 기관 API·양식 제출·조회를 tool call로
    수행
  - 매 step 실행 전 HUMAN-AUTHORITY-GATE 원칙(본인인증·최종제출은
    인간 전속)을 재확인 — K-Execute는 그 경계 앞에서 멈추고 이용자
    행동을 요청한다(대신 완료했다고 말하지 않는다, 기존 SP-21
    금지-2와 동일 원칙을 여기로 옮긴다)
  - 실행 중 실패·거부·추가정보 필요 등은 ORCHESTRATION_SUBTASK_RESULT
    로 즉시 K-Compose 또는 AC에 보고(기존 §0-H 패턴 재사용)
  - **"예외없이 LLM이 수행"의 의미**: atom 하나 실행하는 것 자체가
    항상 LLM 판단을 거친다는 뜻이지, 모든 게 새로 추론된다는 뜻은
    아니다. 예를 들어 `_l1FindProcedureMap`처럼 순수 DB 조회는
    여전히 스크립트가 한다(이건 "판단"이 아니라 "검색"이다) — 다만
    "이 조회 결과를 가지고 지금 무엇을 할지"는 항상 K-Execute(LLM)가
    결정한다. §1-0 원칙("배정된 업무 내에서 판단은 LLM, 배관은 코드")
    과 일치시킨다.

**K-Deliver (SP-21, 축소)** — K-Execute의 실행 결과를 받아 이용자에게
전달하는 것에 집중한다. 지금 갖고 있던 "procedure_maps 갱신"은 K-Execute
결과가 확정된 뒤 그대로 유지(K-Deliver가 캐시를 갱신하는 역할은
변경 없음).

**K-Report (신설, SP-23)** — 여기가 가장 설계 난이도가 높다. 아래
3절에서 따로 다룬다.

### 2-3. 태그 체인 (기존 패턴 확장)

```
[CALL_KINTENT] → [HANDOFF_TO_KCOMPOSE] → [HANDOFF_TO_KEXECUTE](신설)
→ [HANDOFF_TO_KDELIVER] → [HANDOFF_TO_KREPORT](신설) → [ORCHESTRATION_COMPLETE]
```

`call-ai.js`에 `HANDOFF_TO_KEXECUTE`·`HANDOFF_TO_KREPORT` strip 처리와
system 프롬프트 전환 로직을 기존 `HANDOFF_TO_KCOMPOSE`/`HANDOFF_TO_
KDELIVER` 패턴 그대로 복제하면 된다 — 새 배관 자체는 기계적인 작업.

## 3. K-Report — 가장 현실적 검증이 필요한 부분

**요청하신 내용**: "수행 결과를 해당 사건의 모든 이해 당사자의
매체에 기록"(예: 거래 시 국세청·통계청·지자체).

**정직하게 짚어야 할 현실적 한계**: 혼디가 국세청·통계청 시스템에
직접 쓸 수 있는 API 접근권한은 없다(이건 이 프로젝트가 이미 확립한
원칙과도 일치한다 — SP-10_kpublic·jeju-national-agency-catalog가
"국가기관은 지자체 지휘를 안 받는다", "실제 접수 창구는 각 기관
자체 시스템"이라고 여러 차례 명시했다). 따라서 K-Report가 "국세청에
직접 기록"한다는 표현은 문자 그대로는 실현 불가능하다. 대신 아래
셋 중 하나로 현실화해야 한다:

1. **법정 신고 의무가 이미 있는 경우** — 예를 들어 사업자가 어떤
   거래를 했다면 그 사업자 본인이 어차피 부가세 신고·소득 신고를
   해야 한다. 이 경우 K-Report의 역할은 "국세청에 대신 기록"이
   아니라 **"신고에 필요한 데이터(거래일자·금액·상대방 등)를 미리
   구조화해서, 이용자가 실제 신고 시점(홈택스 등)에 바로 쓸 수
   있는 초안으로 만들어두는 것"**이다 — 실제 제출은 여전히 인간
   전속(HUMAN-AUTHORITY-GATE)이거나, 홈택스가 공식 API를 제공하면
   그걸 통해서만 가능하다.
2. **공개 API가 있는 통계·공공데이터 제출** — 일부 통계청 조사는
   실제로 온라인 제출 API/양식이 있을 수 있다(사업체 대상 정기
   통계조사 등). 이 경우는 K-Execute가 이미 하는 "기관 API 호출"과
   본질적으로 같은 작업이라, K-Report가 아니라 K-Execute의 몫으로
   재분류하는 게 맞을 수도 있다.
3. **당사자 간 사적 기록** — 지자체 조례에 따른 신고·거래 당사자
   쌍방에 대한 통지(예: 배달앱 플랫폼이 정산 내역을 셀러에게
   통지) 같은 건 실제로 혼디가 대행 가능한 영역이다.

**제 의견**: K-Report를 "모든 이해당사자 시스템에 실제로 쓰는" 만능
기관으로 설계하면 실현 불가능한 약속을 하게 된다. 대신 K-Report의
실제 역할을 **"이 사건과 관련해 법적·행정적으로 통지·신고 의무가
있는 대상을 식별하고, 각 대상별로 (a) 공식 API가 있으면 그걸 통해
직접 제출, (b) 없으면 제출용 초안을 만들어 이용자에게 넘기고 실제
제출은 이용자 몫으로 안내"로 정의하는 게 정직하고 구현 가능하다고
봅니다. 이러면 결과적으로 "국세청 API가 생기기 전까지는 신고서
초안만 만들어준다"는 게 되는데, 이것도 이용자 입장에선 상당한
가치입니다(신고 자체를 잊거나 데이터를 처음부터 정리하는 수고가
없어짐).

## 4. 10,310건 승격 메커니즘 재설계 — 초안

현재 구조(`worker.js` 확인): `status:'active'`(주피터 승인 완료)가
되면 전체 사용자에게 반환된다 — 즉 지금은 **1건씩 수동 승인**을
전제로 설계돼 있다. 10,310건을 전부 승격하려면 이 전제 자체를
바꿔야 한다. 제안하는 대안 두 가지(택1 또는 혼합):

- **(A) 신뢰도 기반 자동 승격** — eligibility_gate의 confidence
  분포(6절에서 이미 측정: high 93%/medium 5.5%/low 1.5%)를 기준으로,
  confidence:low 조건이 하나도 없는 레코드만 자동 active, 나머지는
  여전히 pending_review로 남겨 표본 검토.
- **(B) 표본 검토 후 일괄 승격** — 10,310건 중 예를 들어 200건을
  무작위 추출해 실제로 사람이 검토, 오류율이 허용 범위(예: 5% 미만)
  면 나머지 전체를 일괄 active 전환하는 스크립트 실행.

두 방법 다 "무조건 전부 승격"보다는 최소한의 품질 게이트를 남기자는
제안입니다 — 다만 최종 판단은 주피터님 몫입니다.
