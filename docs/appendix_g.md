# Appendix G — 이번 회차 결정사항 및 발견사항
**작성** Claude Sonnet 4.6 | 2026-06-22
**범위** 이 대화창에서 새로 결정된 설계·정책(G-D)과 코드/문서 조사로 새로 알게 된 사실(G-F)을 분리해 기록. `MASTER_PLAN_v1.md`(구현 현황)와 짝을 이루는 문서 — 거기는 "무엇을 했는지", 여기는 "왜 그렇게 하기로 했는지/무엇을 발견해서 그렇게 됐는지".

## G-D. 새로 결정한 사항 (Decisions)

**G-D1. entity_type은 person/business 둘만 유지한다.**
institution/org/platform은 별도 타입이 아니라 `business` + `extra.public.identity.entity_subtype`로 흡수. 행정구역 등록처럼 "흡수해도 본질이 같은 것"과, 그림자처럼 "본질적으로 다른 범주"를 구분해 적용한 동일 원칙.

**G-D2. 그림자(agent)는 person/business와 다른 3번째 entity_type이다.**
근거는 "사람/사업자와 그림자의 관계는 그림자(shadow) 관계"라는 비유 — 법인 비유(주주/법인)는 "독립된 법적 주체"라는 속성 때문에 폐기. 그림자는 본체 없이 존재 불가, 양도 불가, 본체 소멸 시 자동 소멸. 데이터모델: `casts_for`(본체 guid) 필드는 **불변**(트리거로 강제), 그림자 생성은 **본체 생성과 동시에 항상 자동**(법인의 "설립"처럼 별도 의사행위가 아님 — 그림자는 빛 아래 서면 항상 생긴다는 비유를 그대로 채택).

**G-D3. 그림자의 키는 본체의 키와 절대 통일하지 않는다.**
본체 키는 non-extractable·자기수탁(브라우저 전용)이라 본체가 오프라인이면 그림자가 서명을 못 한다는 문제가 있었지만, 키를 통일하면 (a) 본체의 진짜 자산(GDC 등) 키가 서버 침해 시 전부 같이 털리고 (b) "본인이 직접 한 말"과 "그림자가 자동으로 한 말"을 구분할 증거가 영구히 사라진다. 대신 **위임 인증서**(G-D4)로 연결성을 확보.

**G-D4. 위임은 키 공유가 아니라 1회 서명 인증서로.**
본체가 가입 시점(온라인일 때) 자기 키로 "이 그림자는 내 대리인이다"를 서명, PDV에 영구기록. 서버는 이 서명을 절대 대신 만들 수 없다(본체 키는 서버에 없으므로) — 따라서 이 단계는 항상 클라이언트(브라우저)에서 발생.

**G-D5. PDV는 구조화 데이터 슬롯을 두지 않는다 — Profile과 역할을 분리.**
"PDV와 Profile의 역할을 먼저 규정하라"는 지적에서 출발. Profile=외부에 보이는 신원/간판(공개), PDV=내부 증거·기억(비공개, 6하원칙 요약+해시만). 업종별 구조화 데이터(`industry_fields`)는 Profile에만 둔다.

**G-D6. 봉투(데이터 구조) 확장은 기존 구조 안의 빈 슬롯으로, 최상위 신규 섹션 추가는 지양.**
Profile의 `identity~finance` 5섹션, PDV의 6하원칙은 "추가만 가능, 삭제·이름변경 영구 금지". `industry_fields`를 5섹션 옆이 아니라 그 안(`public.industry_fields`)에 둔 것도 같은 이유 — 기존 문서·소비자 코드의 "섹션 개수"에 대한 가정을 안 건드리기 위함.

**G-D7. 95개(실제 77개) 업종 스키마는 사람이 전부 미리 쓰지 않고 Tier로 나눠 점진 작성한다.**
Tier1(비규제+빈도높음, 14개)부터, Tier3(규제산업)는 사람 검토 전까지 보류. "1년 베타"라는 시간 여유를 명시적으로 활용 — 한 번에 다 하려 하지 않는다는 원칙을 여러 Phase에서 반복 적용(예: Tier1을 8→14개로 늘릴 때도 "이번 1차 배치"만 먼저).

**G-D8. SP 전달은 "그림자 Profile에서 매번 fresh fetch"로 통일한다.**
처음 설계(이벤트 감지 후 `CFG.system` 교체, "Phase 2.5")를 폐기. 이미 `profile.html`이 쓰고 있던, 캐시 없이 매번 새로 읽는 방식이 실패모드가 구조적으로 적어 그쪽으로 통일. "본인이 자기 비서를 쓰는 것"과 "남이 호출한 비서를 쓰는 것"이 같은 메커니즘이 되도록.

**G-D9. Market(K-Market)은 특별한 지위가 없는 평범한 business다.**
ksic=46(도매 및 상품중개업)으로 등록되는 그냥 하나의 사업자. `commission_rate`는 전역 상수가 아니라 그 Market 인스턴스의 `industry_fields` 값 — 경쟁 Market이 다른 수수료로 들어올 수 있다는 뜻이고, 이건 의도된 결과(시장 경쟁 허용).

**G-D10. 도구 접근권(`tools`)은 Market 전용이 아니라 스키마 포맷의 일반 원칙이다.**
Market의 `search_entities` 호출 권한만 특별 취급하면 "Market이 다시 특별한 business"가 되어버려 G-D9와 충돌. 대신 모든 95개 파일의 헤더 포맷에 선택 필드 `tools`를 추가, 필요한 업종이 필요한 도구를 선언하는 구조로.

**G-D11. 부트스트랩 순환은 라우팅 테이블 하드코딩으로 해소한다(예외로 인정).**
신규 가입자가 Market을 찾으려면 검색이 필요한데 검색은 Market의 일이라는 순환. SP-00-ROUTER에 "공식 Market" handle을 미리 박아두는 것으로 해소 — 데이터 모델상 특별 지위는 아니지만, "처음 진입점을 어떻게 찾는가"라는 별개 문제에 대한 실용적 타협으로 명시.

**G-D12. AI비서 행위의 책임귀속(유한책임 여부)은 의도적으로 보류한다.**
학계에서 논의 중인 주제로 규정, 기술 구현은 이 결론과 독립적으로 진행. 다만 "구분할 수 있는 기술적 증거"(서명, PDV)는 미리 만들어 둔다 — 나중에 어느 쪽으로 결론나든 그 논의 자체가 가능하도록.

**G-D13. 그림자 키 격리(AGENT_KEK)를 별도 Worker로 분리하되, Service Binding 없이는 의미 없다고 명시한다.**
분리만 하고 외부에서 직접 호출 가능하면 공격자가 메인 worker.js 대신 그쪽을 두드리면 그만 — "분리했다"는 사실 자체가 안전을 보장하지 않는다는 걸 계획서에 명시적으로 박음.

**G-D14. signer Worker의 인터페이스는 탈중앙화 방향에 맞춰 지금부터 고정한다.**
"누가 실제로 서명하는지"를 호출자(메인 worker)가 몰라도 되는 인터페이스(`sign(agentGuid, message)`)로 설계 — 지금은 내부적으로 AGENT_KEK 커스터디가 서명하지만, 나중에 "본체 단말이 온라인이면 단말이 직접, 오프라인이면 커스터디 폴백"으로 내부 구현만 교체 가능하게. 오픈해시의 탈중앙 철학을 전면 재설계 없이 지금 적용할 수 있는 가장 저비용 지점으로 판단.

**G-D15. 사용자 유형 선택은 번호 메뉴 대신 AI 비서와의 대화로 추론한다.**
5개 버튼(개인/사업자/공공기관/협회단체/플랫폼)은 그림자가 6번째로 필요해지면서 "완전한 분류가 아니다"가 이미 증명됨 — 분류 체계가 계속 늘어날 걸 메뉴로 대응하지 않고, 대화형 추론으로 전환하기로 함(결과 데이터는 동일, 사용자 경험만 전환).

**G-D16. worker.js 문법 검증은 `node --check`가 아니라 esbuild를 표준으로 쓴다.**
실제 배포 실패(함수 시그니처 줄 소실)를 `node --check`는 못 잡고 wrangler가 쓰는 esbuild만 잡아낸 사건 이후 채택.

## G-F. 새로 알게 된 사실 (Findings)

**G-F1. Ed25519 개인키는 Web Crypto `exportKey('raw', ...)`가 불가능하다.** `pkcs8` 또는 `jwk`만 가능(직접 실행해 확인). 봉투암호화 설계 전체가 이 사실에 의존.

**G-F2. `handleProfileGet`이 `extra` 컬럼 전체를 반환한다(`rows[0]` 그대로).** `extra.public`만 거르지 않음 — 비공개 데이터를 `extra` 안 어디에 둬도(`public` 밖이라도) 공개 API로 노출된다는 뜻. 그림자 키를 별도 테이블(`agent_keys`)로 뺀 직접적 이유.

**G-F3. `user_profiles`에 활성/비활성 컬럼이 원래 없다.** "계정 비활성화" 기능 자체가 시스템에 없었음 — cascade 트리거를 만들기 전에 먼저 발견.

**G-F4. `_sbServiceHeaders`가 `SUPABASE_SERVICE_KEY` 없으면 anon 키로 조용히 폴백한다.** `agent_keys`가 RLS를 진짜로 강제하는 첫 테이블이라 이번에 처음 드러남 — 다른 테이블들도 같은 폴백으로 동작했을 가능성 있음(미점검).

**G-F5. Supabase가 레거시 `anon`/`service_role`(JWT) 키 체계에서 `sb_publishable_...`/`sb_secret_...` 신규 체계로 전환했다.** 신규 secret key는 "브라우저처럼 보이는 요청"이면 자동 401 거부하는 보호장치가 있음(User-Agent 기반) — Cloudflare Workers의 fetch가 여기 걸릴 가능성 있음(미확정).

**G-F6. `node --check`가 못 잡는 esbuild 전용 문법 오류가 있다.** 함수 시그니처 줄이 편집 중 통째로 사라진 사례에서, `node --check`는 OK를 반환했지만 esbuild(=wrangler가 실제 쓰는 도구)는 정확히 잡아냄.

**G-F7. `AGENT-SUPPLIER` 파일은 94개가 아니라 실제 77개(KSIC 2자리 코드 기준)다.** 메모리/추정이 실제와 다를 수 있다는 걸 다시 확인 — 항상 `ls`로 재확인.

**G-F8. OpenHash SCI 논문 초안 자체에 내부 불일치가 있다.** §1 초록은 L1 읍면동 "7,000개", §4.8 본문은 "3,500개" — 원문 오류, 고치지 않고 그대로 보존·기록.

**G-F9. `profile.html`은 이미 "에이전트가 Profile에 부가되는" 메커니즘을 갖고 있었다.** `pub.ai_assistant.system_prompt`를 그대로 페이지 내 채팅창의 LLM 호출에 사용(`_callSellerAI`) — 새로 만들 필요가 없었고, 그림자 SP 합성(Phase 2)이 끝나면 자동으로 맞물림.

**G-F10. `gopang-app.js`의 `loadPersonalAssistantSP()`가 영구 localStorage 캐시 버그를 갖고 있다.** PROFILE_SUBMIT 완료 후에도 갈아탈 코드가 없어, 가입자가 평생 온보딩 SP로 응대받는 상태였음 — 이게 G-D8(SP 전달 통일) 결정의 직접적 계기.

**G-F11. KSIC 46(도매 및 상품중개업)이 K-Market의 정확한 표준산업분류다.** "거래상대를 찾아주는 일"이라는 본질을 그대로 분류체계에 대응시킨 결과 — G-D9의 근거.

**G-F12. KIPO 특허출원(`.hlz`)이 선행출원(10-2026-0018910)의 후속임을 발견했다.** 논문에는 이 계보가 언급되지 않음, 특허 명세서에만 있음.

**G-F13. 특허 청구항은 60개가 아니라 30개다.** 정규식이 `<claim-text>` 하위태그를 중복 집계한 오류 — 직접 정정.

**G-F14. `src/openhash/{plsm,bivm,ilmv,lpbft,importanceVerifier,transactionPipeline}.js`가 이미 존재했지만, 7개 중 실제 호출경로에 살아있던 건 1개(hashChain의 PLSM 경유 호출)뿐이었다.** 나머지는 테스트에서만 호출되는 죽은 코드, 또는(BIVM처럼) 호출자 자체가 죽은 코드. `_submitToLayer`는 정적 GitHub Pages에 `POST /anchor`를 시도하는 구조라 prod/dev 어느 쪽도 실제 L1~L5에 도달 못 하는 상태였음.

**G-F15. `market/webapp.html`의 협상 결과가 결제 팝업에 전달되지 않는다.** `_parseTrade()`가 `seller_guid`/`handle`만 추출해 결제 팝업을 열고, 협상된 금액·할인은 URL로 안 넘어감 — 결제 팝업은 매번 정가로 새로 주문해야 하는 상태였음.

**G-F16. `handlePdvReport`는 6하원칙 요약 문자열과 해시만 영구 저장하고, `what` 객체에 무엇을 넣든 나머지는 버린다.** "PDV는 요약만 기록한다"는 원래 설계 원칙이 실제로 코드로 강제되어 있음을 재확인 — G-D5(PDV/Profile 역할 분리)가 이미 일부 코드 차원에서 지켜지고 있었다는 뜻.

**G-F17. 협회/단체(94)·금융보험(64~66)·전문서비스(71)·교육(85)·보건(86) 같은 규제산업 후보를 Tier 분류 중 식별했고, 87(사회복지서비스)은 Tier2로 1차 분류했다가 재검토 권고로 남겼다.** 어린이집·요양시설처럼 명백한 규제대상 하위업종이 큰 분류 코드 안에 섞여 있어 KSIC 2자리 단위 분류만으로는 안전하지 않을 수 있음.
