서문 — OpenHash의 두 가지 핵심 통찰

블록체인 기술은 탄생 이후 십수 년간 "신뢰할 수 있는 제3자 없이 거래를 기록하는 방법"을 모색해왔다. 그 결과는 PoW의 천문학적 에너지 소비와 PoS의 거버넌스 취약성으로 귀결되었으며, 단 한 국가도 이를 국가 수준의 인증 인프라로 채택하지 못했다.

첫째, "국가"라는 신뢰할 만한 장치가 이미 존재한다.

읍면동사무소, 시청, 광역시도청, 중앙정부는 수백 년에 걸쳐 구축된 제도적 신뢰의 결정체다. 블록체인이 에너지를 태우며 인위적으로 구축하려 한 "탈중앙 신뢰"는, 사실 물리적 공간에 이미 내재되어 있었다. OpenHash는 이 기존의 신뢰 계층을 분산 원장의 앵커로 전용한다.

둘째, "통신망"이라는 신뢰할 만한 소통 메커니즘이 이미 존재한다.

읍면동 기지국에서 시군구 교환국으로, 광역시도 허브를 거쳐 국가 백본망까지 이어지는 통신 인프라는 행정구역과 정확히 일치하는 계층 구조를 갖는다. OpenHash는 이 통신망의 물리적 계층 구조를 그대로 해시 체인의 전파 경로로 활용한다.

국가가 신뢰 계층을 제공하고, 통신망이 전파 경로를 제공한다. 합의 알고리즘이 소모하던 에너지는 사라지고, 기존 인프라의 부가적 효용이 극대화된다.

검증 완료 — 제주도 한림읍 파일럿 2026

Oracle Cloud 단일 VM 위에서 L1(한림읍)부터 L5(Global)까지 전 계층을 구동하고, 14개 읍면동 노드가 상향 전파(L1→L5)와 하향 반환(L5→L1)의 양방향 ILMV를 완성하였으며, leaf_hash 직접 변조 시 즉각 탐지됨을 확인하였다.

이 문서는 두 가지 목적을 가진다. 첫째, OpenHash 시스템의 이론적 설계를 비전문가도 이해할 수 있게 설명한다. 둘째, Oracle Cloud 환경에서 처음부터 재현할 수 있는 완전한 설치 가이드를 제공한다.

독자 대상: 분산 시스템을 이해하는 개발자, 시스템 관리자, OpenHash 생태계 참여를 검토하는 기관

1.1현대 사회가 직면한 문제 — 왜 OpenHash인가

문서의 시대, 인증의 위기

현대 사회는 문서에 의한, 문서를 위한, 문서의 체제다. 한국 사회 하나만 보더라도 운전면허증·건강진단서·보험증·방문증·진료예약서·졸업증명서·주민등록등본·차량등록증 등 각종 기관들이 무수한 증명서를 발급하고, 시민들은 이들 문서를 수천 종의 기관에 제출한다. 세계 전역에서 수십만 종의 인증서가 난립하며, 그 사회적 비용은 천문학적이다.

이 문제를 근본적으로 해결할 수 있는 기술이 블록체인이다. 그러나 지난 십수 년간 블록체인 기술을 국가 수준에서 상용화한 사례가 단 하나도 없다는 사실이 그 한계를 웅변한다.

기존 블록체인의 구조적 한계

1. 에너지 과소비 — Bitcoin의 연간 전력 소비량은 아르헨티나 전체(121 TWh)와 맞먹는다. 어떤 국가도 이 에너지 소비를 감당하며 전 국민의 문서 인증 시스템을 운영할 수 없다.

2. 확장성 한계

시스템처리량합의 방식에너지 효율
Bitcoin7 TPSProof of Work매우 낮음
Ethereum15 TPSProof of Stake낮음
Hyperledger Fabric3,500 TPS허가형 PBFT중간
OpenHash사실상 무제한평상시 없음HTTPS 수준

3. 오염 대응의 비효율성 — 기존 블록체인들은 데이터 오염 발생 시 전체 네트워크를 중단하거나 포크·롤백 방식으로 대응한다. 국가 인증 시스템이 멈춰서는 안 된다.

4. 중앙집중식 기존 시스템의 문제 — 홍길동의 성적표는 학교가 보관하고, 건강 이력은 건강보험공단이 보관하며, 주민등록등본은 행정자치부가 보관한다. 이 구조는 단일 실패점(SPOF), 기관 간 비효율, 데이터 주권 부재, 위변조 검증 어려움의 문제를 낳는다.

OpenHash가 해결하려는 7가지 과제
  1. 블록체인에 준하는 보안, 블록체인의 에너지 없이 — PoW를 완전히 제거하고 에너지 소비를 HTTPS API 호출 수준으로 최소화
  2. 기존 통신 인프라를 그대로 활용하는 계층형 인증 — 읍면동-시군구-광역시도-국가-글로벌의 기존 행정 구조를 신뢰 계층으로 전용
  3. 보안 강도와 효율성의 균형 — 중요한 문서에는 높은 보안을, 일상적 기록에는 경량 검증을 차등 적용
  4. 실시간 오염 탐지 및 선별적 치유 — AI 기반으로 위변조를 즉시 탐지하고, 전체 시스템 중단 없이 오염된 부분만 외과적으로 복구
  5. 통신 인프라 무관성 — 5G, WiFi, 위성통신, 광섬유 등 어떤 통신 방식에서도 동일한 보안 수준 제공
  6. 물리적 분산 저장을 통한 혁신적 보안 — 데이터를 구성원 각자가 보관하고 해시만 공개 원장에 기록
  7. 해시 전용 전송으로 프라이버시 보호 — 원본 문서 대신 SHA-256 해시값만 전송, 137바이트 고정 패킷으로 대역폭 90% 절약

1.2설계 철학 — 블록체인과의 본질적 차이

"오염을 허용하되, 즉시 치유한다"

PoW는 오염을 방지하기 위해 전 세계가 동시에 컴퓨터를 돌린다. 이것이 에너지 낭비의 근원이다. OpenHash는 발상을 전환한다:

"오염된 데이터라도 사용되지 않으면 해가 없고, 사용되면 즉시 치유한다. 따라서 대규모 오염은 발생할 수 없다."

누군가 한림읍 L1 노드의 데이터를 변조하더라도, 그 변조는 상위 L2(제주시)에 이미 anchoring된 Merkle root와 즉시 불일치가 발생한다. 예방이 아닌 실시간 감시와 즉각 치유가 OpenHash의 전략이다.

"모든 기록에 같은 보안을 적용할 필요는 없다"
[일상적 거래] → 경량 ILMV 검증 → 빠른 처리, 최소 에너지
[중요한 계약] → 표준 ILMV + 강화 감사 → 균형 잡힌 보안
[핵심 기록]   → zk-SNARKs + TEE + 스테이킹 → 수학적 무결성 보장
"기록의 신뢰도는 객관적으로 측정된다"

OpenHash는 모든 기록에 객관적인 신뢰도 점수를 부여한다:

신뢰도 점수 = log₂(네트워크 규모) × 계층 가중치 × 서명자 신뢰도 × 시간 가중치

누군가 디지털 운전면허증을 제시하면, 상대방은 그 면허증의 신뢰도를 숫자로 확인할 수 있다. 이것이 OpenHash가 수만 종의 인증서를 대체할 수 있는 근거다.

"기록 그 자체가 신분이 된다"

OpenHash가 완전히 도입된 세계에서는, 구성원의 탄생부터 소멸까지의 학적 기록·직업 기록·자격 기록·거래 기록이 시간순으로 해시 체인에 누적된다. 이 생애 기록의 연속성과 일관성이 곧 신분 증명이 된다. 별도의 졸업증명서, 자격증명서, 경력증명서를 발급받을 필요가 없다.

기존 통념의 극복

기존 통념: "블록체인의 탈중앙화를 위해서는 모든 참여자가 동등한 권한을 가져야 한다."

OpenHash의 반론: 계층별 차등 권한이 오히려 더 강력한 분산성을 달성한다. 이 계층 구조는 단일 실패점을 제거하면서도 각 계층이 독립적으로 운영된다.

1.3기대되는 효과

사회적 효과 — 인증서 체계의 혁신

OpenHash가 전 사회에 도입되면, 수만 종에 달하는 증명서·인증서·신분증·자격증·확인서·예약·거래·계약 등을 단 하나의 시스템으로 대체할 수 있다.

주민등록등본 예시: OpenHash 시스템 내에서 정부가 직접 해당 정보의 생성과 갱신 과정을 관할하면, 종래의 주민등록등초본 관리 체계를 대체하면서 데이터 소유자를 정부에서 개인으로 이관할 수 있다. 내 주민등록 데이터는 내 스마트폰에 보관되고, 정부는 그 해시만 관리한다.

금융 기관 예시: 은행·증권사·보험사 등이 가입자의 계좌 잔고를 갱신하는 과정을 직접 OpenHash에서 관장하면, 기존의 중앙 집중식 금융 원장 체계를 대체할 수 있다.

1.5해시 체인 모델 — 오픈 해시

원본 데이터는 절대로 네트워크에 올라가지 않는다. 오직 원본의 SHA-256 해시값(leaf_hash)만 분산 원장에 기록된다. 원본 데이터는 사용자의 PDV(Personal Data Vault, Supabase 암호화 저장소)에만 보관된다.

[사용자 원본 데이터]
        ↓ SHA-256
    leaf_hash (64자 hex)
        ↓ L1 노드로 전송
  l1_ledger DB 저장
        ↓ Merkle root 계산
  L2로 전파 → L3 → L4 → L5
        ↓ ILMV 하향 반환
  parent_root (L5 root) → L4 → L3 → L2 → L1

각 사용자는 개인 해시 체인을 유지한다:

h_i = SHA-256(h_{i-1} ∥ data_i ∥ height_i)

여기서 height_i는 해시 체인 높이로 논스(nonce) 역할을 하여 재생 공격을 방지한다.

1.6확률적 계층 선택 모듈 (PLSM)

PLSM은 거래의 중요도와 당사자 간 최하위 공통 계층(LCAT)에 따라 기록 계층을 확률적으로 결정한다.

알고리즘: ① 문서 해시와 타임스탬프를 SHA-256으로 이중 해싱 → ② 결과값을 mod 1,000 연산 → 0~999의 정수 r → ③ r과 LCAT, 중요도 점수에 따라 계층 결정

중요도L1L2L3L4L5
저중요도50%30%12%6%2%
고중요도15%20%30%25%10%
표 — LCAT=L1 기준 계층별 선택 확률. 10,000,000회 시뮬레이션에서 χ² 검정 p > 0.99로 균일 분포 확인.

중요도 점수 산출:

score = 0.5 × f_amount(거래금액) + 0.3 × f_type(자산유형) + 0.2 × f_contract(계약유형)

- 고중요도: score ≥ 60  → 강화 모드 (zk-SNARKs, TEE 등)
- 보통중요도: 25 ≤ score < 60 → 표준 모드
- 저중요도: score < 25  → 경량 모드 (ILMV만 적용)

1.7양방향 상호 계층 검증 (ILMV)

ILMV(Inter-Layer Merkle Verification)는 합의 없이 계층 간 데이터 무결성을 실시간으로 검증하는 OpenHash의 핵심 혁신이다.

상향 전파 (L1→L5): 하위 노드가 Merkle root를 계산하여 상위로 전달. 하향 반환 (L5→L1): 상위에서 받은 root를 parent_root로 하위에 반환.

TX 발생 (사용자)
    ↓
L1: leaf_hash 저장 → Merkle root 계산 → L2로 전파
    ↓
L2: child_root 수신 → 자신의 Merkle root 계산 → L3로 전파
    ↓
L3 → L4 → L5 (동일 과정 반복)
    ↓
L5: 최상위 root 확정 → L4로 parent_root 반환
    ↑
L4 → L3 → L2 → L1 (parent_root 하향 전달)
    ↓
L1: parent_root를 DB에 저장

하향 감사 주기: L2→L1 실시간 100% 스트리밍, L3→L2 10분 주기 15% 샘플링, L4→L3 30분 주기 10% 샘플링, L5→L4 1시간 주기 5% 샘플링. 각 계층은 하위 계층을 6개 항목(BLS 서명 유효성, Merkle 증명 일관성, 타임스탬프 신선도, 잔액 불변성, 순서 연속성, 해시 체인 연결)으로 검증한다.

1.8중요도 기반 적응형 무결성 검증

경량 모드 (score < 25): ILMV 실시간 스트리밍만 적용. 소액·빈번 거래에 적합.

표준 모드 (25 ≤ score < 60): ILMV + 결정론적 감사 강화. 일반 상거래에 적합.

강화 모드 (score ≥ 60): 다음 중 하나 이상 추가 적용:

  • zk-SNARKs: 수학적으로 검증 가능한 상태 전이 증명. 세부 데이터 비공개 유지.
  • TEE (신뢰 실행 환경): Intel SGX 등을 통해 노드 운영자도 접근 불가한 검증.
  • 스테이킹/슬래싱: 고중요도 거래 처리 노드에 자산 담보. 위변조 시 즉시 소각.

1.9긴급 경량 PBFT (LPBFT)

OpenHash는 평상시에 합의를 전혀 사용하지 않는다. 합의는 다음 5가지 비상 조건에서만 발동된다:

  1. 동일 노드 연속 2회 검증 실패
  2. 동일 거래에 대한 상충 버전 ≥ 2개
  3. 임의 계층 잔액 불일치 ≥ 10단위
  4. 동일 계층 노드 30% 이상 무응답
  5. 동일 높이 체인 분기 ≥ 2개

3단계 합의(Pre-prepare → Prepare → Commit)를 거친 후 비활성화 조건 4개가 모두 충족되면 자동으로 정상 모드로 복귀한다. L1 기준 합의 레이턴시는 단일 머신 환경에서 평균 0.759ms다.

1.105단계 거래 검증 파이프라인

모든 거래는 다음 파이프라인을 통과한다:

Stage 1: 잔액 확인 — 발신자 잔액 ≥ 거래 금액
Stage 2: 신원 검증 — Ed25519/BLS 서명 검증
Stage 3: 한도 확인 — 단일·일일 누적 한도 검증
Stage 4: 이상 탐지 — Isolation Forest 알고리즘
Stage 5: 규제 준수 — 블랙리스트, 국제 제재 목록

상하 양방향 Hash 연동

4.1단방향 전파의 문제

v1.0 구현에서는 L1이 leaf_hash를 L2로 보내고, L2가 이를 L3으로 보내는 단방향 전파만 존재했다. 이 상태에서 각 노드의 parent_root 필드는 항상 null이었다. ILMV의 진정한 가치는 양방향 연결에 있다. L5의 전지구적 신뢰 앵커가 L1까지 내려와야 계층 간 상호 검증이 완성된다.

4.2ILMV 양방향 설계

L5가 child_root를 받으면:
  → 자신의 Merkle root 계산
  → parentUrl이 없으므로 (최상위) 자신의 root를 parent_root로 반환

L4가 child_root를 받으면:
  → 자신의 Merkle root 계산
  → L5로 전파 (await)
  → L5로부터 parent_root 수신
  → parent_root를 응답에 포함하여 L3로 반환 (ILMV 핵심)

L3 → L2 → L1도 동일한 패턴

L5의 역할이 핵심이다. L5는 상위 노드가 없으므로 자신의 Merkle root를 parent_root로 반환한다. 이것이 하향 반환의 시작점이다.

4.3/push_root 엔드포인트 구현

routerAdd("POST", "/push_root", (c) => {
  // ... 자신의 노드 판별, node_ledger 저장, Merkle 계산 ...

  let parentRoot = null;
  if (self.parentUrl) {
    // 상위에 전파하고 parent_root 수신 (ILMV)
    const resp = $http.send({ url: self.parentUrl + "/push_root", ... });
    parentRoot = JSON.parse(resp.raw).parent_root;
  } else {
    // L5 최상위: 자신의 root를 반환 (ILMV 시작점)
    parentRoot = myRoot;
  }

  // 핵심: parent_root를 하위로 반환
  return c.json(200, {
    ok: true,
    node: self.id,
    merkle_root: myRoot,
    parent_root: parentRoot,  // ← 하위 노드가 이 값을 저장
  });
});

Merkle Tree

5.1Merkle Tree의 역할

Merkle Tree는 다수의 해시값을 하나의 루트 해시로 요약하는 자료 구조다. 단 하나의 leaf라도 변조되면 Merkle root가 완전히 달라진다. 이것이 위변조 탐지의 수학적 기반이다.

leaf_1  leaf_2  leaf_3  leaf_4
  ↓       ↓       ↓       ↓
hash(1+2)         hash(3+4)
       ↓               ↓
      hash(hash(1+2) + hash(3+4))  ← Merkle root

OpenHash에서 Merkle Tree는 두 가지 맥락에서 사용된다: L1에서는 모든 leaf_hash의 Merkle root를 계산하여 L2로 전파하고, L2~L5에서는 하위 노드들의 child_root의 Merkle root를 계산하여 상위로 전파한다.

5.2구현 — PocketBase JS 환경 제약

PocketBase JS 훅 환경(Goja 엔진)은 Node.js가 아니므로 crypto 모듈을 사용할 수 없다. 대신 PocketBase 내장 $security.md5()를 사용한다.

function buildMerkle(hashes) {
  if (!hashes || hashes.length === 0) return "GENESIS";
  if (hashes.length === 1) return hashes[0];
  let layer = [...hashes];
  while (layer.length > 1) {
    const next = [];
    for (let i = 0; i < layer.length; i += 2) {
      const left  = layer[i];
      const right = (i + 1 < layer.length) ? layer[i + 1] : layer[i];
      next.push($security.md5(left + right));
    }
    layer = next;
  }
  return layer[0];
}
MD5 사용에 관하여 현재 구현은 체인 연결(Merkle root 계산)에 MD5를 사용한다. 원본 데이터의 leaf_hash는 사용자가 SHA-256으로 계산하여 전송한다. 프로덕션 환경에서는 SHA-256으로 교체할 것을 권장한다.

5.3Merkle root 갱신 흐름

새 TX 수신 (L1) → l1_ledger에 저장
    ↓
l1_ledger 전체 leaf_hash 조회 → buildMerkle(hashes) 계산
    ↓
myRoot → L2 /push_root로 전송
    ↓ (L2)
node_ledger에 child_root 저장 (upsert: 동일 child_node면 업데이트)
    ↓
node_ledger 전체 child_root 조회 → buildMerkle(childRoots) 계산
    ↓
myRoot → L3 /push_root로 전송 ...

위변조 탐지

6.1탐지 원리

L1의 Merkle root는 항상 L2에 저장된 child_root와 일치해야 한다. L1 DB의 데이터가 변조되면, L1이 새로 계산한 Merkle root는 L2에 이미 anchoring된 child_root와 달라진다. 이 불일치가 위변조의 증거다.

6.2잔액 불변성 검증 (BIVM)

집합 잔액 검증: 불변 계층(LCAT 이상)에서의 순 잔액 변동량 합이 0이어야 한다.

Σδ_k = 0 (LCAT 이상 모든 계층의 순 잔액 변동량 합)

잔액 매핑 무결성 (BMI): 총량은 보존되지만 개별 잔액이 무단 이전되는 내부 위변조를 탐지한다.

BMI_k(T) = MerkleRoot({ balance_u(T) : u ∈ tier_k })

A에서 B로 ε만큼 무단 이전하면 합계는 동일하지만 BMI 루트가 달라진다.

07

이동통신망과 OpenHash 네트워크의 융합

7.1기본 구상

OpenHash의 L1 노드는 읍면동 단위의 물리적 공간에 배치된다. 이동통신 기지국(BTS)은 이미 읍면동 단위로 촘촘히 배치되어 있으며, 안정적인 전력과 네트워크 인프라를 갖추고 있다. 이 인프라를 L1 노드로 활용하면 별도의 인프라 투자 없이 전국 커버리지를 달성할 수 있다.

7.2IPv6 기반 사용자 식별

Gopang 플랫폼에서 모든 사용자는 user.ipv6 필드를 전역 고유 식별자(GUID)로 사용한다. IPv6는 사실상 무한한 주소 공간을 제공하므로, 모든 사용자·기기·기관에 고유 주소를 부여할 수 있다. OpenHash TX에서 from_guid 필드는 이 IPv6 주소를 사용한다.

7.3기지국 ↔ L1 노드 연동 아키텍처

[사용자 단말]
    ↕ 4G/5G
[BTS 기지국 (읍면동 단위)]
    ↕ 광케이블
[L1 노드 서버]  ← OpenHash PocketBase
    ↕ HTTPS
[L2 노드 서버]  ← 시군구 단위
    ↕
[L3 → L4 → L5]

BTS와 L1 노드가 동일한 시설에 위치하면 레이턴시를 최소화할 수 있다. 이동통신 사업자(KT, SKT, LG U+)가 L1 노드 운영 주체가 되면 기존 NOC 인프라를 활용 가능하다.

7.45G MEC와 L1 배치

5G MEC(Mobile Edge Computing)는 기지국 근처에 컴퓨팅 자원을 배치하여 레이턴시를 극소화하는 기술이다. L1 노드를 MEC 서버에 배치하면:

  • 단말↔L1 레이턴시: 1~5ms (MEC) vs 20~50ms (중앙 클라우드)
  • 데이터가 지역을 벗어나지 않아 개인정보보호 강화
  • 5G 네트워크 슬라이싱으로 OpenHash 전용 대역폭 보장 가능

7.5이동통신 사업자 참여 모델

  • 인프라 제공: 기지국 내 서버 공간, 전력, 네트워크
  • 신뢰 계층 역할: 통신 사업자 라이선스에 따른 제도적 신뢰
  • SIM 기반 인증: 기존 SIM 카드 인증 인프라와 Ed25519 서명 연동
  • 수익 모델: TX 처리 수수료를 GDC(₮)로 수령

7.6로밍 사용자의 LCAT 결정

A의 L1: KR-JEJU-JEJU-HANLIM (등록 주소 기준)
B의 L1: KR-SEOUL-JUNG-... (서울 중구)

LCAT(A, B) = min{k : tier_k(A) = tier_k(B)} = L4 (KR — 같은 국가)

→ L4 이상이 불변 계층, L1~L3이 가변 계층
→ L4 노드(한국은행, 금융결제원 등)가 최종 합의

제주도→서울 간 거래는 L4가 anchoring 계층이 된다. 국제 거래는 L5가 LCAT다.

7.7SIM 기반 인증과 Ed25519 서명 연동

[사용자 단말의 SIM]
    ↓ IMSI/MSISDN → 신원 확인
[이동통신 사업자 AAA 서버]
    ↓ 인증 완료 → 세션 키 발급
[OpenHash L1 노드]
    ↓ 세션 키 → Ed25519 서명 키 파생
[TX 서명 및 전송]

SIM의 하드웨어 보안 요소(SE)를 활용하면 서명 키를 단말 외부에 노출하지 않고 안전하게 관리할 수 있다. 사용자가 별도의 키를 관리할 필요 없이 기존 휴대폰 SIM으로 OpenHash에 참여할 수 있게 한다.

8.1/child_root 경로 예약어 충돌

증상: POST /child_root 요청이 항상 400을 반환한다. 핸들러 내부 코드가 실행되지 않는다. 로그에 아무런 오류가 없다.

원인: PocketBase v0.22에서 /child_root 경로는 내부 라우터가 예약하고 있다. routerAdd로 등록해도 PocketBase 내부 라우터가 먼저 가로채며, 훅 핸들러는 절대 실행되지 않는다.

해결: 전체 코드에서 /child_root → /push_root로 교체한다.

sed -i 's|/child_root|/push_root|g' /opt/gopang/pb_hooks/main.pb.js
sed -i 's|/child_root|/push_root|g' /opt/gopang/mock/mock-server.js

8.2전역 함수 정의 불가 (Goja 엔진 제약)

증상: main.pb.js를 배포하면 /health는 정상이지만 /push_root가 400을 반환한다. 로그에 훅 관련 오류가 전혀 없다.

원인: PocketBase v0.22의 JS 훅 엔진은 표준 Node.js가 아닌 Goja(Go 기반 ES5+ 인터프리터)다. 전역 스코프에서 함수를 정의하면 훅 등록 단계에서 내부 예외가 발생한다. 이 예외는 로그에 기록되지 않고 조용히 무시된다.

해결: 모든 코드를 핸들러 내부에 인라인으로 작성한다. 전역 함수/변수를 정의하지 않는다.

// ❌ 잘못된 패턴 — 전역 함수
function getSelf() { ... }
routerAdd("POST", "/push_root", (c) => {
  const self = getSelf();  // 작동하지 않음
});

// ✅ 올바른 패턴 — 핸들러 내 인라인
routerAdd("POST", "/push_root", (c) => {
  const NODE_CONFIG = { ... };  // 핸들러 안에 정의
  const name = $app.dataDir().split("/").pop();
  const self = NODE_CONFIG[name];
});

부록 A

ILMV 위변조 탐지 형식 증명

정리 1 (위변조 탐지 정리): 임의의 경제 단위 u의 데이터 레코드가 위변조된 경우, 최소 1개 이상의 계층에서 잔액 불변성 위반 또는 잔액 매핑 불일치가 탐지된다.

전제 조건: P-1 SHA-256 충돌 저항성, P-2 Merkle 트리 바인딩 속성, P-3 해시 체인 이중 서명 요구, P-4 ILMV 하향 감사 6개 항목 수행.

경우 1 (잔액 변동, ε ≠ 0): u의 잔액이 ε 변경되면 L1 집합 잔액이 ε 변동. 불변 계층에서 Σδ_k ≠ 0이 되어 L2→L1 100% 스트리밍 감사에서 즉시 탐지.

경우 2 (잔액 이전, Σ보존): A에서 B로 ε 무단 이전 시, BMI_k(T)_실제 ≠ BMI_k(T)_예상. P-2(Merkle 바인딩)에 의해 불일치 탐지.

경우 3 (복합 위변조): 임의의 위변조는 경우 1과 2의 합으로 분해 가능하므로 둘 중 하나 이상에서 탐지. ∎
