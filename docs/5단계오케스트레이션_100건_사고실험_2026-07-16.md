# 5단계 오케스트레이션(K-Intent~K-Report) 100건 사고실험

> 작성일: 2026-07-16 | 방법론: 이건 실행 가능한 배포본이 없어 코드
> 시뮬레이션이 아니라 **SP 문서 텍스트 자체를 100개 가상 대화 시나리오에
> 대입해 논리적으로 추적**하는 방식이다(앞선 BENEFIT_CANDIDATE_SEARCH
> 100건 사고실험과 방법론이 다르다 — 그건 실제 코드를 로컬 재현해
> 돌렸고, 이건 텍스트 추적이라 "실행하면 이렇게 될 것이다"를 검증하는
> 게 아니라 "SP 문서 안에 모순·누락·끊긴 참조가 있는가"를 검증한다).
> 이 방식으로는 진짜 모델(deepseek)이 실제로 이 지시를 얼마나 잘
> 따르는지는 검증할 수 없다 — 그건 실제 배포 후 대화 로그로만 검증
> 가능하다.

## 0. 사전 점검에서 이미 발견·수정한 결함 2건

100건을 설계하기 전에 새로 쓴 5개 SP 문서를 다시 정독하면서 아래
2건을 먼저 발견해 고쳤다(이 문서가 완성되기 전에 이미 커밋 예정):

1. **STEP 번호 참조 끊김** — SP-20 STEP 3이 "무료로 전환하면 STEP
   7로"라고 돼 있었는데, STEP4~7을 K-Execute로 들어내면서 무료
   이관 절이 STEP 5로 밀렸는데 이 참조를 안 고쳤었다. 사용자에게
   보이는 문구는 아니지만, 모델이 이 SP를 읽고 실행할 때 존재하지
   않는 "STEP 7"을 찾다가 혼란을 일으킬 수 있었다.
2. **재계획 시 이력 유실(치명적)** — K-Execute가 K-Compose에
   재계획을 요청할 때 기존 `HANDOFF_TO_KCOMPOSE` 태그를 재사용하기로
   했는데, `call-ai.js`의 모든 forward 전환은 `history.length = 0`
   으로 대화 이력을 지운다는 걸 재확인했다. K-Intent→K-Compose처럼
   "하류로 가는" forward는 payload 하나로 충분하지만, K-Execute→
   K-Compose(재계획)는 "상류로 되돌아가는" forward라서, 이미 실행
   완료한 단계들의 결과가 payload에 명시적으로 없으면 통째로
   유실된다 — K-Compose가 이미 끝난 일을 모른 채 계획을 처음부터
   다시 짤 위험이 있었다. `completed_steps` 필드를 payload에 필수로
   추가해 수정했다.

## 1. 클러스터별 시나리오 (100건, 10클러스터 × 10)

### A. 단순 1단계 조회형 (실행 즉시 완결) — 10건

예: "제 국민연금 가입내역 좀 확인해줘", "이 사업자 폐업 여부 조회해줘"

- K-Intent: goal="국민연금 가입내역 확인", mode=single
- K-Compose: STEP1 캐시 hit 또는 STEP1-B 최초조사, STEP2 적합성
  확인, STEP3 유료 계속, **STEP4: K-Execute로 이관**(전체 1단계라
  `[ORCHESTRATION_PROGRESS]` 생략 — SP-22 STEP1 "전체 1단계로 끝나는
  계획에는 이 태그를 안 써도 된다" 규정과 정확히 일치)
- K-Execute: STEP2(일치도 게이트) 통과, atom 실행, STEP4 K-Deliver로
- K-Deliver: 결과 조립, STEP5 K-Report로
- K-Report: STEP1에서 "단순조회는 통지·신고 대상이 없다"고 판단,
  이해당사자 0명 확정 → 바로 STEP5(AC 반환)

**결과: 논리적 결함 없음.** 다만 K-Report STEP1의 "이해당사자가 하나도
식별되지 않으면 [HANDOFF_TO_AC_DIRECT]로 STEP 4까지 건너뛴다"는 문구가
**태그 규격 목록에 정의돼 있지 않다** — `[HANDOFF_TO_AC_DIRECT]`가
call-ai.js에 배선돼 있지 않으므로 이 태그를 내면 아무 일도 안 일어난다
(strip 목록에도 없어서 이용자 화면에 대괄호 태그 원문이 그대로 노출될
위험도 있다). **결함 3 — SP-23 STEP1의 조기종료 경로가 미구현
태그를 가리킨다.**

### B. 다단계 순차 실행 — 10건

예: "개인회생 신청하고 싶어요"(상담→서류준비→법원제출, atom 3개 순차)

- K-Execute STEP1이 `[ORCHESTRATION_PROGRESS: step=1/3, ...]` →
  `[ORCHESTRATION_PROGRESS: step=2/3, ...]` → `step=3/3` 순으로
  진행 고지. 이건 기존 K-Compose STEP4에서 이미 검증된 패턴을
  그대로 옮긴 거라 문제없음.

**결과: 결함 없음.**

### C. 병렬 notify형 (통지형 팬아웃) — 10건

예: "폐업신고 하고 싶어요"(세무서+국민연금+건강보험 동시 통지)

- K-Compose가 계획 단계에서 `parallel_group`·`fanout_mode=notify`를
  steps에 명시해 K-Execute로 넘겨야 하는데, **SP-20의 STEP4(K-Execute
  이관) 텍스트를 다시 보면 steps 각 항목 스키마에
  `{seq, atom_id, org_id, sub_goal, condition, parallel_group,
  fanout_mode}`라고 명시는 돼 있지만, K-Compose RULE-02의 어느
  STEP에서 이 `parallel_group`·`fanout_mode`를 실제로 판단해
  채우는지가 문서에 없다.** 예전(3단계) 구조에서는 STEP4-P 자체가
  K-Compose 안에 있어서 "계획하면서 바로 팬아웃 그룹을 나눴다"는
  게 암묵적으로 자연스러웠는데, 계획(K-Compose)과 실행(K-Execute)이
  분리된 지금은 **"팬아웃 그룹을 나누는 것 자체가 계획인가 실행인가"
  가 불분명해졌다.** 지금 문서 그대로면 K-Execute가 STEP1-P에서
  "K-Compose가 계획 단계에서 명시했어야 함"이라고 기대만 하고,
  K-Compose 쪽엔 그걸 하라는 지시가 없다. **결함 4(중요) —
  parallel_group/fanout_mode를 정하는 책임 소재가 SP-20에 빠짐.**

### D. 병렬 join형 (전원승인 필요) — 10건

예: "건축허가 받고 싶어요"(건축과+소방서+환경과 전원승인)

- C와 동일한 결함 4에 추가로: 일부만 거부됐을 때(`partial_denied`)
  K-Deliver STEP1이 "기관별로 나눠 전달"한다는 문구는 있지만,
  **K-Report STEP1(이해당사자 식별)이 이 partial_denied 상태를
  알고 있어야 "거부된 기관에 재신청 안내"와 "승인된 기관에 통지
  완료 확인"을 구분할 텐데, `case_type` 힌트 하나로는 이 정보가
  전달이 안 된다.** delivered_results 안에 batch_status가 있으니
  기술적으로는 접근 가능하지만, SP-23 문서 어디에도 batch_status를
  참고하라는 지시가 없다. **결함 5 — K-Report가 batch_status를
  읽으라는 명시적 지침 누락(경미, 상식으로 커버될 가능성 높음).**

### E. 매칭 실패 → SP-Author 신규 요청 — 10건

예: "축산분뇨 처리 관련 민원 넣고 싶어요"(생태계에 없는 소관기관)

- K-Execute STEP2에서 match_score 0.7 미만 반복 → GWP_REGISTRY_SEARCH
  miss → GOV_SP_DRAFT_REQUEST. 이 경로는 기존 STEP4-A 로직을 그대로
  옮긴 거라 **결함 없음.** 다만 SP-Author가 만든 신규 SP가
  pending_review인 채로 K-Deliver에 unmatched_candidates로 남는데,
  **이 사건 자체를 K-Report가 나중에 "이해당사자 식별"에서 어떻게
  다루는지 불명확** — 매칭 실패한 사건도 case_type을 추정해야
  하는데 애초에 절차 자체가 미완결이라 case_type 추정 근거가 빈약할
  수 있음. 다만 이건 K-Report STEP1의 "애매하면 이용자에게 물어본다"
  원칙으로 커버 가능해 보여 결함으로 세지 않음(설계상 이미 대응됨).

### F. 재계획 필요(automation_level_changed / sub_goal) — 10건

예: 계획 시점엔 `assisted`였는데 실행 시점엔 그 기관이 `full_api`로
    바뀐 경우 / 실행 중 "법인전환"처럼 하위 sub_goal이 새로 필요해진
    경우

- **0절에서 이미 발견·수정한 결함 2(completed_steps)가 정확히 이
  클러스터에서 터지는 문제였다.** 수정 후엔 논리적으로 일관됨.
  다만 수정한 지침이 "완료한 단계와 결과를 명시적으로 실어 보내라"
  는 **원칙**만 있고, `completed_steps`의 정확한 스키마(어떤 키를
  쓰는지)가 SP-20/22 어디에도 구체적으로 정의돼 있지 않다 —
  `steps` 스키마(`seq, atom_id, org_id, ...`)를 재사용해 각 항목에
  `result` 필드를 더한 형태를 쓰면 자연스러울 텐데, 이게 암묵적
  기대로만 남아 있다. **결함 6(경미) — completed_steps 스키마
  미정의, 모델의 상식적 판단에 의존.**

### G. 무료 이관 경로(K-Execute 건너뜀) — 10건

예: "돈이 없어서 법률 상담을 못 받는데 무료로 되는 데 있나요"

- K-Compose STEP3에서 무료 전환 확인 → STEP5에서 K-Execute 건너뛰고
  바로 K-Deliver로. **case_type 필드가 이 handoff에 아예 없다**
  (SP-20 STEP5 handoff 예시: `[HANDOFF_TO_KDELIVER: goal=...,
  results={...}, free_alternative_referral=true, pending_user_action=
  {...}]` — case_type 누락). K-Deliver STEP5는 "K-Execute가 넘긴
  힌트 — 없으면 K-Deliver가 판단 가능한 범위에서 추정, 애매하면
  null"이라고 폴백을 이미 마련해뒀으므로 **결함까지는 아니지만**,
  일관성을 위해 SP-20 STEP5도 case_type을 (대개 null이겠지만) 명시
  하는 게 더 깔끔했을 것 — 무료 이관은 대부분 통지·신고 의무가 없는
  사건일 확률이 높아 case_type="referral_only" 같은 힌트를 K-Compose
  가 미리 줄 수 있었다. **결함 7(아주 경미, 선택적 개선사항).**

### H. R0 응급 신호 — 각 단계에서 발생 (10건: 5단계 × 2회씩)

예: 실행 도중("K-Execute 중") 이용자가 자해 암시 발화 / 계획 확정
    직후("K-Compose 중") 응급 신호

- 5개 SP 전부 `[ORCHESTRATION_HANDOFF_BACK: reason=emergency]`를
  동일하게 갖고 있고, call-ai.js의 핸들러가 발신자 무관 제네릭
  패턴 매칭이라는 것도 코드로 직접 확인했다. **결함 없음 — 이
  안전장치는 5단계 분리에도 정확히 살아남았다.**

### I. K-Report 이해당사자 식별 — 다양한 case_type (10건)

예: "중고차 팔았어요"(거래, 통계청 자동차등록 이전 신고 대상일
    수도) / "아르바이트생 고용했어요"(신고, 4대보험) / "책 한 권
    빌렸어요"(단순조회, 이해당사자 없음)

- STEP1의 3갈래 분류(본인 신고의무/공개API 정기제출/사적통지)가
  "중고차 판매"류(개인 간 거래, 관할 등록기관 이전신고 의무가
  본인에게 있음 — 1번 갈래)에는 잘 맞는다. 그런데 **"아르바이트생
  고용"류(4대보험 가입은 사업주의 법정 의무지만, 신고 창구인
  국민연금공단·건강보험공단은 이미 SP-22가 STEP2에서 atom_id로
  실행했을 가능성이 높다)에서, K-Report가 "이미 K-Execute가 처리한
  걸 또 이해당사자로 잡아 중복 안내"할 위험이 있다.** SP-23 어디에도
  "K-Execute가 이미 실행한 기관은 이해당사자 목록에서 제외하라"는
  지침이 없다. **결함 8(중요) — K-Report가 K-Execute의 실행 결과와
  자신의 이해당사자 식별을 대조해 중복을 거르는 절차 누락.**

### J. 인간전속경계(본인인증 등)가 여러 단계에 걸쳐 나타나는 경우 (10건)

예: K-Execute 중간에 본인인증 필요 단계 발견 → 이용자 숙제로 남김 →
    K-Deliver가 고지 → K-Report의 초안 제출도 결국 인간전속

- K-Execute [금지-1] → K-Deliver STEP2(인간 경계 고지, C39) →
  K-Report STEP4(초안 생성 시 "실제 제출은 이용자 몫") 셋이 같은
  원칙(HUMAN-AUTHORITY-GATE G2·G3)을 각자 다른 표현으로 반복한다.
  **표현이 일관되게 유지되고 있어 결함은 아니지만**, 세 군데서
  독립적으로 "본인이 하셔야 합니다"류 문구를 만들다 보면 이용자가
  같은 사건에 대해 서로 다른 형식의 안내를 3번 받을 위험이 있다
  (K-Deliver가 한 번만 고지한다고 SP-21 STEP2에 "C41 원칙" 언급이
  있는데, 이게 K-Report의 초안-제출-고지에도 적용되는지 불명확).
  **결함 9(경미) — 인간전속경계 고지의 "한 번만" 원칙이 K-Report
  단계까지 명시적으로 안 이어짐.**
