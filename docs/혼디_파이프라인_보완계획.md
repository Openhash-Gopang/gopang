## Phase 0 — 계측 신설 (선행, 저위험, 즉시 착수 가능) — ✅ 완료

**목표**: `gwp_registry.call_count_30d` 미계측 문제 해소. 뒤 phase들이 쌓일수록 데이터가
늦게 쌓이기 시작하므로 가장 먼저 켠다.

**작업**:
1. `gwp/engine.js`의 PDV 기록 시점(GWP 탭 종료, 6하원칙 요약 직후)에 `POST /gwp-registry/register`를
   증분 방식(`call_count_30d: 기존값+1`)으로 함께 호출.
2. worker.js `handleGwpRegistryRegister`에 "증분 모드" 지원 추가(`increment_call_count: true`
   플래그를 받으면 기존 값을 읽어 +1 하는 분기 신설).
3. 이 요청이 실패해도(gwp_id가 아직 `gwp_registry`에 없는 core 미시딩 서비스 등) 탭 종료 자체를
   막지 않도록 `.catch()` 처리(기존 ESCALATE 생성 실패 처리와 동일 원칙).

**검증**: 탭 종료 시뮬레이션 후 call_count_30d가 실제로 +1 되는지 통합테스트로 확인.

**리스크**: 낮음. 매 탭 종료마다 API 호출이 하나 늘어나는 성능 비용은 있으나 무시할 수준.

## Phase 1 — K-Search 실제 검색 실행 로직 구현 (최우선, 최대 임팩트) — ✅ 완료

**목표**: 패턴 E(전체 시나리오의 ~25%가 2단계에서 끊김) 해소. 이 계획의 핵심.

### 1-A. 태그 문법 확정 (구현 전 필수 선행)
K-Search RULE-02 STEP3([SEARCH]+동의어 확장)이 실제로 내는 파라미터 전체를 SP 원문에서
정확히 뽑아내, 이미 확인된 백엔드 스펙(`p_keyword/q, p_entity_type, p_occupation, p_address,
p_lat/p_lng, p_sort, p_limit/p_offset, ...`)과 1:1 대응하는지 대조. 기존 `type=user` 핸들러와
충돌하지 않도록 구분값(`type=entity`)을 명시적으로 쓰는지 이 단계에서 확정 — 짐작으로 만들지
않는다.

### 1-B. call-ai.js에 신규 핸들러 이식
market/webapp.html의 **이미 실전 검증된** "[SEARCH] 감지 → search_entities RPC 호출 →
`history` 재주입 → `callAI()` 재귀 호출" 로직을 그대로 가져와, gopang의 `call-ai.js`에
`_handleKSearchExecutionTag()`로 이식. 재작성하지 않고 이식하는 이유: 이미 검증된 코드라
새로 짜는 것보다 위험이 낮다.

### 1-C. 두 위임 경로 마무리 연결
- **6-A(AC→K-Search, system 전환)**: `[KSEARCH_HANDOFF]`는 이미 배선 확인됨(`_forwardSwitchSP`).
  1-B가 붙으면 이 경로는 추가 작업 없이 자동으로 완성된다.
- **6-B(도메인 SP→K-Search, nested)**: `[CALL_KSEARCH: query=..., mode=list, caller=...]`는
  market 레포에만 구현 확인. gopang 레벨의 유일한 후보였던 K-Compose는 지난 세션에 만든
  `[GWP_REGISTRY_SEARCH]`(직접 HTTP 호출)로 이미 우회 경로가 있어 **이번 phase에서는 6-B를
  gopang에 새로 만들지 않음**(시급성 낮음, 별도 phase로 미룸).

**완료 후 추가로 필요 확인됨(다음 세션 이월)**: STEP 3-C에서 "그래도 없으면" 분기가 RULE-07
(웹검색)을 먼저 시도하라고 명문화되지 않은 문제 — SP-18 v1.4 필요(사고실험 문서 §3-1 참조).

## Phase 2 — 미등록 업체 온보딩(unclaimed 프로필) — ⏸ 보류

**목표**: 혼디에 아직 없는 소규모 서비스업체를 위한 임시/미청구 프로필 생성 경로.

**보류 사유**: 이건 **코드보다 정책 결정이 먼저**입니다. Hondi의 자기주권 신원모델(TOFU/Ed25519)
핵심과 맞닿아 있고, K-Search SP 자신도 "명의도용 계정 양산 위험"을 경고해뒀습니다. 정식 서명
계정을 전제하지 않은 상태에서 RULE-03 STEP 3 이후(임시 프로필 생성)를 함부로 실행하면, 실제
존재를 확인하지 않은 대상의 프로필이 양산되는 구조가 됩니다.

**대안(실용적 우회)**: 웹검색 폴백(RULE-07, Serper.dev)을 구현해 "혼디엔 없지만 웹엔 이런
정보가 있다"고 정직하게 안내하는 경로를 확보 — 이미 완료. Phase 2 자체(혼디 내 미존재 대상과
실제 핸드셰이크해 거래까지 완결)는 정책 승인 없이는 착수하지 않음.

## Phase 4 — SP 정기갱신 스케줄러 — ✅ 완료 (당초 "별도 결정" 예정이었으나 조기 착수)

**목표**: `gwp_registry` 호출 빈도 기반 주간/월간/분기 갱신 계층화를 실제로 가동.

당초 계획은 "Phase 0 가동 후 실제 데이터량을 보고 결정" — 저장소 밖 스케줄러(GitHub Actions
등)를 실제로 만들지, 수동 조회(`GET /sp-author/refresh-due`)로 충분한지 판단 보류였으나,
2026-07-11 세션에서 GitHub Actions 매일 실행으로 실가동 완료. 로컬 통합테스트(worker.js를
실제 HTTP 서버로 띄워서 검증)로 실제 버그 2건 발견·수정:
(a) tier 재분류만 해도 `next_due_at`이 매번 미래로 밀려나 due 판정이 영원히 안 되던 버그,
(b) 정기갱신 신호의 중복방지가 institution+task로만 판단해 서로 다른 SP 요청이 하나로
뭉개지던 버그(→ target_sp_id 기반으로 수정).
