# PA(profile-assistant) 갱신 로드맵 v1.0
**작성** Claude | 2026-07-17 | **근거** docs/pa_identity_template_thought_experiment_v1.md §5

100인 사고실험에서 코드 결함 3건은 이미 수정·병합 완료(v2.9, 커밋
`afe4dd9`). 이 문서는 그때 "정책 판단이 필요해 코드를 안 건드린" 나머지
항목들을 실제로 처리하기 위한 순서를 정리한다.

## 1번 처리 결과 (완료, 2026-07-17)

`worker.js`의 `TIER3_REGULATED_SCHEMA_IDS`를 열어보니 KSIC **2자리
대분류 코드**만 취급한다(`AGENT-SUPPLIER-NN` 파일명 체계와 동일 단위):
`12,19,21,27,64,65,66,71,85,86,87`. 그런데 사고실험(§2 표)에서 제가
임의로 붙인 예시 코드들은 전부 **4자리**(예: 담배제조업="1200",
법무사사무소="6912")였다 — 실제 시스템 단위와 안 맞는 가짜 코드였다.
2자리로 다시 매핑해 대조한 결과:

| 사고실험 표본 | 실제 2자리 KSIC | TIER3 포함? |
|---|---|---|
| 담배제조업 | 12 | ✅ 포함 |
| 법무사사무소 | 71 | ✅ 포함 |
| 세무사사무소 | 71 | ✅ 포함 |
| 한의원(경영지원) | 86 | ✅ 포함 |
| 동물병원(경영지원) | 86(수의업도 보건업 하위) | ✅ 포함 |
| **주류제조업** | **11(음료 제조업)** | **❌ 미포함** |

5건은 실제로 잘 걸리는데, **주류제조업(11)만 빠져 있다.** 주세법 규제
대상인데 TIER3 목록엔 없다 — 다만 이걸 넣을지는 규제 범위를 어디까지
볼 것인지의 정책 판단이라 제가 임의로 추가하지 않았다. 필요하시면
`TIER3_REGULATED_SCHEMA_IDS`에 `'11'` 한 줄만 추가하면 된다.

## 2·3번 처리 결과 (완료, 2026-07-17)

**2번(KSCO Tier3 게이트)** — `data/ksco_2024_v8.json`에서 24·27 중분류
전체 하위 코드를 뽑아 재분류했다. 상세 근거는
`docs/ksco_schema_tier_classification_v1.md` v1.1 섹션 참조. 결론만
요약하면:

- **Tier3(신규 서버 게이트, `worker.js`의 `TIER3_REGULATED_KSCO_PREFIXES`)**:
  의사(전문·일반)·한의사·치과의사(2411~2414), 수의사(242, 이번에 새로
  판단), 약사·한약사(2431·2432), 판사·검사(2711) — 이 코드들은 `job_ksco`
  가 무엇을 요청하든 서버가 `visibility`를 강제로 `private` 처리한다
  (`industry_fields.schema_id`의 Tier3 게이트와 동일 원칙).
- **Tier1 유지**: 간호사·영양사·임상병리사 등 나머지 24 세분류, 변호사·
  변리사·행정사 등 나머지 27 세분류 — v1.0의 기존 판단 그대로.
- **Tier2(재검토 권고, 이번엔 게이트 미적용)**: 법무사·집행관(2713),
  정부 행정 전문가(2721) — 애매해서 일단 보류만 해뒀다.

**3번(RENEWALING 최소 표본)** — 제 의견은 **3단계(n<3 미확정 / 3≤n<5
잠정 / n≥5 안정)**였고, 그대로 `tools/renew_identity_templates.py`에
반영했다. 100인 사고실험 합성 데이터로 재검증한 결과, 이전엔 82개
조합이 표본 부족한 채로 "권장 필드"를 냈던 게 이번엔 81개가
`insufficient`(추천 없음)로 정확히 걸러졌다. 이 숫자(3, 5)는 실제
가입 속도를 보고 나중에 조정해도 된다 — 스크립트 안에 상수 두 개
(`MIN_SAMPLE_PROVISIONAL`, `MIN_SAMPLE_STABLE`)로 분리해뒀다.

## 순서 제안과 이유

**1번을 가장 먼저** — 유일하게 정책 판단이 필요 없는 순수 검증 작업이라
바로 처리 가능하고, 2번(KSCO Tier3 게이트)을 설계할 때 KSIC 쪽 게이트가
정확히 어떻게 동작하는지 먼저 확인해두는 게 참고가 된다.

**2번과 3번은 병렬 진행 가능** — 서로 다른 파일(worker.js 검증부 vs
tools/renew_identity_templates.py)을 건드리고, 어느 쪽 결정이 먼저
나든 다른 쪽 작업에 영향 없다. 다만 둘 다 **제가 임의로 숫자나 목록을
정하지 않는다** — 2번은 어느 세분류까지 막을지, 3번은 임계값 n을 얼마로
할지 두 가지 다 실제 서비스 판단(가입자 수 증가 속도, 민감직종 범위를
어디까지 볼지)이라 여쭤보고 진행하는 게 맞다고 본다.

**4번은 후순위** — 사고실험 표에서 "PASS(제한적)"으로 나온 12건 전부
가상 시나리오였고, 실제 사용자가 이 삭제를 몇 번이나 요청하는지 아직
데이터가 없다. 지금 범용 태그부터 만들기보다는, PA가 임시방편으로
알아서 처리하게 둔 채로 실사용 데이터가 쌓이면 그때 패턴을 보고
설계하는 게 낫다고 본다(RENEWALING과 같은 "쌓이면 보인다" 원칙).

**5번은 사실상 "하지 않음"을 기본 결론으로 제안** — 사고실험 §5에서도
적었듯, 세분화 자체가 민감정보를 구조화된 필드로 만드는 셈이라 지금의
뭉뚱그림이 오히려 의도치 않게 안전한 설계일 가능성이 높다. 다른 강한
이유가 없으면 이 항목은 로드맵에서 그냥 닫아도 된다고 생각한다.

## 진행 방식 제안

1·2·3번 모두 처리 완료했습니다. 4번(개인 프로필 범용 필드 삭제)은
실사용 데이터가 쌓일 때까지 보류를 제안하고, 5번(`work_domain` 'other'
세분화)은 아예 닫는 걸 제안합니다 — 이견 있으시면 말씀해 주세요.
