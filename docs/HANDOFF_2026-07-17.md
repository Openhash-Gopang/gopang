## 0. 작업 방식 확인 (매우 중요)

이 세션은 `bash_tool` 샌드박스에서 `github.com`/`raw.githubusercontent.com`/`codeload.github.com`은
접근 가능하지만 **git push 인증 정보가 없다.** 따라서:

1. `git clone https://github.com/Openhash-Gopang/gopang.git`로 로컬에 저장소를 받는다.
2. 코드를 수정·커밋한다(`git config user.name "jejuro"`, `git config user.email "tensor.city@gmail.com"`).
3. `git format-patch <base>..HEAD --stdout > 파일명.patch`로 patch 생성.
4. **patch 생성 시 base 커밋 지정에 항상 주의** — `git am`은 커밋을 재작성해서 해시가 바뀐다.
   그래서 `origin/main..HEAD`처럼 상대 비교하면 이미 반영된 커밋까지 다시 포함되는 사고가
   이번 세션에서 여러 번 있었다. **반드시 `git fetch origin` 먼저 하고, 실제 origin/main 해시를
   직접 확인한 뒤, 필요하면 `git log --oneline`으로 정확한 "새 커밋 시작점"을 특정해서
   `git format-patch <내_로컬_이전_커밋_해시>^..HEAD` 형태로 명시적으로 만들 것.**
5. patch를 `present_files`로 전달하고, 실제 clone에 `git am`으로 적용해서 **에러 없이 붙는지
   미리 검증**한 뒤에 사용자에게 전달할 것(이번 세션에서 검증 없이 드렸다가 충돌 나서 다시
   만든 적이 여러 번 있었음).
6. 사용자는 Windows PowerShell을 쓴다. `PS C:\...>` 프롬프트 문자열을 포함해서 복사-붙여넣기
   하는 실수가 반복됐다 — 명령어 안내 시 "프롬프트 문자열은 빼고 명령어만 붙여넣으세요"를
   명시할 것.
7. 저장소 로컬 경로: `C:\Users\주피터\Downloads\gopang` (사용자 PC 기준).

## 2. 이번 세션에서 발견·수정한 실제 프로덕션 버그 목록 (전부 gopang main에 반영 완료)

1. `src/pdv/evidencePackage.js` — `anchor()` 구 API 호출로 법원 제출용 증거 패키지 생성이
   항상 실패하고 있었음(스토킹/가정폭력 시나리오 영향)
2. `src/ai-secretary/phase6.js` — 같은 클래스의 버그, AI 비서의 모든 대화 기록(PDV+앵커링)이
   항상 실패
3. `src/ai-secretary/phase2.js`/`phase4.js` — K-Law classifier가 순수 민사분쟁(협박/사기 어휘
   없는 임대차 분쟁 등)을 절대 못 잡던 구조적 갭. Phase 2 게이트 제거 + Phase 4에 P2 단독
   고신뢰도 감지 시 최소 S1 문턱 보장 로직 추가
4. `src/network/layerClient.js` — "config import 제거"라는 자기 주석대로 import는 지웠는데
   참조는 안 지워서 이 파일을 import하는 순간 항상 크래시
5. `worker.js`의 push 발송 관련 — 테스트 자체는 안 고쳤지만(픽스처 버그였음), 실측 과정에서
   `_sendWebPush`가 실제로 조용히 항상 실패하고 있었다는 것도 확인(원인은 테스트 픽스처의
   placeholder 키였음 — 프로덕션 코드는 정상으로 판명)

이 외에 **테스트 파일만의 드리프트**(프로덕션은 정상, 테스트가 낡아서 실패)를 10건 이상
발견·수정했다 — 각 커밋 메시지에 원인이 상세히 적혀있으니 필요하면 `git log`로 확인.

## 3. 신설한 기능

`src/ai-secretary/phase7.js` — LLM 기반 "종합 위법 가능성 판단"(2단계). Phase 1~5(규칙기반,
비용 0)가 이미 S0가 아니라고 판정한 대화에만 LLM을 호출해서 토큰 비용을 최소화하는 설계.
`runPipeline()`에 4번째 선택 인자 `llmCaller`로 주입식(DI) — 실제 운영에서 쓰려면 이 함수를
worker.js 등에서 실제 LLM 호출부(`deepseek-client.js` 등)와 연결하는 배선 작업이 아직 안 되어
있다(테스트에서는 목으로만 검증함) — **필요하면 이 배선 작업이 남은 과제.**
