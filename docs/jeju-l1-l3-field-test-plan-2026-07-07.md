# 제주도 OpenHash L1~L3 필드 테스트 계획서 (2026-07-07)

**범위**: 제주특별자치도 한정 시험. 전국(L4)·글로벌(L5) 확장은 이번 계획 범위 밖.

## 0. 배경 — 지난 사고실험에서 확인된 선행 과제

이 계획 이전에 진행한 사고실험(단일 L1 기준)에서, 여러 L1이 실제로 늘어나면
반드시 풀어야 할 문제 네 가지를 확인했다. 이 계획서는 그 네 가지를 실제
인프라로 풀어내는 것이 목적이다.

| 사고실험에서 발견한 문제 | 이 계획서에서의 대응 |
|---|---|
| ①구매자 소속 L1 차감은 성공, 판매자 소속 L1 크레딧이 네트워크 오류로 실패 → GDC 소실 | §5 브릿지 트랜잭션의 outbox 패턴(재시도 큐) |
| ②크레딧 응답 유실 후 재시도 → 중복 크레딧 | §5 tx_hash 기준 멱등성 |
| ③L2가 검증하는 시점이 브릿지 진행 중일 때 오탐 | §6 in-flight 브릿지 레지스트리 |
| ④"이 guid가 어느 L1 소속인지" 조회 불가 | §4 guid→L1 소속 레지스트리 |

## 1. 토폴로지 정의

| 계층 | 정의 | 개수 |
|---|---|---|
| **L1** | 읍면동(邑面洞) 단위 — 개별 PocketBase OpenHash 서버, 각자 `blocks`/`gdc_keys` 등 실제 GDC 원장 보유 | **43개**(제주시 26 + 서귀포시 17, `prompts/Jejudo/05-emd/emd-master-data.json` + `05-emd/hallim/hallim-data.json` 기준) |
| **L2** | 행정시(行政市) 단위 — 소속 L1들의 Merkle 루트·GDC 총량을 집계·검증 | **2개**(제주시, 서귀포시) |
| **L3** | 제주도 전체 — L2 두 곳을 최종 집계, 도 단위 총량 보존 검증의 최상위 루트(이번 시험의 최상위) | **1개** |

## 2. 인프라 준비

### 2.1 현재 상태 재확인

서버(168.110.123.175, Oracle Cloud)에 이미 6개 PocketBase 프로세스가 떠 있다:
`hanlim`(8091, 공개, 실제 GDC 원장 보유) / `l2-jeju`(8092) / `l2-seogwipo`(8093) /
`l3-jejudo`(8094) / `l4-kr`(8095) / `l5-global`(8096) — 단, 8092~8096은
지금까지 Merkle 루트 집계 전용이었고 `blocks`/`gdc_keys` 컬렉션이 없다.

**이번 계획에서 정리**: 8092(`l2-jeju`)·8093(`l2-seogwipo`)는 원래 설계대로
**진짜 L2 역할**(집계 전용, GDC 원장 없음)로 유지한다 — 지난 대화에서 검토했던
"L2 프로세스를 임시로 L1 역할 겸용"안은 이번 43개 전체 구축 계획에서는
**철회**한다(역할이 섞이면 나중에 헷갈린다). 대신 **새 포트 43개
(8101~8143 등)에 L1 전용 PocketBase 인스턴스를 하나씩 새로 띄운다.**

### 2.2 리소스 산정 — 확인 필요

- PocketBase는 가볍지만(Go 바이너리, 인스턴스당 수십 MB 수준) 43개를 한
  서버에 다 띄우면 누적 메모리·디스크 IO가 상당해진다. 배포 전에 반드시:
  ```bash
  free -h
  df -h /
  ```
  로 여유 확인 — 지난 SSH 세션에서 본 `Memory usage: 43%`, `Usage of /: 9.7%
  of 44.96GB`는 프로세스 6개 기준이라, 43개로 늘리면 재확인이 꼭 필요하다.
- **권장**: 서버 한 대에 다 몰아넣지 말고, 최소 파일럿(§7 Phase 1)에서
  실제 리소스 사용량을 측정한 뒤 전체 확장 여부·서버 증설 여부를 결정한다.

### 2.3 템플릿화

`hanlim`의 `pb_data` 스키마(컬렉션 정의: `blocks`, `gdc_keys`, `l1_ledger`,
`node_ledger`, `profiles`, `seller_products` 등)를 **PocketBase 마이그레이션
파일**로 export해서 `sql/`(또는 신설 `l1-migrations/`) 디렉터리에 버전관리
한다 — 이번에도 "SQL/스키마가 리포에 없어서 벌어진 사고"(6월 문서화 누락,
`market_purchase` RPC 등)를 반복하지 않기 위해 **최초 배포부터 버전관리**한다.

각 신규 L1은 다음만 다르면 된다: 포트, `pb_data` 경로, `NODE_ID`(emd_code).
`main.pb.js` 훅 파일은 **전부 동일한 파일을 공유**한다(이미 그렇듯
`--hooksDir=/opt/gopang/pb_hooks` 공용 — 훅 하나 고치면 전체가 재시작된다는
점은 §8 배포 절차에서 그대로 유의사항으로 남긴다).

### 2.4 도메인/라우팅

`l1-{emd_code 소문자}.hondi.net` 패턴(예: `l1-hallim.hondi.net`,
`l1-aewol.hondi.net`)으로 43개 서브도메인을 준비하거나, 초기 파일럿
단계에서는 서브도메인 없이 `hondi-proxy` Worker가 `L1_NODE_MAP`에 포트를
직접 매핑해 내부적으로만 라우팅해도 된다(사용자에게 노출되는 API는 항상
Worker 경유이므로, L1 자체가 공개 도메인일 필요는 없다 — 오히려 `hanlim`
외 나머지는 `127.0.0.1` 바인딩으로 막아두는 게 지금 L2~L5처럼 더 안전하다).

## 3. 데이터 모델 — 각 L1에 필요한 것

기존 `hanlim`과 동일:

| 컬렉션 | 용도 |
|---|---|
| `blocks` | 이 L1 내부 거래 원장(기존과 동일) |
| `gdc_keys` | 이 L1 소속 사용자의 공개키 등록소 |
| `l1_ledger` / `node_ledger` | Merkle 앵커링(기존과 동일) |
| **`bridge_out`(신설)** | 이 L1에서 다른 L1로 나가는 거래의 outbox 큐 — §5 |
| **`bridge_in`(신설)** | 다른 L1에서 이 L1으로 들어오는 거래의 멱등성 체크용 처리 이력 — §5 |

`profiles`/`seller_products`는 L1마다 안 두고 **hanlim에만 유지**한다(사용자
프로필·판매 카탈로그는 지역 소속과 무관하게 전국 검색 대상이어야 하므로,
지금처럼 단일 소스로 두는 게 맞다 — GDC 잔액/원장만 지역별로 분산한다).

## 4. guid → L1 소속 레지스트리 (선행 필수 과제)

### 4.1 왜 필요한가

지금 가입 흐름(`_registerToL1`)은 무조건 `hanlim`에만 등록한다. 43개 L1이
생기면 "이 사용자가 어느 읍면동 L1에 지갑을 두는지"를 정해야 하고, 브릿지
거래 시 상대방(판매자)의 소속 L1도 조회할 수 있어야 한다.

### 4.2 설계

- **가입 시점에 사용자의 읍면동을 확정**한다 — 이미 `register-flow.js`에
  "내 위치" 버튼(Kakao 역지오코딩)이 있으니, 그 결과(주소)를 `emd_code`로
  매핑하는 조회만 추가하면 된다. 위치 정보를 안 주면 기본값(`hanlim`)으로
  폴백하거나, 읍면동을 직접 선택하게 한다.
- **레지스트리 저장 위치**: 새 레지스트리 컬렉션을 따로 만드는 대신,
  **L3(제주도 전체) 노드에 `guid_home_l1`이라는 단일 컬렉션**을 두고
  `{guid, emd_code, registered_at}`만 기록한다 — 전체 43개 L1 중 어디에
  물어봐도 되는 게 아니라 "정답이 하나뿐인 조회"이므로, 계층 최상위(L3)에
  두는 게 자연스럽다(이 계획 범위에서 L3가 곧 최상위 루트이기 때문).
- Worker의 `/gwp/register-key`가 지금은 `hanlim`에 고정 등록하는데, 이걸
  `emd_code` 파라미터를 받아 해당 L1로 라우팅하고, 동시에 L3의
  `guid_home_l1`에도 기록하도록 확장해야 한다.

## 5. 브릿지 트랜잭션 프로토콜 (크로스-L1 거래)

지난 사고실험에서 설계한 2단계 브릿지를 실제 컬렉션 기반으로 구체화한다.

```
① 구매자(L1-A 소속)가 판매자(L1-B 소속)에게 구매
   → L1-A: 정상 차감(기존 로직) + bridge_out에 레코드 생성
     {tx_hash, target_emd, amount, status:'pending'}

② Worker(허브)가 bridge_out을 폴링(또는 즉시 트리거) →
   L1-B의 새 엔드포인트 POST /api/bridge-in 호출
   {tx_hash, source_emd, guid, amount}

③ L1-B: bridge_in에 tx_hash가 이미 있으면(멱등) 그대로 200 반환하고 아무
   것도 안 함. 없으면 판매자에게 크레딧(block_type:'bridge_in' 블록 생성)
   + bridge_in에 {tx_hash, status:'applied'} 기록 → 200 반환

④ Worker가 ③의 성공 응답을 받으면 L1-A의 bridge_out 레코드를
   status:'completed'로 갱신. 실패하거나 응답이 없으면 status는
   'pending'으로 남고, **재시도 큐(cron 또는 다음 폴링 때 재시도)**가
   ②부터 다시 시도한다 — ③이 멱등이므로 여러 번 재시도해도 안전하다.
```

### 5.1 미완료 브릿지의 취급

`status:'pending'`인 `bridge_out` 레코드가 일정 시간(예: 5분) 넘게 남아있으면:
- 자동 재시도를 계속하되,
- 그 이상(예: 1시간) 넘어가면 **보상 트랜잭션**(구매자에게 환불) 후
  `status:'refunded'`로 마감 — 이 경우 원 거래는 실패로 처리되고 구매자에게
  알림이 필요하다(UI/PDV 보완은 이번 계획 범위 밖, 별도 작업으로 남김).

## 6. 총량 보존 검증의 계층 전파

### 6.1 L1 단위 (이미 구현됨)

`GET /api/supply/verify` — 이 L1의 발행 총량 vs 잔액 합. 43개 각각에
동일하게 배포한다(§2.3 템플릿에 이미 포함).

### 6.2 L2 단위 (신설)

`GET /api/supply/verify`(L2, 예: `l2-jeju`)를 새로 만든다:
```
Σ(소속 L1들의 total_minted) == Σ(소속 L1들의 total_balance)
   + (진행 중인 bridge_out 미완료분은 "일시적 불일치"로 별도 표시,
      §5.1의 재시도 유예 시간 안이면 오류로 보고하지 않음)
```
소속 L1 목록은 `emd-master-data.json`의 `행정시명` 필드로 정적으로 고정한다
(제주시 26개/서귀포시 17개, 통폐합 전까진 안 바뀜).

### 6.3 L3 단위 (신설)

`GET /api/supply/verify`(L3, `l3-jejudo`) — L2 두 곳(제주시/서귀포시)의
결과를 다시 합산. **이번 계획의 최종 검증 지점**이다 — "제주도 전체
GDC 총량은 항상 발행 총량과 같다"를 여기서 확인한다.

### 6.4 알려진 한계 (재확인)

- 브릿지가 in-flight인 순간의 일시적 불일치를 "오류 아님"으로 구분하는
  로직(§5.1의 유예시간 기준)이 없으면 오탐이 반복된다 — 이건 이번 계획의
  §5에서 이미 다뤘지만, L2/L3 검증 함수 구현 시 반드시 이 유예시간을
  실제로 반영해야 한다(빠뜨리기 쉬운 지점이라 별도로 강조).

> **2026-07-07 Phase 1 실측 결과로 갱신**: 실제로는 sentinel 계정
> (`bridge-out:{target}` / `bridge-in:{source}`) 설계 덕분에, 로컬 L1
> 단위 보존식이 브릿지 진행 상태와 무관하게 **항상** 성립한다는 게
> 실증됐다. 즉 여기서 우려한 "유예시간 로직"은 필요 없었다 — §11.3
> 시나리오5 결과 참고.

## 8. 배포 절차상 유의사항 (기존 교훈 재확인)

- `--hooksDir` 공유 구조상 **`main.pb.js`를 고치면 43개+6개 프로세스가
  전부 동시 재시작**된다 — 사람 트래픽 적은 시간대에, 그리고 Phase별로
  프로세스 수가 늘어날수록 재시작 자체의 영향 범위도 커진다는 점을
  감안해야 한다(Phase 4 시점엔 50개에 육박하는 프로세스가 한 번에
  재시작됨 — 이 시점부터는 `main.pb.js` 변경을 더 신중하게 다뤄야 한다).
- 이번에도 **콜백 바깥 최상위 함수/상수 선언 금지**(Goja 제약, 이번
  세션에서 두 번 실제로 걸렸던 문제) — `bridge_out`/`bridge_in` 관련
  신규 라우트 작성 시 반드시 지킬 것.
- 새 컬렉션(`bridge_out`, `bridge_in`, `guid_home_l1`)의 필수 필드
  (`required:true`) 여부를 스키마 생성 시 미리 정확히 정의할 것 — `mint`
  때 `buyer_guid` required 필드를 빈 문자열로 넣어 저장 실패했던 사고를
  반복하지 않기 위해, **스키마부터 먼저 `curl`로 조회해서 확인 후 코드
  작성**하는 순서를 지킨다.

## 9. 테스트 시나리오 (Phase 1 집중)

1. **동일 L1 내 거래** — 이미 검증됨(기존 hanlim E2E), 회귀 확인용으로만 재실행.
2. **인접 L1 간 정상 거래** — hanlim 소속 구매자 → 애월 소속 판매자, 정상 완료 확인.
3. **의도적 장애 주입 — ②(bridge-in 호출) 실패** — 애월 L1을 일부러
   내려놓고 거래 시도 → `bridge_out`이 `pending`으로 남는지, 재시도가
   실제로 도는지, 최종적으로 완료되거나(애월 복구 시) 보상 처리되는지(유예
   시간 초과 시) 확인.
4. **중복 재시도** — ③을 의도적으로 두 번 호출(같은 tx_hash) → 두 번째
   호출이 크레딧을 중복 반영하지 않는지(멱등성) 확인.
5. **L2 검증 시점 경합** — 브릿지 진행 중(①은 됐고 ③은 아직)인 순간에
   `l2-jeju`의 `/api/supply/verify`를 호출 → 오탐(즉시 `valid:false`)이
   아니라 유예시간 내 "미완료"로 구분되는지 확인.

**→ 5개 시나리오 전부 2026-07-07 실측 완료. §11.3 참고.**

## 10. 이번 계획에 포함하지 않는 것

- L4(전국)·L5(글로벌) — 제주도 시험 범위 밖.
- 국세청 AI 모듈 연동 — 별도 요청사항으로 이번 계획 밖.
- 개인키 완전 분실 시 복구 절차 — 기존에 남겨둔 미해결 항목, 이번 계획과 무관.
- 손익계산서(수익/비용) 계정의 서버측 재대사 — 기존에 남겨둔 미해결 항목.

### 11.4 남은 항목

- **서버 리소스**: 현재 인스턴스가 OCI Always Free `VM.Standard.E2.1.Micro`(고정 1GB RAM, 리사이즈 불가)로 추정됨(`free -h` 총량 956Mi가 정확히 일치). Phase 2(제주시 26개 전체)를 위해서는 서버 증설이 선행돼야 함 — 별도 트랙으로 분리(옵션: Ampere A1.Flex 무료 이전, PAYG 유료 전환, 또는 L2 경계 기준 인스턴스 분산).
- **크론 스윕 미검증**: `worker.js`의 `scheduled()`(주기적 `_sweepBridgeOutbox`)는 이번 세션에서 전부 수동 트리거로만 검증됐고, 실제 Cloudflare Cron Trigger 주기로 동작하는지는 미확인.
- **테스트 데이터 잔재**: `sim-*` guid들의 테스트 잔액이 hanlim/애월 원장에 남아있음(실사용자 영향 없음, 필요 시 정리).
- **`/api/tx`의 `buyer_sig` 비검증**: L1이 서명을 암호학적으로 검증하지 않는다는 사실이 이번 시뮬레이션 과정에서 드러남(register-key와 달리). 실거래 전환 전 별도 보완 필요 — 이번 계획 범위 밖으로 남김.
