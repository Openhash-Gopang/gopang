# 웹푸시 알림이 폰이 아니라 PC로 가는 문제 (2026-07-21)

device-link(기기 간 지갑 이전) 승인 요청을 보냈는데, 웹푸시 알림이 폰이
아니라 **요청을 보낸 PC 자신**에게 도착하는 문제를 조사·수정한 기록입니다.

## 1. 원인

`profiles.push_subscription`은 **계정(guid)당 딱 하나만** 존재하는 필드입니다
(기기별이 아님). 그런데 "웹푸시 구독을 모든 가입자의 기본값으로 보장"하는
자동 로직(`gopang-app.js`, 2026-07-20 신설)이 **이 기기가 실제로 그 계정의
지갑 키(Ed25519, IndexedDB)를 갖고 있는지는 확인하지 않고**,
`localStorage.gopang_user_v4`에 그 guid 세션이 남아있기만 하면 조용히
그 기기를 push 대상으로 재등록해버렸습니다.

이번 경우: 개발 PC가 예전에 이 계정으로 로그인했던 이력이 있어
`gopang_user_v4`에 해당 guid가 남아있었고, device-link 승인을 "기다리는 중"인
바로 그 PC에서 klaw/hondi.net 아무 페이지나 열려 있던 사이 백그라운드에서
이 로직이 실행돼 PC 자신을 push 대상으로 덮어썼습니다. 그 결과 폰으로
가야 할 device-link 알림이 폰에는 오지 않고 PC 자신에게 도착·재생됐습니다.

## 2. 수정

`gopang-app.js`의 자동 구독 로직에 `GopangWallet.exists()`(이 기기
IndexedDB에 실제 서명키가 있는지 확인) 체크를 추가했습니다. 지갑 키
생성은 원래 휴대폰 전용 관례("PC는 이 키를 생성하지 않음")이므로, 이
체크로 "세션만 있고 진짜 지갑은 없는 기기"(= 아직 device-link 승인 전인
PC)를 걸러낼 수 있습니다.

```js
if (typeof window.GopangWallet === 'undefined' || !(await window.GopangWallet.exists())) {
  console.info('[Push] 이 기기에 지갑 키가 없어 기본 구독 보장을 건너뜁니다(세션만 있는 기기로 추정).');
  return;
}
```

## 3. 이미 잘못 저장된 구독을 지금 당장 되돌리려면

이 패치는 **앞으로의 재발만** 막습니다. 이미 서버에 PC의 구독으로 덮어써진
상태는 저절로 안 돌아오므로:

- **가장 간단한 방법**: 폰에서 gopang/klaw 앱(웹)을 한 번 열기 — 폰은
  진짜 지갑 키를 갖고 있으므로 이 체크를 통과해 자동으로 자신을 다시
  push 대상으로 등록합니다.
- 또는 폰의 설정 화면에서 알림 토글을 껐다 켜서 즉시 강제 재구독.

## 4. 근본적으로 남는 설계 한계 (후속 개선 후보, 이번엔 수정 안 함)

`push_subscription`이 계정당 1개뿐인 구조 자체는 그대로입니다. 즉 사용자가
여러 기기(폰 + 태블릿 등)를 동시에 정당하게 쓰는 경우, 나중에 로그인한
기기가 이전 기기의 구독을 덮어쓰는 건 이번 수정 이후에도 여전합니다 —
다만 최소한 "지갑 키 없는 기기가 가로채는" 가장 위험한 경우는 막았습니다.
기기별로 구독을 여러 개 저장하고 device-link 알림은 항상 "지갑 키를 실제로
보유한 기기 전체"로 보내는 방식이 근본적 해결책이며, 별도 작업으로
분리하는 걸 권장합니다.
